## Problem

On RGB-LED boards (QT Py NeoPixel, `LED_RGB`) the beat LED is hardcoded green
(`rgbLedWrite(pin, 0, 40, 0)`). The web/touch UI already has BEAT + BAR1
colour pickers (LNK-036, `dot_beat_color`/`dot_accent_color`, NVS-persisted,
live-applied via /live) — but they only drive the touch-display dot and web
dot. The physical LED ignores them, and shows no bar-1 accent at all.
KitchenSync's strip (P4-019) already does colour + accent from the web UI.

## Solution

- Pure `led_flash_rgb()` in `led_phase.{h,c}` (host-tested): pick accent vs
  beat colour from bar phase (int part 0 = downbeat, same rule as the touch
  dot), scale channels by brightness/255 (40 keeps the old NeoPixel level).
- `led_set()` RGB path flashes a stored current colour; `led_task` sets it
  per flash from `g_config` + `tempo_source_phase(quantum_beats)`. Fallback
  beat / WiFi-down blink use the beat colour (no bar info there).
- Non-RGB boards unchanged (plain GPIO ignores colour).
- Web UI hint updated: colours drive dot + beat LED.

## Deliverable

`led_phase.{h,c}` + tests; `X32Link.ino` LED wiring; hint text. Live-applies
with no reboot since `g_config` is read at each flash.
