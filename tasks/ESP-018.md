## Problem

The Touch's DIN clock (GPIO11) intermittently **stalls and then bursts to catch up** --
the same signature as P4-033, which meant DROPPED clock pulses on the P4.

Caught on the analyzer during the nudge test (30s capture, 132.5 BPM):

```
t=16.862s   interval = 50.070 ms   <- ~2.6 clock periods missing
t=17.514s   interval =  7.855 ms   <- compressed catch-up
```

Baseline spacing is 18.87 ms. A clean 12-second capture earlier showed **zero** stalls
and zero bursts, so this is intermittent -- it appeared while the web server was being
hammered with `/status` polls and `/nudge` POSTs.

## Why it matters

`MAX_BURST` caps the catch-up, so a stall does not merely delay pulses -- it can DROP
them. This is exactly what P4-033 turned out to be, and the RC-505 is the customer's
box: an audible stutter mid-set is the one failure this product cannot have.

## Suspects

- The MIDI writer is a **pinned task on core 1** (`ktouch_midi_out.cpp`), while the web
  server and the display tick run in `loop()` on **core 0**, so a naive "web blocks the
  clock" story does NOT obviously hold. That is what makes this interesting.
- The Touch's writer does NOT call `ESP_LOGx`, so it is not a straight P4-033 repeat.
- Candidates worth measuring: WiFi/lwIP work landing on core 1; the LovyanGFX SPI
  flush; the AXS5106L I2C reads (now with up to 3 extra re-reads per release, ESP-016);
  `Serial` writes; flash/NVS access stalling the cache.

## Approach

Do what worked for P4-033: **instrument, do not guess.**

Add the same tick-health probe to the Touch's writer task -- `gap` (previous tick end ->
this tick start) vs `work` (time inside the tick):

- large **gap** => the task was not scheduled: the culprit is OUTSIDE the writer
  (starvation, another core-1 task, a flash-cache stall)
- large **work** => something in the writer blocked

Surface `max_gap` / `max_work` / `overruns` in `/status` (the Touch already exposes
`tfail`/`tzero`/`ccancel` telemetry -- same pattern).

**Rate-limit the probe's own logging.** On the P4 the first version logged every
overrun, and the log delayed the next tick, which logged again -- 3945 phantom
"overruns" in 40s. It manufactured the fault it measured.

## Acceptance criteria

- [ ] Tick-health probe in the Touch writer task, reported via `/status`
- [ ] Root cause named with a measurement, not a hypothesis
- [ ] Analyzer capture on GPIO11 over >=30s under web load: **0 stalls, 0 bursts**
- [ ] Clock jitter characterised. Today the Touch is **533 us stdev (min 15.4 / max
      24.4 ms)** vs the P4's **331 us (+-1 ms)** -- the S3 is measurably looser and we
      do not yet know why. Decide whether that is acceptable for the RC-505 or needs
      fixing too.
