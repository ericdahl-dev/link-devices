## Goal

Get the P4 onto WiFi through its **ESP32-C6 co-processor** (SDIO) and joining an
Ableton **Link** session as a native peer — reusing `link_protocol.c` +
`link_listener` + `link_measurement*` unchanged; only the WiFi/UDP glue is new.
This is the P4's tempo source; P4-005 (clock out) and P4-006 (metronome) both
depend on a live tempo.

## Why

The P4 SoC has no native WiFi radio — it's provided by the onboard ESP32-C6 over
the hosted-WiFi stack. Everything downstream needs a tempo, and Link-over-WiFi is
the primary source (matching the shipped S3 behaviour).

## Open questions (the real work is here)

- **Is the onboard C6 pre-flashed** with the ESP-Hosted slave firmware on the
  Waveshare P4-NANO, or does it need flashing first? (First thing to confirm.)
- **Stack choice:** `esp_wifi_remote` + `esp-hosted` — how it changes the WiFi
  bring-up vs the S3's native `WiFi.begin()`. Multicast join for Link
  (`224.76.78.75:20808`) over the hosted transport — does IGMP/multicast pass
  through esp-hosted cleanly?
- **The `WiFi.setSleep(false)` invariant** (modem-sleep drops buffered multicast
  and Link silently never receives — see AGENTS.md) — how it maps onto the hosted
  stack.
- Arduino vs ESP-IDF for the hosted WiFi glue on the P4.

## Reuse

`link_protocol.c` (our ~100-line gossip parser), `link_listener`,
`link_measurement.c` + `link_measurement_session.c` (LNK-031 pure orchestrator),
`link_phase.c` — all Arduino-free, should port unchanged. Only the UDP socket +
multicast-join glue is re-implemented against the hosted-WiFi API.

## Deliverable

P4 joins a Link session over WiFi and reads session BPM/phase (verify against
Ableton Live or the S3 unit on the same network), reusing the pure Link modules.

## WiFi bring-up spike (2026-07-04) — PROVEN ✅ (open question retired)

The biggest unknown is answered: **the onboard C6 is pre-flashed with the
ESP-Hosted slave firmware and WiFi works out of the box.** ESP-IDF `station`
example (`scratchpad/p4_wifi_link`) built for `esp32p4` with the managed
components `esp_hosted 2.12.9` + `esp_wifi_remote 1.6.1`, slave target
`esp32c6`:

- SDIO transport came up unmodified: **Slot 1, 4-bit, 40 MHz, CLK[18] CMD[19]
  D0-3[14-17], slave-reset[54]** — matches the esp_hosted defaults, so **no
  Waveshare-specific pin remap** needed.
- `Identified slave [esp32c6]`, capabilities WLAN + BLE.
- **Associated + DHCP in ~6 s → IP 192.168.1.201**, gw 192.168.1.1.

Config that mattered (in `sdkconfig.defaults`): `CONFIG_SLAVE_IDF_TARGET_ESP32C6=y`
plus the P4 v1.3 min-rev gate (`ESP32P4_SELECTS_REV_LESS_V3` / `REV_MIN_100`).

**Follow-up (minor):** C6 slave reports an old version (`Host 2.12.0 >
Co-proc 0.0.0`) — backward-compatible and worked, but Espressif recommends a C6
OTA upgrade to avoid RPC timeouts. Non-blocking.

**Remaining for P4-004:** add the Link multicast listener — join
`224.76.78.75:20808`, disable WiFi power-save on the remote
(`esp_wifi_remote` PS-none — the S3's `WiFi.setSleep(false)` invariant: modem
sleep drops buffered multicast and Link silently never receives), and parse
gossip with the pure `link_protocol.c` to read session BPM. **Verifying it needs
a live Link peer on the `Skeyelab` network (Ableton Live or the S3 X32Link unit).**
