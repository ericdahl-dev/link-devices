## Problem

`g_cfg_gen` (`ks_main.c:65`, `volatile uint32_t`) is bumped in the `/live` handler
(`ks_web.cpp:432`) **after** a non-atomic multi-field write including a 5-int
`s_cfg->clock[o] = cand.clock[o]` struct copy (`ks_web.cpp:431`). The clock task reads
those same fields directly every tick with no consistency guard (`ks_main.c:129-131`) —
a real torn-read window (new `ppqn` with old `phase`). It looks like a seqlock (writer
bumps a counter) but the reader never checks it. On a dual-core P4 the `volatile` gives
visibility but not ordering, so field-writes and the counter bump can be observed out
of order.

Separately, "which fields are live-safe" is encoded twice: the copy list in
`ks_web.cpp:423-431` and the checkbox pre-clear in `ks_form.c:50-54`.

## Solution (deepen)

Make it a real seqlock (bump to odd before write, even after, with a release barrier;
reader samples gen, reads, re-checks with an acquire barrier, retries on change) — or
double-buffer the config with a pointer swap. Give `ks_config` a single **live-safe
field mask** both the `/live` copy and the form pre-clear consult, so the "which fields
apply live" contract has one owner.
