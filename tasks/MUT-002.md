## Goal

Implement the guard: maintain a desired mute state for a configurable set of
**locked** channels, and whenever the mixer reports a locked channel's
`/ch/NN/mix/on` changing away from the desired value, **send it back**.

## Acceptance Criteria

- [ ] `mute_guard.{cpp,h}` added to `esp32/X32SafeMutes/`.
- [ ] Holds a desired-state table for locked channels. On startup, reads current
      `/ch/NN/mix/on` for each locked channel (query or first observed value)
      and adopts it as the desired value.
- [ ] On `/ch/NN/mix/on ,i <v>` for a **locked** channel where `v` ≠ desired:
      after `MUT_RESTORE_DELAY_MS` debounce, send `/ch/NN/mix/on ,i <desired>`.
- [ ] Restore covers `/ch`, `/bus/N`, and `/dca/N` mute nodes (DCA mute is a bit
      mask — see cheat-sheet `grp/mute`; handle channel `mix/on` first, others
      if straightforward).
- [ ] No fight loop: the self-issued restore must not re-trigger another restore
      (ignore the echo of our own write, e.g. by expecting it within a window).
- [ ] **Lock/unlock from the web UI:** per-channel lock toggles; locking adopts
      the current state as desired; mask persisted in NVS via AppConfig.
- [ ] Pure-logic decision (`should_restore(locked, desired, observed)` →
      bool/value) is host-tested under `esp32/test/` (Unity).
- [ ] Demo on the emulator: lock ch 1 muted; a second client unmuting ch 1 is
      reverted within ~`MUT_RESTORE_DELAY_MS`; unlocked channels move freely.
      Record in Progress.

## Notes

- Inverted semantics again: desired "muted" = `mix/on 0`.
- Debounce exists so a deliberate operator change to an *unlocked* channel is
  never touched, and rapid toggles on a locked one settle to desired without a
  storm of OSC writes.
- Keep the restore send on `osc_out` (already model-port aware).
