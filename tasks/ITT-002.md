## Goal

Implement the rules engine: a configurable table mapping **triggers** (inbound
MIDI events, or observed OSC changes) to **actions** (send an OSC command, or
send a MIDI message). This is the "If This Then That" core.

## Acceptance Criteria

- [ ] `ittt_rules.{cpp,h}` added to `esp32/MidiOscIttt/`.
- [ ] Rule model supports at least:
  - **MIDI → OSC**: `NoteOn ch/n` or `CC ch/n` → OSC address + value, with a
    value map (MIDI 0..127 → OSC float/int, linear or fixed).
  - **OSC → MIDI**: observed OSC address (e.g. `/ch/01/mix/on`) crossing a
    condition → emit Note/CC/PC.
- [ ] Pure-logic evaluation (`ittt_eval(rule, event) -> action`) is host-tested
      under `esp32/test/` (Unity): table-driven cases for both directions,
      including value scaling and no-match.
- [ ] Rules are editable from the **web UI** and persisted (NVS, or a small
      JSON/CSV on LittleFS); capacity `ITT_MAX_RULES`.
- [ ] Loop guard: an OSC action that itself echoes back must not retrigger the
      rule that sent it (suppress self-originated echoes within a window).
- [ ] Worked examples documented in the task Progress and demoed on the
      emulator:
  - MIDI CC7 on ch 1 → `/ch/01/mix/fader ,f <scaled>` (MIDI fader → channel level).
  - MIDI NoteOn 60 → `/ch/01/mix/on ,i 0` (note as a mute button).
  - OSC `/ch/01/mix/on` → NoteOn/Off out (mixer mute drives an external LED/MIDI).

## Notes

- Value scaling uses the encodings in
  [`docs/xr18-xair-osc-cheatsheet.md`](../../docs/xr18-xair-osc-cheatsheet.md)
  (linear/log/fader). For a MIDI fader → channel level, map 0..127 → 0.0..1.0
  and send as the raw OSC float (the mixer applies its own fader taper).
- Keep the engine allocation-free at eval time; build the rule table once at
  load / on web save.
- This generalises X32Midi2OSC (MIDI→OSC) and X32Commander (OSC→MIDI/OSC) from
  the upstream toolset into one bidirectional firmware.
