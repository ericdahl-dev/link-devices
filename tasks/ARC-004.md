## Goal

Give the constants both firmwares must agree on a single home, so they can't
drift. Each `config.h` keeps only genuinely firmware-specific values.

## Problem (from the architecture review)

`X32Link/config.h` and `X32MidiClock/config.h` each hand-duplicate
`MCK_CLOCK_WINDOW`, `MCK_PPQN`, `MCK_BPM_THRESHOLD`, `MCK_POLL_MS`,
`MCK_CLOCK_TIMEOUT_MS`, and the `LINK_*` timings. Change one, remember the
other — a manual-sync hazard. (The MIDI-stability fix had to be applied to both
files for exactly this reason.)

## Acceptance Criteria

- [ ] A shared header (e.g. `midi_link_timing.h`) holding the
      protocol/timing constants both firmwares share, real in one sketch dir and
      symlinked into the other per the existing convention (`esp32/AGENTS.md`).
- [ ] Both `config.h` files include it and retain only firmware-specific values
      (e.g. `MCK_DEFAULT_BPM`, `MCK_SEND_INTERVAL_MS`, `MCK_REFRESH_BARS` for
      X32MidiClock; FX slot / LNK-specific for X32Link).
- [ ] No shared constant defined in two places; both firmwares compile;
      `check_docs.sh` and host suites green.

## Notes

- **Respect the real-file/symlink discipline** in `AGENTS.md` — don't fight it;
  add the shared header the same way the `osc_*` / `midi_*` files are shared.
- Low-risk mechanical change; do after the behavioural refactors.
