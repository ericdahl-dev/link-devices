## Why this is a ship gate

Dan's unit is a **KitchenSync Touch**, and the Touch is the one firmware in this repo with
**no OTA path at all**. `grep` for `Update.h` or `/update` across `KitchenSyncTouch/` returns
nothing.

That means the moment the unit leaves the bench, every future fix -- including the three
silent-drop bugs found this cycle, any of which could have shipped -- requires **physical
access to the device and a USB cable**. For a box that lives in someone else's rig in another
city, that is not a support story. It is the absence of one.

Both other firmwares already have it. Shipping the Touch without it is the gap.

## This is mostly wiring, not building

The hard parts already exist and are already shared:

- `X32Link/web_config.cpp` implements the full flow: `#include <Update.h>`, a `/update` POST
  handler that streams the `.bin` straight into the inactive OTA app slot, and a footer link.
- The **page itself is already shared** -- `ui_update_page()` lives in `ui_chrome` and is
  explicitly used by both X32Link and KitchenSync (ARC-017: *"shared CSS/JS + result & update
  pages"*). The Touch already links `ui_chrome`.
- The Touch already has the **partition table for it**: `KitchenSyncTouch/partitions.csv` was
  added with `app0`/`app1` OTA slots (4 MB each) and an `otadata` partition. The slots are
  sitting there, unused.

So this is: mount the handler, render the shared page, add the footer link. Not a new
subsystem.

## The one thing that is NOT free

**The partition table can only be written over USB, never over OTA.** Any Touch unit currently
running the old 4 MB default table (i.e. flashed before `partitions.csv` + `PartitionScheme=custom`
landed) has **no app1 slot to receive an OTA into**, and never can, over the air.

So: **Dan's unit must be flashed over USB with the custom partition table before it ships.**
If it goes out on the old table, it is permanently un-updatable, and this ticket's entire
purpose is defeated on the one device that needs it. Verify the table on the physical unit
before it leaves.

## Scope

- [ ] `/update` GET renders `ui_update_page()` (already shared -- do not write a second one)
- [ ] `/update` POST streams to the inactive OTA slot via `Update.h`, mirroring `web_config.cpp`
- [ ] Footer link to `/update`, matching the other two firmwares
- [ ] Success/failure result page, then reboot into the new slot
- [ ] **A failed or aborted upload must leave the running firmware intact** -- OTA writes the
      *inactive* slot, so a truncated upload must simply fail to validate and roll back, never
      brick. Test this deliberately: kill the upload halfway and confirm the device still boots.
- [ ] **Verify Dan's physical unit has the custom partition table before shipping** (see above)
- [ ] Confirm an OTA while a Link session is running does not stall the MIDI clock -- flash
      writes suspend the cache and freeze both cores. If it does stall, the update page must
      say "stop your set before updating" rather than pretending otherwise. Measure it; this
      codebase has been wrong about flash/RT interaction twice (P4-033, ESP-018).

## Parity (ESP-017)

X32Link and KitchenSync both have OTA. The Touch is the outlier. This closes the gap using the
*same shared page*, which is the parity principle working as intended: one implementation, three
devices.
