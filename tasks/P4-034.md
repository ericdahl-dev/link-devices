## Problem

**Multi-output USB-MIDI does not work.** Only ONE output's clock ever reaches USB gear,
no matter how many outputs are enabled. The P4's headline feature — 4 independent clock
outputs (P4-010: per-output division, phase, swing) driving USB gear — silently
delivers a single stream.

Caught on the logic analyzer, not in code review. Probing a Midihub's MIDI A DIN OUT
(pin 5) against the P4's own DIN mirror (GPIO20):

- out0 and out1 **both enabled @ 24 PPQN, both routed to cable A**
- expected MIDI A to carry **2x** the pulses of the DIN reference
- measured ratio: **1.004** (546 vs 544 pulses) — exactly ONE output's worth

The same bug explains an earlier confusion: with out0 on cable B and out1 on cable A,
MIDI A was **completely silent** — out0's send (first in the loop) won, out1's was
dropped.

## Root cause

`usb_midi_host.c`:

```c
void usb_midi_host_send(const uint8_t* data, int len)
{
    if (!s_ready || len <= 0 || len > 64) return;
    if (s_out_busy) { s_dropped++; return; }   /* previous packet still in flight */
    ...
}
```

**One in-flight bulk transfer, no queue. Busy → silently drop.**

`ks_main.c`'s fan-out calls `usb_midi_host_send()` **once per output, back-to-back in
the same 1 ms tick**. A USB full-speed bulk transfer occupies ~1 ms (one frame), so the
first call submits and the rest hit `s_out_busy` and are discarded. With 4 outputs
enabled, ~3 of every 4 packets are thrown away.

DIN was unaffected (UART driver has a TX ring), which is exactly why this hid for so
long — the DIN mirror always looked perfect.

`s_dropped` has been counting the carnage the whole time; nothing ever surfaced it.

## Fix

Batch the tick's events into **one** transfer instead of N. A USB-MIDI event packet is
4 bytes and a bulk transfer carries up to 64 — i.e. **16 events**, far more than the 4
outputs need. Bonus: every output's clock then lands in the **same USB frame**, so the
outputs stay tightly aligned to each other.

New pure, host-tested module (ADR-0003) alongside the existing pure `usb_midi_pack`:

```c
typedef struct { uint8_t buf[64]; int len; uint32_t dropped; } UsbMidiBatch;
bool usb_midi_batch_add(UsbMidiBatch*, const uint8_t pkt[4]);  /* false + count if full */
void usb_midi_batch_reset(UsbMidiBatch*);
```

Glue (`ks_main.c` + `usb_midi_host.c`):

- accumulate all outputs' packets for the tick into one `UsbMidiBatch`
- submit it as a single `usb_midi_host_send(batch.buf, batch.len)`
- if the endpoint is still busy, **carry the batch to the next tick** rather than
  dropping it (16 events is huge headroom: 4 outputs x 24 PPQN @ 168 BPM is only
  ~0.27 events/ms)
- expose `s_dropped` in `/status` so this class of bug can never hide again

## Behaviors to test (pure)

1. add 1 packet -> len=4, bytes match
2. **add 4 packets (4 outputs) -> len=16, all present, none dropped**  <- tracer bullet
3. fill to 16 events -> 17th rejected, `dropped`=1, buffer intact
4. reset -> len=0, `dropped` preserved (running counter)
5. glue: endpoint busy -> batch carries to next tick, nothing lost

## Verification

Re-run the analyzer comparison: out0 + out1 both @24 PPQN on cable A must give a
MIDI-A / DIN pulse ratio of **2.0** (currently 1.004). Then step out1's division
(÷2, x2, 1/16) and confirm the ratio tracks the configured PPQN.
