## Symptom

Under sustained HTTP traffic, the 24-PPQN MIDI clock **stops for 0.6-0.8 s**, then either
drops the missed pulses permanently or dumps a catch-up burst. Measured on a logic analyzer
(GPIO20 DIN MIDI TX + GPIO21 bar strobe), against a live Ableton Link session at 120 BPM.

Both outcomes are stage-ending: downstream gear either falls permanently behind, or is hit
with ~28 clock bytes back-to-back (which it reads as a violent tempo spike).

## Evidence (60 s captures, Saleae Logic @ 2 MS/s)

| condition | clock period sd | worst gap | drift end-to-end | verdict |
|---|---|---|---|---|
| idle, no HTTP client | 0.478 ms | 24.98 ms | +1.41 ms | clean |
| full 33 KB page loaded once, mid-capture | 0.447 ms | 29.97 ms | +1.34 ms | clean |
| browser page open, polling `/status` at 1 Hz | 0.603 ms | 38.99 ms | +3.42 ms | one ~18 ms stall |
| **10 Hz connection churn (new TCP per req)** | **13.970 ms** | **766.12 ms** | **+741.58 ms** | **~35 pulses LOST, never caught up** |
| **10 Hz connection churn (repeat run)** | **11.185 ms** | **611.01 ms** | +1.39 ms | **stall + 28-byte catch-up burst** |

Nominal period is 20.833 ms (120 BPM, 24 PPQN). **These two stall events are real and
unambiguous, but they are INTERMITTENT — see the correction under Fix. They are not
reproducible on demand, and the trigger was never isolated.**

**The clock engine itself is sound.** At idle every bar is *exactly* 96 clocks (29/29 bars,
nothing dropped or doubled), mean 20.834 ms = 119.997 BPM, phase-locked with bounded error.
The ~0.5 ms sd at idle is NOT jitter — it is the 1 kHz tick quantising a 20.833 ms period into
a 20/21 ms mix (histogram: 20 ms x535, 21 ms x2281, averaging to 20.833). That is
`CONFIG_FREERTOS_HZ=1000` behaving exactly as P4-006 designed. Do not "fix" it.

## Why this surfaced now (and why P4-037 is not the cause)

Pre-P4-037 firmware looks *clean* under the same hammer — but only because its socket-exhaustion
bug (P4-037) **wedges the web server within seconds**, so the device never actually serves the
load. The old bug was accidentally protecting the clock by refusing to work.

P4-037 fixed the server. The device now stays up under load, actually processes it, and the
clock stalls. So P4-037 did not introduce this defect, it **exposed** it. Rolling P4-037 back
would only re-hide it behind a dead web server.

This is urgent because **ESP-019** just made the web UI a *performance* transport surface —
the whole point is that the page is open in front of you mid-set.

## Root cause: FOUND — KitchenSync never got the ESP-018 fix

`clock_out_task` was created at **priority 6, unpinned** (`ks_main.c`). lWIP's `tcpip_task`
runs at **18**. The network stack could legally preempt the clock generator — and did.

This is not a new bug. It is **ESP-018**, which was diagnosed on the Touch (same silicon,
identical writer, losing MIDI pulses under web load) and fixed there by moving the writer to
**priority 19, core 1**. ARC-024 then ported that fix to X32Link. KitchenSync — the firmware
ARC-019 calls "the deep exemplar" — is the one that never got it:

| firmware | clock task | had the fix? |
|---|---|---|
| Touch (`ktouch_midi_out.cpp:177`) | `xTaskCreatePinnedToCore(..., 19, NULL, 1)` | yes (ESP-018) |
| X32Link (`midi_clock_out_io.cpp:124`) | priority 19, core 1 | yes (ARC-024) |
| **KitchenSync (`ks_main.c:381`)** | **`xTaskCreate(..., 6, NULL)`** | **NO** |

## Fix

`xTaskCreatePinnedToCore(clock_out_task, "clock_out", 4096, NULL, 19, NULL, 1)` — the same
sandwich the other two use, for the same reasons (above lwIP 18 so the web UI cannot deschedule
the clock; below WiFi 23 because starving the radio loses the Link session).

**Measured after the fix**, same 10 Hz churn, same rig: worst gap **43 ms** (was 611-766 ms),
phase-locked, drift +1.45 ms, no pulses lost.

**What is NOT proven:** that the catastrophic stall is gone. The 611/766 ms events were
**intermittent** — 5 minutes of identical churn produced none even BEFORE the fix, and the
tick-health probe never fired during a clean run. So a single clean 60 s run after the fix is
weak evidence. The priority inversion was real and is now corrected; whether it was the *only*
cause of those two events needs a long soak to confirm. A 43 ms gap still shows up post-fix,
which a flash-cache stall would also explain (that freezes both cores regardless of priority).

**Correction to an earlier claim in this ticket:** the stalls were first written up as
"reproduced 2/2 under churn". That was wrong — two real events happened to land in two load
runs, but the same load later ran clean for 5 minutes. The events are real (unambiguous on the
analyzer); the *trigger* was never pinned down.

## Remaining (the fix landed; the ticket has not)

The priority inversion is corrected. What is still open is the ability to SEE the next
stall without an analyzer clipped on at the right moment, and a soak long enough to say
whether the rare catastrophic event is actually gone.

## Acceptance criteria

- [x] `clock_out` runs above lwIP (priority 19, core 1), matching Touch + X32Link
- [ ] Publish the tick-health probe (max_gap/max_work/overruns) in `/status` — it currently only LOGS on overrun, so a stall that happens while nobody is watching serial is invisible (ARC-024 parity)
- [ ] No clock gap > 25 ms under 10 Hz connection churn for 60 s, on the analyzer
- [ ] Every bar is exactly 96 clocks under that load
- [ ] Missed-pulse policy is a deliberate, documented choice (drop vs burst), not an accident

## Notes

Captures saved (`.sal`, openable in Logic 2). Rig: ch0 = GPIO20 (DIN MIDI TX), ch1 = GPIO21
(downbeat strobe, rising edge = Link bar boundary), ch2 = MIDI OUT cable off the USB port-A
interface. Both MIDI paths show the identical stall, so it is upstream of the cable — it is the
clock engine's scheduling, not USB or DIN.

One 10 Hz keep-alive (no churn) run also disturbed the clock, but that run was confounded (the
device was answering `/status` in ~1.3 s and reconnecting) — do not draw conclusions from it
without a clean re-run. Whether the trigger is connection setup/teardown specifically, or just
sustained request handling, is still **open**.
