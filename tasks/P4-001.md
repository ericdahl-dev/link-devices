## Goal

Scope a **second firmware target** on the **ESP32-P4-NANO** — the "pro/hub" tier
alongside the shipped X32Link (ESP32-S3) bridge. The S3 stays as-is (small, cheap,
focused tempo→OSC/USB-MIDI-device box); the P4 does *more*, using its capabilities
the S3 lacks. This ticket is research/feasibility + a decision to spin up a real
firmware dir, **not** implementation.

## Why (what the S3 can't do)

The S3's single USB-OTG is device-only on its one USB-C, so it can't act as a USB
**host**, and it has no audio out. The ESP32-P4-NANO (confirmed from Waveshare
docs) removes both limits:
- **Type-A USB 2.0 OTG High-Speed = a real host port** (sources VBUS) → enumerate
  and drive USB-MIDI gear (e.g. a **Blokas MIDIHub**) directly and stream Link
  clock to it, no computer in the path.
- A **separate Type-C** for flashing/serial (USB-to-UART) → hosting doesn't cost
  the programming port (unlike the S3's host-or-device either/or).
- **ESP32-C6 co-processor** (SDIO) for WiFi 6 / BT5 → Ableton Link over WiFi works.
- **Onboard speaker** → audible metronome.
- Also: MIPI-DSI (bigger display), RMT (precise pulse gen for analog sync),
  TWAI/CAN, Ethernet + PoE, SD.

## Features to scope

1. **Link → USB-MIDI host.** Enumerate a USB-MIDI device (MIDIHub) as a host and
   stream 24-PPQN clock to it — reuse the pure `clock_ticker` engine (LNK-027/028)
   for the tick schedule; the glue is a USB-MIDI **host** write instead of a device write.
2. **Audible metronome.** Click on each beat (accent on the bar-1 downbeat) out the
   speaker, driven by the phase pipeline (`tempo_source_phase()` / `beat_synth` /
   `clock_ticker`). Configurable on/off + accent. Self-contained, showcases the P4 audio.
3. **(later)** bigger display, multi-USB-device routing, PoE-powered install.

## Reuse (the ADR-0003 payoff)

The pure modules — `link_protocol.c`, `link_measurement.c` +
`link_measurement_session.c`, `link_phase.c`, `clock_ticker.c`, `beat_synth.c`,
`bpm_*`, `app_config.c` — are Arduino-free C and should port to the P4/ESP-IDF
**unchanged** + keep their Unity suites. Only the glue is re-implemented per
platform: WiFi (via the C6 hosted stack, e.g. `esp_wifi_remote`/esp-hosted), USB
(host stack instead of TinyUSB device), display, and audio out.

## Open questions

- **Toolchain:** ~~Arduino vs ESP-IDF?~~ **ANSWERED (2026-07-04 spike):** Arduino
  works. Arduino-ESP32 3.3.8 ships an `esp32:esp32:esp32p4` (ESP32P4 Dev Module)
  profile; a bare sketch **compiled + flashed + verified** on the P4-NANO
  (`/dev/cu.usbmodem5ABA0782971`), RISC-V toolchain already present. So the P4 hub
  can share X32Link's Arduino toolchain and reuse the pure modules. ESP-IDF is also
  installed (`~/esp/esp-idf`) as a fallback if a feature needs it. Serial readback
  was flaky (native-USB, `CDCOnBoot=default`) — use `USBMode=hwcdc,CDCOnBoot=cdc`
  like the S3 build, or the web `/status` pattern.
- **WiFi-via-C6:** the hosted-WiFi setup (esp-hosted / `esp_wifi_remote`) — effort + how it changes the WiFi glue.
- **USB-MIDI host stack:** ESP-IDF `usb_host` + a MIDI class driver, or TinyUSB host. VBUS/power path on the Type-A port (external 5V / PoE).
- **Audio path:** the speaker — onboard codec / I2S DAC / PWM? What driver.
- **Build/dir layout:** new sketch/app dir in this repo; how much of the build + shared-module wiring is common with the S3.
