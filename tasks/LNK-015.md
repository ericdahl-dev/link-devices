## Goal

Build the actual on-device UI on top of LNK-014's display/touch bring-up:
a **status screen** (live tempo readout) and a **settings screen** that lets
the unit be fully configured from the touchscreen alone — no phone or
browser required. Both screens read/write the same `g_config` and call the
same `config_save()` / `config_validate()` as `web_config.cpp`, so the touch
UI and web UI stay in sync and either can be used interchangeably.

> **Rendering stack: raw LovyanGFX, NOT LVGL.** This task originally assumed
> LVGL, but LNK-014 shipped on raw **LovyanGFX** — LVGL was dropped as overkill
> (and the full firmware is already at **85% flash**; LVGL would push it over).
> So there are no `lv_obj_t` screens, no `lv_scr_load()`, no `lv_keyboard`, and
> no `lv_timer_handler()`. Screens are drawn directly with LovyanGFX primitives
> (`fillRoundRect`/`drawString`/`fillCircle`), and touch is handled by
> **hit-testing** the AXS5106L coordinates (with LNK-014's rotation-4 Y-flip)
> against each screen's tap rectangles. The `lvgl` library stays installed but
> unused. The biggest new piece is an on-screen keypad (see Notes) — LVGL's
> `lv_keyboard` is no longer available to lean on.

## Screen 1 — Status (default screen on boot)

Mirrors the live readout in `web_config.cpp`'s `.scr` panel:

| Field | Source |
|---|---|
| BPM, large numeric readout | `tempo_source_bpm()` / `g_current_bpm` |
| Phase wheel / bar-position indicator (sweeps through the bar, not just a beat blink) | `tempo_source_phase(g_config.quantum_beats)` (LNK-019) |
| Beat indicator (blinks per beat, available even before phase is valid) | `tempo_source_beat()` |
| Tempo source label ("Ableton Link" / "USB MIDI Clock") | `g_config.input_source` |
| Mixer connection state (connected/lost) | WiFi status + whether BPM updates are arriving (`tempo_source_active()`) |
| OSC target | `/fx/{g_config.fx_slot}/par/01` → `g_config.mixer_ip` |

### Phase indicator — real bar-quantized phase, via LNK-019

This is **true Link phase**: a bar-quantized, session-synced beat/bar
position (`tempo_source_phase()`), not the older locally-synthesized beat
blink. It's backed by `LNK-017`'s full `Timeline` parse (`beatOrigin`/
`timeOrigin`), `LNK-018`'s unicast ping/pong clock-offset measurement
against a real peer, and `LNK-019`'s `tempo_source_phase()` API that unifies
Link and MIDI behind one call.

Render it as a small radial sweep/wheel (0→`quantum` maps to a full
revolution) rather than a binary blink — that's the whole point of having
real phase instead of a beat pulse. While `tempo_source_phase_valid()` is
`false` (no completed measurement yet, e.g. right after a peer appears),
fall back to showing the plain `tempo_source_beat()` blink and a small "sync
ing…" label instead of the wheel — don't show a phase wheel built on invalid
data.

For MIDI, phase is pulse-counted and effectively exact (no clock-sync
estimation involved — see `LNK-019`'s notes); it becomes valid after one
full bar of observed pulses.

## Screen 2 — Settings (reachable by tapping the status screen)

One-to-one with the web form fields in `web_config.cpp`:

Each "widget" below is drawn with LovyanGFX primitives and backed by a tap
rectangle the touch handler hit-tests against (no LVGL objects).

| Field | Widget | Notes |
|---|---|---|
| Tempo Source | 2 tap rects styled as a segmented control (Link / MIDI) | matches `.seg#segSrc` |
| Mixer Model | 2 tap rects (XR-Series / X32) | matches `.seg#segModel`; changes FX slot max (4 vs 8) |
| FX Slot | grid of tappable slot rects, 1–4 or 1–8 | matches `.slots` |
| Mixer IP | on-screen numeric keypad (hand-drawn LovyanGFX buttons) or an IP-quad stepper | matches `.fld[name=mixer_ip]` |
| Quantum (beats/bar) | stepper or slot-grid (1-16), default 4 | new — `g_config.quantum_beats` (LNK-019). Drives the phase wheel's bar length. |
| WiFi SSID | text field + hand-drawn on-screen keyboard (see Notes) | matches `.fld[name=wifi_ssid]` |
| WiFi Password | masked text field + on-screen keyboard | matches `.fld[name=wifi_pass]`; leave blank = keep current, same as the web form |
| Write & Reboot | button | calls `config_validate()` then `config_save()` + `ESP.restart()`, identical to `handle_save()` in `web_config.cpp` |

Validation failure (bad IP / out-of-range slot) shows an inline error instead
of saving — mirror `handle_save()`'s `config_validate()` gate, don't
duplicate validation logic.

## Implementation

- Extend `touch_display.{h,cpp}` (from LNK-014) with a small screen-state
  machine (`enum { SCREEN_STATUS, SCREEN_SETTINGS, SCREEN_KEYBOARD }`), each
  with a `draw_*()` (full LovyanGFX redraw on entry) plus a set of tap
  rectangles. Switch screens by changing the state and redrawing. Tapping the
  status footer opens settings; a "back"/"cancel" rect on settings returns to
  status without saving.
- **Touch input is hit-testing, not an LVGL input device.** `touch_display_tick()`
  (from LNK-014) already reads AXS5106L coordinates; extend it to apply the
  rotation-4 mapping (X direct, `y = 319 - y`, per LNK-014's Progress) and
  dispatch a tap to the active screen — walk that screen's tap rects, and on a
  hit run the field's action (toggle a segment, pick a slot, push a keypad
  digit, fire Write & Reboot). Debounce on finger-up so one press is one tap.
- Read `g_config` directly for initial field values (same source the web
  form template substitution reads from in `build_html()`).
- Update interface:
  ```cpp
  void touch_display_begin(void);   // from LNK-014, now also draws the status screen
  void touch_display_tick(void);    // from LNK-014, now also dispatches taps (hit-test)
  void touch_display_update(void);  // call once per loop() iteration to refresh just
                                     // the status screen's live fields (BPM, beat,
                                     // connection state) — redraw only changed regions
  ```
- Call `touch_display_update()` from `X32Link.ino`'s `loop()` under
  `#ifdef HAS_TOUCH_DISPLAY`, same place `web_config_handle()` is called.
- Watch the **flash budget**: LNK-014 is already at 85%. Keep the UI lean (no
  LVGL; reuse LovyanGFX's built-in font, avoid large glyph sets).
- Update `esp32/AGENTS.md`'s module table to add `touch_display.*` once this
  lands (table note: "the directory is the source of truth for the file
  list" — keep the one-line description accurate, don't let it go stale).

- [n/a] WiFi password "keep current" on save — no WiFi field in the touch UI
      (deferred to web, above).

      measurement issue tracked separately, **not** this UI.

## Notes

- **Rendering stack** — see the callout under Goal: this task is raw LovyanGFX,
  not LVGL. The `lv_*` widget mentions throughout were reconciled to
  LovyanGFX-drawn tap-rects + hit-testing after LNK-014 shipped without LVGL.
- Don't fork config validation/save logic into `touch_display.cpp` — call
  the existing `config_validate()` / `config_save()` in `app_config.c` /
  `app_config_nvs.cpp`. The touch UI is a second front-end, not a second
  source of truth.
- **On-screen keyboard is the biggest new build** (no `lv_keyboard` to lean
  on). Draw a compact full-width keypad in LovyanGFX for SSID/password entry on
  the 172×320 panel: a grid of character tap-rects + shift/backspace/done,
  writing into a small char buffer. Keep it minimal (lowercase + digits + a
  shift layer) rather than a full soft keyboard. For Mixer IP, a digits-only
  keypad (or an IP-quad stepper) avoids the alpha layout entirely. If the
  keyboard proves too heavy for the 85%→ flash budget, defer SSID/password
  entry behind "keep current" and keep first-time WiFi setup on the existing
  web captive portal.
- This task does not change `bpm_publisher.c`, `osc_out.c`,
  `link_protocol.c`, or `link_measurement.*` — it only consumes
  `tempo_source_phase()` (LNK-019's finished API). Pure UI + config wiring
  on top of existing seams.
