## Goal

Build the actual on-device UI on top of LNK-014's display/touch bring-up:
a **status screen** (live tempo readout) and a **settings screen** that lets
the unit be fully configured from the touchscreen alone — no phone or
browser required. Both screens read/write the same `g_config` and call the
same `config_save()` / `config_validate()` as `web_config.cpp`, so the touch
UI and web UI stay in sync and either can be used interchangeably.

## Screen 1 — Status (default screen on boot)

Mirrors the live readout in `web_config.cpp`'s `.scr` panel:

| Field | Source |
|---|---|
| BPM, large numeric readout | `tempo_source_bpm()` / `g_current_bpm` |
| Phase wheel / bar-position indicator (sweeps through the bar, not just a beat blink) | `tempo_source_phase(g_config.quantum_beats)` (LNK-019) |
| Beat indicator (blinks per beat, available even before phase is valid) | `tempo_source_beat()` |
| Tempo source label ("Ableton Link" / "USB MIDI Clock") | `g_config.input_source` |
| Mixer connection state (connected/lost) | WiFi status + whether BPM updates are arriving (`tempo_source_active()`) |
| OSC target | `/fx/{g_config.fx_slot}/par/01` → `g_config.mixer_ip` |

### Phase indicator — real bar-quantized phase, via LNK-019

This is **true Link phase**: a bar-quantized, session-synced beat/bar
position (`tempo_source_phase()`), not the older locally-synthesized beat
blink. It's backed by `LNK-017`'s full `Timeline` parse (`beatOrigin`/
`timeOrigin`), `LNK-018`'s unicast ping/pong clock-offset measurement
against a real peer, and `LNK-019`'s `tempo_source_phase()` API that unifies
Link and MIDI behind one call.

Render it as a small radial sweep/wheel (0→`quantum` maps to a full
revolution) rather than a binary blink — that's the whole point of having
real phase instead of a beat pulse. While `tempo_source_phase_valid()` is
`false` (no completed measurement yet, e.g. right after a peer appears),
fall back to showing the plain `tempo_source_beat()` blink and a small "sync
ing…" label instead of the wheel — don't show a phase wheel built on invalid
data.

For MIDI, phase is pulse-counted and effectively exact (no clock-sync
estimation involved — see `LNK-019`'s notes); it becomes valid after one
full bar of observed pulses.

## Screen 2 — Settings (reachable by tapping the status screen)

One-to-one with the web form fields in `web_config.cpp`:

| Field | Widget | Notes |
|---|---|---|
| Tempo Source | 2-way LVGL segmented control (Link / MIDI) | matches `.seg#segSrc` |
| Mixer Model | 2-way segmented control (XR-Series / X32) | matches `.seg#segModel`; changes FX slot max (4 vs 8) |
| FX Slot | grid of tappable slot buttons, 1–4 or 1–8 | matches `.slots` |
| Mixer IP | text field + on-screen numeric keypad (`lv_keyboard` in number mode, or a custom IP-quad widget) | matches `.fld[name=mixer_ip]` |
| Quantum (beats/bar) | stepper or slot-grid (1-16), default 4 | new — `g_config.quantum_beats` (LNK-019). Drives the phase wheel's bar length. |
| WiFi SSID | text field + `lv_keyboard` (text mode) | matches `.fld[name=wifi_ssid]` |
| WiFi Password | text field (masked) + `lv_keyboard` | matches `.fld[name=wifi_pass]`; leave blank = keep current, same as the web form |
| Write & Reboot | button | calls `config_validate()` then `config_save()` + `ESP.restart()`, identical to `handle_save()` in `web_config.cpp` |

Validation failure (bad IP / out-of-range slot) shows an inline error instead
of saving — mirror `handle_save()`'s `config_validate()` gate, don't
duplicate validation logic.

## Implementation

- Extend `touch_display.{h,cpp}` (from LNK-014) with the two screens as
  `lv_obj_t*` screens, swapped via `lv_scr_load()`. Tap anywhere on the
  status screen's footer to open settings; a "back"/"cancel" button on
  settings returns to status without saving.
- Read `g_config` directly for initial field values (same source the web
  form template substitution reads from in `build_html()`).
- Register the touch input device with LVGL now (deferred from LNK-014).
- Update interface:
  ```cpp
  void touch_display_begin(void);   // from LNK-014, now also builds both screens
  void touch_display_tick(void);    // from LNK-014, unchanged
  void touch_display_update(void);  // call once per loop() iteration to refresh
                                     // the status screen's live fields (BPM, beat,
                                     // connection state) — cheap, no full redraw
  ```
- Call `touch_display_update()` from `X32Link.ino`'s `loop()` under
  `#ifdef HAS_TOUCH_DISPLAY`, same place `web_config_handle()` is called.
- Update `esp32/AGENTS.md`'s module table to add `touch_display.*` once this
  lands (table note: "the directory is the source of truth for the file
  list" — keep the one-line description accurate, don't let it go stale).

## Acceptance Criteria

- [ ] Status screen shows live BPM, beat blink, source label, and connection
      state with no perceptible lag (~10 Hz update is enough — this is not
      the 47fps spectrum redraw `tsv_display.cpp` does).
- [ ] `lv_timer_handler()` (via `touch_display_tick()`) keeps running
      smoothly while `bpm_task`/`tempo_source_poll()` and `web_config_handle()`
      run concurrently — no visible UI starvation.
- [ ] Settings screen pre-populates from `g_config` and round-trips through
      `config_validate()` + `config_save()` + `ESP.restart()` exactly like
      the web form.
- [ ] Changing Mixer Model on the touch settings screen updates the FX slot
      grid's max (4 vs 8), matching the web form's JS behavior.
- [ ] Device can be fully configured (WiFi, mixer IP, model, slot, tempo
      source) via touch alone, with **no network client connected** —
      confirms the touch UI doesn't depend on WiFi/web_config being up.
- [ ] Existing web UI (`web_config.cpp`) still works unmodified and
      concurrently — both UIs read/write the same `g_config` / NVS.
- [ ] WiFi password field: leaving it blank on save keeps the existing
      stored password (matches `handle_save()`'s "keep current" behavior).
- [ ] Phase wheel renders only once `tempo_source_phase_valid()` is true;
      before that, the screen shows the plain beat blink + a "syncing…"
      state instead of a wheel built on invalid data.

## Notes

- Don't fork config validation/save logic into `touch_display.cpp` — call
  the existing `config_validate()` / `config_save()` in `app_config.c` /
  `app_config_nvs.cpp`. The touch UI is a second front-end, not a second
  source of truth.
- LVGL's built-in `lv_keyboard` widget is sufficient for SSID/password/IP
  entry on a 172×320 panel if laid out full-width; don't build a custom
  keyboard widget.
- This task does not change `bpm_publisher.c`, `osc_out.c`,
  `link_protocol.c`, or `link_measurement.*` — it only consumes
  `tempo_source_phase()` (LNK-019's finished API). Pure UI + config wiring
  on top of existing seams.
