## Goal

Assess what it takes to support the **Behringer Wing** as a mixer target for
tempo/FX control, the way we support XR18/X32 today. The Wing is **not** an
X32-family device — different control protocol — so this is its own research
track, deliberately **later** than the X32/XR18 FX work in LNK-029. Output is a
findings doc + a `MODEL_WING` feasibility recommendation, **not** code.

## Why separate from LNK-029

LNK-029 (X32/XR18 FX-slot introspection + multi-slot tempo control) assumes the
**X32-family address tree** (`/fx/N/par/MM`, ports 10023/10024). The Wing uses a
different address/param model, so its findings won't port. It's also farther down
the roadmap — keep it from blocking the X32/XR18 work. Reuse LNK-029's *method*
(introspect FX type → map tempo param → fan out to N targets with per-target note
division), just re-answered for the Wing.

## What we know today (grounding)

- Model support is XR18/X32 only: `MODEL_XR18=1`, `MODEL_X32=2`
  (`app_config.h`); `config_model_port` → X32:10023 / XR18:10024;
  `config_model_slot_max` → X32:8 / XR18:4.
- The OSC layer (`osc_out`/`osc_in`) is X32-dialect-specific. Adding a Wing means
  either parameterizing that layer or adding a **separate Wing OSC dialect
  module** behind the existing `tempo_source`→OSC seam.

## Research questions

- **Protocol dialect.** The Wing speaks its own control protocol (OSC-style plus a
  text/`$`-style command channel) with a different address tree, node/param model,
  and encoding. Confirm: control UDP port(s); whether plain OSC suffices or the
  node protocol is required; how FX params are addressed and encoded
  (name-addressed nodes vs numeric par indices; value units).
- **FX topology.** Confirm the Wing's FX slot count and effect set (more slots and
  a different "premium" effect set than X32's verb/delay), which effects are
  tempo-syncable, and how each tempo param is expressed (ms vs beat-division vs a
  sync/tap node).
- **Introspection.** Does the Wing expose richer read-back (effect type/name per
  slot, param metadata) than X32's `/fx/N/type`? If so, auto-detection may be
  *cleaner* on Wing.
- **Firmware integration.** What `MODEL_WING` needs: a new model id + port in
  `config_model_port`/`config_model_slot_max`, and how much of LNK-029's Q1–Q4
  (introspection, param catalog, multi-slot fan-out, per-target divisions) is
  reusable vs Wing-specific. Estimate: parameterize `osc_out`/`osc_in` vs a new
  dialect module.
- **Discovery/connection.** Confirm a Wing is reached the same way as the X32
  (user-set mixer IP) or needs different discovery.

## Deliverables (this is research)

- [ ] Wing control port(s) + address/encoding model documented (with real dumps
      from a Wing if one is available).
- [ ] Wing FX topology: slot count, tempo-syncable effects, tempo-param encoding.
- [ ] Introspection capability assessment (vs X32 `/fx/N/type`).
- [ ] `MODEL_WING` feasibility + integration recommendation: parameterize the
      existing OSC layer vs a separate Wing dialect module behind the seam; how
      much of LNK-029 is reusable.
- [ ] Follow-on Wing implementation ticket(s) filed from the findings.

## Notes

- Do the X32/XR18 research (LNK-029) first; use its structure as the template.
- Needs a Wing on the bench (or its documented protocol) to dump `/fx` state.
- Keep any tempo→param unit math in a pure, host-tested module, same as the rest.
