## Goal

Store several WiFi networks and try each in turn before giving up to the
SoftAP config portal. A device that moves between two sites should just join
the one that is there.

## Why

The developer moves the hardware between an office network (`test123`) and a
home network (`Skeyelab`). The firmware stores exactly one SSID, so at the
other site it burns the whole 45 s connect budget (`WIFI_CONN_TIMEOUT_US`),
falls to SoftAP, and needs credentials re-entered through the captive portal.
Every trip, in both directions.

Found 2026-07-10 the hard way: the P4 had `test123` in NVS while the
compile-time seed said `Skeyelab`, so it quietly joined a different network
than everyone assumed and appeared on an unexpected subnet. Twenty minutes
went into "is the routing broken" before anyone checked which SSID it had
actually joined. The single-credential model made a normal situation look
like a fault.

`X32_emulator.ino` already does the crude version -- `tryConnect(HOME)` then
`tryConnect(GUEST)`. The product firmware never learned it.

## Design

This lands almost entirely in a module that already exists and is already
pure. `X32Link/wifi_conn_policy.c` (ARC-013) is a host-tested state machine
whose whole job is "when do we give up to AP". Adding "try the next known
network first" is a new state, not a rewrite:

    WCS_CONNECTING(slot n) --timeout--> WCS_CONNECTING(slot n+1) --exhausted--> WCS_AP

Keep the per-slot budget well under today's 45 s total, or a three-slot device
takes over two minutes to reach the config portal. Suggest ~15 s per slot.

`KsConfig` grows N credential slots (3 is plenty). **This is a layout change:
bump `KS_CONFIG_VERSION`** -- the `_Static_assert` on `sizeof(KsConfig)` will
force it, and P4-014's decode guard will reset any stale blob to defaults
rather than loading garbage. That is the mechanism working as designed.

Slot 0 keeps today's `wifi_ssid`/`wifi_pass` semantics so nothing else has to
change; the compile-time seed (`CONFIG_KS_WIFI_SSID`) still only fills an
empty slot 0.

## Scope

- N credential slots in `KsConfig` (SSID + password), N = 3.
- `wifi_conn_policy` walks the slots, then falls to AP. Pure, host-tested:
  the per-slot budget, the exhaustion edge, and "a slot with an empty SSID is
  skipped, not attempted".
- Web UI: list saved networks, add one, forget one. Blank password on an
  existing slot means "keep current", same rule `ks_config_set` already has.
- Log which slot/SSID was joined, next to the IP. The whole confusion above
  came from not knowing which network the board picked.

## Non-goals

- Scanning to pick the strongest known network. Faster and more correct when
  neither or both are present, but it costs a scan at every boot and is a
  bigger change. Revisit if blind ordering proves annoying.
- Roaming between APs mid-session. This is about boot-time selection.

## Acceptance Criteria

- [x] Device with two saved networks joins whichever is present, at either site
- [x] Empty slots are skipped, not attempted
- [x] All slots exhausted -> SoftAP, and not slower than ~45 s total
- [x] Joined SSID logged alongside the IP at boot
- [x] Web UI can add and forget networks; blank password keeps the stored one
- [x] Slot walk + exhaustion covered in `test/test_wifi_conn_policy.c`
- [x] `KS_CONFIG_VERSION` bumped; an old blob resets to defaults (P4-014)

## Notes

- Pairs with ESP-012 (mDNS): knowing the *name* stops mattering which address
  DHCP hands out, and this stops mattering which network it joined.
- X32Link has the identical problem and shares `wifi_conn_policy.c`. Once the
  pure policy is widened, its glue is a small follow-up.

## Done 2026-07-10 (PR #51)

Two criteria did not land exactly as written. Both deserve saying out loud.

**"an old blob resets to defaults"** -- it does not. It MIGRATES. Writing this AC,
I had P4-014's guard in mind and treated the reset as the safe outcome. On the
bench it is the opposite: resetting to defaults forgets the network and strands
the device in the SoftAP portal, which is the precise ritual this ticket exists
to abolish. So `ks_config_decode()` grew a v1 -> v2 migration (frozen v1 layout,
its own `_Static_assert`, duplicated independently in the test). The guard is
not weakened: a migrated blob faces `ks_config_valid()` like any other, a v1
blob of the wrong size is still defaults, and an unversioned blob that happens
to be v1-sized is still legacy.

**"joins whichever is present, at either site"** -- verified as *slot 0
unreachable, slot 1 real*: the walk fired, the STA re-pointed, an IP arrived on
slot 2/2. NOT yet verified by physically carrying the board between the office
and home networks. The state machine is right; the two-real-sites case is
untested.

Two bugs surfaced only on hardware, neither reachable from the host suite:

- `/save` parsed its body from a 1024-byte STACK local. Three url-encoded
  SSID/pass pairs overflow it, so I grew it to 3072 -- straight into a
  stack-protection panic, because the `httpd` task's whole stack is ~4 KB. It
  presented as "save silently does nothing", since the crash-reboot is
  indistinguishable from the intended post-save reboot. Body is heap-allocated
  now, and an over-long body is refused rather than truncated and half-applied.
  Recorded in `AGENTS.md`.
- SSID was interpolated into an HTML `value=` unescaped (pre-existing; this
  tripled the count). `attr_esc()` added.

CI caught a third: `X32Link.ino` read `connect_start_us`, which this renamed.
The host suite does not compile `.ino` files -- compile X32Link with
`arduino-cli` locally before pushing.

Follow-ups, neither blocking:

- X32Link still passes `nslots=1`; widening its glue is a small change.
- `ks_web.cpp` remains untested (ARC-017), which is why both hardware bugs lived
  there. The stack overflow in particular is exactly the class of defect a host
  test never sees.
