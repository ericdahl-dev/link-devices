## Goal

Bring up a new board target for X32Link: the **Waveshare ESP32-S3-Touch-LCD-1.47**.
Get the display + touch controller + LVGL initialized and rendering a static
splash screen. No interactive UI yet — that's LNK-015. No touch event wiring
beyond confirming raw coordinates print to Serial.

This is the hardware foundation for an on-device touch UI that gives X32Link
full feature parity with the web config UI (LNK-015), so the unit can be
configured and monitored without a phone/browser on the same network.

## Target board

**Waveshare ESP32-S3-Touch-LCD-1.47** — https://www.waveshare.com/esp32-s3-touch-lcd-1.47.htm

| | |
|---|---|
| SoC | ESP32-S3R8, dual-core LX7 @ 240MHz |
| RAM | 512KB SRAM + 8MB PSRAM (stacked) |
| Flash | 16MB |
| Display | 1.47" IPS, 172×320, 262K color, **JD9853** driver (SPI, ST7789-compatible command set) |
| Touch | Capacitive, I2C — **chip identity is unconfirmed, see note below** |
| Storage | TF card slot |
| USB | Type-C, native USB-CDC |

TF card SPI pins are not needed for this task (no SD usage planned) — skip
bringing those up.

### ⚠️ Touch controller chip discrepancy — resolve on first boot, don't guess

Waveshare's own documentation disagrees with itself across pages: the
product wiki lists the touch chip as **AXS5106L**, while an ESP-IDF demo
snippet for the same board references **CST816D** (I2C addr 0x15, R=0x2A,
W=0x2B). Do not pick one from docs. Instead:

1. On first bring-up, run a raw I2C bus scan on BOTH candidate buses —
   `Wire.begin(8, 9)` (Waveshare example code) and `Wire.begin(42, 41)`
   (in-session subagent claim) — since the sources conflict; log every
   responding address to Serial and note which bus the touch chip answers on.
2. Match the address against the AXS5106L and CST816D datasheets to
   identify which chip is actually populated on the unit in hand.
3. Bring up the matching driver only. Note the confirmed chip in this
   task's Progress section so LNK-015 doesn't have to re-derive it.

## Implementation

- **Display**: use `Arduino_GFX_Library` (`Arduino_ESP32SPI` bus +
  `Arduino_ST7789` panel class — JD9853 accepts the same init command set).
  Register it as an LVGL display driver (`lv_init()` + flush callback), same
  shape as `tsv_display.cpp`'s LVGL setup but with a raw `Arduino_GFX`
  backend instead of the Waveshare 4.3" BSP (no vendor BSP library exists
  for this board's exact chip pairing — write the panel/touch glue directly).
- **Touch**: bring up whichever I2C touch driver LNK-014 step 1–3 confirms.
  Register it as an LVGL input device once confirmed; for this task it's
  sufficient to print raw `(x,y)` touch coordinates to Serial on contact —
  LVGL input-device registration can land in LNK-015 alongside the first
  screen that needs it.
- **New module**: `esp32/X32Link/touch_display.h` / `touch_display.cpp` with
  a minimal interface for this task:
  ```cpp
  void touch_display_begin(void);  // panel + touch + LVGL init, draws splash
  void touch_display_tick(void);   // lv_timer_handler(); call from loop()
  ```
- **Board gating**: add `#define HAS_TOUCH_DISPLAY` to this board's
  `build_opt.h` (new board profile, does not affect existing Super
  Mini/XIAO builds). Guard `touch_display_begin()` / `touch_display_tick()`
  calls in `X32Link.ino` with `#ifdef HAS_TOUCH_DISPLAY` so firmware for
  the existing screenless boards is unaffected.
- **Splash**: a static LVGL screen — "X32·SYNC" wordmark + "booting…" — close
  in spirit to the web UI's panel header (`web_config.cpp`'s `.brand` /
  `.wordmark`), proving the full display pipeline (SPI init → LVGL → flush)
  works before any live data is wired in.

## Notes

- No vendor BSP/component library wraps display+touch init together for
  this exact board (unlike the 4.3" board's `Waveshare_ESP32_S3_Touch_LCD_43`
  library) — write the glue directly against `Arduino_GFX_Library` + the
  confirmed touch driver, LVGL-registered by hand.
- PSRAM: allocate the LVGL draw buffer in PSRAM
  (`heap_caps_malloc(..., MALLOC_CAP_SPIRAM)`), same pattern as `tsv_display.cpp`.
- Keep `touch_display.*` independent of `web_config.*` — both read/write the
  same `g_config` / `config_save()`, neither owns the other.

## Progress

- 2026-07-02 — **`touch_display` module built, splash + touch working on hardware**
  (raw **LovyanGFX, no LVGL** — see the decision in LNK-015; LVGL dropped as
  overkill for this UI). Files: `X32Link/touch_display.{h,cpp}` (+ `axs5106l.{c,h}`
  pure parser, host-tested). Gated by `HAS_TOUCH_DISPLAY`; `X32Link.ino` calls
  `touch_display_begin()` (splash, pre-WiFi) + `touch_display_tick()` (loop).
  Full X32Link + LovyanGFX = **85% flash** (over the 80% budget note; LVGL would
  be worse). Gotchas found the hard way, for LNK-015:
  * **Backlight must be driven manually** — `pinMode(46,OUTPUT)+digitalWrite HIGH`.
    LovyanGFX `Light_PWM` (LEDC) drove the backlight fine in a standalone sketch
    but left it DARK in the full firmware (LEDC conflict). Don't use Light_PWM here.
  * **AXS5106L read**: write reg 0x01 then a STOP (`endTransmission(true)`) then
    read — a repeated-start (`false`) returned no data. Reference frame is **6
    bytes** `[gesture,num,xH,xL,yH,yL]`; the current code over-reads 14 and works
    (chip pads) but should be tightened to 6 (matches `axs5106l.c`; parser already
    decodes from the first 6). Chip also has a soft-reset (`reg 0xF0 <- {B3,55,AA,
    34,01}`) that isn't currently sent — add if reads ever get flaky.
  * **Touch→screen mapping** (rotation 4): X maps direct, **Y is inverted**
    (`dy = 319 - y`). That's the whole transform for the current orientation.
  TODO before "done": tighten read to 6 bytes, drop the on-screen debug readout in
  `touch_display_tick()`, register touch as a real input device for LNK-015.
- 2026-07-02 — **Display validated on glass** (LovyanGFX, not Arduino_GFX —
  LovyanGFX 1.2.21 + lvgl 9.4.0 were already installed; Arduino_GFX was not).
  Working config: `lgfx::Panel_ST7789` (JD9853 speaks the ST7789 command set) on
  `Bus_SPI` SPI2 @ 40MHz, pins SCK=38/MOSI=39/DC=45/CS=21/RST=40, backlight
  `Light_PWM` on BL=46. Panel: memory 240x320, visible 172x320,

  variant — plain rotation 0 rendered the text horizontally mirrored; JD9853
  MADCTL default differs from ST7789). Colors/offset/text all correct. NOTE:
  LovyanGFX **1.2.7 fails to compile** on esp32 core 3.3.8
  (`gpio_hal_iomux_func_sel` renamed in Bus_RGB.cpp) — must use >= 1.2.21.
- 2026-07-02 — **Touch chip confirmed: AXS5106L @ I2C 0x63** (7-bit), via an
  on-hardware I2C scan on the schematic-verified touch bus (SDA=42/SCL=41, TP_RST
  GPIO47 pulsed). Resolves the AXS5106L-vs-CST816D doc conflict: the Waveshare
  wiki was right (AXS5106L), the ESP-IDF demo snippet wrong (CST816D=0x15 not
  present). Report format (from ChipSourceTek part + toto04/axs5106l ref): write
  reg 0x01, read 14 bytes; data[1]=finger count; point i at 2+i*6 as
  [status,x_hi,x_lo,y_hi,y_lo,pressure]; X/Y 12-bit, coord high nibble in low 4
  bits of x_hi/y_hi (status flags in high nibble). Pure parser +
  host tests landed: X32Link/axs5106l.{c,h}, test/test_axs5106l.c (5 tests green).

- Hardware note for LNK-015/LNK-021: **this board has NO onboard WS2812 /
  addressable LED.** The "Colorful RGB LED" in Waveshare marketing belongs to
  the *non-touch* ESP32-S3-LCD-1.47 USB-A dongle (a different PCB, RGB on
  GPIO38). On THIS board GPIO38 is LCD_SCL; the only LEDs are a charger-status
  LED and an always-on power LED, neither MCU-driven. A per-beat indicator here
  must use the LCD itself or the backlight (GPIO46). (An interim build drove
  GPIO46 as a beat strobe but that's not independently confirmed; the shipped
  headless build uses LED_NONE and relies on the web /status beat dot.)
