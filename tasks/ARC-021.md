## Problem

PR #65 pulled the touch-edge *dispatch* (press / release / cue tap arbitration)
out of `ktouch_display.cpp` into a pure, host-tested `ktouch_touch_step`
(`ktouch_ui.{c,h}`). That was the tap half of the display deepening. The
**render-side derivations are still smeared into the LovyanGFX draw calls** in
`KitchenSyncTouch/ktouch_display.cpp` (verified on master), where they are
untestable by construction:

- **sync-wheel phase angle** — `fmodf(phase / (float)g_config.quantum_beats, 1) * 360`
  mixed into `fillCircle`/`cosf`/`sinf` (`ktouch_display.cpp:247`).
- **status line** — AP setup text vs mDNS name vs "no wifi", branched inline on
  `g_ap_mode` / `WiFi.status()` (`ktouch_display.cpp:237-238`).
- **BPM formatting** — inline `printf("%3.0f")`.
- **sync-state** — `phase_valid ? 1 : (active ? 0 : -1)` derived inline.

X32Link already solved exactly this: `X32Link/touch_ui.c` isolates the identical
logic as pure functions (`ui_phase_angle`, `ui_bpm_str`, `ui_hit`) with host tests
(`test_touch_ui`). Touch re-derives the same math on the wrong side of the seam —
one deep module's worth of logic living inside draw code that no test can reach.

**Deletion test:** move these derivations into `ktouch_ui` and the complexity
concentrates in one tested module; `ktouch_display.cpp` shrinks to draw calls.
Nothing genuinely display-specific is lost — the LovyanGFX calls stay; only the
math leaves.

## Solution

Extend the pure `ktouch_ui` module (the one PR #65 established) with the
render-side derivations — sync-wheel angle, BPM string, sync-state, and the
status-line selection — mirroring X32Link's `touch_ui.c`. `ktouch_display.cpp`
becomes a thin adapter: it calls the pure functions and draws their results.
Reuse X32Link's `ui_phase_angle` / `ui_bpm_str` directly where the logic is
identical (they are already compiled into the host suite), rather than adding a
parallel Touch copy — this is also an ARC-018-style "reference, don't fork" call.
