## Problem

ADR-0006/0007 say the shared pure C lives in one place and is compiled into each
firmware by path (KitchenSync does exactly this via `X32LINK_DIR` in
`KitchenSync/main/CMakeLists.txt`). **KitchenSyncTouch does not follow it for the
timing core** — it carries byte-identical *copies* of the shared engines:

| Forked copy (Touch) | Source of truth (X32Link) |
|---|---|
| `KitchenSyncTouch/clock_ticker.{c,h}` | `X32Link/clock_ticker.{c,h}` |
| `KitchenSyncTouch/link_phase.c` | `X32Link/link_phase.c` |
| `KitchenSyncTouch/midi_clock_out.c` | `X32Link/midi_clock_out.c` |
| `KitchenSyncTouch/link_protocol.c` | `X32Link/link_protocol.c` |
| `KitchenSyncTouch/transport_launch.c` | `X32Link/transport_launch.c` |

`diff` reports every pair IDENTICAL today. That is the danger, not the comfort:
this is **one deep module presented as two adapters that must be hand-synced**. A
fix to `clock_ticker.c`'s catch-up-burst suppression, or to `transport_launch`'s
arm/fire state machine, must be applied twice, and the copies can drift silently —
CI compiles both but never compares them. The pure engines are host-tested once
(`test/test_clock_ticker.c`, `test_transport_launch.c`, `test_link_phase.c`), so
the tests protect the X32Link source while the Touch copy rides on faith.

**Deletion test:** delete the Touch copies and point the build at the single
source — complexity vanishes, nothing reappears. This is duplication, not
divergence.

Touch already reuses the Link stack + `fw_version` from X32Link by committed
relative symlink (per the ESP-016 foundation, PR #60). These five files were
forked instead of joining that mechanism — an oversight, not a decision.

## Solution

Compile the single X32Link source into KitchenSyncTouch the same way the Link
stack already arrives — committed relative symlinks (mode 120000) into
`../X32Link/`, or the arduino-cli sketch build's existing shared-path scheme —
and delete the forked copies. No behaviour change: the bytes are identical today,
so this is a build-graph change plus five deletions.

Confirm first that no copy carries a Touch-only edit (grep + `diff` in the
acceptance criteria). If one does, that edit is either a bug-fix that belongs
upstream in X32Link or a genuine divergence that earns a comment — resolve it
explicitly, don't preserve it by keeping the fork.

## Acceptance Criteria

- [ ] `diff` confirms each forked file is byte-identical to its X32Link source
      *before* removal; any delta is triaged (upstreamed or documented) not lost.
- [ ] `KitchenSyncTouch/{clock_ticker.{c,h},link_phase.c,midi_clock_out.c,link_protocol.c,transport_launch.c}`
      are gone; the sketch builds the X32Link source by symlink/path instead.
- [ ] KitchenSyncTouch compiles (arduino-cli, Waveshare S3 board) — flash % noted.
- [ ] `test/Makefile` still compiles these engines from the X32Link path (it
      already does for the host suite); host suite green.
- [ ] `AGENTS.md` module map + `check_docs.sh` reflect the single source.
- [ ] Hardware: Touch still boots, joins Link, and ticks DIN clock (no bench
      board today — mark NOT VERIFIED if unavailable, like ARC-017).

## Notes

- **Under-application of ADR-0006/0007, not a contradiction.** No ADR reopens.
- Legitimately per-firmware and NOT in scope: the tick *task* glue
  (`ktouch_midi_out.cpp` DIN-only writer vs `ks_tick.c` vs
  `midi_clock_out_io.cpp` USB+DIN). Those emit differently for real hardware
  reasons — see ARC-019 for unifying the *derivation* they wrap.
- Unblocks ARC-019: you cannot route three tick tasks through one shared
  derivation module while that module is itself forked.
