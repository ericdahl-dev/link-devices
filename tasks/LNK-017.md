## Goal

Extend `link_protocol.c`'s TLV parser to read the **full** `tmln` (Timeline)
payload — currently it only reads the first 8 bytes (`microsPerBeat`) and
ignores the rest. The full payload carries `beatOrigin` and `timeOrigin`,
which are required to compute actual beat/bar **phase**, not just BPM. Also
parse the `mep4` TLV (a peer's unicast measurement-service endpoint), needed
by `LNK-018` to run the ping/pong clock-sync exchange against a real peer.

This is pure protocol-parsing work — no networking changes, no UDP sockets.
Everything here is host-testable in `esp32/test/`, same as the existing
`link_protocol.c`/`test_link_protocol.c` split.

Reference: the real Ableton Link wire format was confirmed by reading the
vendored SDK source at `esp32/lib/link/include/ableton/link/{Timeline,PeerState,PayloadEntries}.hpp`
and `esp32/lib/link/include/ableton/discovery/NetworkByteStreamSerializable.hpp`
(submodule — run `git submodule update --init esp32/lib/link` if not checked
out locally). We are **not** linking that SDK at runtime (see LNK-002/LNK-003
— its networking layer depends on asio, which doesn't build for arduino-esp32).
We're using it only as a byte-format spec to hand-roll the same wire protocol
in plain C, exactly as `link_protocol.c` already does for the `tmln`
tempo-only field.

## Wire format (confirmed from SDK source)

### `tmln` (Timeline) — currently parsed as 8 bytes, actually 24 bytes

```
key  = 0x746d6c6e ('tmln')
size = 24
value:
  bytes  0-7  int64 BE   micros_per_beat       (what we already parse)
  bytes  8-15 int64 BE   beat_origin_micro     (fixed-point: beats = value / 1e6)
  bytes 16-23 int64 BE   time_origin_us        (microseconds, in the session's
                                                 shared GHostTime domain — see LNK-019)
```

### `mep4` (MeasurementEndpointV4) — new TLV to parse

```
key  = 0x6d657034 ('mep4')
size = 6
value:
  bytes 0-3  uint32 BE   IPv4 address
  bytes 4-5  uint16 BE   UDP port (the peer's measurement/ping-pong service)
```

Absence of `mep4` in a given Alive/Response packet is valid (e.g. an
IPv6-only peer never sends it, per the zero-size-entries-omitted TLV rule) —
treat a peer with no recorded `mep4` as "not measurable," don't error.

## Implementation

- Extend the TLV loop in `link_protocol.c`'s `link_proto_parse()`:
  - `tmln`: read all 24 bytes into a new `static Timeline s_timeline` (or
    equivalent), not just `s_bpm`. Keep `link_proto_bpm()` working unchanged
    (still derived from `micros_per_beat`).
  - `mep4`: record `(ip, port)` on the matching `Peer` entry in `s_peers[]`
    (extend the `Peer` struct with `uint32_t mep4_ip; uint16_t mep4_port;
    bool has_mep4;`).
- New accessors in `link_protocol.h`:
  ```c
  typedef struct { int64_t micros_per_beat; int64_t beat_origin_micro; int64_t time_origin_us; } LinkTimeline;
  bool link_proto_timeline(LinkTimeline* out);     // false if no tmln seen yet
  bool link_proto_peer_endpoint(int index, uint32_t* ip, uint16_t* port);  // false if no mep4
  ```
- `link_proto_reset()` / `link_proto_tick()` (peer TTL expiry) must also
  clear/drop `mep4` state for expired peers — don't leave stale endpoints
  pointing at a peer that already timed out.

## Notes

- Don't touch `link_listener.cpp` in this task — it's pure `link_protocol.c`
  parsing work. `LNK-018` wires the new endpoint data into an actual UDP
  ping/pong client.
- `beatOrigin`'s fixed-point encoding (`micro-beats`, i.e. `llround(beats * 1e6)`)
  is exactly how the SDK serializes it — don't reinterpret it as a plain
  int64 beat count or a double; divide by `1e6` when converting to beats.

## Progress

Done. Built via TDD vertical slices (one test, minimal impl, repeat) in
`esp32/test/test_link_protocol.c`.

**What was built:**

- `link_protocol.h`: added `LinkTimeline` struct (`micros_per_beat`,
  `beat_origin_micro`, `time_origin_us`, all `int64_t`) and two new
  accessors: `link_proto_timeline(LinkTimeline*)` and
  `link_proto_peer_endpoint(int index, uint32_t* ip, uint16_t* port)`.
- `link_protocol.c`:
  - `Peer` struct extended with `mep4_ip`, `mep4_port`, `has_mep4`.
  - TLV loop now reads all 24 bytes of `tmln` (was 8): `micros_per_beat`
    still drives `s_bpm` unchanged; `beat_origin_micro`/`time_origin_us`
    read from bytes 8-23 into `s_timeline` when `size >= 24`. A new
    `s_timeline_seen` flag backs `link_proto_timeline()`'s false-until-seen
    contract.
  - New `mep4` TLV (key `0x6d657034`, 6-byte value) parsed and attached to
    the peer matched by NodeId (`find_peer`) already upserted earlier in
    the same `link_proto_parse()` call.
  - `upsert_peer()` explicitly zeroes `mep4_ip/port/has_mep4` when
    allocating a brand-new peer slot, since slots are reused via
    swap-remove on expiry/ByeBye and could otherwise leak a previous
    occupant's stale endpoint into a new peer at the same index.
  - `link_proto_reset()` clears `s_timeline`/`s_timeline_seen` too.
  - `link_proto_tick()`/`remove_peer()` needed no new code for mep4
    cleanup — dropping a peer via swap-remove already discards its whole
    `Peer` struct (mep4 fields included); the slot-reuse zeroing above is
    what actually prevents stale data, and is covered by a dedicated test.
