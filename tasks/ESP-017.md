## Design principle (decided 2026-07-11)

**KitchenSync (P4) and KitchenSync Touch (S3) are the same product.** They must be
feature-equivalent. They differ ONLY in:

1. **hardware** — what the box physically has (4 USB-MIDI cables vs 1 DIN jack; a
   speaker vs no speaker; a WS2812 strip vs an LCD; a mic vs no mic), and
2. **exposure** — how a feature is surfaced (a web dropdown vs a touch gesture).

A feature must never be missing from one device just because nobody wired it up. If a
feature cannot exist on a device, that must be a HARDWARE fact, written down here --
not an accident of implementation.

This is a standing rule for every future feature: build it once in the pure layer
(ADR-0003), then expose it on both.

## The bug this principle catches

The Touch has no swing. The reason is not that swing is hard -- it is that **the Touch
opted out of the shared clock engine**.

The pure, host-tested `clock_output_due()` already applies **division + phase + swing**
in one call, and the P4 uses it:

```c
int clock_output_due(ClockTicker* t, double beats_now, int ppqn,
                     int phase_mbeats, int swing_mbeats, int max_burst);
```

`ktouch_midi_out.cpp` instead calls the simpler `midi_clock_out_ticks_due()` at a
hardcoded 24 PPQN, and the Touch's nudge was HAND-ROLLED on top of it
(`clk_beats = beats + nudge/1000.0`) -- reimplementing a subset of an engine that
already existed and was already tested.

So the Touch is not missing swing. It is missing `clock_output`.

## Fix: adopt clock_output on the Touch

Switching `ktouch_midi_out` to `clock_output_due()` delivers, in one change:

- **swing** (`swing_mbeats`) -- free, already tested (P4-013)
- **division / rate** (`ppqn`: 24, x2, /2, 1/16 ...) -- free, already tested (P4-010)
- **phase nudge** (`phase_mbeats`) -- and DELETES the hand-rolled version

Then add the fields to `AppConfig` + the web form, mirroring the P4's controls.

## Parity audit (2026-07-11)

### Software gaps -- Touch should have these, no hardware reason not to

| feature | P4 | Touch | notes |
|---|---|---|---|
| **swing** | yes | **no** | pure `swing.c` already shared |
| **division / ppqn** | yes | **no** (fixed 24) | pure `clock_output.c` already shared |
| **phase nudge** | yes | yes (hand-rolled) | fold into `clock_output` |
| **multi-network WiFi** | 3 slots | 1 SSID | pure `wifi_conn_policy` already shared |
| **follow Link transport** | yes (per-output `follow_link`, ESP-011) | **no** (always self-masters) | the Touch cannot currently follow a Link session's transport |
| **OTA `/update`** | yes | **no** | `ui_update_page` already shared |

### Hardware-differentiated -- NOT gaps, but the feature should still exist, exposed differently

| feature | P4 | Touch | how Touch expresses it |
|---|---|---|---|
| clock outputs | 4 (USB cables) | 1 (DIN) | one output; DIN is the product |
| audible metronome | speaker | none | n/a -- no speaker |
| visual metronome | WS2812 strip | LCD | the sync wheel IS the Touch's visual metronome |
| tempo-follow (mic) | yes | none | n/a -- no mic |
| brightness | LED strip | LCD backlight | same idea, different surface |

### Touch-only (fine -- an interaction, not a feature gap)

- `play_on_release` (turntable feel) -- a touch gesture; the P4 has no touchscreen.

## Acceptance criteria

- [ ] `ktouch_midi_out` uses `clock_output_due()`; the hand-rolled nudge is deleted
- [ ] Touch web UI exposes SWING and RATE, mirroring the P4's controls
- [ ] Swing verified on the analyzer on the Touch's DIN (GPIO11): the flat 0xF8 spacing
      breaks into the alternating long/short 12-pulse groups, as it did on the P4
- [ ] Division verified on the analyzer: measured PPQN tracks the configured rate
- [ ] Multi-network WiFi + OTA `/update` on the Touch (or an explicit written decision
      that they are out of scope for the Dan wedge)
- [ ] This parity table kept current -- any NEW feature lands on both, or states here
      which hardware fact prevents it

## Notes

- The pure engines are already shared and host-tested. This is almost entirely a
  wiring + exposure job, not new logic -- which is exactly what ADR-0003 was for.
