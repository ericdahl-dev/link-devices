## Problem

P4Hub currently ships with the ESP-IDF default single-app partition table — no
OTA slot, so every update is a USB-C cable + `idf.py flash`. Same pain as
X32Link (LNK-034), and once the P4-NANO is mounted at the hub end of a rack
with only its USB-A host port exposed (occupied by the downstream MIDI gear),
re-cabling is worse than an annoyance.

## Solution

- **Partition table**: switch to the built-in `CONFIG_PARTITION_TABLE_TWO_OTA`
  (`sdkconfig.defaults`) — `otadata` + `ota_0` + `ota_1` (~1.5 MB each), no
  factory slot. Every OTA push targets whichever slot isn't currently running.
- **`GET /update`** (`p4hub_web.cpp`): a themed page with a file input + a
  small fetch-based uploader (`fetch('/update',{method:'POST',body:file})` —
  the file's raw bytes as the request body, no multipart parsing needed on
  either side).
- **`POST /update`**: `esp_ota_get_next_update_partition()` picks the inactive
  slot; the request body is streamed in 4 KB chunks straight into
  `esp_ota_write` (so the whole image never sits in RAM), then `esp_ota_end` +
  `esp_ota_set_boot_partition` commit it and the device reboots. Any failure
  (`esp_ota_begin`/`write`/`end`/image validation) aborts the write and reports
  a plain-text 500 with the reason; the currently-running slot is never
  touched until the new one is verified and committed.
- `main/CMakeLists.txt`: added `app_update` + `esp_partition` to
  `PRIV_REQUIRES` for `esp_ota_ops.h`.

## Deliverable

`P4Hub/sdkconfig.defaults` (`CONFIG_PARTITION_TABLE_TWO_OTA=y`),
`P4Hub/main/CMakeLists.txt` (component requires), `P4Hub/main/p4hub_web.cpp`
(`/update` GET + POST handlers, registered in `p4hub_web_start`), README
capability table + Configuration section updated. `idf.py build` (esp32p4)
covers it in CI; on-device flash-and-boot-into-new-slot verification pending
hardware access.
