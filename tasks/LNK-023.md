## Goal

Root-caused live on real hardware: `X32Link.ino`'s `setup()` only creates
`bpm_task`/`led_task` *after* `wifi_try_connect()` succeeds
(`if (!wifi_try_connect()) start_config_ap(); // never returns on AP path`,
then task creation). `start_config_ap()` loops forever servicing the captive
portal and never returns — so if WiFi fails (e.g. factory-default SSID
`"X32-Emulator"` doesn't match the real network), `bpm_task`/`led_task`
**never get created at all**. Result: the LED can't blink, and
`tempo_source_phase()`/`LNK-021`'s phase-lock can't run, even for **USB MIDI
clock** — which doesn't need WiFi at all. Confirmed live: device receiving
real MIDI clock from Ableton, stuck in AP fallback, zero LED activity.

Two changes:
1. Make tempo/LED processing work **independent of WiFi outcome**.
2. Add a **WiFi-down status indicator**: an occasional triple-blink on the
   same LED, so there's a visible "not connected" signal without a screen.

## Part 1 — decouple task creation from WiFi

Current `setup()` order (`X32Link.ino`):
```
tempo_source_select(); tempo_source_pre_net();
if (!wifi_try_connect()) start_config_ap();   // BLOCKS FOREVER on failure
tempo_source_begin(); osc_sender_begin();
xTaskCreatePinnedToCore(bpm_task, ...);
xTaskCreatePinnedToCore(led_task, ...);
web_config_begin();
```

Target order:
```
tempo_source_select(); tempo_source_pre_net();

g_bpm_mutex = xSemaphoreCreateMutex();
xTaskCreatePinnedToCore(bpm_task, ...);   // moved up — unconditional
xTaskCreatePinnedToCore(led_task, ...);   // moved up — unconditional

bool wifi_ok = wifi_try_connect();
if (wifi_ok) {
    tempo_source_begin();   // Link multicast join — only meaningful with an IP
    osc_sender_begin();
    web_config_begin();
} else {
    start_config_ap();      // now NON-BLOCKING: WiFi.mode(AP)+softAP+web_config_ap_begin()
                             // only, then returns. Track `s_ap_mode = true`.
}
```

`start_config_ap()` rewrite: strip its `for (;;) web_config_handle();` —
`loop()` already calls `web_config_handle()` every iteration, and
`web_config_handle()` is already AP/STA-agnostic (`if (s_captive)
s_dns.processNextRequest(); server.handleClient();` — no change needed
there). Just stop blocking; let `setup()` fall through to `loop()`.

`loop()` needs an `s_ap_mode` check so its existing STA-reconnect block
doesn't fight AP mode:
```c
if (!s_ap_mode && WiFi.status() != WL_CONNECTED) {
    wifi_connect();          // existing STA-reconnect logic, untouched
    tempo_source_begin();
    osc_sender_begin();
}
web_config_handle();
```

**Verify before relying on it** (defensive — don't assume, check): with
`tempo_source_begin()` not yet called (AP-fallback / still-connecting
state), confirm `tempo_source_poll()`/`tempo_source_active()`/
`tempo_source_bpm()` don't crash for the Link adapter — `link_listener_poll()`
calls `WiFiUDP::parsePacket()` on a socket that hasn't called
`beginMulticast()` yet. Arduino's `WiFiUDP` is expected to no-op/return 0 in
that state, but confirm on real hardware, don't assume from reading the
library source alone. MIDI adapter is unaffected either way (`tempo_source_poll()`
is already a no-op for MIDI — pulses are interrupt/task-driven independent
of WiFi).

**Out of scope**: retrying STA from AP-fallback without a reboot. Current
behavior (AP mode is terminal until the user reconfigures via the captive
portal and it reboots via `handle_save()`'s `ESP.restart()`) is unchanged —
only the *blocking* nature of getting there changes.

## Part 2 — WiFi-down triple-blink

When `WiFi.status() != WL_CONNECTED` (covers both "still attempting initial
connect" and "parked in AP fallback"), `led_task` fires three quick blinks
every few seconds, distinct from the single per-beat flash:

```c
#define WIFI_DOWN_BLINK_ON_MS   60
#define WIFI_DOWN_BLINK_GAP_MS  80
#define WIFI_DOWN_BLINK_COUNT   3
#define WIFI_DOWN_BLINK_INTERVAL_MS 4000   // how often the triple-blink repeats
```

Add to `led_task()`'s loop: track `last_wifi_blink_ms`; when
`WiFi.status() != WL_CONNECTED` and the interval has elapsed, run the
3x-on/off sequence (blocking the task briefly, ~420ms total — acceptable,
`led_task` is low-priority and dedicated). This can collide in time with a
single beat-flash from `tempo_source_phase()`/`tempo_source_beat()` (e.g.
MIDI clock running while WiFi is down, the exact case found live) — accept
the occasional overlap rather than building priority/queueing logic; the
two patterns (one 30ms flash vs. three 60ms flashes) are visually distinct
enough not to confuse the operator. Document this as a known limitation,
don't over-engineer it.

## Implementation

- Extract the trigger decision as a pure, host-testable function (mirror
  `LNK-021`'s `led_phase_should_flash()` pattern) — e.g.
  `bool wifi_down_blink_due(uint32_t now_ms, uint32_t last_blink_ms, uint32_t interval_ms, bool wifi_connected)`
  in a small new module (or alongside `led_phase.c` if that's a natural
  fit — your call). Put the actual GPIO sequencing in `led_task()` itself
  (Arduino-only, not host-testable, same as the rest of that task).
- `s_ap_mode` (or equivalent) needs to be visible to both `setup()`/`loop()`
  — a simple static/global in `X32Link.ino` is fine, no need to expose it
  elsewhere.

## Notes

  scope, keep this ticket's blast radius to "don't block LED/MIDI on WiFi."
- Don't try to coordinate `led_task`'s two blink behaviors (beat-flash,
  WiFi-down triple-blink) through anything fancier than "both live in the
  same task's sequential loop, occasional visual overlap is acceptable."
