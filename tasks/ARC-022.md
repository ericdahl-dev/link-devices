## Problem Statement

The user changes a setting in the web UI. It takes effect immediately -- they hear it, they
see it. They reboot the device. **It is gone, reverted to whatever was last written by an
explicit Save.** No warning, no "unsaved changes" marker. The UI showed the new value right
up until the power cycle, so there was nothing to tell them the setting was never real.

The owner hit this from the user side and reported it as a display bug:

> "my metronome voice settings are not reflected in the UI when i reboot"

The dropdown was *also* failing to render the saved value, and that was fixed. But the
underlying report was pointing at something much bigger, and fixing the dropdown hid it: on
the P4, `ks_config_save()` is called from **exactly one place** -- the `/save` handler.
The `/live` handler copies the incoming values into the running config via
`ks_config_live_safe_copy()` and **never writes NVS at all.**

So *every* field in the live-safe set is ephemeral. That set is not a corner of the product.
It is:

- `clock_out_enable`
- metronome `accent`, `volume`, `voice`
- LED `enable`, `brightness`, `mode`, `fade`, `beat_color`, `accent_color`
- **the entire `clock[o]` struct, for all four outputs** -- enable, cable, PPQN division,
  phase nudge, swing, and `follow_link`

On the Touch, the same shape: `/nudge` and `/bright` mutate `g_config` and return. Only the
`/save` form path reaches NVS.

### The sharpest instance

`follow_link` -- the per-output transport master built in this cycle (P4-010 / ESP-011) --
lives in `clock[o]`, which `ks_config_live_safe_copy()` copies wholesale. Its toggle in the
web UI is a `.live` control.

**So: set an output to manual transport, reboot, and it is back to following Link.** The
feature works perfectly and then forgets itself. It is the newest thing in the product and it
is already subject to the oldest bug in it.

### Why this is worse than it sounds

The failure is **invisible and delayed**. The user configures four outputs -- divisions,
swing, nudge, which ones follow Link -- gets the rig exactly right, plays a set, powers down,
and arrives at the next one with a device that has quietly forgotten the whole arrangement.
There is no error to search for and nothing on screen ever looked wrong. The natural
conclusion is "this thing doesn't save its settings," which is a product-trust failure, not a
config-layer one.

And there is a trap on the way out: an explicit Save posts the **form**, so it writes the
form's idea of the world. A user who spent ten minutes on live sliders and then hits Save can
have those live values **overwritten by the stale form state** -- the save actively destroys
the edits it looks like it is preserving. (This is exactly what the metronome-voice dropdown
did before it was fixed: it rendered the wrong value, and then Save wrote that wrong value
back.)

## Solution

**A live edit is a real edit. It persists.**

The user stops having to know the difference between a control that writes through and one
that does not -- a distinction that exists only because of how the handlers were wired, and
which the UI never communicated. They change a setting; it survives the power cycle. That is
the entire user-visible contract.

Persisting on *every* `/live` POST is not an option: config is stored as one blob
(`nvs_set_blob(handle, KEY, cfg, sizeof(*cfg))` -- there is no per-field write), and a single
slider drag emits dozens of POSTs. That would blob-write flash tens of times per gesture and
wear NVS for no benefit.

So the write is **debounced**: edits mark the config dirty, and the blob is written once the
edits stop. A drag becomes one write. The debounce is not an implementation detail we are
tolerating -- it is what makes write-through safe, and it is the part with real logic in it,
so it is the part that gets tested.

## User Stories

1. As a performer, I want the outputs I set to *not* follow Link to still be set that way
   after a power cycle, so that my rig comes up the way I left it.
2. As a performer, I want per-output swing to survive a reboot, so that a groove I dialled in
   at soundcheck is still there at the downbeat.
3. As a performer, I want per-output phase nudge to survive a reboot, so that I do not
   re-align every device against every other device at every gig.
4. As a performer, I want per-output PPQN division to survive a reboot, so that the box I set
   to half-time is still in half-time tomorrow.
5. As a performer, I want per-output cable assignment to survive a reboot, so that output 2
   still goes to the machine I patched it to.
6. As a performer, I want which outputs are enabled at all to survive a reboot.
7. As a performer, I want my metronome voice to survive a reboot, so that I do not get a
   Tone click when I chose Wood. (The originating report.)
8. As a performer, I want metronome volume and accent to survive a reboot.
9. As a performer, I want LED brightness to survive a reboot, so that a unit I dimmed for a
   dark stage does not come back up at full brightness in front of an audience.
10. As a performer, I want LED mode, fade, and beat/accent colours to survive a reboot.
11. As a KitchenSync Touch owner, I want my nudge setting to survive a reboot, so that the
    device stays aligned to the gear it drives.
12. As a KitchenSync Touch owner, I want my screen brightness to survive a reboot, so that a
    unit dimmed for a dark room does not come back up glaring.
13. As a user, I want a live control and a form control to mean the same thing, so that I
    never have to learn which parts of the UI are real.
14. As a user, I want a slider drag to cost the device one flash write and not fifty, so that
    I am not silently wearing out the hardware by adjusting a knob.
15. As a user dragging a slider, I want the value I *land* on to be saved -- not every value I
    passed through on the way there.
16. As a user, I want hitting Save after making live edits to preserve them, so that the act
    of saving never destroys the settings it appears to be saving.
17. As a user who pulls the power right after a change, I want the change to have been written
    already if a moment has passed, so that the window in which an edit can be lost is small
    and understood rather than permanent.
18. As a user, I never want to see an "unsaved changes" nag, because there should be no such
    state to warn me about.
19. As a KitchenSync owner *and* a Touch owner, I want both devices to behave identically
    here, so that what I learn about one transfers to the other (ESP-017).
20. As a developer, I want one persistence policy shared by both firmwares, so that fixing it
    on one device cannot leave the other broken.
21. As a developer, I want the "should this write now?" decision to be pure and host-tested,
    so that its timing behaviour is provable without a bench, a device, or a power cycle.
22. As a developer adding a new live-editable field, I want it persisted automatically by
    virtue of being live-safe, so that the next field added cannot silently reintroduce this
    exact bug.
23. As a developer, I want the debounce to survive a burst of edits arriving faster than the
    debounce window, so that a continuous drag defers the write rather than starving it
    forever.
24. As a developer, I want a config write to never happen from the 1 ms MIDI writer task, so
    that a flash operation can never stall the clock (P4-033, ESP-018 -- this codebase has
    already shipped that bug twice).
25. As a support-minded owner, I want a user's report of "it forgot my settings" to be
    impossible rather than triaged.

## Implementation Decisions

### A new pure module in the shared layer

`X32Link/` is the shared pure layer: KitchenSync points its CMake at it
(`set(X32LINK_DIR ...)`), and KitchenSyncTouch symlinks its files in individually. **A pure
module placed there lands in both firmwares by construction** -- parity here is structural,
not a promise someone has to remember to keep (ESP-017).

Add a **config-persist policy** module there. It owns one question and holds no config of its
own:

> Edits have been arriving. Given the current time, should the blob be written *now*?

Deliberately shaped so the module knows nothing about `KsConfig`, `AppConfig`, NVS, or FreeRTOS
-- it is a timer state machine over `(dirty, last_edit_time, now)`. That is what makes it pure,
shared, and testable off-device.

The interface is small enough to state exactly, because the *shape* is the decision:

```c
typedef struct {
    uint32_t last_edit_ms;
    bool     dirty;
} ConfigPersist;

void config_persist_reset(ConfigPersist* p);
void config_persist_mark(ConfigPersist* p, uint32_t now_ms);   // an edit arrived
bool config_persist_due (ConfigPersist* p, uint32_t now_ms);   // write the blob now?
```

`config_persist_due()` returns true **once** per settled burst and clears `dirty` -- so the
caller's job is reduced to "if it says yes, write the blob," and a re-write cannot be issued
twice for the same burst. Two constants, both in the pure module so both devices inherit them:
a **quiet window** (edits must stop for this long) and a **maximum deferral** (a continuous
drag must not defer the write forever -- story 23).

### Wiring, per device

- **KitchenSync (P4):** the `/live` handler already calls `ks_config_live_safe_copy()`. It
  additionally marks the persist state dirty. A **low-priority** task -- the existing
  `status_task` (priority 2) is the natural host, since it already exists to own slow,
  blocking work away from the RT path -- polls `config_persist_due()` and calls
  `ks_config_save()` when it fires.
- **KitchenSyncTouch (S3):** `/nudge` and `/bright` mark dirty; the Arduino `loop()` polls and
  calls the existing NVS save. Same policy object, same constants.

**The write must never happen on the MIDI writer task.** A flash write suspends the cache and
stalls both cores; that is a clock stall, and this codebase has shipped the "blocking call in
the 1 ms RT loop" bug twice already (P4-033: `ESP_LOGx` in the clock task; ESP-018: the writer
running below lwIP). The persist poll belongs on a low-priority task by construction, and the
review of this change should confirm that it is.

### The live-safe field set stays the single owner

ARC-016 established `ks_config_live_safe_copy()` as the one authority on which fields may
change without a reboot. This PRD does **not** introduce a second list of "and which of those
should persist." The answer is *all of them* -- if a field is safe to change live, it is safe
to store. Any future field added to the live-safe set is therefore persisted automatically
(story 22), and no one has to remember to add it in two places. A second list is exactly how
this bug would come back.

### Save must not clobber live state

The `/save` form path writes the form's view of the world, which can be **staler than the live
values the user just set**. Once live edits persist, the form must render from the *current*
running config so a Save cannot silently revert them. The metronome-voice dropdown was one
instance of this and was fixed by rendering `selected` from `s_cfg`; the same discipline has to
hold for every live-editable control on the form.

### Not doing: an "unsaved changes" indicator

That treats the symptom as a communication problem. The right number of settings that are
applied-but-not-saved is zero (story 18).

## Testing Decisions

**A good test here asserts observable behaviour through the public interface, and would survive
a rewrite of the internals.** It says *"a burst of edits produces one write, after the edits
stop"* -- not *"`dirty` was set to true."* The debounce module's public surface is
`mark`/`due`/`reset` and a clock the test controls, which is the whole point of making time an
input rather than something the module reads: the timing behaviour is provable with no device,
no flash, and no power cycle.

**Module under test:** the new pure config-persist policy in the shared layer. Host-tested,
under `test/`, alongside the existing pure suites. Behaviours:

- A single edit does not write immediately; it writes once the quiet window elapses.
- A burst of edits within the quiet window produces **exactly one** write, after the last one
  -- the slider-drag case (stories 14, 15).
- A continuous stream of edits that never leaves a quiet gap still writes at the maximum
  deferral bound -- the debounce defers, it does not starve (story 23).
- `due()` returns true exactly once per settled burst; polling again without a new edit returns
  false, so a poll loop cannot re-write the same blob repeatedly.
- A fresh/reset state is not dirty and never asks for a write. (`clock_ticker.reset()` taught
  this lesson directly: a counter that `reset()` did not zero read stack garbage, because
  callers stack-allocate. Assert the zero state.)
- Time is injected. No `millis()`, no `esp_timer`, no wall clock.

**Prior art**, and it is close: `test_ks_config.c` already covers `ks_config_live_safe_copy()`
and the validate/set grammar; `test_ktouch_config.c` covers the Touch's `app_config_set()`.
`clock_ticker`'s tests are the model for a pure module whose entire job is a decision over
injected time. Follow those.

**What is NOT unit-tested, and must therefore be checked on the bench:** the glue -- the
handler marking dirty, and the low-priority task performing the write. Every bug found this
cycle (P4-033, P4-034, ESP-018) lived in the glue, which carries 0% coverage. So the
acceptance test is a **power cycle on real hardware**, and it is not optional:

- [ ] Set an output to **not** follow Link. Reboot. It still does not follow Link. *(The story
      that motivates the whole ticket.)*
- [ ] Change swing, nudge, division, and cable on an output. Reboot. All four survive.
- [ ] Change metronome voice. Reboot. It survives. *(The originating report.)*
- [ ] Change LED brightness on the P4 and screen brightness on the Touch. Reboot. Both survive.
- [ ] Drag a slider end to end and confirm it results in **one** write, not one per step.
- [ ] Make live edits, then press Save. The live edits survive -- Save does not revert them.
- [ ] With a Link session running and the clock on the analyzer, hammer live edits and confirm
      **zero** stalls, bursts, or dropped pulses (`/status` already reports all three). This is
      the P4-033/ESP-018 regression check: it proves the flash write is not on the RT path.

## Out of Scope

- **Per-field NVS keys.** Config is one blob today and stays one blob. Splitting the schema is
  a much larger change and this bug does not require it.
- **The `/save` + reboot path.** It works. It keeps working. WiFi credentials and other
  reboot-required fields (the non-live-safe set) still go through it, unchanged.
- **Config versioning/migration.** No schema change here -- no new fields, no reordering. The
  existing `KS_CONFIG_VERSION` and its `_Static_assert` on `sizeof(KsConfig)` stay as they are.
- **Any UI redesign.** The controls stay exactly where they are; they simply start meaning what
  they appear to mean.
- **Persisting transport/play state across reboot.** A device must not come up playing. Out of
  scope, and deliberately so.

## Further Notes

This bug is the same shape as the three found this cycle, and the pattern is now hard to
dismiss: **the pure layer is at 97.2% coverage across 467 tests, the glue is at 0%, and every
single defect has been in the glue.** `ks_config_live_safe_copy()` -- pure, tested, correct --
does its job perfectly. The handler simply never calls save afterwards. No pure-layer test
could ever have caught that, because the omission is not *in* the pure layer.

That is the argument for putting the debounce policy in the shared pure module rather than
writing a timer in each handler: it moves the only part with real logic in it to where the
tests are, and leaves the glue with nothing to get wrong but a single call.

Worth flagging for whoever picks this up: `follow_link` is only exposed to this bug because
`ks_config_live_safe_copy()` copies `clock[o]` **wholesale**. That is the right call -- a
field-by-field copy would be one more list to forget to update -- but it does mean the
live-safe set silently grows every time a field is added to `ClockOutputCfg`. Which is
precisely why persistence must follow the live-safe set automatically, rather than being a
second list maintained by hand.
