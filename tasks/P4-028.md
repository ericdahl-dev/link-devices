## Problem

Observed on real hardware (2026-07-07, P4 + Live + Ableton Note + S3 on one Link
session — see [[p4-phase-validated-aligned]]): when a **new peer joins**, the P4's
GhostXForm is invalidated for ~500 ms and the clock/LED free-runs from beat 0
before re-acquiring. Steady-state phase is correct (validated against the S3's
known-good phase dot); this is a transient tied specifically to the peer-join
event.

Evidence from the PHASE_DEBUG trace (clock_out_task) at the moment Ableton Note
joined:

```
I (93236) PHASE lock=1 beats=49.798 | xform=1 intercept=-67321476   ← locked
I (93486) PHASE lock=0 beats=0.165  | xform=0 intercept=0           ← xform invalid
I (93664) link peers 2 ... phase free
I (93736) PHASE lock=0 beats=0.191  | xform=0 intercept=0
I (93987) PHASE lock=1 beats=51.508 | xform=1 intercept=-67326638   ← re-acquired
```

`xform.valid` dropped to 0 for ~500 ms (2 samples), `beats` reset toward 0,
`play` briefly toggled, and one transient sample showed junk origins
(`beat0=-16243 t0=544860`). It was the **only** de-lock in a 55 s capture, so it
is not routine drift — it is the re-measurement/re-target that fires on peer
count change. The `phase free` status label was therefore *accurate*, not
cosmetic.

## Fix (proposed — investigate first)

Root-cause where the peer-join path invalidates the xform. Likely in the P4 glue
(`KitchenSync/main/link_measure_io.c`) or the shared session policy
(`link_measurement_session.c`): a new peer triggers a fresh measurement and the
current xform is cleared before the new one lands, instead of being **held as
last-known-good** across the re-measurement window.

- Prefer: keep serving the previous GhostXForm (and keep `bs.locked` true) until
  a new valid measurement replaces it — no free-run gap. Only invalidate if the
  measurement genuinely fails past a timeout.
- Guard the junk-origin transient (`beat0=-16243`) — don't let a half-formed
  timeline reach beat_source.
- Add/keep a host test in `test/` for the session state machine across a
  peer-count change (pure `link_measurement_session.c`), then verify on hardware
  by joining/leaving a peer and confirming no LED/phase glitch.

## Notes

- Surfaced by the LNK-026 phase instrumentation; the `PHASE_DEBUG` block in
  `ks_main.c` (now gated off) is the tool to reproduce — flip it on to re-trace.
- Real-world impact: peers join/leave mid-set, so each event is a ~half-second
  phase glitch on outputs. Matters for the "phase-accurate" product promise
  ([[p4hub-product-vision]]) but is bounded and not a steady-state error.
