## Problem

Observed on real hardware (2026-07-07, P4 + Live + Ableton Note + S3 on one Link
session — see [[p4-phase-validated-aligned]]): when a **new peer joins**, the P4's
GhostXForm is invalidated for ~500 ms and the clock/LED free-runs from beat 0
before re-acquiring. Steady-state phase is correct (validated against the S3's
known-good phase dot); this is a transient tied specifically to the peer-join
event.

Evidence from the PHASE_DEBUG trace (clock_out_task) at the moment Ableton Note
joined:

```
I (93236) PHASE lock=1 beats=49.798 | xform=1 intercept=-67321476   ← locked
I (93486) PHASE lock=0 beats=0.165  | xform=0 intercept=0           ← xform invalid
I (93664) link peers 2 ... phase free
I (93736) PHASE lock=0 beats=0.191  | xform=0 intercept=0
I (93987) PHASE lock=1 beats=51.508 | xform=1 intercept=-67326638   ← re-acquired
```

`xform.valid` dropped to 0 for ~500 ms (2 samples), `beats` reset toward 0,
`play` briefly toggled, and one transient sample showed junk origins
(`beat0=-16243 t0=544860`). It was the **only** de-lock in a 55 s capture, so it
is not routine drift — it is the re-measurement/re-target that fires on peer
count change. The `phase free` status label was therefore *accurate*, not
cosmetic.

## Root cause (found — firmware review 2026-07-07)

It is NOT the re-measurement path. `link_session_on_trigger` (peer-join / re-measure)
emits only FLUSH_RX/START_ATTEMPT/SEND_PING and `link_measurement_attempt_begin/end`
explicitly preserve a committed xform — last-known-good is already held there.

The real mechanism is a **false-positive epoch reset**:
- `link_protocol.c:131` stores a single global `s_timeline`, last-writer-wins, with
  no session-leader arbitration. It is **shared with the S3** (same latent bug).
- A joining peer, before adopting the session timeline, gossips its own fresh-boot
  timeline with a tiny `time_origin_us` (the trace's `t0=544860` ≈ 0.5 s since that
  peer booted).
- `link_measure_io_poll` → `link_session_on_timeline` (session.c:20) → the
  `prev(~67M) − new(544860) > 1 s` backward-jump test in
  `link_phase_timeline_epoch_reset` (link_phase.c:23) fires → emits `LS_RESET_XFORM`
  → `link_measurement_reset` (measurement.c:212) zeroes the xform → `beat_source_step`
  falls to the free-run accumulator restarting near 0 → the ~500 ms glitch.
  `next_measure_us=0` forces an immediate re-measure that re-locks ~250 ms later.

This looks identical to a genuine transport re-origin at the `on_timeline` interface,
which is why `test_epoch_reset_on_backward_jump` didn't catch it.

## Fix (proposed)

- **(A) Debounce the epoch decision** in `link_session_on_timeline`
  (`link_measurement_session.c:15-23`, shared pure — stays host-tested): record a
  small backward origin as a *candidate* and only emit `LS_RESET_XFORM` if it
  persists across a ~150–250 ms settle window (≥2–3 gossip intervals). A joining
  peer's junk origin is replaced by the real session origin within one gossip cycle,
  so it never confirms; a genuine re-origin persists and confirms. Requires threading
  `now_us` into `on_timeline`.
- **(B) Hold-last-good timeline** so the junk `tl` never reaches `beat_source` for
  even one tick (else a wild one-tick `beats` can emit up to `MAX_BURST=96` spurious
  0xF8). Cleanest at the `link_proto_timeline()` boundary in
  `KitchenSync/main/link_measure_io.c` / `ks_main.c`. Fixing the last-writer-wins
  store in the shared `link_protocol.c:131` fixes **both boards**.
- **(C, optional)** For genuine re-origins, hold old xform + `bs.locked` until
  `attempt_end(true)` atomically overwrites, instead of an immediate reset.
- **Test**: in `test/test_link_measurement_session.c`, prime origin 510000000, feed
  one junk `on_timeline(200000)` → assert NO reset; feed the small origin across the
  full settle window → assert exactly one `LS_RESET_XFORM`.

Original framing (kept for context): the xform is "cleared before the new one lands."
That's true but the trigger is the junk gossip timeline, not the re-measurement.

- Prefer: keep serving the previous GhostXForm (and keep `bs.locked` true) until
  a new valid measurement replaces it — no free-run gap. Only invalidate if the
  measurement genuinely fails past a timeout.
- Guard the junk-origin transient (`beat0=-16243`) — don't let a half-formed
  timeline reach beat_source.
- Add/keep a host test in `test/` for the session state machine across a
  peer-count change (pure `link_measurement_session.c`), then verify on hardware
  by joining/leaving a peer and confirming no LED/phase glitch.

## Status (2026-07-07)

- **(A) DONE — debounce, host-tested.** `link_session_on_timeline` now threads
  `now_us` and holds a backward jump as a candidate, only emitting `LS_RESET_XFORM`
  after it persists `LINK_SESSION_EPOCH_SETTLE_US` (500 ms); a corrective gossip
  clears it. Tests: `test_epoch_reset_confirmed_after_settle`,
  `test_epoch_transient_join_no_reset`, `test_epoch_intermittent_junk_never_confirms`
  (all green). Both glues (`KitchenSync` + `X32Link`) pass `now_us()`. P4 firmware
  builds. This stops the false xform reset (the sustained free-run-from-0).
- **(B) REMAINING — hold-last-good timeline.** Even with the xform held, the junk
  `tl` still reaches `beat_source` (via `ks_main`/the S3 equivalent) for the ticks
  it sits in the store → a one-tick wild `beats`. Fix at the `link_proto_timeline()`
  read (or the shared `link_protocol.c:131` last-writer-wins store) so both the
  epoch detector and `beat_source` see only settled timelines. Touches the S3's
  golden path — **validate on hardware on BOTH boards** (reproduce the Ableton Note
  join on the P4, confirm no de-lock; confirm the S3 phase dot still perfect) before
  landing.
- **Also validate the 500 ms settle** against real peer-joins; tune if the corrective
  gossip lands sooner (the firmware review suggested 150–250 ms may suffice).

## Notes

- Surfaced by the LNK-026 phase instrumentation; the `PHASE_DEBUG` block in
  `ks_main.c` (now gated off) is the tool to reproduce — flip it on to re-trace.
- Real-world impact: peers join/leave mid-set, so each event is a ~half-second
  phase glitch on outputs. Matters for the "phase-accurate" product promise
  ([[p4hub-product-vision]]) but is bounded and not a steady-state error.
