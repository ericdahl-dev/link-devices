> PRD. Feasibility + phased plan backing this task:
> `docs/plans/2026-07-08-p4-link-audio-feasibility.md`. Scope here is the
> **proof-of-concept (Tiers 1–3)**: prove KitchenSync can publish a Link Audio
> channel that Ableton Live receives, in sync, using the onboard mic as the
> source. USB Audio Class ingest (Tier 4) and the MIDI↔Link bridge are out of
> scope (see below) but the seams are shaped so they drop in later.

## Problem Statement

As of Live 12.4, Ableton Link can carry **audio** over the LAN (Link Audio), not
just tempo. Today a KitchenSync (ESP32-P4-NANO) is a tempo-only Link peer: it
reads/writes BPM and phase, but it cannot put sound into a Link session. A user
running Live who wants audio from a KitchenSync-connected source has no path —
the device speaks the Link timeline but is silent on the network. Before we
commit to building a shippable Link Audio box, we don't yet know the one thing
that decides whether the product is real: **can Ableton's Link Audio stack run on
the P4 at all, and can we feed it beat-accurate audio so Live receives it in
sync?**

## Solution

A proof-of-concept firmware for the P4-NANO that publishes a single Link Audio
channel (a `LinkAudioSink`) sourced from the **onboard ES8311 microphone**, wired
into the existing Link timeline so the audio arrives in Ableton Live sample-
accurately aligned to the session beat. From the user's perspective: they open
Live 12.4 on the same wired network, see a Link Audio channel named after the
KitchenSync appear, subscribe to it, and hear the room mic in time with the Link
session — no laptop driver, no USB interface, one Ethernet cable (PoE).

The PoC deliberately uses the onboard mic (not a USB audio interface) because it
removes the only genuinely new *driver* work (a USB Audio Class host) from the
critical path and reuses the ES8311 I2S/codec bring-up already done in **P4-006**.
What remains is the actual product risk — **beat-accurate timestamping of audio
buffers** — which the PoC exists to prove.

## User Stories

1. As a Live 12.4 user on the same wired LAN, I want a KitchenSync to appear as a
   Link Audio channel, so that I can receive audio from it without any driver
   install.
2. As a Live user, I want the KitchenSync channel to carry a human-readable name
   (peer + channel), so that I can tell which device I'm subscribing to.
3. As a Live user, I want the audio I receive from the KitchenSync to be
   sample-accurately aligned to the Link session beat, so that captured sound
   lands in time with my project.
4. As a Live user, I want the KitchenSync to send audio **only when I subscribe**
   to its channel, so that idle devices consume no network bandwidth.
5. As a KitchenSync user, I want the device to publish its mic over one Ethernet
   cable with PoE, so that a single wired run gives me power and a rock-solid
   (jitter-free) audio link.
6. As a KitchenSync user, I want the audio channel to ride the same Link session
   the device already joins for tempo, so that audio and tempo are one connection,
   not two.
7. As a KitchenSync user, I want the mic audio to stay in sync when the Link host
   changes tempo, so that alignment holds across a live session, not just at a
   fixed BPM.
8. As a KitchenSync user, I want the audio to re-align cleanly when the Link host
   stops/starts transport (re-origins the ghost timeline), so that a transport
   bounce doesn't leave the audio a beat off. (Same re-origin hazard LNK-026
   surfaced for phase.)
9. As a firmware developer, I want the beat-timestamp logic to be a pure,
   host-tested module, so that I can prove sync correctness on the dev box without
   flashing hardware.
10. As a firmware developer, I want audio input to sit behind a single
    `audio_source` seam (mirroring `tempo_source`), so that swapping the mic for a
    USB Audio Class interface later is a drop-in behind the seam.
11. As a firmware developer, I want the Link Audio SDK's ESP32 `IoContext` port to
    be validated first with base Link (tempo) before adding audio, so that a
    platform-layer break is isolated from an audio-layer break.
12. As a firmware developer, I want a "silent sink" milestone (a published channel
    carrying silence that Live can see) before wiring real audio, so that network
    plumbing is proven independently of the mic path.
13. As a firmware developer, I want the ESP-IDF glue for `LinkAudioSink` to be a
    thin `*_io` adapter over the pure timestamp module, so that it matches the
    existing `link_measurement_io` / `midi_clock_out_io` pattern.
14. As a firmware developer, I want the PoC to run on the P4-NANO's early silicon
    (rev v1.3) under current ESP-IDF v5.x, so that it builds on the hardware we
    actually have.
15. As a firmware developer, I want the mic-capture path to reuse the P4-006
    ES8311 bring-up (I2C 0x18, SCL=GPIO8/SDA=GPIO7), adding only the ADC/record
    side, so that I don't re-solve the codec.
16. As a product owner, I want the PoC to answer a single yes/no — "does
    recognizable, in-sync audio arrive in Live from the P4" — so that we can
    decide whether to fund the shippable product.
17. As a product owner, I want the licensing gate (GPLv2 / Ableton commercial Link
    license, Link Audio SDK beta) flagged as a prerequisite, so that engineering
    isn't blocked late by a business veto.
18. As a KitchenSync user, I want an on/off control for publishing the mic channel,
    so that I can leave the device on the network without always broadcasting.

## Implementation Decisions

- **Reuse the Link Audio SDK as-is (`github.com/Ableton/link`).** The
  `link_audio/*` layer is templated on `platform::IoContext` and dependency-
  injected the same way base Link is — no raw threads, no hardcoded OS calls. The
  port is a platform-layer job, not a rewrite. Audio transport is uncompressed
  16-bit interleaved PCM (`Codec::kPCM_i16`); no codec CPU cost.
- **Three sequenced milestones** (each a shippable checkpoint):
  1. **Base Link on P4 over Ethernet** — resurrect the SDK's `examples/esp32`
     (experimental, WiFi, Xtensa, IDF v4.3.1) onto **P4 RISC-V / IDF v5.x /
     Ethernet**, honoring the early-silicon min-rev flags
     (`CONFIG_ESP32P4_SELECTS_REV_LESS_V3`, `CONFIG_ESP32P4_REV_MIN_100`). Exit:
     P4 is a tempo peer in a live session.
  2. **Silent Link Audio sink** — swap `Link`→`LinkAudio`, publish one
     `LinkAudioSink` of silence; validate the ESP32 `IoContext` carries the extra
     `Receivers`/`MainProcessor`/`SinkProcessor` load. Exit: the channel appears
     in Live 12.4.
  3. **Mic → sink** — feed `LinkAudioSink::BufferHandle::write(samples,
     beatsAtBufferBegin, numFrames, numChannels, sampleRate)` from the ES8311 mic.
     Exit: in-sync mic audio in Live.
- **New `audio_source` input seam** — one interface delivering PCM frames + sample
  rate, dispatching by source (mic now, USB later). Mirrors the existing
  `tempo_source` seam that dispatches Link vs MIDI. The mic implementation is the
  ES8311 I2S RX path.
- **New pure module `link_audio_stamp`** (name TBD) — maps `(frame position at
  buffer start, sample rate, tempo state)` → `beatsAtBufferBegin`. Consumes the
  existing **`tempo_snapshot`** atomic `{bpm, phase, valid, quantum}` (ARC-001) as
  its tempo source; does not introduce a new one. This is where sync correctness
  lives. Must handle the **re-origin case** (LNK-026): when the ghost timeline
  epoch jumps, the mapping re-primes rather than emitting a beat offset.
- **New thin glue `link_audio_sink_io`** — ESP-IDF adapter owning the `IoContext`
  instance and the `LinkAudioSink`, calling `write()` on each captured buffer with
  the beat stamp from the pure module. Mirrors `link_measurement_io.cpp` /
  `midi_clock_out_io.cpp`. Not unit-tested; exercised on hardware.
- **Mono, codec-native sample rate.** ES8311 is mono; the PoC publishes 1 channel
  at whatever rate the codec is clocked (16k or 48k), passed through as the
  `sampleRate` arg. Fidelity is explicitly PoC-grade.
- **Publish-on-subscribe honored for free** by the SDK (a sink sends only when a
  source exists) — no extra gating logic needed for story 4.
- **Config** (on/off for the mic channel) lives in the web UI per ADR-0004
  (real-time controls vs web config split); no new touch-screen control for the
  PoC.

## Testing Decisions

- **What a good test is here:** external-behavior only. Given a tempo state
  (`bpm`, `phase`, `valid`, `quantum`) and a buffer's frame position + sample
  rate, assert the **computed `beatsAtBufferBegin`** — never internal fields or
  call order. Sync correctness is a pure function of inputs, so it is fully
  host-testable without hardware or a network.
- **Module under test:** `link_audio_stamp` (the beat-timestamp mapper), including
  its re-origin / epoch-reset behavior. Any pure decision logic in `audio_source`
  (e.g. source selection, buffer sizing) is host-tested too; the I2S capture
  itself is not.
- **Harness / prior art:** Unity suites in `test/`, run with `cd test && make`,
  matching the existing pure-logic suites — `test_link_phase.c` (the LNK-026
  epoch-reset tests are the closest prior art), plus the `clock_ticker`,
  `beat_synth`, and `midi_bpm_calc` suites. New suite: `test_link_audio_stamp.c`.
- **On-hardware validation (not unit tests):** milestone exits are verified on the
  P4-NANO against a real Live 12.4 peer — channel visibility (silent-sink
  milestone) and audible in-sync mic (mic milestone). This mirrors how Link
  measurement/phase work is verified live against Ableton peers.

## Out of Scope

- **USB Audio Class host ingest (Tier 4).** No UAC driver in this PoC. The
  `audio_source` seam is designed so this drops in later; building it is a
  separate task. (P4-NANO USB-host gotcha for whoever picks it up: the interface
  must wire directly to the Type-A port — no hub, the host lacks TT support.)
- **MIDI ↔ Link bridge.** The other half of the eventual product; not part of
  proving the audio thesis.
- **Multi-channel / stereo / high-fidelity audio.** PoC is mono, codec-native
  rate.
- **Receiving Link Audio (a `LinkAudioSource`) on the P4.** PoC only *publishes*.
- **Shippable packaging, enclosure, PoE hardware productization.**
- **The commercial licensing agreement itself** — flagged as a prerequisite to
  building, but negotiating it is not an engineering deliverable here.

## Further Notes

- **Do the licensing email first.** Link is GPLv2; closed hardware needs Ableton's
  commercial Link license, and the Link Audio SDK is still young. It can veto the
  product independent of engineering, and it's free to ask — so ask before
  investing the milestone work.
- **The novel risk is one thing: `beatsAtBufferBegin` correctness.** Everything
  else is reuse (the SDK's IoContext seam, the P4-006 codec) or a bounded platform
  port. The mic was chosen specifically because its steady I2S clock is the
  easiest possible input to prove that timestamp against — easier than an async
  USB interface's clock domain.
- **Sequencing rationale:** milestones are ordered so a failure is attributable —
  a platform-layer break (milestone 1) can't be confused with an audio-layer break
  (milestone 2) or a timestamp bug (milestone 3).
- When ready to build, split the three milestones into their own `P4-0xx` tasks
  with `depends_on` chaining; this PRD is the umbrella.
