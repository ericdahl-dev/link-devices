## Problem

Traced live while scoping CLK-001's master-clock epic. KitchenSync's actual clock
engine does **not** stop when the last Link peer disappears: `session_timeline.c`'s
settled timeline and `link_measurement.c`'s committed `GhostXForm` are only cleared
on an explicit re-origin (`link_proto_reset()`), never on peer loss. `beat_source_step()`
gates `active` on `have_session` (has a timeline ever settled), not on live peer count —
so `plan.pulses[o]` (the MIDI/analog clock) keeps flowing off the frozen timeline
indefinitely after peers hit 0.

**Correction (verified against `ks_main.c` before writing tests):** the LED standby
render and the metronome are NOT affected — `ks_main.c:176`'s
`tp_playing = !link_proto_start_stop_seen() || link_proto_playing()` already fails
safe: when `start_stop_seen` resets false on peer loss, the `!seen` fallback flips
`tp_playing` back to true ("assume playing when the session never told us"). That
logic is correct and stays as-is.

The actual bug is narrower, in `/status` (and the web UI's PLAYING/STOPPED badge,
`ks_web.cpp:213,251`, which reads it straight through):
`ks_main.c:302,305` sets `s_stat.bpm = link_proto_bpm()` and
`s_stat.playing = link_proto_playing()`, both reading `link_protocol.c` variables
that `remove_peer()`/`link_proto_tick()` zero the instant peer count hits 0
(`link_protocol.c:86-87,111-112`). But a second, already-correct representation of
tempo exists right next to it: `in->tl.micros_per_beat` — the settled timeline
`ks_tick_step` already receives via `wifi_link_timeline()` — which is **never**
cleared on peer loss (confirmed: only `link_proto_reset()`, an explicit re-origin,
touches it). Status is reading the wrong one of two values that already exist.
`playing` has no such frozen twin — that part genuinely needs new held state.

**The device is not silent when solo — `/status` and the web UI's transport badge
misrepresent a clock that's still running.** Anything plugged into the clock output
keeps ticking while the dashboard says `bpm: 0` / `STOPPED`.

## Fix

Scoped entirely to the pure, host-tested `ks_tick_step` core (not the untested
`ks_main.c` glue loop):

- `KsTickPlan` gains `bpm` (derived from `in->tl.micros_per_beat` while `bs.active`,
  instead of a separate zeroed variable).
- `KsTickPlan`'s `playing` latches the last known value once `in->start_stop_seen`
  has ever been true, instead of snapping to false the moment it resets (which, per
  the current code, only ever happens on peer loss or an explicit reset).
- `ks_main.c`: `/status` reads `plan.bpm`/`plan.playing` instead of calling
  `link_proto_bpm()`/`link_proto_playing()` directly.

No new module — the held state lives in `KsTickState`, alongside `beat_source`'s own
state, following the same pattern.

## Explicitly out of scope

- Originating a clock when the device has **never** had a session (that's P4-040).
- Any change to `X32Link`'s or the LED-only variant's behavior — this only touches
  `KitchenSync/main/ks_tick.c` and `ks_main.c`, not the shared `link_protocol.c`/
  `session_timeline.c`/`beat_source.c` engine those firmwares also compile against.
  Their "no session → silence" behavior is correct for a follow-only device and stays
  unchanged.

## Acceptance criteria

- [ ] Host test: once locked (`bs.active=true`), `plan.bpm` tracks
      `60e6 / in->tl.micros_per_beat`, independent of any separate peer-loss zeroing.
- [ ] Host test: `in->start_stop_seen` goes from true to false (peer loss) after having
      reported `playing=true` — `plan.playing` stays `true` (held), not `false`.
- [ ] Host test: `in->start_stop_seen` has never been true (fresh state, no session ever
      seen transport) — `plan.playing` reports `false` unchanged; no false "playing" on
      a device that never had a session. This is the P4-040 case, not a regression here.
- [ ] Hardware-validated: pull the only Link peer, confirm `/status` and the web UI's
      transport badge keep reporting the held tempo/playing state instead of `0`/`STOPPED`
      while the clock output continues.

## Resolution

Shipped in #97 ("KitchenSync stops lying when solo, can originate a clock alone").
The ticket sat at `todo` afterwards; corrected 2026-07-13 during the Ordna audit.

`master_clock` has since moved out of `KitchenSync/main/` into the shared home (ESP-027) --
the Touch is the clock box too, and a clock box that goes silent when the last peer leaves is
indefensible. Its header used to say "KitchenSync (P4) only".
