## Resolution (2026-07-15) — done, split across two tickets

Delivered in two parts and fully verified:
- **The wire** (`/status`, `/config.json`, form grammar, `KsCaps`) converged earlier
  (ESP-029/030 pt work): the Touch serves the shared shape, a single-element `clock[]`,
  passwords as `pass_set` only, and capability flags so absent hardware is never faked.
- **The storage + data model** converged in **ESP-042** (PR #130): the Touch adopted
  `KsConfig`, `AppConfig` was retired, and the migration-first hazard this ticket flags in
  bold was honoured exactly — the AppConfig→KsConfig migration was written and host-tested
  FIRST (`test_ktouch_migrate`), and the bench Super Mini kept its WiFi (`Skeyelab`) + a
  non-default 143.000 BPM across a real upgrade (confirmed by NVS dump). `quantum_beats`
  was promoted to the shared schema per item 4.

Every acceptance criterion is met. See ESP-042 for the migration and hardware evidence.

## Why the original plan was wrong

This task used to say *"share ks_config_json.c, like ESP-029 shared ks_status.c."* **That was
reasoning by analogy, and it does not hold.**

`ks_status_json()` takes **plain parameters** — a device just feeds it values, so sharing it was
free. `ks_config_json()` takes a **`const KsConfig*`** — sharing it means sharing the **data model**,
not the formatting. And the two devices do not currently have the same data model:

| | KitchenSync (`KsConfig`) | Touch (`AppConfig`) |
|---|---|---|
| WiFi | **3 slots** | **1 slot** |
| Clock outputs | **4**, each cable/ppqn/phase/swing/follow | **1** DIN out, no per-output cfg |
| Metronome | 4 fields | absent (no speaker fitted) |
| LED strip | 6 fields | absent (no strip fitted) |
| Follow-beat | yes (mic) | absent |
| `quantum_beats` | — | **Touch has it** |
| `play_on_release` | — | **Touch has it** |
| `brightness` (LCD) | — | **Touch has it** |

Serving `/config.json` from `KsConfig` on a Touch would have forced one of two bad outcomes:
either **fake the hardware** (report a metronome and an LED strip that do not exist — the exact lie
ESP-028 exists to prevent), or **omit fields** and make the iOS app's decoder throw, since every
`KsConfig` field there is required.

## The decision (2026-07-14)

**Converge the configs.** They should be very much the same, and the only real difference should be
**the number of outputs**. Both devices get **multiple WiFi slots** — Touch's single slot is a
deficiency, not a design choice.

## ⚠️ THE HAZARD: Touch DISCARDS its config on a version mismatch

This is the thing that must not go wrong, and it will bite on the first boot after flashing.

- **KitchenSync MIGRATES.** `ks_config_nvs.c` reads the old version and upgrades it, and its own
  comment is explicit about why: *"A v1 blob is MIGRATED, not discarded — a version bump that
  silently forgot [wifi creds]…"*
- **Touch DOES NOT.** `app_config.h` returns `CFG_DECODE_DEFAULTED` for an
  *"absent/unversioned/wrong-version/wrong-size/invalid"* blob. It **throws the config away and
  defaults.**

So changing Touch's config struct without a migration means: **the device loses its WiFi
credentials, falls back to the `KitchenSync-Setup` SoftAP, and drops off the network.** On a bench
unit that is an annoyance. On a unit at a venue it is a dead box.

**Write the migration FIRST, host-test it FIRST, and only then change the struct.**

## Approach

1. **Migration first.** `AppConfig` v1 blob → `KsConfig`, preserving `wifi_ssid`/`wifi_pass` into
   `wifi[0]`, and `nudge_mbeats` → `clock[0].phase`. Host-tested against a byte-exact v1 blob
   fixture before any struct changes land. Follow `ks_config_nvs.c`'s pattern — it already does
   exactly this, including writing the upgraded blob back so every boot doesn't re-migrate.
2. **`KS_CLOCK_OUTPUTS` becomes per-target** — 4 on the P4, 1 on the Touch. `KsConfig.clock[]` and
   `/status`'s `launch[]` both take their length from it. ESP-029 already made `launch[]`
   length-aware; do the same for `clock[]`.
3. **Touch adopts `KsConfig`**, sharing `ks_config.c` / `ks_config_json.c` / `ks_form.c` the way it
   already shares 46 other modules. Its web form and display code move to the shared model.
4. **Find a home for the fields that are genuinely device-specific.** `brightness` is an LCD the P4
   does not have; `play_on_release` is a touch-screen behaviour. `quantum_beats` is arguably useful
   on BOTH (launch quantisation is a Link concept, not a screen concept) — consider promoting it to
   the shared schema rather than treating it as Touch-only.
5. **Hardware a device does not have still must not be faked.** If Touch carries `metronome_enable`
   in a shared struct, the DEVICE must not claim it can click, and the app must not draw a control
   for a speaker that is not fitted. Decide this explicitly — a capability flag, or simply not
   emitting those keys — rather than letting it happen by accident.
6. Then `/config.json` is nearly free, and the iOS settings screen lights up with no app change.

## Acceptance criteria

- [ ] **A Touch flashed with the new firmware KEEPS its WiFi credentials.** Verified on real
      hardware: the bench unit must still be on the network after the upgrade, not sitting in SoftAP.
- [ ] Migration is host-tested against a byte-exact v1 `AppConfig` blob fixture
- [ ] Both devices have multiple WiFi slots
- [ ] `KS_CLOCK_OUTPUTS` is per-target; `clock[]` and `launch[]` lengths both follow it
- [ ] `GET /config.json` returns 200 on the Touch with the shared shape
- [ ] The `clock` array has ONE element on a Touch — never padded to four
- [ ] Passwords are never emitted — only `pass_set`
- [ ] No device reports a capability it does not physically have
- [ ] The iOS settings screen opens against a real Touch and a `POST /save` round-trips without
      clobbering anything invisible
- [ ] Touch's own web UI still works

## Notes

The client already treats a `/config.json` 404 as an attributable state
(`ConfigAvailability.unsupportedByFirmware`) and says *"firmware older than 2026-07-14 has no
/config.json"*, pointing the user at an update. **So shipping this lights up the settings screen with
no app change at all** — except T-015 (drive the output cards off the array length rather than a
hardcoded 4), which is already filed.

This is the largest single change in the project so far, and it touches config persistence on a
device that is in use. Do it in the order above. The migration is the risky part; everything else is
mechanical.

See ESP-032: if Touch eventually moves to a P4 board, a converged config is exactly what makes that
migration a non-event.
