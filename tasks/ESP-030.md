## Why

**A client cannot safely change a Touch's settings, and never will be able to until this exists.**
Verified against real hardware (`kstouch-dfd0`, 2026-07-14, on current `master` firmware):
`GET /config.json` → **404**.

This is not cosmetic. `POST /save` is a **full form** — it sends every field and the device writes
all of them. Without a read there is no read-modify-write, so a client that wanted to change one
setting would have to POST a full form built from **fabricated defaults**, silently clobbering
every setting it could not see. That is worse than doing nothing, so the iOS app correctly refuses:
on a 404 it disables the settings screen and tells the user their firmware is too old.

**So today a Touch is, from any client's perspective, read-only for configuration — permanently.**
The only way out is a firmware update, which is exactly why the app deliberately keeps its OTA path
reachable even when `/config.json` fails.

## Approach: share the module, don't reimplement the document

Same discipline as ESP-029, and for the same reason: a second implementation is what caused the
`/status` drift in the first place.

`KitchenSync/main/ks_config_json.c` already produces this document. Move it to `X32Link/` (the
shared-module home) and symlink it into `KitchenSyncTouch/`, exactly like the **46 modules Touch
already shares**. Then `ktouch_web.cpp` registers `/config.json` and calls it. One implementation,
one wire shape, drift impossible.

The contract is already pinned from the other side: the iOS app decodes it in
`Sources/KitchenSyncApp/Models/KsConfig.swift`, with tests asserting the exact wire format.

```json
{
  "clock_out": true, "metronome": false, "metro_accent": true,
  "metro_vol": 80, "metro_voice": 0,
  "led": false, "led_bright": 60, "led_mode": 0, "led_fade": 55,
  "led_beat": "#00B400", "led_accent": "#DC6E00",
  "follow_beat": false,
  "wifi": [{"ssid":"Backline","pass_set":true}, ...],
  "clock": [{"en":true,"cable":0,"ppqn":24,"phase":0,"swing":0,"follow":true}]
}
```

Three rules the KitchenSync implementation already follows and which sharing preserves for free:

1. **The password is never emitted** — only `pass_set` (bool). The device's own web form doesn't
   echo a saved password back either (`ks_web.cpp`'s `build_wifi()`), and the client's model has no
   field to put one in. A blank password on `POST /save` means "keep current" (`ks_config_set`'s
   documented no-op). That is what makes write-only credentials work.
2. **Colors are `"#RRGGBB"` strings** (`#%06X`), not ints. Tests pin this.
3. **The `clock` array's length is the output count.** Touch emits ONE element. Do not pad to four —
   a client renders a card per element and would draw three dead outputs. Same rule as `launch[]`.

**Report honestly what the hardware has.** If a Touch has no LED strip, omit the field rather than
reporting a default — a config screen showing an LED strip on a device with no LED strip is the
same class of lie as ESP-028's `sync:1` over a dead wire.

## Acceptance criteria

- [ ] `ks_config_json.c` lives in `X32Link/` and is symlinked into `KitchenSyncTouch/`
- [ ] `ktouch_web.cpp` serves `GET /config.json` by CALLING it — no second implementation
- [ ] Returns 200 with the P4-041 shape
- [ ] `clock` array length == actual clock-output count (1 on Touch)
- [ ] WiFi passwords are NEVER in the response — only `pass_set`
- [ ] `led_beat` / `led_accent` are `#RRGGBB` strings
- [ ] Fields for hardware the Touch lacks are omitted, not faked
- [ ] The iOS app's settings screen opens against a real Touch, shows its ACTUAL settings, and a
      `POST /save` round-trips without clobbering anything invisible
- [ ] `KitchenSync-iOS_App`'s `KsConfigDecodingTests` fixtures decode a real Touch response
      unmodified

## Notes

The client already treats a 404 here as a distinct, attributable state
(`ConfigAvailability.unsupportedByFirmware`) rather than a generic failure — it says *"firmware
older than 2026-07-14 has no /config.json"* and points the user at an update. **So shipping this
makes the Touch's settings screen light up with no app change at all.**

Touch's `POST /save` already exists and its form handler already works (its own web UI posts to it).
This task is the missing *read* half.

See **ESP-032**: if Touch eventually moves to a P4 touch board, the shared pure module is exactly
the code that survives. The Arduino wrapper does not. Another reason not to hand-roll a second
implementation here.
