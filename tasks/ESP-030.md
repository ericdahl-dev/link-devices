## Why

**A client cannot safely change a Touch's settings, and it never will be able to until this
exists.**

`GET /config.json` (P4-041, KitchenSync, 2026-07-14) is how a client reads a device's ACTUAL
current settings. The Touch 404s on it — verified against real hardware (`kstouch-dfd0`,
2026-07-14).

This is not a cosmetic gap. `POST /save` is a **full form**: it sends every field and the device
writes all of them. Without a read, there is no read-modify-write — a client that wanted to change
one setting would have to POST a full form built from **fabricated defaults**, silently clobbering
every setting it could not see. That is worse than doing nothing, so the iOS app correctly refuses:
on a 404 it disables the settings screen entirely and tells the user their firmware is too old.

So today a Touch is, from a client's perspective, **read-only for configuration forever**. The only
way out of that state is a firmware update — which is exactly why the app deliberately keeps its
OTA path reachable even when `/config.json` fails.

## What to build

Mirror `KitchenSync/main/ks_config_json.c`. The contract is already specified — the iOS app decodes
it in `Sources/KitchenSyncApp/Models/KsConfig.swift`, and there are tests pinning the exact wire
shape.

```json
{
  "clock_out": true, "metronome": false, "metro_accent": true,
  "metro_vol": 80, "metro_voice": 0,
  "led": false, "led_bright": 60, "led_mode": 0, "led_fade": 55,
  "led_beat": "#00B400", "led_accent": "#DC6E00",
  "follow_beat": false,
  "wifi": [{"ssid":"Backline","pass_set":true}, ...],
  "clock": [{"en":true,"cable":0,"ppqn":24,"phase":0,"swing":0,"follow":true}]
}
```

Two rules the KitchenSync implementation already follows and this MUST too:

1. **The password is never emitted.** Only `pass_set` (bool). The device's own web form doesn't
   echo a saved password back either (`ks_web.cpp`'s `build_wifi()`), and the client's model
   (`WifiSlot`) has no field to put one in. A blank password on `POST /save` means "keep current"
   (`ks_config_set`'s documented no-op) — that is what makes write-only credentials work.
2. **Colors are `"#RRGGBB"` strings**, `#%06X` — not ints. The client parses hex with a leading
   `#` (and tolerates its absence). Tests pin this.

**The `clock` array's length is the output count.** Touch emits ONE element. Do not pad to four —
a client renders a card per element and would draw three dead outputs. (Same rule as `launch[]` in
ESP-029.)

Whatever Touch does not have, omit the field or report it honestly — do NOT report a hardcoded
default for hardware that isn't there. A config screen that shows an LED strip on a device with no
LED strip is the same class of lie as ESP-028's `sync:1` over a dead wire.

## Acceptance criteria

- [ ] `GET /config.json` returns 200 with the P4-041 shape
- [ ] `clock` array length == actual clock-output count (1 on Touch)
- [ ] WiFi passwords are NEVER in the response — only `pass_set`
- [ ] `led_beat` / `led_accent` are `#RRGGBB` strings
- [ ] Fields for hardware the Touch lacks are omitted, not faked
- [ ] The iOS app's settings screen opens against a real Touch, shows its actual settings, and a
      `POST /save` round-trips without clobbering anything invisible
- [ ] `KitchenSync-iOS_App`'s `KsConfigDecodingTests` fixtures decode a real Touch response
      unmodified

## Notes

The client already treats a 404 here as a distinct, attributable state
(`ConfigAvailability.unsupportedByFirmware`) rather than a generic failure — it will say *"firmware
older than 2026-07-14 has no /config.json"* and point the user at an update. So shipping this makes
the Touch's settings screen light up with no app change at all.

Depends on ESP-029 only for `fw` in `/status` — a client wants to know what it's talking to before
it decides what to ask for. Not a hard blocker; they can land in either order.
