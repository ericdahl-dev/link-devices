## Goal

Start/stop each MIDI output from the keyboard. **`1`..`4` toggle Clock Out 1..4**; the
existing per-output toggle is what the key drives, so the key and the click are the same
action with the same semantics (START bar-quantized, STOP immediate).

This is a performance tool. A mouse hunting for a toggle mid-set is the wrong instrument.

## Why number keys

Direct: one key = one output, no modifier, no chord, no mode. It maps to how the outputs
are already labelled on screen (Clock Out **1**..**4**), so there is nothing new to learn.

## Behaviour

- **`1`..`4`** -> toggle that output's transport. Identical to clicking its toggle:
  stopped -> PLAY (arms, fires on the bar line); armed/playing -> STOP (immediate).
- **`space`** -> the "all" transport (the existing `out=all` PLAY/STOP pair).
- A key press on an output that **follows Link** must be **inert** -- Link owns that
  output's transport, the toggle is already greyed, and the key must behave the same way
  (see ESP-011). It must not silently do nothing while *looking* live: flash the greyed
  toggle, or briefly surface the existing "Link owns transport for outputs set to follow
  it" note, so the user learns *why* rather than concluding the feature is broken.
- A key for a **disabled** output does nothing.

## The guard that actually matters

**A number typed into a text field must never fire a transport.** The config form has a
WiFi SSID and password on the same page. Typing a passphrase containing `1` and having a
drum machine launch is exactly the kind of "why did my gear start" failure this codebase
has already been burned by once (ESP-011's transport arbitration exists for the mirror
image of it).

So: ignore the key when the event target is an `input`, `select`, `textarea`, or anything
`contenteditable`. This is not a nice-to-have; it is the feature's correctness condition.

Related: the KitchenSync Touch deliberately has **no settings screen on the LCD**, because
"an accidental tap must never pop a config screen mid-performance" (ESP-016). Same
instinct applies here in reverse -- an accidental keystroke must never move transport.

## Discoverability

A shortcut nobody knows about is not a feature. Put the key on the control: a small
`1`..`4` badge on each output's toggle (and `space` on the all-transport row). No help
modal, no cheat sheet -- the affordance lives on the thing it triggers.

## Scope / parity (ESP-017)

P4 web UI only, and that is not a parity violation: **transport already exists on both
devices, exposed differently.** The Touch's transport surface is its full-screen touch
toggle (touch is transport-ONLY, by design, ESP-016) -- it has no per-output web transport
to bind a key to. If the Touch's web UI ever grows transport controls, it inherits the
same key map.

## Acceptance criteria

- [ ] `1`..`4` toggle Clock Out 1..4; `space` toggles all
- [ ] Key and click go through the SAME path (`/transport?out=N&play=…`) -- one behaviour,
      not two implementations that can drift
- [ ] A key is INERT while focus is in any text input / select / textarea
- [ ] A key on a Link-owned output does not act, and SAYS why (does not fail silently)
- [ ] Key badges shown on the toggles
- [ ] Verified on the wire: pressing `1` emits `0xFA` on the bar line, exactly as the
      click does (the analyzer rig from ESP-011/P4-034 already proves this path)

## Notes

- The JS already has the toggle handler and `/transport` plumbing; this is a key listener
  that reuses it, not a new transport path. Keep it that way.
