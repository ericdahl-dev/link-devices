## Goal

Start/stop each MIDI output from the keyboard. **`1`..`4` toggle Clock Out 1..4**; the
existing per-output toggle is what the key drives, so the key and the click are the same
action with the same semantics (START bar-quantized, STOP immediate).

This is a performance tool. A mouse hunting for a toggle mid-set is the wrong instrument.

## Why number keys

Direct: one key = one output, no modifier, no chord, no mode. It maps to how the outputs
are already labelled on screen (Clock Out **1**..**4**), so there is nothing new to learn.

## Behaviour

- **`1`..`4`** -> toggle that output's transport. Identical to clicking its toggle:
  stopped -> PLAY (arms, fires on the bar line); armed/playing -> STOP (immediate).
- **`space`** -> the "all" transport (the existing `out=all` PLAY/STOP pair).
- A key press on an output that **follows Link** must be **inert** -- Link owns that
  output's transport, the toggle is already greyed, and the key must behave the same way
  (see ESP-011). It must not silently do nothing while *looking* live: flash the greyed
  toggle, or briefly surface the existing "Link owns transport for outputs set to follow
  it" note, so the user learns *why* rather than concluding the feature is broken.
- A key for a **disabled** output does nothing.

## The guard that actually matters

**A number typed into a text field must never fire a transport.** The config form has a
WiFi SSID and password on the same page. Typing a passphrase containing `1` and having a
drum machine launch is exactly the kind of "why did my gear start" failure this codebase
has already been burned by once (ESP-011's transport arbitration exists for the mirror
image of it).

So: ignore the key when the event target is an `input`, `select`, `textarea`, or anything
`contenteditable`. This is not a nice-to-have; it is the feature's correctness condition.

Related: the KitchenSync Touch deliberately has **no settings screen on the LCD**, because
"an accidental tap must never pop a config screen mid-performance" (ESP-016). Same
instinct applies here in reverse -- an accidental keystroke must never move transport.

## Discoverability

A shortcut nobody knows about is not a feature. Put the key on the control: a small
`1`..`4` badge on each output's toggle (and `space` on the all-transport row). No help
modal, no cheat sheet -- the affordance lives on the thing it triggers.

## Scope / parity (ESP-017)

P4 web UI only, and that is not a parity violation: **transport already exists on both
devices, exposed differently.** The Touch's transport surface is its full-screen touch
toggle (touch is transport-ONLY, by design, ESP-016) -- it has no per-output web transport
to bind a key to. If the Touch's web UI ever grows transport controls, it inherits the
same key map.

## Acceptance criteria

- [x] `1`..`4` toggle Clock Out 1..4; `space` toggles all
- [x] Key and click go through the SAME path (`/transport?out=N&play=…`) -- one behaviour,
      not two implementations that can drift
- [x] A key is INERT while focus is in any text input / select / textarea
- [x] A key on a Link-owned output does not act, and SAYS why (does not fail silently)
- [x] Key badges shown on the toggles
- [x] Verified on the wire: pressing `1` emits `0xFA` on the bar line, exactly as the
      click does — **measured on the logic analyzer: +0.338 ms from the bar strobe.**
      See "On the wire" below.

## Notes

- The JS already has the toggle handler and `/transport` plumbing; this is a key listener
  that reuses it, not a new transport path. Keep it that way.

## Outcome (2026-07-11)

Landed in `KitchenSync/main/ks_web.cpp` — page CSS (`.kbd` badge, `nak`/`nakrow` flash),
the `1`..`4` badge in `build_outputs()`, the `space` badge on the transport row, and one
`keydown` listener.

The listener posts nothing itself: it resolves the key to the button the mouse would hit
and calls `.click()` on it. So "key and click are one path" is structural, not a promise —
there is no second `fetch('/transport')` to drift. A greyed (Link-owned) button no-ops its
`click()` in the browser for free, which is what makes the inert case fall out rather than
needing its own branch.

**Verification.** The page JS is embedded in a C++ raw string with no host-test seam, so it
was extracted verbatim and driven in a real browser (Chromium, real `KeyboardEvent`s, real
`disabled`-button click semantics) against a DOM mirroring `build_outputs()`. 20/20 green,
including the condition this ticket says decides correctness: `1` typed into the WiFi
password field posts nothing. Also covered: SSID/select/textarea/contenteditable/number
inputs inert; disabled output and clock-out-off inert; Link-owned output inert but flashes
the toggle and pulses the owner note; a non-followed output still keys while Link owns
another; `space` flips PLAY/STOP off the last `/status` and is `preventDefault`-ed;
ctrl/meta/alt and auto-repeat do not move transport.

`idf.py build` is clean for the P4; host suite and `check_docs.sh` pass.

### On hardware (flashed over USB, 2026-07-11)

Flashed to the P4 (ESP32-P4 rev v1.3, 16MB). Device serves the listener, both guards, the flash
keyframes and all five badges. Verified against a **live Ableton Link session** (peers 1,
120.00 BPM, phase locked), driving the real device page in Chromium with **real OS-level
keystrokes**:

- **The key drives transport, end to end.** Pressing `1` on an enabled, manual output posts
  `/transport?out=0&play=1` and the device goes `launch:[0,0,0,0]` -> `[1,0,0,0]` (ARMING) ->
  `[2,0,0,0]` (PLAYING) — i.e. it armed and fired on the bar line, from a keypress. Pressing
  `1` again stops it immediately (`[2,..]` -> `[0,..]`). START quantized, STOP immediate,
  exactly as the click does.
- **Link-owned output is inert AND says why.** With an enabled output set to follow Link and
  the session owning transport, `3` posts nothing, flashes the greyed toggle (`.nak`) and
  pulses the owner note. Confirmed on the device page.
- **Disabled output is inert.** Keys for outputs with their enable switch off post nothing.
- **The guard holds on real hardware.** With focus in a form control, real keystrokes are
  ignored (caught this live: keys correctly went dead while focus sat in a checkbox).
- **One keypress = exactly one POST.** Measured on a freshly loaded page: 1 keydown, 1 POST.
  (An earlier reading of "hundreds of POSTs" was an artifact of stacked `fetch` wrappers in the
  test instrumentation, not the firmware. Re-measured clean.)

### On the wire (logic analyzer, 2026-07-11) — the last AC, and ESP-015's "never measured"

Saleae Logic, 3 channels @ 2 MS/s, against a live Link session at 120 BPM:
`ch0` = GPIO20 (DIN MIDI TX), `ch1` = GPIO21 (downbeat strobe, rising edge = Link bar
boundary), `ch2` = the MIDI OUT cable off the USB port-A interface (the gear-facing path).
A **real keystroke** pressed `1` mid-capture; Async Serial decoded at 31250 8N1.

```
bar strobe: 5 rising edges, period mean 1.9980 s (expect 2.0000 s @120 BPM 4/4)

DIN  : 0xFA at 2.667299 s | bar line 2.666961 s  ->  START +0.338 ms from the bar line
USBA : 0xFA at 2.667487 s | bar line 2.666961 s  ->  START +0.526 ms from the bar line
USB-A lags the DIN wire by +0.188 ms (USB frame scheduling)
next bar is 1999.7 ms later -- it landed on THIS bar, not a neighbouring one
```

**+0.338 ms is not merely "small", it is the structurally correct value.** The strobe rises on
the downbeat tick (`ks_main.c:207`), then the bar's clock byte is shifted out (one byte at
31250 baud = **0.320 ms**), then the START. 0.320 ms + ~18 us of CPU = 0.338 ms. So the `0xFA`
is emitted **on the bar tick, immediately behind that bar's `0xF8`** — exactly what the clock
engine claims. The keypress also *waited* 0.667 s for the grid rather than firing on press,
which is the quantized launch doing its job.

Capture saved (`.sal`, openable in Logic 2). One keypress -> one `0xFA`, on the bar line, on
both the DIN wire and the gear-facing USB-A cable.

**Observation for a future ticket (not ESP-019's business):** the 24-PPQN clock itself carries
real jitter — mean 20.833 ms (= 120.00 BPM exactly) but sd 0.558 ms with outliers at
min 16.898 / max 27.977 ms, on BOTH the DIN and USB paths (so it is the clock engine, not the
cable). Worth a look before anyone tries to shave sub-millisecond timing corrections elsewhere:
this is the jitter floor they would be measured against.

### Found while verifying: P4-037

Hardware verification was blocked by a **pre-existing** bug it flushed out: the web UI wedged
permanently after one desktop-browser visit (LWIP socket exhaustion, `accept (23)`). Filed and
fixed as **P4-037** — required, because ESP-019 makes this web UI a performance transport
surface. Also worth knowing for the next person: **`/transport` returning 200 does NOT mean the
transport armed** — with no Link session there is no beat grid, `ks_tick.c:44` early-returns,
and the intent is silently dropped. Check `launch[]` in `/status`, not the HTTP status.
