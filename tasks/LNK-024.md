## Goal

`LNK-010` (done, 2026-06-29) already merged Link + USB-MIDI-clock input into
one firmware: `X32Link` compiles both `TempoSource` adapters in and selects
the active one at runtime via the `input_source` NVS field (`TEMPO_SRC_LINK`
/ `TEMPO_SRC_MIDI`), toggleable from the web UI's segmented control. There is
no functional reason left to flash `X32MidiClock` as a separate board image.

`LNK-010`'s own acceptance criteria included "Old `X32Link/` and
`X32MidiClock/` dirs retired (or clearly deprecated)" — left unchecked, its
Progress note calls retiring `X32MidiClock/` "optional, non-blocking
cleanup." This ticket is that cleanup, finally scoped out as its own task.

**Not a code-change ticket.** Filed ticket-only per coordinator instruction —
do not implement. Mid live-debug on an unrelated real MIDI-not-received
issue with the user right now; hold this until that's resolved and
explicitly picked up.

## Why this isn't a trivial `rm -rf X32MidiClock/`

Checked the symlink graph fresh this session (`esp32/`, both directions):

`X32MidiClock` is currently the **canonical source** for the MIDI parsing
modules — `X32Link` symlinks them IN, not the other way around:

```
X32Link/midi_clock.h     -> ../X32MidiClock/midi_clock.h
X32Link/midi_clock.cpp   -> ../X32MidiClock/midi_clock.cpp
X32Link/midi_bpm.h       -> ../X32MidiClock/midi_bpm.h
X32Link/midi_bpm.cpp     -> ../X32MidiClock/midi_bpm.cpp
X32Link/midi_bpm_calc.h  -> ../X32MidiClock/midi_bpm_calc.h
X32Link/midi_bpm_calc.c  -> ../X32MidiClock/midi_bpm_calc.c
```

Deleting `X32MidiClock/` today, with no other changes, breaks `X32Link`'s
build. Meanwhile `X32MidiClock/` itself reverse-symlinks 13 shared modules
back from `X32Link/` (`app_config.*`, `osc_*`, `bpm_tracker.*`,
`web_config.*`, `web_status_json.*`) — those are fine to lose, they're
copies of files that live canonically in `X32Link` already.

Checked: nothing outside these two dirs symlinks to `X32MidiClock/` or to
any `midi_*` file — blast radius is contained to `X32Link/` + `X32MidiClock/`
+ `esp32/test/`.

## Retirement plan (for whoever picks this up)

1. **Flip canonical ownership of the MIDI modules**: move
   `midi_clock.{h,cpp}`, `midi_bpm.{h,cpp}`, `midi_bpm_calc.{h,c}` so they
   live for-real in `X32Link/` (current symlink targets, today living in
   `X32MidiClock/`), then re-point anything that needs them the other way —
   or just delete `X32MidiClock/`'s copies once `X32Link/`'s become real
   files instead of symlinks.
2. **Delete `X32MidiClock/`**: the sketch dir itself (`X32MidiClock.ino`,
   `build_opt.h`, `config.h`) plus its 13 reverse-symlinks into `X32Link/`'s
   shared modules.
3. **Fix `esp32/test/Makefile`**: `MIDICALC = ../X32MidiClock/midi_bpm_calc.c`
   points at a file inside the dir being deleted — repoint to
   `../X32Link/midi_bpm_calc.c` (or wherever step 1 lands the canonical
   file). Run `cd esp32/test && make` after — all suites, especially
   `test_midi_bpm_calc`, must still build and pass (15 tests currently).
4. **Check `MCK-*` tickets/docs for standalone-board assumptions** before
   actually deleting anything — `tasks/MCK-001.md` through `MCK-006.md`
   describe `X32MidiClock` as "a separate device from X32Link" for a
   USB-MIDI-only board with no Link/WiFi-multicast capability requirement.
   If there's a real hardware reason to keep a smaller, Link-free,
   USB-MIDI-only binary around (e.g. a board variant, or a deliberately
   simpler/cheaper unit with no Link feature), don't blindly delete — either
   keep `X32MidiClock/` deliberately (and update `LNK-010`'s/this ticket's
   AC to "clearly deprecated" instead of "retired"), or confirm with the
   user that the standalone option is genuinely unused before removing it.
   Also grep `esp32/AGENTS.md` and any other top-level docs for
   `X32MidiClock` references that would go stale.
5. **Re-run the full host suite + `arduino-cli compile` for all surviving
   sketches** (`X32Link`, `X32FaderDisp`, and whatever else exists by the
   time this is picked up — check current sketch list fresh, don't assume
   it's still just these two) before calling it done.
