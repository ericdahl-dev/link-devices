## Goal

Route both config editors — `web_config.cpp` `handle_save()` and `touch_ui.c`
`ui_apply_settings_tap()` — through **one pure per-field edit function** so the
clamp/dependency rules live in a single host-tested place. Both already share the
final `config_validate()`/`config_save()` gate, but the *edit semantics* are
duplicated and have drifted.

## Problem (includes a live bug)

- **Divergent clamps:** touch clamps `fx_slot` to the new model's max on model
  change and clamps quantum live (`ui_apply_settings_tap`); web does neither
  (relies on `config_validate` to reject). Same edit, two behaviors.
- **Live bug from non-locality:** `handle_save()` can set `midi_clock_out_enable`,
  but `touch_ui.h`'s `UI_F_*` enum has no entry for it — so the **touch panel
  physically cannot toggle MIDI clock out** (the LNK-027 "touch toggle deferred"
  gap, now shown to be structural). A field added in one editor is silently
  missing from the other.

## Design (not binding)

- A pure `config_apply_field(AppConfig* cfg, int field_id, long value)` (or
  promote/extend the already-pure `ui_apply_settings_tap`) that owns every
  per-field clamp + dependency (model→slot re-clamp, quantum bounds, enum ranges).
- `web_config.cpp` `handle_save()` shrinks to string→value parsing
  (`server.arg().toInt()` / `toCharArray`) then calls the shared edit fn per field.
- Touch taps call the same fn.
- Then add the `midi_clock_out` control to the touch settings screen (needs the
  screen-layout pass noted in LNK-027's follow-on — do it here or file-split; at
  minimum the *field/enum + apply logic* lands so the gap is closed structurally).

## What the refactor actually found (2026-07-04)

On close reading, the "duplicated edit logic" is thinner than the review implied,
and part of the divergence is **intentional and documented**:
- **Quantum:** web deliberately does *not* clamp (`web_config.cpp:292-295` comment)
  — it lets `config_validate()` be the single source of truth and 400-rejects an
  out-of-range post; touch clamps live because inc/dec buttons naturally clamp.
  These are modality-appropriate; forcing them identical would override a
  documented decision. **Left as-is.**
- **Model→slot dependency** ("slot must be ≤ the model's max"): genuinely shared,
  and worth one home. **Extracted** to pure `config_set_model()` — both editors
  route through it now.
- **`midi_clock_out` touch gap:** real, but it's a **screen-layout** problem (the
  172×320 settings screen is full — LNK-027 follow-on), not a duplication one.
  Adding an un-drawn field/apply case would be dead code. **Deferred** to the
  touch-layout follow-on; no dead code added.

## Acceptance Criteria

- [x] The one genuinely-shared dependency rule (model→slot) lives in a pure,
      host-tested `config_set_model()`; `test_app_config.c` covers keep-valid-slot,
      clamp-on-downshift, and ignore-invalid-model.
- [x] Both editors use it: `web_config.cpp handle_save()` and `touch_ui.c
      ui_apply_settings_tap()` — web now inherits the same slot re-clamp.
- [x] `midi_clock_out` touch divergence: **won't-do (intentional)** — resolved by
      **ADR-0004**: the web UI is the complete config surface; the touch screen
      carries only real-time controls, and `midi_clock_out_enable` is a
      reboot-gated config flag. Not a bug, so no touch control is added.
- [x] Quantum reject-vs-clamp divergence: intentionally preserved (documented).
- [x] Full host suite green (19 suites); firmware compiles 86%.
