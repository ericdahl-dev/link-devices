## Why

X32Link's `handle_save` (`X32Link/web_config.cpp`) reads every int field **unconditionally**:

```cpp
app_config_set(&cfg, ACF_INPUT_SOURCE,   server.arg("input_source").toInt());
app_config_set(&cfg, ACF_MIDI_CLOCK_OUT, server.arg("midi_clock_out").toInt());
app_config_set(&cfg, ACF_PHASE_DISPLAY_MODE, server.arg("phase_flash").toInt() == 1 ? 1 : 0);
```

`WebServer::arg()` returns `""` for an absent key, and `String("").toInt()` is `0`. `config_validate`
only rejects a `0` that is *out of range*. For the three fields whose `0` is a **legal** value —
`input_source` (0 = Link), `midi_clock_out_enable` (0 = off), `phase_display_mode` (0 = sweep) — a
**partial POST silently resets them**. The iOS app saving a Wi-Fi network (a subset POST) would
disable the MIDI clock and reset the tempo source every time.

Both other firmwares already fixed this and X32Link adopted **neither**:
- **Touch** (`ktouch_web.cpp`): a `full_form` hidden sentinel + `hasArg` guards — "absent = off" only
  when the full form was posted.
- **P4** (`ks_web.cpp`): delegates to pure, host-tested `ks_form_resolve()` (`ks_form.c`); absent
  value fields keep their base value, and `/live` uses `ks_form_apply()` (a patch that doesn't clear
  checkboxes).

Second problem: X32Link's `/config.json` speaks the **shared** dialect (`clock_out`, `led_beat`,
`wifi[]`) but its `/save` reads **bespoke** keys (`midi_clock_out`, `dot_beat`, flat `wifi_ssid`), so
a client **cannot round-trip** it — read `clock_out:true`, but must POST `midi_clock_out=1`.

## What to build

Route X32Link's `/save` (and `/live`) through the pure `ks_form_resolve` / `ks_form_apply` the P4
already uses, in the shared form grammar. The deletion test passes: X32Link's bespoke `AppConfig`
save path is a shallow re-implementation of what `ks_form` already owns deeply and tests.

## Notes

Touches `web_config.cpp`. **Decision:** kept X32Link's own `AppConfig` (it's a mixer bridge, a
separate product — `mixer_ip`/`model`/`fx_slot`/`fdr_*` have no honest home in the clock-box
`KsConfig`, and bloating `KsConfig` with them would be the wrong shape). Instead gave `AppConfig`
the same treatment `KsConfig` has: a shared-vocabulary string setter (`app_config_set_kv`) + a pure
partial-vs-full-form merge (`x32_form_merge`), both host-tested — so all three firmwares now share
ONE form discipline without X32Link adopting `KsConfig`. `handle_live` was already `hasArg`-guarded
(partial-safe), so it was left as-is.
