## Problem

`p4hub_config_load` (`P4Hub/main/p4hub_config_nvs.c`) reads the raw `P4HubConfig`
NVS blob straight into the struct via `nvs_get_blob`. When the struct **layout
changes** — a field added or (worse) inserted mid-struct — the old blob misloads:
fields after the change point read shifted/garbage bytes until a full re-save.

This bit us repeatedly as the config grew: P4-010 (`clock[]` array), P4-011
(`tempo_source`), P4-012 (`metronome_volume`/`voice`). Workarounds so far:
append-at-end (P4-011) and "do a Write & Reboot after flashing to rewrite the
blob" (P4-012). Both are manual and fragile.

## Fix

Version the persisted config so a layout change safely falls back to defaults
instead of loading garbage:

- Add a `uint32_t version` (or magic + version) as the **first** field of
  `P4HubConfig`, bumped on every struct change.
- In `p4hub_config_load`: after `nvs_get_blob`, if the **stored blob size !=
  `sizeof(P4HubConfig)`** OR the loaded `version` != the current version, call
  `p4hub_config_defaults()` (discard the stale blob). Then keep the existing
  compile-time-SSID seed.
- Optionally also reset if `p4hub_config_valid()` fails after load (belt + braces).

Net: flashing a firmware with a changed config struct boots to clean defaults
(no garbage clocks/metronome), and the user just re-enters settings once — no
"must Write & Reboot to un-corrupt" dance.

## Alternative

Per-key NVS storage (one `nvs_set_*` per field) instead of a single blob is
inherently layout-robust, but more code and a bigger change. The version+size
guard is the cheaper, sufficient fix.

## Deliverable

Struct-layout changes never load garbage config; verified by flashing a
size-changed struct over an old NVS blob and seeing clean defaults, not garbage.
