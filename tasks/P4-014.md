## Problem

`p4hub_config_nvs` persists the `P4HubConfig` struct as a raw blob. Any change to
the struct layout — a new field, or (worse) a field inserted mid-struct — shifts
the byte offsets, so an old blob loads **garbage** into the shifted fields after a
flash. This has bitten repeatedly:

- P4-012 inserted `metronome_volume`/`voice` before `clock[]` → the whole clock
  array misloaded until a full web-UI **Write & Reboot** rewrote it.
- P4-013 appended `swing_mbeats` to `ClockOutputCfg` → the `clock[]` stride
  changed → same garbage-until-rewrite.

The current mitigation is "do one Write & Reboot after flashing," which is easy to
forget and reads as a bug when the outputs come up wrong.

## Solution

A small versioned envelope around the blob:

- Prepend a header: a magic, a `version` int, and `sizeof(P4HubConfig)`.
- On load: if magic/version/size don't match what the firmware expects, **discard
  the blob and fall back to `p4hub_config_defaults()`** instead of loading garbage
  (then optionally re-save so it's clean).
- Bump `version` whenever the struct layout changes; treat any mismatch as "start
  from defaults."

Optionally, migrate known-older versions field-by-field instead of resetting —
but even the plain reset-on-mismatch removes the garbage-load footgun.

## Deliverable

Versioned load/save in `p4hub_config_nvs` (or a thin pure header codec that's
host-testable): a mismatched/short/absent blob yields defaults, never garbage. A
matching blob round-trips unchanged. Document the "bump version on layout change"
rule next to the struct. Then the post-flash Write & Reboot is no longer required.
