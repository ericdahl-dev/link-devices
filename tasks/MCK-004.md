## Goal

Wire up WiFi and OSC output in the new firmware. This task is mostly
integration — the shared modules (`app_config`, `osc_sender`, `osc_out`,
`web_config`) already implement everything; we just need to call them from the
right place in `X32MidiClock.ino` and confirm they compile and link cleanly
with the USB MIDI additions.

## Acceptance Criteria

- [ ] `wifi_try_connect()` and `start_config_ap()` copied/adapted from `X32Link.ino`
      (identical logic — AP fallback if credentials missing or connect fails in 30 s)
- [ ] `osc_sender_begin()` and `osc_send_bpm()` called correctly from the BPM task
- [ ] `web_config_begin()` / `web_config_handle()` running so config page stays live
- [ ] Factory reset (BOOT button hold) works identically to X32Link
- [ ] WiFi reconnect loop in `loop()` works (same pattern as X32Link)
- [ ] Firmware compiles with both USB MIDI and WiFi active simultaneously

## Notes

**USB + WiFi coexistence**: On ESP32-S3 the USB peripheral and WiFi radio are
independent — they don't conflict. Both can run at full speed simultaneously.
WiFi stays on the 2.4 GHz radio; USB-OTG uses the dedicated USB DP/DM pins.

**Serial debug**: With `ARDUINO_USB_CDC_ON_BOOT=0` in `build_opt.h`, `Serial`
becomes the USB CDC port (only works after host connects). Switch to `Serial0`
(UART, pins 43/44) for boot messages — or keep a `#define DEBUG_SERIAL Serial0`
alias so it's easy to change.

**No changes to shared modules**: `osc_sender.cpp`, `web_config.cpp`, and
`app_config_nvs.cpp` should compile and link without modification. If a
compile error appears, fix it in the shared file (which benefits both firmwares)
rather than forking.

**AP SSID for this firmware**: Change the AP name to `"X32MidiClock-Config"` so
the user can distinguish it from an X32Link device during setup.
