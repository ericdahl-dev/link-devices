## Goal

The other direction: derive tempo from **incoming MIDI clock** (from a USB-MIDI
device on the host port, or DIN) and **publish it into the Ableton Link session**,
so a hardware sequencer / drum machine can be the tempo master and pull Link (and
everything on it) along. Makes the P4 a two-way bridge, not just Link→MIDI.

## Reuse

- BPM-from-clock math already exists, pure + host-tested: `midi_bpm_calc.c`
  (24-PPQN pulse intervals → BPM), used by the S3 USB-MIDI-device path.
- USB-MIDI **host** IN reception is already wired (P4-005 reads the bulk IN
  endpoint for the loopback counter) — feed those 0xF8 bytes to `midi_bpm_calc`.

## Scope

- Feed host-IN 0xF8 timing to `midi_bpm_calc` → BPM.
- Publish tempo into the Link session (requires bidirectional Link — the P4
  becoming a tempo-setting peer, not just a listener; ties to the broader
  full-peer Link work).
- Handle source arbitration: Link vs MIDI-in as tempo master (web UI selectable).

## Deliverable

A MIDI-clock source on the P4's host port sets the Link session tempo; other Link
peers follow. Source selection in the web UI (P4-007).

## Notes

- The Link-publish side is the hard part (full peer / tempo-setting), bigger than
  just reading clock. Low priority until the read/clock-out spine is mature.

## Progress (embedded-fw-agent)

Shipped **tempo propagation** (MIDI clock IN → BPM → published into the Link
session). Phase-accurate master is scoped out as a follow-up (see below).

### Link tempo-set/adoption protocol (researched from the Ableton/link SDK)
- A peer's Timeline replaces the session's **only when its `beatOrigin` is greater
  than the current session `beatOrigin`** (`Sessions.hpp` `updateTimeline`:
  `if (timeline.beatOrigin > session.timeline.beatOrigin)`). `beatOrigin` is a
  monotonic priority, not "the current beat".
- Link's own `setTempo` builds the new timeline with
  `newBeatOrigin = max(session.toBeats(now), session.beatOrigin + 1)`
  (`ClientSessionTimelines.hpp`) — preserving the beat at `now` (continuity) while
  strictly out-ranking the prior timeline. `midi_link_master.c` mirrors this exactly.
- `Timeline` (`Timeline.hpp`): `key='tmln'` (0x746d6c6e); fields `tempo`
  (microsPerBeat), `beatOrigin` (Beats, wire = micro-beats i64), `timeOrigin`
  (µs i64) — matches what our `link_protocol.c` already parses.
- `SessionMembership` (`SessionId.hpp`): `key='sess'` (0x73657373), value = the
  8-byte sessionId (a NodeId). A peer's timeline is only eligible for adoption when
  it carries the session's sessionId, so we learn the observed session's id and
  claim it in our ALIVE (`link_proto_session_id` + `wifi_link_send_alive`).
- Cross-session merge (which session wins when two meet) is driven by **ghost time
  from ping/pong measurement**, sessionId as tiebreaker (`Sessions.hpp`
  `handleSuccessfulMeasurement`, `SESSION_EPS = 500 ms`). We do NOT run a
  PingResponder, so we can't force a session merge — hence the phase caveat.

### Pure logic (red-green TDD'd)
- `link_protocol.c`: `link_proto_build_alive()` (builds magic+header + `sess` +
  `tmln`, mirroring the parser) and `link_proto_session_id()` (+ `sess` parse).
  Round-trip test: build → `link_proto_parse` → timeline/session/bpm match.
- `midi_clock_in.c` (P4Hub/main): 24-PPQN pulse ring + stop-timeout → BPM via the
  **reused** `midi_bpm_calc.c` (window MUST be 24: the calc bakes in
  "window intervals == one beat").
- `midi_link_master.c` (P4Hub/main): BPM (+ observed session) → Link `LinkTimeline`
  preserving beat continuity and guaranteeing strictly-increasing `beatOrigin`.
- `p4hub_config.c`: `tempo_source` field (Link-follow default / MIDI-master) +
  `tempo_src` web field. **Appended at struct end** — the config is a raw NVS blob,
  so a new field mid-struct corrupts an in-place upgrade (caught + fixed on-device).
- Reused unchanged: `midi_bpm_calc.c`, the `link_protocol.c` parser, `beat_clock`,
  `clock_ticker`, `metronome`, `usb_midi_pack`, `transport`, `clock_output`.

### Glue
- `usb_midi_host.c`: `usb_midi_host_set_clock_cb()` — per-0xF8 IN callback feeds
  `midi_clock_in_pulse()` (was: loopback counter only).
- `wifi_link.c`: random per-boot NodeId; `wifi_link_send_alive()` multicasts the
  built ALIVE on the socket it already owns; `wifi_link_set_master_timeline()`
  override so clock-out/metronome are source-agnostic.
- `p4hub_main.c`: `midi_master_task` (master mode only) derives BPM, installs the
  master timeline, and re-broadcasts (periodic keep-alive + on tempo change). The
  clock-out task never phase-locks in master mode (its ghost xform is a peer's
  domain — would garble beats); free-run gives correct tempo, best-effort phase.

2. Feed 24-PPQN MIDI clock into the P4's USB host port (drum machine / sequencer).
3. P4 serial shows `MIDI-master: in-clk RUN bpm <X> … alive-tx <n>` (tempo derived

4. In Ableton Live (Link on), watch the session tempo follow `<X>`.
   - If Live does NOT adopt: it likely gates a peer's session timeline on having
     measured that peer (we run no PingResponder). That's the deferred phase-master

     Live merges sessions. Tempo-read + best-effort publish is what shipped.
