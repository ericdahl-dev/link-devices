## Goal

Make the P4 act as a USB **host** on its Type-A OTG port: enumerate a connected
USB-MIDI device (e.g. a **Blokas MIDIHub**) and open a MIDI IN/OUT pipe to it.
This is the capability the S3 physically can't do (single device-only OTG) and is
the foundation for P4-004 (Link → USB-MIDI-host clock OUT).

## Why

The headline P4 use case: stream tempo clock straight to USB-MIDI gear with no
computer in the path. That needs a working USB host + MIDI class driver first.

## Approach (staged)

1. **Enumeration spike (do first, cheap):** ESP-IDF `usb_host_lib` example
   (officially supports ESP32-P4) — build for `esp32p4`, flash, and confirm the
   connected MIDIHub enumerates: VID/PID, device/interface class, endpoints, and
   that the Type-A port sources VBUS. Proves host port + power + enumeration on
   this board with real gear attached. *(Toolchain proven, example builds — see
   session 2026-07-04.)*
2. **MIDI class driver:** no ESP-IDF MIDI-host example exists (only `cdc`/`hid`).
   Options: (a) TinyUSB host + `midi_host` class, (b) hand-roll a USB-MIDI (Audio
   Class 1.0 / bulk MIDI streaming) driver over `usb_host` bulk transfers. Scope
   the effort; the MIDI streaming class is small (bulk IN/OUT, 4-byte USB-MIDI
   event packets).
3. **Bridge into the app:** expose a `midi_host_send()` glue that P4-004's
   clock-out task writes 0xF8 to — the pure `clock_ticker` schedule is unchanged.

## Open questions

- TinyUSB host MIDI class vs hand-rolled over `usb_host` — which is less work on
  Arduino-ESP32 3.3.8? (Arduino core is device-mode TinyUSB; host may need the
  ESP-IDF `usb_host` component directly.)
- VBUS/power path on the Type-A port — self-powered hub vs bus-powered device;
  external 5V / PoE for higher-draw gear.
- Does the Arduino-ESP32 P4 build expose `usb_host` cleanly, or does this feature
  push KitchenSync toward an ESP-IDF component build?

## Deliverable

Enumeration spike result (does the MIDIHub enumerate as a host?) + a go/no-go and
effort estimate on the MIDI class driver, then the driver + `midi_host_send()`.

## Spike results (2026-07-04) — host stack proven, hub is the blocker

Built + flashed ESP-IDF `usb_host_lib` (v5.5.4) for `esp32p4`. Confirmed:

- **USB host works.** `USB Host installed with peripheral map 0x1`; Type-A port
  sources VBUS; enumeration runs. **Serial console works** over the P4 UART port
  (unlike the S3's flaky native-USB readback) — big dev-loop win.
- **Blocker: an intermediate USB hub.** With a USB hub in the path the stack logs
  `HUB: Connected device is FS, transaction translator (TT) is not supported` and
  disables the port. The ESP32-P4 host (IDF 5.5) does **not** support a
  full-speed hub's TT, so nothing behind a FS hub enumerates. Downstream ports
  `[1:3]`/`[1:4]` appearing at all = a multi-port hub is present.
- **Fix = connect the USB-MIDI device *directly* to the Type-A port** (no
  intervening hub). A FS MIDI device wired straight to the HS host just downshifts
  to FS — no TT needed. ✅ **Confirmed: direct connection enumerates cleanly.**

### ✅ Direct-connection enumeration (2026-07-04) — full descriptor readout

Blokas **Midihub MH-24V00BD** enumerated as a host device on the Type-A port:

- **idVendor `0x16d0`, idProduct `0x0c9c`**, bcdUSB 1.10, Full speed,
  bDeviceClass `0xef` (Misc/IAD composite), bMaxPower 500mA, 4 interfaces.
- Manufacturer=`Blokas`, Product=`Midihub MH-24V00BD`, Serial=`MH-24V00BD`.
- **Two functions (two IADs):**
  1. **CDC serial** (IAD class `0x02`): if0 CDC-control (INT EP `0x85`) + if1
     CDC-data (BULK EP `0x04` OUT / `0x83` IN). Blokas config/editor channel — not
     needed for clock out.
  2. **USB-MIDI** (IAD class `0x01` Audio): if2 AudioControl + **if3
     MIDIStreaming (class `0x01` / subclass `0x03`)** with **BULK EP `0x01` OUT**
     (host→device — clock write target) and **BULK EP `0x82` IN** (device→host),
     wMaxPacketSize 64.

**Driver is now well-scoped (feeds P4-005):** open interface 3, claim bulk EP
`0x01` OUT / `0x82` IN, exchange 4-byte USB-MIDI 1.0 event packets. A 24-PPQN
clock tick (`0xF8`) is the single-byte packet `[0xCF, 0xF8, 0x00, 0x00]` where the
high nibble `C` = cable number (0–3) and low nibble `F` = CIN (single byte).
Standard USB-MIDI class — either TinyUSB host `midi_host` or a small hand-rolled
bulk driver over `usb_host`.

**MIDIStreaming jack topology (decoded on-device, all 4×4 visible):** added a
class-specific descriptor decoder to the spike firmware (`usb_print_config_descriptor`
skips CS descriptors). The P4 sees the full Midihub 4×4:
- **4 EMBEDDED MIDI IN jacks** (id 1–4) = 4 USB virtual ports the **host sends to**
  → the clock-OUT targets, selected by cable number 0–3.
- **4 EMBEDDED MIDI OUT jacks** (id 9–12) = 4 USB virtual ports the **device sends
  to the host**.
- **4 EXTERNAL IN** (id 5–8) + **4 EXTERNAL OUT** (id 13–16) = the physical DIN
  sockets. USB-virtual ↔ physical-DIN wiring is set inside the **Blokas Midihub
  Editor**, so a clock-out proof needs a route from the USB port the P4 writes →
  a DIN OUT (or an observable destination) configured in the Editor.
- Bumping `CONFIG_USB_HOST_CONTROL_TRANSFER_MAX_SIZE` 256→1024 did **not** clear
  the `CHECK_SHORT_CONFIG_DESC FAILED` on `[1:4]` — that failure is collateral of
  the hub/TT issue, not descriptor size. Keep the 1024 setting anyway (MIDI-class
  config descriptors can exceed 256).

### ESP-IDF build config needed for this P4-NANO (early silicon, chip rev v1.3)

IDF 5.5 defaults min P4 rev to v3.01; this board is **v1.3** (`esp32p4-eco2`),
and v3.x vs v1.x P4 silicon have "huge hardware difference" (Kconfig help). Must
set in `sdkconfig.defaults`:

```
CONFIG_ESP32P4_SELECTS_REV_LESS_V3=y
CONFIG_ESP32P4_REV_MIN_100=y
CONFIG_USB_HOST_CONTROL_TRANSFER_MAX_SIZE=1024
```

(Arduino-ESP32's P4 build already tolerates v1.x — this stricter min-rev gate is
ESP-IDF-only, but KitchenSync may need the `usb_host` component from ESP-IDF regardless.)

## Done (2026-07-04)

Host bring-up proven end-to-end: enumeration (full 4×4 jack topology), interface
claim (if3), and **bulk read + write** all working on hardware. The clock-OUT
write path + loopback read are validated in the P4-005 spike (`scratchpad/
p4_midi_clock`): TX/RX `0xF8` track 1:1 with zero errors. The reusable
`midi_host_send()` glue + Link-driven tempo are folded in under P4-005 / P4-002.
