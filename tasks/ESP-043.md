## Why

The headless S3 Super Mini has no screen and (yet) no buttons, so it gives the operator NO
local feedback: you cannot tell from the box whether it is stopped, armed, running, or
locked to a Link session — you have to open the app. The board has one unused onboard
addressable RGB (WS2812 on GPIO48, per the Arduino esp32s3 variant's `PIN_RGB_LED`). Drive
it as a status light so a screenless box says what it is doing across a room.

Surfaced when the operator noticed "the onboard LED doesn't blink in time" — correct: nothing
drove it (`KSTOUCH_HAS_LED` is the external WS2812 STRIP, off here; `transport_led` is the
DevKit's lit buttons). This adds the onboard single RGB as a distinct capability.

## What

- `config.h`: `HAS_STATUS_RGB` + `STATUS_RGB_GPIO 48`, gated to `BOARD_S3_SUPERMINI` only.
  NOT the Waveshare (GPIO48 is the panel's there) or the DevKit.
- `ktouch_status_rgb.{c,h}`: the STATE -> COLOUR decision as a PURE, host-tested function —
  it does not live in `loop()` (same rule that keeps tempo/transport mapping out of views):
  - stopped -> dark (the light reflects transport; a stopped box says nothing even while
    its clock free-runs)
  - armed -> amber blink at the beat rate (cocked hammer)
  - running, free-run -> green beat flash
  - running, Link-locked -> cyan beat flash (blue channel is the "locked" tell; free-run
    keeps blue at zero so it never reads as a dim cyan)
  - bar-1 downbeat -> a brighter accent of the SAME hue (so "one" reads without masking the
    free/Link signal)
  - the flash is HARD-EDGED (bright the first ~14% of each beat, dark the rest) — a hammer,
    not a "loading" throb
- `.ino`: a thin `status_rgb_tick()` seam samples `ktouch_transport_state()` /
  `ktouch_midi_locked()` / `ktouch_midi_beats()` / `quantum_beats`, calls the pure function,
  and writes the LED only when the colour changes (loop runs every ~5 ms; `rgbLedWrite`
  reconfigures RMT each call).

## Acceptance criteria

- [x] The state->colour mapping is a pure, host-tested function (`test_status_rgb.c`, 5 cases:
      stopped-dark, flash-on/off, green-vs-cyan, downbeat brighter, armed amber).
- [x] Gated to the Super Mini build; the Waveshare and DevKit builds are unaffected.
- [x] Verified on the bench Super Mini (OTA): amber while armed, cyan beat-flash while
      running + Link-locked (peers:1, clk:locked), green free-run, dark when stopped.
- [x] Host suite green (580/0); both Touch variants compile.

## Follow-on (future)

The operator has purchased **buttons with an RGB LED inside each**. The Super Mini's transport
surface is coming (config.h already flags `HAS_BUTTONS` as expected). Those per-button RGBs
should reuse THIS colour vocabulary (`ktouch_status_rgb` or a sibling) so the onboard light
and the button lights say the same thing — one state->colour owner across the box. New ticket
when the buttons are wired. See also `transport_led.c` (the DevKit's on/off button lamps),
which the RGB buttons would supersede.
