## Goal

Validate `LNK-017`–`LNK-019`'s phase pipeline, **and** `LNK-021`'s
phase-locked beat LED, against real Link sessions and a real MIDI clock
source — the same way `LNK-006` validated the original BPM-only firmware and
`LNK-016` will validate the touch UI. This is the gate before any
phase-consuming feature (touch UI phase indicator in `LNK-015`, the beat LED
in `LNK-021`, or a future `docs/ideas.md` A1 bar-quantized scene recall) can
be trusted.

## Test plan

1. **Single-peer Link accuracy**: join an Ableton Live session (or AUM /
   another Link app) on the same LAN as the device's only other peer. Let
   `LNK-018`'s measurement complete. Compare `tempo_source_phase(4.0)`
   crossing zero against Live's own visible downbeat (e.g. its transport
   beat indicator, or an audio click recorded simultaneously). Record the
   observed offset/jitter in milliseconds.
2. **Re-measurement on peer churn**: quit and relaunch the Link app (or
   toggle Link off/on) while the device is running. Confirm
   `tempo_source_phase_valid()` correctly drops to `false` during the gap
   and a fresh measurement completes against the new/returning peer without
   a restart.
3. **Multi-peer session**: join 2+ Link peers simultaneously (e.g. Live +
   a phone app). Confirm the device locks onto one reference peer and stays
   stable (no oscillating offset from switching reference peers
   mid-session) — `LNK-018` intentionally doesn't implement multi-peer
   consensus, confirm that's not a problem in practice for a single
   listening device.
4. **MIDI clock accuracy**: drive USB MIDI clock from a DAW at a known
   tempo and time signature. Confirm `tempo_source_phase(4.0)` tracks bar
   position exactly (pulse-counted, should have effectively zero jitter
   compared to Link) and `tempo_source_phase_valid()` becomes true after
   one full bar of pulses, not before.
5. **Tempo change mid-session**: change tempo in the DAW/Live session while
   running. Confirm phase doesn't glitch or jump incorrectly across the
   tempo change (re-check the `beatOrigin`/`timeOrigin` math in `LNK-017`
   handles a `Timeline` update correctly — a tempo change publishes a new
   `Timeline` with a new `beatOrigin`/`timeOrigin` pair, not just a new
   `tempo`).
6. **Stability soak**: run 30+ minutes with a stable Link session. Confirm
   the `GhostXForm` offset doesn't drift or get corrupted over time (no
   periodic re-measurement was required by LNK-018 — confirm that's
   actually fine in practice, or note it as a follow-up if drift appears).
7. **Beat LED phase-lock** (`LNK-021`): with a Link session running,
   visually compare the physical GPIO48 LED's flash against the session's
   actual downbeat (same reference as step 1 — DAW transport indicator or a
   recorded click). Repeat with MIDI clock as the source. Confirm the LED
   falls back to the legacy free-running blink (not darkness) during the
   pre-sync gap right after boot/peer-join, and snaps to phase-locked
   flashing with no visible double-flash once `tempo_source_phase_valid()`
   goes true.

## Acceptance Criteria

- [ ] All seven steps pass against real Link peers and a real MIDI clock
      source.
- [ ] Measured Link phase offset/jitter recorded in this file's Progress
      section, in milliseconds, with enough detail (peer app used, network
      conditions) to be reproducible.
- [ ] MIDI phase confirmed effectively exact (no measurable jitter beyond
      pulse-interval quantization).
- [ ] Tempo-change-mid-session (step 5) does not produce a visible glitch;
      if it does, file a follow-up task against `LNK-017`/`LNK-019` rather
      than papering over it here.
- [ ] Multi-peer reference-peer stability (step 3) confirmed acceptable for
      this firmware's single-reference-peer design, or a follow-up filed if
      it isn't.
- [ ] Beat LED (step 7) visually tracks the real downbeat for both sources,
      with a clean (non-stuttering) transition out of the pre-sync fallback
      blink. **MIDI side confirmed 2026-06-30** (during LNK-023 field work,
      including the WiFi-down case); Link side and the fallback-transition
      check still open.

## Notes

- This is hardware-in-the-loop, like LNK-006/LNK-016 — host tests don't
  cover real network timing/jitter.
- If jitter is unacceptably large for "land on the downbeat" use cases
  (`docs/ideas.md` A1), the likely next lever is increasing LNK-018's
  sample count before trusting the median, or adding periodic
  re-measurement to correct for any local clock drift over long sessions —
  scope either as a follow-up task here rather than guessing in advance.

## Progress

**2026-06-30, partial/incidental:** Step 4 (MIDI clock accuracy) got a real
data point during `LNK-023` field debugging, not a dedicated LNK-020 pass —
recording it here so it isn't lost. Live Ableton session over USB MIDI
clock, 120bpm: `tempo_source_phase(4.0)` via `/status` tracked correctly
once Ableton's transport was actually playing (`2.875 → 1.0 → 3.125 → 1.25
→ 3.4167` across ~1s polls, matching expected +2.0/sec mod 4), and the beat
LED visually confirmed in sync by the user. Step 1 (Link accuracy), step 2
(re-measurement on peer churn), step 3 (multi-peer), step 5 (tempo change
mid-session), step 6 (stability soak), and step 7's WiFi-down-specific
sub-case are all still unrun — this ticket stays `todo`. Worth noting: the
phase reading initially appeared "frozen" and looked like a bug until we
confirmed Ableton's transport was stopped (clock pulses pause when not
playing) — not a defect, but a good reminder for whoever runs the full
validation pass to explicitly start DAW playback before judging phase
behavior "stuck."
