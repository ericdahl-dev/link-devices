## Goal

Rewire the physical beat LED (`led_task()` in `X32Link.ino`, GPIO48) to flash
on real phase-zero crossings instead of free-running off BPM. Today it's
**not** synced to anything: it computes `beat_ms = 60000/bpm` and flashes
every `beat_ms` measured from an arbitrary local reference (whenever
`led_task` started looping) — right tempo, arbitrary phase offset. This task
makes it phase-locked, using `LNK-019`'s `tempo_source_phase()` — which is
source-agnostic, so this works identically for **both** Link and MIDI
without `led_task()` needing to know which source is active.

Also note: `tempo_source_beat()` exists today with a doc comment implying it
was meant to drive this LED ("true once per beat, self-clears (LED)") but is
currently **dead code** — nothing calls it. This task is also where it
finally gets used (as the pre-sync fallback, see below).

## Implementation

Replace `led_task()`'s `beat_ms`/`last_flash_ms` free-running timer with a
phase-wrap detector:

```c
static void led_task(void*) {
    led_setup();
    float prev_phase = -1.0f;
    for (;;) {
        if (tempo_source_active()) {
            if (tempo_source_phase_valid()) {
                float phase = tempo_source_phase(1.0f);  // fractional-beat phase
                // Flash on wraparound: phase dropped from near-1 back near-0
                // since the last poll. (prev_phase < 0 means "first reading
                // since sync" — skip flashing on that one, just seed it.)
                if (prev_phase >= 0.0f && phase < prev_phase) {
                    led_set(true);
                    vTaskDelay(pdMS_TO_TICKS(LED_FLASH_MS));
                    led_set(false);
                }
                prev_phase = phase;
            } else {
                // Not synced yet (e.g. mid-measurement on Link, or <1 bar of
                // MIDI pulses observed) — fall back to the legacy free-running
                // blink so the LED doesn't go dark during the sync gap.
                prev_phase = -1.0f;  // re-seed once phase becomes valid again
                // ... existing beat_ms/last_flash_ms free-running logic here ...
            }
        }
        vTaskDelay(pdMS_TO_TICKS(5));
    }
}
```

Use `tempo_source_phase(1.0f)` (quantum = 1 beat), not the bar-level quantum
the touch UI's phase wheel uses — this preserves the LED's existing
**per-beat** blink cadence, just phase-locked instead of free-running. (A
downbeat-only LED — flashing once per bar instead of once per beat — is a
one-line change to use `LINK_QUANTUM_BEATS` instead of `1.0f`; worth keeping
in mind for `docs/ideas.md` idea A1/A3 later, but out of scope here — don't
change the LED's visible behavior beyond fixing its phase alignment.)

**Polling resolution**: the task loop polls every 5ms, so wrap detection has
up to ~5ms of slop — call this out in Notes rather than trying to eliminate
it; it's well within any reasonable "looks synced" tolerance and matches the
existing loop's granularity.

## Acceptance Criteria

- [x] `led_task()` flashes on `tempo_source_phase(1.0f)` wraparound once
      `tempo_source_phase_valid()` is true, for **both** `input_source`
      settings (Link and MIDI) — no source-specific branching inside
      `led_task()` itself; it only calls the source-agnostic `tempo_source_*`
      API, same as the rest of `X32Link.ino`.
- [x] While `tempo_source_phase_valid()` is false (sync gap), the LED falls
      back to the legacy BPM-derived free-running blink — it never goes dark
      just because phase isn't ready yet. (Deviation: fallback uses
      `tempo_source_beat()`, not the literal old `beat_ms`/`last_flash_ms`
      timer — see Progress.)
- [x] Transition from fallback blink to phase-locked blink (once
      `tempo_source_phase_valid()` flips true) doesn't produce a visible
      double-flash or stutter — `prev_phase` is correctly re-seeded rather
      than comparing against a stale fallback-mode value.
- [x] `tempo_source_beat()` is now actually called somewhere meaningful (the
      fallback path) — or, if the fallback path is rewritten to not need it,
      update its doc comment in `tempo_source.h` so it no longer claims to be
      LED-consumed when it isn't. Don't leave the comment lying about who
      uses it either way.
- [ ] On real hardware: LED flash visibly aligns with the actual downbeat of
      a Link session, and separately with a MIDI clock source's beat — fold
      this check into `LNK-020`'s hardware validation pass (add it as a test
      step there) rather than duplicating a hardware-validation task here.
      (Blocked on hardware — see Progress; LNK-020 owns this check.)

## Notes

- This task only touches `X32Link.ino`'s `led_task()` — no changes to
  `tempo_source.*`, `link_protocol.c`, or `link_measurement.*`.
- Keep the existing `LED_FLASH_MS` (30ms on-time) and active-high/active-low/
  RGB board-variant handling (`led_set()`) untouched — only the *timing
  trigger* changes, not the flash mechanics or board portability.

  phase-wrap decision `bool led_phase_should_flash(float prev_phase, float
  phase, bool valid)`. Host-tested. Did **not** put this in `link_phase.c`
  (LNK-019) — that file is explicitly Link-adapter-only beats/timeline math
  (`LinkTimeline`-dependent); this decision has nothing to do with Link or

  flashes, first reading since (re)sync seeds without flashing, increasing
  phase doesn't flash, wraparound flashes, equal phase doesn't flash,
  `prev_phase == 0.0` is a real prior reading (not the `-1.0f` sentinel).
  Wired into `esp32/test/Makefile` (`test_led_phase` target + `all`/`clean`).
- `esp32/X32Link/X32Link.ino` `led_task()` rewritten per the ticket's
  pseudocode: phase-wrap detection via `tempo_source_phase(1.0f)` +
  `led_phase_should_flash()` when `tempo_source_phase_valid()`, fallback to
  `tempo_source_beat()` when not valid, `prev_phase` re-seeded to `-1.0f` on
  every invalid->valid transition (and also when `tempo_source_active()`
  goes false, so a dropped-then-resumed source doesn't compare against a
  stale reading either — small addition beyond the literal pseudocode, same
  spirit). Dropped the `g_bpm_mutex`/`g_current_bpm` read from `led_task()`
  — no longer needed since neither the phase path nor `tempo_source_beat()`
  takes a BPM input; `g_bpm_mutex`/`g_current_bpm` are still used elsewhere
  (`bpm_task`, web status) so left untouched there.

Deviation from ticket's pseudocode (documented per instructions): the
fallback branch does not reimplement the old `beat_ms`/`last_flash_ms`
free-running timer inline. Instead it calls `tempo_source_beat()` directly.
Ticket AC explicitly offered this as one of two valid options ("or, if the
fallback path is rewritten to not need it...") — chose this because it
satisfies AC2 (LED never dark during sync gap; `tempo_source_beat()` is
still a BPM/clock-paced per-beat flag, same visible cadence as the old
timer) and AC4 (`tempo_source_beat()` finally gets a real caller, matching
its doc comment) in one move, and removes a second piece of duplicated
beat-timing logic from `led_task()` rather than leaving the legacy timer
math sitting next to the new phase math. `tempo_source.h`'s doc comment on
`tempo_source_beat()` already correctly describes this exact usage ("beat()
stays useful as a fallback before phase() has enough data to be valid. See
LNK-019.") — it was speculative when LNK-019 wrote it; it's accurate now,
so no comment edit was needed, and `tempo_source.*` was left untouched per
the ticket's explicit scope note.

Per LNK-019 implementer's note: confirmed by reading `tempo_source.cpp`
fresh — `tempo_source_phase_valid()` gates MIDI validity on
`g_config.quantum_beats` (a full bar of pulses, `midi_phase_valid()` in
`midi_bpm_calc.c`), not on the `1.0f` quantum `led_task()` passes to
`tempo_source_phase()`. Net effect: on MIDI, the LED stays in fallback mode
until a full bar (not just one beat) of clock pulses has been observed,
even though once valid, `phase(1.0f)`'s per-beat wrap math is correct
regardless of bar boundary (`pulse_count % 24`). This is more conservative
than strictly necessary but is existing, untouched `tempo_source.*`
behavior — in scope per the ticket note, not a bug to fix here.

Polling-resolution slop (~5ms, per ticket's Notes) not eliminated, as
instructed — not an issue for a "looks synced" LED.
