## Symptom

"When starting all from stopped, the drum machine running off KitchenSync falls behind."

Reported by ear, confirmed on the analyzer. Every start, on every slave, forever.

## Cause

**The MIDI spec is explicit: a slave begins playing on the FIRST CLOCK IT RECEIVES AFTER a
Start.** So `0xFA` must reach the wire *before* the `0xF8` that marks the downbeat.

`ks_main.c` ran the clock fan-out loop **before** the transport loop, so at the downbeat tick
the bytes left in the order `0xF8, 0xFA`. The slave had not started when that clock arrived, so
it discarded it and began on the NEXT one — **exactly one 24-PPQN tick late**:

| tempo | one tick | how late every start was |
|---|---|---|
| 120 BPM | 20.83 ms | 20.8 ms |
| 145 BPM | 17.24 ms | 17.2 ms |

The cruel part: ESP-019's acceptance criterion **measured this and passed it**. The capture said
`0xF8` at +0.018 ms from the bar strobe, `0xFA` at +0.338 ms — the Start was correct to the
microsecond, and in the wrong ORDER. "Lands on the bar line" was the wrong question; the right
one is "does the slave's FIRST CLOCK land on the bar line".

## Fix

Swap the two loops in `ks_main.c`: transport bytes first, then the clock fan-out. Both go into
the same USB batch and the same UART in the order added, so the swap is the whole fix. Stop
(`0xFC`) is order-insensitive and rides along.

## Verified on the analyzer (145 BPM, ch0 = GPIO20 DIN, ch1 = GPIO21 bar strobe)

Before:
```
bar strobe -> 0xF8 at +0.018 ms -> 0xFA at +0.338 ms
   the slave discards that clock, starts on the next one: +17.2 ms LATE
```
After:
```
bar strobe -> 0xFA at +0.018 ms -> 0xF8 at +0.339 ms   (0.320 ms = one MIDI byte @31250)
   the slave's FIRST clock IS the downbeat: +0.339 ms from the bar line
```

## Also found while chasing this (config, not a bug)

`clk0_phase` was set to **+30 milli-beats** on the bench unit. Positive = fires earlier, so the
clock grid sat **12.4 ms early** off the bar line at 145 BPM — visible on the analyzer as the
downbeat clock landing 12.0 ms BEFORE the bar strobe, with the next one 5.0 ms after it
(12.0 + 5.0 = 17.2 = one tick, exactly as it should). Zeroed on the unit. Not a code defect, but
it stacked with the byte-order bug and made the sync error worse.

## Acceptance criteria

- [x] `0xFA` precedes the downbeat `0xF8` on the wire
- [x] A slave's first clock after Start IS the downbeat (measured: +0.339 ms from the bar line)
- [ ] Confirmed BY EAR on the drum machine: start from stopped, hit lands on the beat
      — owner check, the analyzer cannot hear it

## Notes

- The Touch (`ktouch_midi_out.cpp:115`) emits `0xFA` from the same writer that emits `0xF8`.
  Check its ordering has the same property before assuming parity.
- The audio-out capture attempted for this was **invalid**: a Logic 8 is digital-only, its
  threshold is ~1.4 V, and line-level audio is ±1 V AC — the "hits" were zero-width noise
  glitches. To measure audible onset, use a trigger/gate output, or record the audio and an
  attenuated GPIO21 strobe on two channels of an audio interface.
