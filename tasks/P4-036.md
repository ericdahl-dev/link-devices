## Problem

The web flasher (`tools/build-flasher-site.sh`, published by `.github/workflows/release-pages.yml`
on a `v*` tag) builds **five targets, and every one of them is an ESP32-S3**: four X32Link board
variants plus KitchenSync Touch. Every manifest declares `"chipFamily": "ESP32-S3"`.

**KitchenSync -- the P4, the flagship, the four-output box -- cannot be flashed from the web page
at all.** It is the only product in the repo that still requires a terminal, a toolchain, and
`idf.py`.

## Why it was left out (and why that reason needs re-testing, not repeating)

Two real obstacles, neither of them proven fatal:

1. **ESP Web Tools / esptool-js support for the ESP32-P4 is unverified.** `chipFamily` is the one
   thing the browser can genuinely check by itself -- it reads the chip ID out of the ROM
   bootloader and refuses a mismatched image. That is a *feature* (it is what makes a
   wrong-firmware flash impossible), but it means the P4 only works if esptool-js knows the P4's
   chip ID. **Nobody has actually tried it.** That is a 20-minute experiment, not a design
   decision.
2. **The P4 is an ESP-IDF build, not Arduino.** The flasher script drives `arduino-cli compile`
   and harvests the Arduino export directory. The P4 needs `idf.py build` and a different set of
   artefacts. The CI job would need an IDF toolchain alongside `arduino-cli`.

## Why it matters more than it looks

The flasher exists so a user -- Dan, or the next one -- can plug a board in and get firmware
without a build environment. The P4 is the device with **four independent clock outputs**, the one
with the most going on, and therefore the one most likely to be handed to someone who does not
want to install ESP-IDF.

It is also the device whose firmware has changed the most this cycle (P4-033, P4-034, per-output
transport master). Shipping fixes to a P4 owner currently means "clone the repo."

## Scope

- [ ] **First, settle question 1 empirically.** Build a P4 image by hand, hand-write a manifest
      with the P4's chipFamily, and try to flash it from a browser. If esptool-js does not support
      the P4, everything below is moot and this ticket becomes "wait for upstream" (with the
      version we tested recorded, so the next person does not re-run the same experiment blind).
- [ ] Add an ESP-IDF build path to `build-flasher-site.sh` alongside the `arduino-cli` one.
- [ ] Emit a manifest with the correct P4 `chipFamily`, and the correct offsets -- the existing
      validator (`tools/validate_manifests.py`) already checks JSON validity, that files exist,
      that offsets do not overlap, and **that the app actually fits the slot declared in its own
      partition table**. The P4 target must pass the same gate, not a relaxed one.
- [ ] Install the IDF toolchain in the release workflow (`release-pages.yml`, tag-triggered --
      deliberately kept out of `ci.yml` so it does not slow every push).
- [ ] Confirm the page's chip-detection copy is honest about what it can and cannot tell: chip
      *family* is auto-detected and enforced, but the **board within a family is not detectable**
      (Super Mini and QT Py are silicon-identical). That caveat is already true for the S3 targets
      and stays true here.

## Reassurance worth writing down

A wrong-board flash **cannot brick an ESP32.** The ROM bootloader lives in mask ROM and is always
recoverable via download mode. (This was stated incorrectly once during this cycle and corrected;
recording it here so the anxiety does not get re-litigated when someone reviews the flasher's risk
surface.)

## Hard constraint

**LinkAudioPoC is GPLv2-quarantined and must never appear in any public-distribution flasher
target.** It is not in the five current targets. It must not become the sixth.
