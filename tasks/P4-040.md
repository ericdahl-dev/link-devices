## Problem

CLK-001's "master-clock option" needs the device to be its own tempo origin. P4-039
fixes the case where a session *was* settled and then lost — the clock engine already
free-runs through that (see P4-039 for the trace). But `beat_source_step()`'s `active`
output is gated on `have_session`, which starts `false` and stays `false` until a Link
timeline has settled at least once (`session_timeline.c`). **A KitchenSync that boots
alone, or that a user wants to run standalone on purpose, currently cannot output a
clock at all** — there is no path to `active=true` without a Link peer ever having
existed.

Confirmed via `tools/linkcli` (#94): the firmware is receive-only at the Link protocol
level (`tempo_source_begin()` only joins the multicast group; there is no Alive/Response
encoder anywhere in the tree). So "master" here can only ever mean *driving this box's
own outputs* from an internal source — not broadcasting a tempo that other real Link
apps/peers would adopt. That's a separate, larger ticket if we ever want it (an outbound
gossip encoder, plus real session-merge semantics — see CLK-001's open questions).

## Scope: KitchenSync (P4) only

`X32Link` (X32-mixer bridge, a special-use device) and the LED-indicator-only variant
are follow-only by design and keep today's behavior unchanged. They share
`link_protocol.c`/`session_timeline.c`/`beat_source.c` with KitchenSync, so this
capability must NOT be added to that shared engine — it lives entirely in
`KitchenSync/main/`, wired in only there.

## Design

New pure module, `master_clock.c` (host-tested, same shape as `beat_source.c`):

- Tracks tap-tempo and manual-BPM state, set from the web UI. Independent of Link.
- Produces a `LinkTimeline`-shaped basis (just `micros_per_beat`, no real
  `beat_origin`/`time_origin` needed) that `ks_tick_step` can feed through the same
  `beat_source_step()` free-running-accumulator path Link's un-locked case already
  uses — no change to `beat_source.c` itself.
- `master_clock_active()` becomes true the first time the user sets a tempo (tap or
  manual BPM), independent of `have_session` — this is what makes a never-joined P4
  able to output a clock at all.

**Auto-promote:** the instant `wifi_link_peers() == 0`, `ks_main.c` seeds the internal
source from whatever tempo was last running (P4-039's held value) if the user hasn't
already set one — continuity, no audible jump. Tap/BPM controls are live from that
point on.

**Rejoin:** the instant peers > 0 again, defer back to the Link-driven basis. No merge
arbitration needed — this device never broadcasts, so there is no competing session to
resolve; going solo was local bookkeeping only, not a second session on the wire. (Real
Link's actual merge tie-break needs an active ghost-time measurement between two
broadcasting peers — see `LinkAudioPoC/third_party/link/ableton/link/Sessions.hpp:131-181`
— which doesn't apply here since we were never a session to begin with.)

## Acceptance criteria

- [x] Host test: `master_clock_arbiter` with no Link session ever seen and a tap-tempo
      set — reports `have_session=true`, free-running at the tapped tempo
      (`test_master_clock.c`; `ks_tick_step`/`beat_source.c` need no changes at all —
      the arbiter just feeds them a synthetic `LinkTimeline` on the same free-run path
      Link's un-locked case already uses).
- [x] Host test: session settles, peers drop to 0, no tap has occurred — internal source
      auto-seeds from the last Link tempo on the peers>0→0 edge, continuous across the
      transition (`test_arbiter_seeds_from_link_on_solo_edge`).
- [x] Host test: while solo-promoted, a peer reappears with a real timeline — output
      defers back to Link immediately, no merge logic
      (`test_arbiter_rejoin_defers_to_link`). Also covered: a live tap taken while
      already solo isn't clobbered by a later tick's arbiter call
      (`test_arbiter_does_not_clobber_live_tap_while_solo`).
- [ ] Web UI: tap-tempo / manual BPM control, and an indicator for "running on internal
      clock" vs. "following Link" (`MasterArbiterOut.on_internal` is already wired
      through `ks_main.c`; needs an HTTP endpoint + JS control, ties into CLK-001's
      source-selector UI item). Not yet done.
- [ ] Hardware-validated on the logic analyzer: solo → tap a new tempo → rejoin a real
      Link session, confirm no glitch in the pulse train at either transition. Blocked
      on the web UI control above (no way to tap a tempo without it yet).

## Explicitly out of scope

- Broadcasting our tempo onto the real Link multicast group so other Link apps/hardware
  adopt it ("Link as leader" in CLK-001). Needs an outbound Alive/Response encoder and,
  eventually, real session-merge semantics if we want to compete as a legitimate second
  session rather than just falling back silently. Separate, larger ticket.
- MIDI-in and audio-follow as tempo sources feeding this same arbiter (P4-011, ESP-007) —
  this ticket is internal/tap only. The general multi-source arbiter is CLK-001's job.
- `X32Link` / LED-only variant behavior changes — none.

## Resolution

Shipped in #97 ("KitchenSync stops lying when solo, can originate a clock alone").
The ticket sat at `todo` afterwards; corrected 2026-07-13 during the Ordna audit.

`master_clock` has since moved out of `KitchenSync/main/` into the shared home (ESP-027) --
the Touch is the clock box too, and a clock box that goes silent when the last peer leaves is
indefensible. Its header used to say "KitchenSync (P4) only".
