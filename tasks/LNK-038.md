## Problem

Deployed devices cannot report what firmware they run. Today we had a live
X32Link at 192.168.1.55 and no way to tell whether it was current, short of
feature-sniffing the web UI HTML — there is no version string anywhere in the
source. This gets worse once devices are distributed: support, OTA upgrade
decisions ("is this .bin newer than what's on the box?"), and bug reports all
need "what version are you on?" to be answerable by the device itself.

## Solution

- One shared `fw_version.h` (in `lib/`) exposing `FW_VERSION` (checked-in
  semver string, bumped on release) and `FW_BUILD` (`__DATE__ " " __TIME__`
  compile stamp — free from the compiler, distinguishes dev builds).
- Optionally overridable later via a `-DFW_VERSION=...` build property fed by
  `git describe --tags` in CI; the checked-in string is the v1 mechanism.
- Surface it in four places, in both firmwares (X32Link + KitchenSync):
  1. Serial boot banner and the periodic `[X32Link]` status line.
  2. Web UI footer on `/`.
  3. `/status` JSON field (`"fw"`) — machine-readable, lets a script audit a
     fleet of devices.
  4. `/update` OTA page — show the running version before flashing over it.
- Tag releases (`git tag vX.Y.Z` ↔ `FW_VERSION`) so any distributed .bin can
  be traced back to source.

## Deliverable

`lib/fw_version.h` + wiring in `X32Link.ino` / `web_config.cpp` /
`web_status_json` and the KitchenSync equivalents; boot banner prints it;
`/status` returns it; `/update` shows it; release-tagging process noted in
`AGENTS.md`. Host suite: `web_status_json` builder test asserts the `fw` field.

## Notes (done)

`fw_version.h` lives in `X32Link/` rather than `lib/` to match the existing
sharing pattern: KitchenSync gets it via its `X32LINK_DIR` include path
(ADR-0003), X32MidiClock via symlink. Started at FW_VERSION "2.1.0" —
continues the story of the web UI's old hardcoded "FW 2.0" badge, which this
replaces. No dedicated `/version` endpoint: the `"fw"` field in `/status` is
the machine-readable surface (one fewer shallow route, per the ARC-005/010
pass-through-glue principle).
