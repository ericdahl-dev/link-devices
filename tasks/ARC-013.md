## Problem

The P4 connection policy is 4 module-statics (`s_ap_mode`, `s_got_ip_once`,
`s_link_started`, `s_connect_start_us` — `wifi_link.c:82-85`) whose transitions live
interleaved with radio I/O in `on_wifi_event` (`wifi_link.c:120-160`). Only the 30 s
deadline is pure/tested (`wifi_fallback.c`). The rules "retry forever after first IP,"
"ignore stray STA events in AP mode," "spawn listener once," and the reconnect backoff
are untestable because they only exist next to `esp_wifi_*` calls.

The *same* policy is hand-reimplemented on the S3 in a different (blocking) shape
(`X32Link/X32Link.ino:181-233`), with the 30 s budget as a raw `> 30000` literal.
`wifi_fallback.h:16` even documents "must stay in sync by hand." The "retry forever
after first IP" rule has no S3 analogue — behaviour already differs.

Deletion test: delete the 4 statics and nothing reconstructs the policy — it's a state
machine that wants to be a module, not scattered flags.

## Solution (deepen)

A pure `wifi_conn_policy`: `step(state, event, now_us, connect_start_us) → {next, action}`
with `action ∈ {connect, backoff_connect, give_up_to_ap, spawn_listener, ignore}` and
`state ∈ {CONNECTING, GOT_IP, RETRYING, AP}`. The 4 booleans collapse to one enum. The
P4 event handler translates `esp_event → event → step → execute`; the S3 loop translates
`WiFi.status() → event → step`. One tested owner of the give-up budget + AP-fallback
rule (subsumes `wifi_fallback`).

## Notes

- Host-test the full transition table (`test/test_wifi_conn_policy.c`); the fallback
  test already proves the pattern.
- Migrate both targets; removes the "keep in sync by hand" seam. Behaviour on hardware
  should be unchanged — smoke-test STA connect, mid-session drop (retry-forever), and
  first-boot/no-AP fallback on the P4; the blocking connect + AP fallback on the S3.
