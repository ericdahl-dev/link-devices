## What to build

The reusable core of the RGB LED output subsystem, established by migrating the P4's
**existing** visual metronome onto it with **zero behavior change**:

- A pure `fill(out[0..npix))` renderer contract; `metro_strip_render` conforms as-is.
- A pure `led_output` dispatcher: given the snapshot + a per-output config + the board
  descriptor, it fills each output's pixel buffer, applies **brightness once** (removed
  from `metro_strip_render` — currently double-scales), and hands the buffer to the glue.
- A phase-accurate `LedSnapshot { beats, quantum, transport, link_locked }` published by
  the clock task (one-writer/many-reader, same beat the MIDI clock uses).
- The ESP-IDF glue generalized to `write(output, px, npix)` over RMT `led_strip`, driven
  by a **separate core-0, low-priority LED task** — the render+push must leave the 1 ms
  clock writer (pri 19, core 1) entirely, and must not block it (bounded RMT wait, no
  `portMAX_DELAY`).

No config-model change and no app change in this slice — the P4 keeps its current LED
config; only the render path moves. This is the foundation ESP-047 and ESP-050 build on.

## Acceptance criteria

- [ ] `metro_strip_render` no longer applies `cfg->bright`; the dispatcher owns brightness (single application, verified by a host test that the two are not multiplied).
- [ ] Pure `led_output` dispatcher + `LedSnapshot`, host-tested (Unity), no Arduino/ESP-IDF deps.
- [ ] P4 LED render+push runs on a dedicated core-0 low-priority task, not inside `clock_out_task`.
- [ ] Clock-task tick health unchanged under LED activity (no new `dropped`/`overruns`/`max_work` regression on the analyzer bench).
- [ ] P4 visual metronome looks identical on hardware (same colors/pattern/brightness) — behavior-preserving refactor.
- [ ] All existing host suites green, including `test/test_metro_strip.c`.
