## Goal

Two bugs were found on the **Touch** and fixed there (ESP-030 pt3). Both live in code
the P4 also has, or in a contract the P4 also publishes. The P4 has **not been checked**
— its hardware was not on the bench — and this task exists so nobody closes that gap by
reading the source and calling it verified. That is exactly how both bugs shipped.

### 1. Is `link_owns` actually a live claim on the P4?

`ks_status.h` defines it as one: *"link_owns is true once a Link peer has published
StartStopState ... when link_owns is true the manual PLAY/STOP buttons are ignored, so
the UI greys them."* The iOS app takes that literally and **stops sending the tap**.

The Touch was passing `link_proto_start_stop_seen()` — a STICKY flag ("has any peer EVER
published a StartStopState", latched until the last peer leaves) — while
`ktouch_transport.c` never consulted Link start/stop at all. So it announced "I ignore
your play button" and then obeyed it, and play/stop were dead in the app for as long as
any Link peer had ever pressed play.

The P4 is believed to genuinely defer to Link via `ks_tick` arbitration, which would make
its `link_owns` honest and its greyed buttons correct. **Believed is not verified.**

### 2. Does a partial POST to `/save` silently switch things off?

An unchecked HTML checkbox sends nothing, so "absent = off" is the only way a device's own
page can turn one OFF — and it also means any client posting a SUBSET disables every box
it never knew to send.

On the Touch this was not hypothetical: a POST carrying only `wifi`/`clk0_*` set
`clock_enable` AND `transport_enable` to 0, killing play/stop on the touchscreen. And the
iOS app's `saveFormFields` does not emit every P4 checkbox either — if `ks_web.cpp`'s save
handler has the same absent-means-off behaviour, **saving WiFi from the app will disable
whatever the app has never heard of.**

The Touch's fix: absent = off applies ONLY when the sender declares the whole form
(`full_form`, which the device's page posts). Silence from a partial patch means "leave it
alone", not "turn it off".

## Acceptance Criteria

- [ ] On a real P4, with a Link peer that has pressed play: `GET /status` reports
      `link_owns:true` **and** `POST /transport?play=1` is genuinely IGNORED. If the device
      obeys while reporting true, `link_owns` is a lie and must be fixed the way the Touch's
      was — report what the arbiter actually decides, never "we once saw one".
- [ ] With NO peer having published StartStopState: `link_owns:false` and transport obeys.
- [ ] Audit `ks_web.cpp`'s save handler for absent-checkbox-means-off. If present, adopt the
      `full_form` sentinel so a partial POST cannot disable a field the sender never sent.
- [ ] `POST /save` carrying only `wifi_ssid` leaves every checkbox-backed setting unchanged
      (measured on hardware, before and after `GET /config.json`).
- [ ] The iOS app's `saveFormFields` is diffed against the P4's full form, and any field the
      app omits is either emitted or provably safe to omit.

## Notes

The recurring error class, now five for five: **two situations collapsed into one signal.**
T-009 (404 ≠ dead network), T-010 (dropped poll ≠ disconnection), T-016 (decode failure ≠
unreachable), T-018 (lifetime counter ≠ live fault), ESP-028 (`sync:1` on a wire dead for
138 s) — and now `link_owns` ("seen once" ≠ "owns now").

Do NOT close this from a reading of the source. The app's own test suite proved the point:
`RealTouchStatusTests` decodes a VERBATIM `/status` capture and caught `link_owns` instantly,
while every hand-written fixture in the repo had been passing for weeks. Capture real P4
bytes and add them the same way.
