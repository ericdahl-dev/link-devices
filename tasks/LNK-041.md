## Problem

CI compiles X32Link with **no `PartitionScheme`**, so the ESP32 core falls back to
its `default` table: a **1.25 MB** app slot. Real units do **not** run that table.

AGENTS.md (hardware-verified 2026-07-09): **`PartitionScheme=min_spiffs` is required
for OTA** — the QT Py FQBN's default is `tinyuf2_noota`, which has no second app slot,
so `/update` fails with `Partition Could Not be Found`. The deployed unit
(192.168.1.55) runs `min_spiffs`. So the shipping table is `min_spiffs` (**1.9 MB**
app slot), and CI measures something else entirely.

Measured on 2026-07-11 (same commit, same sketch):

| Scheme | App slot | X32Link uses |
|---|---|---|
| `default` — **what CI compiles** | 1,310,720 | **82%** |
| `min_spiffs` — **what ships** | 1,966,080 | **55%** |

Two consequences, and the second is the dangerous one:

1. **The 82% is fiction.** It reads like X32Link is nearly out of flash. It isn't —
   it's at 55% of the table it actually runs on. Someone will eventually refuse to
   add a feature, or go on a size hunt, over a number that describes no real device.
2. **CI never compiles the table that ships.** The size gate is measuring a
   partition layout no unit boots. That is the same class of mistake ESP-016 already
   hit on the Touch: the core silently fell back to the 4 MB `default` table and the
   sketch quietly reached 87% full before anyone noticed the table was wrong.

## Solution

Pin the shipping scheme in CI so the number means something:

```
FQBN: esp32:esp32:esp32s3:USBMode=hwcdc,CDCOnBoot=cdc,FlashSize=16M,PSRAM=opi,PartitionScheme=min_spiffs
```

Touch already does this correctly (`TOUCH_FQBN` carries `PartitionScheme=custom`);
X32Link's `FQBN` is the one still unpinned. This is a one-line CI change — no
firmware change, no partition change on any device.

## Acceptance Criteria

- [ ] X32Link's CI FQBN carries `PartitionScheme=min_spiffs` (all four X32Link
      compile steps: headless, Waveshare touch, QT Py NeoPixel, QT Py + battery).
- [ ] The X32 emulator step is left alone — it has its own FQBN/`default_8MB` and is
      a different board.
- [ ] CI reports X32Link flash usage against the 1.9 MB slot (expect ~55%).
- [ ] A comment in `ci.yml` says WHY, the way the `TOUCH_FQBN` comment does, so nobody
      "simplifies" it back out.

## Notes

- **Does not change any device.** This only makes CI compile the table the units
  already boot. No reflash, no NVS impact.
- Related but separate: **LNK-042** (give X32Link a real 16 MB table on the Waveshare
  board). This ticket is the cheap, zero-risk half; LNK-042 is the one with a
  hardware deadline.
- Do NOT reach for `PartitionScheme=custom` here: X32Link has no `partitions.csv`, so
  `custom` breaks its build outright (the `ci.yml` comment on `TOUCH_FQBN` already
  says so).
