## Goal

A standalone, laptop-free **S3 touch box** that grabs Ableton Link over WiFi and
drives external gear with MIDI clock + transport over a DIN jack, with a
touchscreen for hands-on transport (PLAY/STOP). First target: **Dan's Roland
RC-505** loop station. This is the GTM **wedge** product -- a cheap S3 brick to a
named customer -- made concrete.

Product line: **KitchenSync Touch** (this, S3, tactile) sits below **KitchenSync
Pro** (the P4 hub). The repo target is the **X32Link** firmware (S3), not the
KitchenSync P4 firmware; "KitchenSync Touch" is the product brand.

## Why

- The approved GTM (`docs/kitchensync-gtm.md`) calls for a cheap S3 brick to a
  named customer (Dan); the P4 is flagged "wrong price/BOM" for this. This ticket
  is that wedge.
- Dan's RC-505 has DIN MIDI IN + USB-MIDI but **no native Ableton Link**. Bridging
  Link -> MIDI clock is exactly this device's job.
- The touchscreen is the differentiator: physical transport, no laptop. It is an
  *added* feature over the P4's web-only UI, which is why the name is "-touch",
  not "-lite".

## What already exists (most of it)

- **Link receive** -- X32Link `link_protocol.c` (clean-room, receive-only).
- **Touch display + UI** -- Waveshare **ESP32-S3-Touch-LCD-1.47** (AXS5106L touch
  @ I2C 0x63, LovyanGFX), gated `HAS_TOUCH_DISPLAY` (LNK-014/015). `touch_ui.c` is
  pure + host-tested (`ui_hit` rect hit-testing, settings-field slots). Screens
  today: status / settings / IP-keypad.
- **Clock + transport engine** -- shared pure `clock_ticker` / `midi_clock_out`
  (0xF8) and `transport` / `transport_launch` (0xFA/0xFC, bar-quantized START),
  already compiled into both firmwares.

## What is new (the lift)

1. **DIN MIDI OUT on the S3** -- port `midi_uart_out.c` (the ESP-015 pattern: UART
   TX @ 31250 8N1). Simpler than the P4 (WiFi on-die, `esp_read_mac` works, no C6).
   **`#define MIDI_TX_GPIO 11`** (see pin map below). Route both the clock (0xF8)
   and transport (0xFA/0xFC) to it -- and clock regardless of USB (the ESP-015 ungate).

   **Pin decision (confirmed against the Waveshare pinout, 2026-07-10):** MIDI TX =
   **GPIO11** -- right header, under the touch pins, next to 3V3 + GND so all three
   DIN wires come off one header. Circuit (3.3 V MIDI-OUT): GPIO11 ->[10R]-> DIN
   pin 5, 3V3 ->[33R]-> DIN pin 4, GND -> DIN pin 2 (pins 1/3 unused; 220R on each
   also works). Free header pins if GPIO11 is inconvenient: 1,2,4,5,6,7,8,9,10.
   Taken: LCD 21/38/39/40/45/46, touch 41/42/47/48, SD 13-18, console 43/44, USB
   19/20, strapping GPIO3. Full map saved to auto-memory (s3-touch-lcd-147-pinout).
2. **A transport touch screen** -- a new `touch_ui` screen with big PLAY / STOP hit
   rects -> a transport intent (reuse the `transport_intent` mailbox pattern) ->
   `transport_launch` (START quantized to the bar) -> 0xFA/0xFC on the DIN wire.
   The pure launch/quantize logic already exists (ESP-011); this adds the on-screen
   surface + wiring.

## Connector decision

- Use the RC-505's **5-pin DIN MIDI IN**. Set the RC-505 MIDI sync to external/AUTO.
- The RC-505's USB-MIDI is a **device** port (USB-B) -> using it would force the S3
  to be a USB *host* (more firmware, VBUS, awkward connector) for no benefit when a
  DIN IN is present. DIN wins. (USB-host only earns its keep on gear with no DIN.)

## Acceptance Criteria

The criteria below originally named the RC-505 in every line. **We do not own one** -- it
is Dan's box -- so as written they were unfalsifiable in-house and would have held this
ticket open forever. Rewritten to say what can actually be established here, and validated
against the real DIN MIDI slave that IS on the bench: a **Korg ER-1**.

The ER-1 is the harder test, not a substitute for a weaker one. A drum machine slaved to
MIDI clock puts every timing flaw straight into the audio -- a stall is an audible stutter,
jitter smears the hats, a dropped pulse is a flam. A loop station AVERAGES incoming clock
over many pulses and forgives precisely the faults ESP-018 was chasing. Passing on an ER-1
is stronger evidence than passing on an RC-505 would have been.

- [x] S3-Touch board grabs Link tempo and shows it on-screen (with the tenth -- a live
      132.5 BPM session renders as 132.5, not 132)
- [x] DIN MIDI clock (0xF8) out at the Link tempo -- decoded on the analyzer at GPIO11:
      132.44 BPM measured against a 132.5 session, zero junk bytes
- [x] Touch PLAY/STOP sends 0xFA/0xFC, START bar-quantized, STOP immediate -- proven on the
      wire: two STARTs exactly 2 bars apart (3.630s vs 3.625s predicted), a STOP mid-bar at
      0.53 bars (i.e. NOT quantized)
- [x] Nudge and swing verified on the wire (nudge to 0.22ms; swing an exact 3:2 shuffle)
- [x] No stalls, no bursts, no dropped pulses under sustained web load (ESP-018)
- [x] Touch decision logic host-tested -- `ktouch_touch_step`, 12 tests (moved out of the
      glue; the original criterion named test_touch_ui.c, but the Touch has its own sketch)
- [x] **Drives real DIN MIDI gear correctly: a Korg ER-1 follows Link tempo and transport,
      confirmed by ear on the bench**
- [ ] **RC-505 specifically** -- cannot be closed in-house. The RC-505 is a standard MIDI
      clock slave, so the engineering risk is retired; what remains is integration, and
      **Dan is that test.** Ask him for one thing back: set MIDI sync to AUTO/external, hit
      play, and report whether it drifts over ten minutes.
- [ ] **Confirm Dan's unit is the ORIGINAL RC-505, not the mkII** -- a $0, one-message check
      that could invalidate the whole DIN-connector decision. Do it before soldering a cable.

## Decision 2026-07-10: dedicated sketch (not a build variant)

KitchenSync Touch is its own GTM product (Link -> MIDI/DIN + touch transport, **no
OSC, no X32 mixer**), so it gets a **dedicated `KitchenSyncTouch/` Arduino sketch**,
NOT an X32Link build variant. X32Link stays the X32-mixer product. The sketch reuses
X32Link's shared modules by committed relative symlink (ADR-0006; the proven
X32_emulator / LoraLink mechanism) and keeps its OWN `config.h` + `app_config` to
sever the mixer/OSC coupling. Board flag `BOARD_WAVESHARE_S3_TOUCH_LCD_147` lives in
`config.h` (not build_opt.h) so it can't go stale in the arduino-cli cache.

Full design + symlink list + increment plan + footguns:
`docs/plans/2026-07-10-kitchensynctouch-sketch-design.md` (with the Embedded
Firmware Engineer).

Increments: (1) boots + Link + DIN/USB clock + tempo display; (2) transport touch
screen (PLAY/STOP -> bar-quantized 0xFA/0xFC); (3) trimmed web + captive portal +
OTA; (4) polish (mDNS/ESP-014, quantum, colours).

Bench findings so far (2026-07-10): flashed the S3-Touch; the blank screen was a
build-target bug (build_opt.h was `-DBOARD_QTPY_ESP32S3`, no display) -- fixed by
the Waveshare flag; the display comes up, the board grabs Link (peers:1, 125 BPM)
and OSC/clock work. OSC is now dead weight for this product -> the dedicated sketch
drops it. DIN 0xF8 on GPIO11 not yet analyzer-verified (wrong leads on the bench).

## Constraint 2026-07-10: touch is TRANSPORT-ONLY

Hard product rule: the touch screen does ONLY transport (status + PLAY/STOP).
**No settings/config screens on the LCD** -- an accidental tap must never pop a
setting mid-performance. ALL config (WiFi, quantum, clock/transport enable) lives
on the **web UI** (Inc3). This drops the X32Link touch settings/keyboard screens
entirely; `touch_ui`'s settings-tap half is not reused. Simpler + safer.

## Decision 2026-07-10: DIN-only, no USB-MIDI (bench-diagnosed)

On the bench the S3's USB serial/upload port kept dropping. Root cause: enabling
the clock made X32Link call USB.begin() with a USB-MIDI interface -> the S3 became
a composite CDC+MIDI device, and that re-enumeration knocks the native USB-CDC
serial port off the Mac (only the ROM download mode brought it back). The cable
was fine (used all day).

Fix + product call: **KitchenSync Touch is DIN-only.** Dan's RC-505 is driven over
a 5-pin DIN jack, so there is no reason to be a USB-MIDI device -- and being one
breaks the serial. Dropped the USB-MIDI path (ktouch_midi_out emits 0xF8 on DIN
only; no USB.begin/midi_clock composite). USB stays plain CDC -> serial stable.
USB-MIDI can be an opt-in feature later if wanted.

Verified on hardware: KitchenSync Touch boots, joins WiFi, grabs Link (125 BPM,
phase valid, beats advancing -> DIN clock task ticking), and the serial port
survives the boot (no composite drop).

## Display UX (2026-07-10, from bench)

- **Landscape** 320x172 (setRotation, held horizontally) -- room for big BPM, the
  sync wheel, a readable name, and (Inc2) fat side-by-side PLAY/STOP buttons.
- Reach by **name**: show `<host>.local` (mDNS, per-unit kstouch-XXXX), not a faint
  IP. Big + white -- must be legible across a stage.
- **Mode-aware bottom line (Inc3):** in AP/setup mode show the setup Wi-Fi details
  (join `KSTouch-Config` -> 192.168.4.1) so a new user knows how to connect; once
  joined to the user's Wi-Fi, show the mDNS name / IP. Inc1 has no AP mode yet
  (compile-time creds), so only the connected path shows today.
- Sync wheel: sweeping phase marker (beat position in the bar).

## Transport UI: full-screen toggle (2026-07-10, from bench)

Two states (play/stop) = ONE full-screen toggle, not two buttons. Tap anywhere:
STOPPED -> ARMING (amber, next bar) -> PLAYING (green); tap again -> STOPPED. One
huge target (fat-finger-proof), the screen colour IS the state, and it moots any
left/right touch-mirror calibration. Pure logic is `ktouch_toggle_intent(state)`
(host-tested); the writer publishes launch state for the display to colour + toggle.

**Trigger mode (web setting, Inc3):** `play_on_release` -- fire the toggle on
touch-down (digital-DJ) or on release (turntable-DJ: hold, release to let the beat
play). Both have merits. Config field added (default = on touch); the web toggle
lands with the Inc3 config UI. Per the touch-is-transport-only rule, this setting
lives on the web, never on the LCD.

## Web UI brought to the P4 gold standard (2026-07-10, Inc3b)

The Inc3b config page shipped functional but visibly below the P4's finish: flat
stacked toggles, no section grouping, and help text built by misusing the `.cap`
heading class with inline `style="…color:var(--mut)"` overrides. Realigned it to
`ks_web.cpp` (the P4 page, our brand reference) — `ktouch_web.cpp` only:

- `.frow.head` header rows (title + LED-dim tick + master toggle).
- `.grp` groups with `.sect` accent rails that reveal a feature's settings when
  its toggle flips on (the P4's `syncSect`; JS-driven on load, so no new server
  tokens — fields stay in the DOM and still submit).
- One `.hint` class for all help text; zero inline-style captions remain.
- Responsive desktop treatment (two-column form + 4-across status meter bridge).

Form contract unchanged (field names, ids, `%TOKENS%`, `/nudge` `/bright` wiring),
so `handle_save`/`handle_status` and `test_ktouch_ui.c` are unaffected. The shared
`ui_chrome.{h,c}` was **not** touched. This convention is now **ADR-0008**: the P4
page is the reference, per-firmware pages mirror it, and the shared chrome is never
edited to fix one page.

Open call for a future increment: the reveal is JS-on-load (one-frame flash of a
disabled feature's rail before it collapses). The P4 server-renders the hide state
via `%CLKSECT%`-style tokens; port that to Touch if the flash ever matters.

## Touch-edge dispatch extracted (2026-07-11, PR #65)

`ktouch_touch_step` — the press / release / cue tap arbitration that used to live
inline in `ktouch_display_tick` — is now a pure, host-tested module
(`ktouch_ui.{c,h}` + `test_ktouch_ui.c`). This is the tap half of the display
deepening (see the architecture review's candidate #5).

**Still smeared into the LCD draw calls** (`ktouch_display.cpp`, on master) and
untestable — the *render-side* derivations:
- sync-wheel phase angle — `fmodf(phase / quantum, 1) * 360` at `:247`,
- AP-line vs mDNS status text — `g_ap_mode` / `.local` at `:237-238`,
- BPM formatting and the sync-state (`phase_valid ? 1 : active ? 0 : -1`).

X32Link already isolated the equivalent logic in a tested `touch_ui.c`
(`ui_phase_angle` / `ui_bpm_str`); Touch should mirror it. Filed as **ARC-021**.
