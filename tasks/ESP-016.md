## Goal

A standalone, laptop-free **S3 touch box** that grabs Ableton Link over WiFi and
drives external gear with MIDI clock + transport over a DIN jack, with a
touchscreen for hands-on transport (PLAY/STOP). First target: **Dan's Roland
RC-505** loop station. This is the GTM **wedge** product -- a cheap S3 brick to a
named customer -- made concrete.

Product line: **KitchenSync Touch** (this, S3, tactile) sits below **KitchenSync
Pro** (the P4 hub). The repo target is the **X32Link** firmware (S3), not the
KitchenSync P4 firmware; "KitchenSync Touch" is the product brand.

## Why

- The approved GTM (`docs/kitchensync-gtm.md`) calls for a cheap S3 brick to a
  named customer (Dan); the P4 is flagged "wrong price/BOM" for this. This ticket
  is that wedge.
- Dan's RC-505 has DIN MIDI IN + USB-MIDI but **no native Ableton Link**. Bridging
  Link -> MIDI clock is exactly this device's job.
- The touchscreen is the differentiator: physical transport, no laptop. It is an
  *added* feature over the P4's web-only UI, which is why the name is "-touch",
  not "-lite".

## What already exists (most of it)

- **Link receive** -- X32Link `link_protocol.c` (clean-room, receive-only).
- **Touch display + UI** -- Waveshare **ESP32-S3-Touch-LCD-1.47** (AXS5106L touch
  @ I2C 0x63, LovyanGFX), gated `HAS_TOUCH_DISPLAY` (LNK-014/015). `touch_ui.c` is
  pure + host-tested (`ui_hit` rect hit-testing, settings-field slots). Screens
  today: status / settings / IP-keypad.
- **Clock + transport engine** -- shared pure `clock_ticker` / `midi_clock_out`
  (0xF8) and `transport` / `transport_launch` (0xFA/0xFC, bar-quantized START),
  already compiled into both firmwares.

## What is new (the lift)

1. **DIN MIDI OUT on the S3** -- port `midi_uart_out.c` (the ESP-015 pattern: UART
   TX @ 31250 8N1). Simpler than the P4 (WiFi on-die, `esp_read_mac` works, no C6).
   **`#define MIDI_TX_GPIO 11`** (see pin map below). Route both the clock (0xF8)
   and transport (0xFA/0xFC) to it -- and clock regardless of USB (the ESP-015 ungate).

   **Pin decision (confirmed against the Waveshare pinout, 2026-07-10):** MIDI TX =
   **GPIO11** -- right header, under the touch pins, next to 3V3 + GND so all three
   DIN wires come off one header. Circuit (3.3 V MIDI-OUT): GPIO11 ->[10R]-> DIN
   pin 5, 3V3 ->[33R]-> DIN pin 4, GND -> DIN pin 2 (pins 1/3 unused; 220R on each
   also works). Free header pins if GPIO11 is inconvenient: 1,2,4,5,6,7,8,9,10.
   Taken: LCD 21/38/39/40/45/46, touch 41/42/47/48, SD 13-18, console 43/44, USB
   19/20, strapping GPIO3. Full map saved to auto-memory (s3-touch-lcd-147-pinout).
2. **A transport touch screen** -- a new `touch_ui` screen with big PLAY / STOP hit
   rects -> a transport intent (reuse the `transport_intent` mailbox pattern) ->
   `transport_launch` (START quantized to the bar) -> 0xFA/0xFC on the DIN wire.
   The pure launch/quantize logic already exists (ESP-011); this adds the on-screen
   surface + wiring.

## Connector decision

- Use the RC-505's **5-pin DIN MIDI IN**. Set the RC-505 MIDI sync to external/AUTO.
- The RC-505's USB-MIDI is a **device** port (USB-B) -> using it would force the S3
  to be a USB *host* (more firmware, VBUS, awkward connector) for no benefit when a
  DIN IN is present. DIN wins. (USB-host only earns its keep on gear with no DIN.)

## Design question for Dan

- Tempo sync (0xF8) only, or transport too (0xFA/0xFC so loops launch on the
  downbeat)? The touch screen implies transport -> build both.
- Confirm the unit is the original RC-505 (has the DIN jack; the USB-B strongly
  suggests it).

## BOM (hardware)

- Waveshare **ESP32-S3-Touch-LCD-1.47** (~$15) -- or a QT Py S3 + a separate panel.
- 5-pin DIN jack (or TRS-A) + **2 resistors** (~220R from 5V logic; ~33R/10R from
  3.3V). That is the entire MIDI-out bill.

## Acceptance Criteria

- [ ] S3-Touch board grabs Link tempo (existing) and shows it on-screen
- [ ] DIN MIDI clock (0xF8) out to the RC-505 MIDI IN; RC-505 follows Link tempo
- [ ] Touch PLAY/STOP sends 0xFA/0xFC (START quantized to the bar); RC-505 starts/stops in time
- [ ] New transport-screen hit-rects host-tested in `test/test_touch_ui.c`; clock/transport reuse the existing pure engines
- [ ] Verified on hardware: DIN out decoded on the analyzer (same rig as ESP-015), RC-505 syncs

## Non-goals / later

- USB-host path (unneeded; DIN is present).
- A bigger touch panel (2.1" / 4.3") for a fuller transport surface -- driver swap;
  price it only if the 1.47" feels cramped for 2-3 buttons.
- Per-track / scene control on the RC-505 (CC/SysEx territory, separate ticket).

## Decision 2026-07-10: dedicated sketch (not a build variant)

KitchenSync Touch is its own GTM product (Link -> MIDI/DIN + touch transport, **no
OSC, no X32 mixer**), so it gets a **dedicated `KitchenSyncTouch/` Arduino sketch**,
NOT an X32Link build variant. X32Link stays the X32-mixer product. The sketch reuses
X32Link's shared modules by committed relative symlink (ADR-0006; the proven
X32_emulator / LoraLink mechanism) and keeps its OWN `config.h` + `app_config` to
sever the mixer/OSC coupling. Board flag `BOARD_WAVESHARE_S3_TOUCH_LCD_147` lives in
`config.h` (not build_opt.h) so it can't go stale in the arduino-cli cache.

Full design + symlink list + increment plan + footguns:
`docs/plans/2026-07-10-kitchensynctouch-sketch-design.md` (with the Embedded
Firmware Engineer).

Increments: (1) boots + Link + DIN/USB clock + tempo display; (2) transport touch
screen (PLAY/STOP -> bar-quantized 0xFA/0xFC); (3) trimmed web + captive portal +
OTA; (4) polish (mDNS/ESP-014, quantum, colours).

Bench findings so far (2026-07-10): flashed the S3-Touch; the blank screen was a
build-target bug (build_opt.h was `-DBOARD_QTPY_ESP32S3`, no display) -- fixed by
the Waveshare flag; the display comes up, the board grabs Link (peers:1, 125 BPM)
and OSC/clock work. OSC is now dead weight for this product -> the dedicated sketch
drops it. DIN 0xF8 on GPIO11 not yet analyzer-verified (wrong leads on the bench).
