## Why

**A real Touch on a real network is invisible to the iOS app, and the app's own discovery heuristic
is why.**

We advertise **generic** `_http._tcp` with no TXT record and no service subtype
(`wifi_link.c:125`, ESP-012):

```c
mdns_service_add(NULL, "_http", "_tcp", 80, NULL, 0);
```

With nothing to match on, a client has only the hostname. So `KitchenSyncDiscovery` browses every
`_http._tcp` responder on the LAN and filters:

```swift
names.filter { $0 == "kitchensync" || $0.hasPrefix("kitchensync-") }
```

On 2026-07-14 a real Touch appeared on the bench as **`kstouch-dfd0`**. It does not match that
prefix. **The app cannot see it at all** — not "shows it wrong", cannot see it.

The heuristic has the mirror-image failure too: any stranger's box named `kitchensync-anything` on a
venue's network gets adopted straight into the fleet. Fine at home; not fine at a show.

Guessing identity from a hostname was always a heuristic. It has now failed in both directions, on
the first real device. Advertise the truth instead.

## Second problem this fixes: cross-target OTA

`POST /update` accepts **any** image and cannot tell a cross-target `.bin` from a good one. The
fleet is multi-target (ESP-026). Nothing in the client can check either — the firmware does not
publish its chip target — so the iOS app's OTA sheet is reduced to **asking the user to promise**
the binary matches, via a checkbox. A filename convention is not a safety mechanism.

Publishing `target` lets a client refuse a mismatched flash instead of trusting a tick-box.

## What to build

In `mdns_advertise()` (`KitchenSync/main/wifi_link.c`), and the Touch's equivalent:

```c
mdns_txt_item_t txt[] = {
    { "dev",    "kitchensync"      },   /* identity — match on THIS, not the hostname */
    { "model",  "touch" },              /* or "p4" — which product, for a client that cares */
    { "target", CONFIG_IDF_TARGET  },   /* e.g. "esp32p4" — the chip this .bin was built for */
    { "fw",     FW_VERSION         },   /* LNK-038; lets a client know if /config.json exists */
};
mdns_service_add(NULL, "_http", "_tcp", 80, txt, sizeof(txt) / sizeof(txt[0]));
```

`mdns` copies these, so the array need not outlive the call. You will also need
`#include "fw_version.h"` in `wifi_link.c`, which it does not currently have.

**⚠️ The block above has never been compiled.** It was drafted on a machine with no `idf.py`, so it
is a starting point, not working code. Verify the `mdns_txt_item_t` field names and the
`mdns_service_add` signature against the ESP-IDF version actually in use before trusting it — the
API has changed shape across IDF releases.

`fw` is worth including even though `/status` already reports it: it lets a client know whether a
unit has `/config.json` (ESP-030) *before* it calls anything.

## Acceptance criteria

- [ ] Both firmwares advertise TXT: `dev`, `model`, `target`, `fw`
- [ ] A client can identify a unit WITHOUT matching on the hostname
- [ ] A non-KitchenSync host named `kitchensync-whatever` is no longer adopted into the fleet
- [ ] `kstouch-dfd0` — a real Touch — is discovered by the iOS app
- [ ] `target` matches the actual build target and is usable to refuse a cross-target OTA
- [ ] Verified with `dns-sd -B _http._tcp local` + `dns-sd -L <instance> _http._tcp` showing the
      TXT record on real hardware

## Notes

**Client-side must degrade gracefully.** Units in the field have no TXT record and will not get one
until they are reflashed. So the app should match on `dev` when TXT is present and **fall back to
the hostname prefix when it is absent** — otherwise this change makes every un-updated device
disappear, which is a worse bug than the one it fixes. That fallback is tracked app-side and should
land before, or with, this.

**Do not** add a KitchenSync-specific service type (`_kitchensync._tcp`) instead of TXT. The devices
genuinely serve HTTP, browsers and the device's own web UI depend on that, and a subtype would
strand every existing client. TXT is additive; a service-type change is not.
