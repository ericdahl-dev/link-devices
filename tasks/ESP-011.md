## Goal

Play/Stop buttons on KitchenSync's web UI that send MIDI Start (0xFA) /
Stop (0xFC) to the connected USB-MIDI device, quantized to the Link grid --
**per clock output**. Start the rig from the box or a phone, without a DAW in
the room, and launch each piece of gear on its own bar.

## Why this is cheap (unlike ESP-010)

Everything downstream already exists and is host-tested:

- `X32Link/transport.h` -- pure Start/Stop edge state machine (P4-008)
- `X32Link/usb_midi_pack.h` -- encodes 0xFA/0xFC as USB-MIDI packets
- `KitchenSync/main/ks_main.c:128` -- already fans a `TransportAction` out to
  every enabled clock output cable

Today the only thing that *drives* that machinery is Link's `isPlaying` flag.
This ticket adds a second driver: the web UI. No Link protocol changes, no
packet transmission, **no Ableton licensing gate** -- unlike ESP-010 (propose
transport to the Link session), which is blocked on it. We are talking to our
own MIDI device, not to the session.

Prefer this ticket over ESP-010. It delivers most of the user-facing value
("start my rig from the web UI") for a fraction of the work and none of the
legal risk.

## Per-output transport

Each of the 4 clock outputs already owns its division, phase and swing
(P4-010). It should own its transport too: arm output 1 now, output 3 in two
bars, stop the drum machine while the synth keeps running. That is the
Multiclock-class behavior this product is chasing, and it falls out of the
same mechanism as global transport.

Today `ks_main.c:127-136` fans ONE `plan.transport` to every enabled cable,
driven by ONE `Transport` state machine in `KsTickState`. Per-output means
both become arrays, exactly the move `clock_ticker` already made for
`plan.pulses[]`:

    TransportAction transport;            ->  transport[KS_CLOCK_OUTPUTS]
    Transport       tr;   (in KsTickState) ->  tr[KS_CLOCK_OUTPUTS]

`transport_update()` itself is already per-instance and needs no change --
it is a pure edge detector over one `Transport`. Feed it a per-output intent
instead of the session-wide one and the pure module is done. This is a small,
contained widening of `ks_tick_step`, and its host tests widen with it.

Keep a master control in the UI (start/stop all) on top of the per-output
buttons -- the common case is still "start everything".

## The good bit: quantized launch

We keep *listening* to Link for the timeline (beats + phase), and use it to
time the manual Start. Press Play mid-bar and the device does not lurch in
immediately -- it arms, waits for the next downbeat, and emits 0xFA exactly
on the grid. The rig starts in time with the Link session even though we
never send transport back to it.

This is the whole reason to prefer this over a dumb remote button, and the
P4 already has everything needed: `ks_tick_step` runs every millisecond with
the current beat position, and `beat_source` already reports phase-locked
beats. Arming is "emit START when `beats` crosses the next quantum boundary".

Same for Stop -- either immediate (musicians expect stop to be instant) or
quantized. Recommend: **Start quantized, Stop immediate.** Ableton's own
launch-quantization behaves this way and it is what the muscle memory expects.

The quantum should reuse the metronome's (`KS_TICK_METRO_QUANTUM`, 4 beats)
rather than inventing a second one; a future ticket can expose it in the UI
alongside the meter work (P4-022).

## Design

The interesting question is who owns transport when both Link and the web
want it. Options, cheapest first:

1. **Manual only when Link isn't driving.** If no StartStopState has been seen
   (`link_proto_start_stop_seen() == false`), the web buttons own transport.
   Once a peer publishes transport, Link wins and the buttons grey out. Simple,
   no conflict, matches how the metronome already behaves.
2. **Manual always overrides**, until the next Link transport edge. More
   flexible, but two masters -- the classic source of "why did my gear stop".

Recommend (1) for v1. The arbitration decision belongs in a pure module
(a `transport_source` selector, mirroring `beat_source`, ADR-0003) with the
web handler feeding it an intent, not poking `usb_midi_pack` directly.

Note `transport_update()`'s priming rule already prevents a spurious Start on
join -- reuse it rather than emitting raw bytes from the web handler.

## Scope

- POST endpoint (`/transport?out=N&play=1|0`, `out=all` for the master),
  same live-apply shape as `/live`.
- Per-output Play/Stop buttons in each Clock Out block, plus a master
  start/stop. Each shows its own state: stopped / armed / running.
- Only enabled outputs participate; a disabled output ignores transport.
- Metronome / LED follow automatically -- `ks_tick.c:56` gates on
  `tp_playing`, so feeding the manual state through the same flag gets the
  audible + visual behavior for free.

## Acceptance Criteria

- [ ] Each output has independent transport state (stopped/armed/running)
- [ ] Master start/stop drives all enabled outputs at once
- [ ] Outputs can run while others are stopped, each on its own cable
- [ ] Web Play ARMS; 0xFA is emitted on the next downbeat, not immediately
- [ ] Emitted Start lands within a few ms of the Link grid (bench-measured
      against the metronome click / a scope, same rigor as P4-009)
- [ ] Web Stop sends 0xFC immediately (no quantize)
- [ ] Arming state visible in the UI ("starting on next bar...")
- [ ] Pressing Play when no Link session exists: starts immediately (nothing
      to quantize to) rather than hanging armed forever
- [ ] No spurious Start when a device is connected mid-play
- [ ] Defined, tested behavior when Link also publishes transport (option 1)
- [ ] Arbitration AND quantized-launch timing host-tested as pure modules
- [ ] `ks_tick_step`'s widened per-output transport covered in
      `test/test_ks_tick.c`
- [ ] Verified on hardware against the Midihub / a real MIDI device
- [ ] Metronome + LED follow the manual transport state

## Notes

- Pairs with ESP-009 (standby heartbeat): the box shows it is waiting, and
  this gives you the button to start it -- on the beat.
- The quantized-launch machinery is the same shape P4-025 (transport modes /
  quantize / resync) wants. Build it here, generalize there; consider closing
  P4-025 into this one rather than keeping two tickets for one mechanism.
- Does NOT start Ableton -- that is ESP-010, which needs the Link send path
  and is licensing-blocked. Worth stating plainly in the UI so nobody expects
  the DAW to follow.
