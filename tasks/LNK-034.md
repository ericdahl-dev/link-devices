## Problem

The only way to get new firmware onto a deployed X32Link unit is a USB cable +
`arduino-cli upload`. Once a box is racked behind a mixer that's a real
inconvenience — every fix or feature needs it pulled and re-cabled.

## Solution

Add a `/update` endpoint to the existing `web_config.cpp` server (the same rack-
panel HTTP server that already serves `/`, `/status`, `/save`):

- **`GET /update`** — a small themed page with a plain `<input type=file>` +
  `<form method=POST enctype=multipart/form-data>`. No JS needed — a normal
  browser file upload.
- **`POST /update`** — `Update.h` (bundled with the esp32 Arduino core) streams
  the uploaded `.bin` into the inactive OTA app slot via the standard
  `HTTPUpload` callback (`Update.begin/write/end`), then reboots into it on
  success. Failure (bad image, interrupted upload) leaves the running firmware
  untouched and reports a 500 with the reason, mirroring `handle_save()`'s
  result-page shape.

No new library dependency — `Update.h` ships with the esp32 Arduino core
already required by this sketch. Relies on the board's default OTA-enabled
partition scheme (app0/app1), already implied by the `FlashSize=16M` FQBN used
in CI and documented in `AGENTS.md`.

## Deliverable

`X32Link/web_config.cpp`: `/update` GET (upload page) + POST (`HTTPUpload`
handler + result page, reboot on success) wired into `web_config_begin()`.
Footer link from the main config page. Host suite unaffected (no pure logic
here — this is glue, like the rest of `web_config.cpp`); `arduino-cli compile`
covers it in CI.
