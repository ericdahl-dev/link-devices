## Goal

The Touch and the P4 keep their config in TWO different structs — `AppConfig`
(KitchenSyncTouch/app_config.h) and `KsConfig` (X32Link/ks_config.h) — with two NVS
migration lineages, two `/live` handlers, and two persist paths. Converge the Touch onto
the shared `KsConfig` so there is ONE config struct across the fleet, and retire
`AppConfig`.

## Why (the DRY bug that proves it)

ESP-037 (settable tempo) had to add `tempo_mbpm` to BOTH structs, with a separate
`vN->vN+1` blob migration in each. And it surfaced a real bug: the P4's `/live` copies
only the fields in `ks_config_live_safe_copy`'s allowlist, and tempo was missing from it
— so the P4 parsed, validated, then SILENTLY DROPPED a tempo edit. The Touch's `/live`
applies fields directly, so it worked there. One feature, two code paths, one of them
quietly broken. That is the divergence tax, and it recurs on every new field.

## What is already shared (do not re-do)

- `/status` — `ks_status.c` (ESP-029), symlinked into the Touch.
- `/config.json` — `ks_config_json.c` + `KsCaps` (ESP-030).
- The form grammar — `ks_config_set()` keys (ESP-030).
- `master_clock`, `clock_output`, `beat_source`, the tick engine.

So the WIRE is one shape; only the CONFIG STORAGE and web handlers drifted. The Touch
already builds a transient `KsConfig` to emit `/config.json` — making `KsConfig` its
ACTUAL storage is the natural next step, not a new direction.

## Why it is not trivial

- The Touch's NVS blob format changes (AppConfig -> KsConfig), so it needs ONE more
  migration — the last one, reading the final `AppConfig` layout and writing `KsConfig`.
  Same freeze-and-migrate discipline as ESP-030/037; the bench board's WiFi must survive.
- `KsConfig` carries fields the Touch does not use (mixer, 4 clock outputs vs 1). `KsCaps`
  already hides them on the wire; the Touch carries the full struct and populates the
  subset — a few bytes of NVS to delete a whole parallel config system.
- The Touch's `handle_live`/`handle_save` fold onto the P4's `ks_form_apply` +
  `ks_config_live_safe_copy` path, so the ONE live-safe definition governs both and the
  ESP-037 bug class cannot recur.

## Acceptance Criteria

- [x] The Touch stores `KsConfig`; `AppConfig` and its migrations are deleted.
- [~] A single, final AppConfig->KsConfig NVS migration preserves WiFi + all settings.
      The migration is WRITTEN and host-tested (test_ktouch_migrate: v1..v4 + WiFi
      survival + rejects). Hardware smoke on the bench Super Mini (WiFi survives a real
      upgrade) is the one remaining hands-on step.
- [x] One `/live` path and one `ks_config_live_safe_copy` govern both firmwares.
- [x] A future config field means editing ONE struct, ONE migration, ONE live-safe list.
- [~] Host suite green; P4 (esp32p4) + both Touch builds + X32Link all COMPILE. Hardware
      smoke on the Super Mini and the P4 still pending.
