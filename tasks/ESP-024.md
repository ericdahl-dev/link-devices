## UPDATE 2026-07-13 — the baseline exists now, measured on the wire

The bench rig (ESP-025) + Saleae on the DIN TX gave the number this ticket was written to obtain:

```
mean   20833.5 us   (120.012 BPM)
stdev    373.4 us   <-- THE NUMBER
spread  1124.0 us   peak-to-peak
```

**The spread is one millisecond.** That is the 1 ms writer task rendered visible on the wire —
the edge can only land on a tick boundary, so the jitter IS the tick quantisation. Every jitter
figure previously in this repo was the task's own view of itself; this is the first taken from
the DIN wire, and it confirms the thesis below outright.

Also now true, and it changes the plan:
- **ESP-026 blocks this.** Do not tune the clock on the Arduino core (fixed sdkconfig, no
  `menuconfig`, so IRAM/cache/interrupt knobs are unreachable) and then again on the P4.
- **The analyzer is scriptable** from the agent (Logic 2 MCP on `127.0.0.1:10530`) and transport
  is scriptable (`POST /transport`), so timing runs need no human finger.
- The ER-1 floor measurement (first acceptance criterion) is still **not done** and is still the
  cheapest possible first experiment.

## Problem

**Every jitter number we have was measured on software-timed edges.** The clock edge is
placed by a 1 kHz FreeRTOS task (`clock_output_step()`, the 1 ms writer in
`midi_clock_out_io.cpp`). A 1 ms task cannot place an edge better than ~1 ms, so the
numbers we keep quoting are measuring *the tick rate*, not the silicon:

| Board | measured | how the edge is placed |
|---|---|---|
| KitchenSync Touch (S3) | **533 us stdev** (min 15.4 / max 24.4 ms) | 1 ms software task |
| KitchenSync (P4) | **331 us** (+-1 ms) | 1 ms software task |

ESP-018 closed with exactly this left open: *"the S3 is measurably looser and we do not
yet know why. Decide whether that is acceptable for the RC-505 or needs fixing too."*

We have never measured a **properly-architected** S3. Until we do, we do not know whether
the S3 is the problem or the 1 ms task is.

## Why it matters

**This blocks the PCB spin.** The board decision (S3 alone vs. S3 + a ~$1 companion MCU
as a dedicated edge generator) hinges on one number we have never taken. Getting it wrong
in either direction is expensive:

- Fab an S3 board that can't hold the clock -> respin.
- Add a second chip we didn't need -> BOM, a second clock domain to discipline, and
  complexity in the one part of the product that must not be flaky.

And it is the *only* thing that can justify leaving ESP32 at all. Switching MCU families
means writing an Ableton Link `IoContext` platform layer from scratch — Ableton ships
exactly one embedded platform layer and it is ESP32. Do not trade that away on a guess.

## The ceiling to keep straight

DIN MIDI is 31250 baud, so a clock byte takes **320 us to serialise**. Two things follow,
and they are easy to conflate:

- That 320 us is **fixed latency, not jitter.** It is constant, so it does not widen the
  distribution. The receiver clocks off the **start bit** — so start-bit jitter below
  320 us is still real and still worth removing.
- It becomes a **quantiser** the moment a clock byte queues behind other MIDI traffic in
  the UART FIFO. Then the clock byte is delayed by up to a full byte time, per byte
  ahead of it. That is an architecture fix (clock bytes jump the queue), not a chip fix.

The analog/Eurorack triggers have no such floor — they are bare GPIO edges, and that is
where hardware timing earns its keep outright.

## Approach

**Measure, change one thing, measure again.** Do not batch the changes.

### 0. Fixture (do this first, it invalidates everything else if skipped)
- `tools/linkcli` running and in-session with the device — the firmware is a Link
  *listener* and reports `peers:0` alone on the wire.
- **Read the tempo from `/status`. Do not hardcode it in the analysis script.**
- Analyzer on the DIN TX pin (Touch: GPIO11) and on the analog trigger pin.
- Capture >= 10 min, and repeat the whole run **under web load** (`/status` polling +
  `/nudge` POSTs) — that is the condition that produced the ESP-018 stalls.

### 1. Baseline
Re-take the Touch's numbers on the current build so the before/after is same-rig,
same-tempo, same-session. Report stdev, p99, min/max, and stall/burst counts.

### 2. Analog edges -> RMT
The trigger outputs are pure pulses; RMT places them with ~50 ns precision, fully
offloaded from the scheduler. Cheapest, cleanest win.

Budget note: the S3 has **4 RMT TX channels and the WS2812 LED owns one**
(`2026-07-10-analog-sync-board.md`), so at most 3 outputs get hardware-shaped edges.

### 3. DIN byte handoff -> IRAM'd hardware timer
RMT cannot drive a UART, so the DIN path needs a different fix:

- Hand the `0xF8` to the UART from a **GPTimer ISR marked `IRAM_ATTR`**, so a flash cache
  miss can never delay the edge. ISR latency is a few us and deterministic.
- **Give clock bytes priority over other MIDI traffic** in the FIFO — this is the
  quantiser named above, and it is invisible on an idle wire but ugly under real traffic.

If that is not enough, the escalation is to **encode the MIDI frame as an RMT waveform**:
a UART byte is just 10 bit-times of 32 us, which RMT can emit with ~50 ns placement,
removing the UART and the FIFO entirely. More invasive (all MIDI must then route through
RMT), so hold it in reserve.

### 4. Re-measure and decide

## The gear reality check (this is what decides it, not a spec sheet)

**We do not own an E-RM Multiclock.** It is the product's *market* comparison, not a
bench reference — we cannot measure against it and must not quote its spec as our bar.

What is actually on the bench is a **Korg ER-1**. That is better than a spec sheet,
because it answers the only question that matters: *does any of this reach real gear?*

The two output paths are **not** equivalent, and must not share a threshold:

- **DIN MIDI feeds gear that has its own clock handling.** The ER-1 averages/PLLs the
  incoming F8 stream. Its internal floor may simply **absorb** an improvement from 533 us
  to 10 us — in which case the DIN work is below the floor that matters and gets killed,
  no matter how pretty the analyzer trace is. **We do not currently know the ER-1's clock
  floor. Measuring it is step 1, not an assumption.**
- **Analog / Eurorack triggers have no PLL to absorb anything.** The module fires on the
  edge. Edge quality passes straight through to the sound. This path can be worth RMT
  even if the ER-1 cannot see the difference at all.

**The ER-1 test:** clock the ER-1 over DIN, record its audio out, onset-detect the drum
hits, and measure onset jitter against the Link timeline — before and after. If the ER-1's
hits do not land tighter, the DIN change did not reach the customer.

## Kill criterion

Per the standing rule: **a timing correction below the jitter floor gets killed, not
shipped.** Written down before the work starts, so the result cannot be rationalised:

- **DIN path — KILL** if the ER-1's measured onset jitter does not improve, even if the
  analyzer shows the wire got dramatically tighter. A wire nobody downstream can resolve
  is not a product improvement. (Take the ER-1's own floor first: if its onset jitter is
  already wider than our *current* 533 us, the DIN work is dead on arrival and we have
  saved ourselves the effort.)
- **Analog path — KILL** if RMT does not move the trigger edge decisively below
  **10 us stdev**. There is no PLL downstream to hide behind here, so if hardware timing
  cannot deliver on a bare GPIO pulse, edge placement was never the bottleneck.
- **SHIP the S3 for the PCB** if the analog path clears its bar and the DIN path either
  clears the ER-1 test or is credibly killed as below-the-floor — with **0 stalls and 0
  dropped pulses over 10 min under web load** in both cases (ESP-018's failure mode is
  dropped pulses, which is audible regardless of jitter).
- **Escalate to the two-chip fallback** (S3 keeps the network + Link; a ~$1 MCU generates
  every edge from a "downbeat at T, tempo X" message over SPI) *only* if a kill fires.
  Not before, and not by switching MCU families.

One honest limit on the ER-1 as an oracle: it is **one** device, and a forgiving one is
not proof the fleet is forgiving. A null result on the ER-1 kills the *DIN* work for now;
it does not prove the RC-505 (Dan's actual box, which we also do not own) agrees.

## Acceptance criteria

- [ ] **ER-1's own clock floor measured first** — onset jitter while slaved to the
      *current* build. If it is already wider than 533 us, say so and kill the DIN work
      before writing any of it
- [ ] Baseline re-taken on the current build with `linkcli` in-session and the tempo read
      from `/status` (not hardcoded)
- [ ] Analog trigger edges generated by RMT; measured on the analyzer
- [ ] DIN byte handed off from an `IRAM_ATTR` hardware-timer ISR; clock bytes prioritised
      in the UART FIFO; measured on the analyzer **and** at the ER-1's audio out
- [ ] Before/after table: stdev, p99, min/max, stalls, dropped — idle **and** under web
      load, >= 10 min each
- [ ] The kill criterion is applied out loud, per path: ship / kill / escalate, with the
      number
- [ ] The PCB board decision (S3 alone vs. S3 + companion MCU) recorded with that number
      as its evidence
