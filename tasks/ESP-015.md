## Goal

Emit MIDI (clock + transport) on a GPIO as a real DIN-style UART byte stream,
mirroring what the P4 already sends over USB, plus a one-pulse-per-bar strobe
GPIO. PR 1 of 3 in the "GPIO MIDI/clock I/O alongside USB" feature (design:
docs/plans/2026-07-10-gpio-midi-clock-io-design.md). Siblings: MIDI IN (P4-011),
analog sync pulses (P4-021 / LNK-028).

## Why

Two payoffs from one small change:

1. **Measurement.** A digital logic analyzer cannot decode USB, so the P4's
   USB-MIDI timing has never been measured. A UART MIDI mirror at 31250 8N1 is
   decodable by Logic 2's Async Serial analyzer -- real 0xF8/0xFA/0xFC bytes with
   timestamps. With the downbeat strobe as a trigger, this finally measures
   ESP-011's claim ("does the emitted 0xFA land on the Link downbeat within a few
   ms?") and the FREERTOS_HZ=1000 -> <=1ms jitter assumption, both unmeasured.
2. **Feature.** A UART MIDI OUT is the firmware half of DIN MIDI out -- drive a
   hardware synth (with a jack + 2x 220-ohm resistors), not just a USB host.

## What shipped (PR)

- Pure: `ks_tick` gains `plan.downbeat`, computed via the existing `bar_reset_due`
  + a `BarReset` in `KsTickState`. Fires once per bar; primes (no false fire) on a
  re-origin. Host-tested in `test/test_ks_tick.c` (fires-once-per-bar,
  no-downbeat-without-session).
- Glue: `midi_uart_out.c` -- UART1 TX @ 31250 8N1, one-byte fire-and-forget
  (`uart_write_bytes`, no queue, no `uart_wait_tx_done`), plus the strobe GPIO.
- `ks_main.c` emit loop taps output 0's 0xF8 / 0xFA / 0xFC onto the wire from the
  same code point as the USB send, and strobes `plan.downbeat` each tick.

## Pins (CANDIDATES -- confirm on the P4-NANO header before wiring the analyzer)

- MIDI TX = GPIO20, downbeat strobe = GPIO21. CONFIRMED on the P4-NANO header
  (ADC1_CHANNEL4 / ADC1_CHANNEL5), free, and boots clean with WiFi/mDNS up.
- MIDI IN (PR2) = GPIO22 (ADC1_CHANNEL6): free on the header.
- WARNING for PR3: GPIO24-27 are the USB1 PHY (USB1P1_*), the USB-MIDI host port
  -- not free. Analog-sync candidates are GPIO23/32/33/36/45-48.

## Notes

- Realtime MIDI is single bytes -- no `usb_midi_pack` on this path.
- Mirrors exactly one output (0) because realtime bytes are cable-agnostic;
  mirroring all four would duplicate bytes. A `mirror_output` config field (which
  bumps KS_CONFIG_VERSION) was deferred to keep this PR migration-free.
- Drives the analyzer, not a synth: a real DIN jack needs the connector + 2
  resistors (receiver side has the opto). Not needed to measure.

## Measured on hardware (2026-07-10, Logic 8ch @ 4 MHz)

**Wiring confirmed** with a temporary 0xF8-every-500ms diagnostic: CH0 decoded 8x
0xF8 at exactly 31250 8N1, 304us/byte (10-bit frame @ 31250 = 320us). CH0 <->
GPIO20 and the decode chain are correct.

**Design fix found while measuring:** the 0xF8 clock was gated on `usb_ready`, so
the DIN wire stayed silent with no USB host attached. Wrong for a hardware MIDI
out -- clock is a musical decision, not an I/O-availability one. Ungated in
`ks_tick` (pulses computed from clock_out_enable + output enable + active); the
USB *send* is now gated in the glue instead. Test flipped
(`test_clock_not_gated_by_usb_ready`).

**With the fix, clock streams from Link tempo, no USB device (125 BPM, 24 PPQN):**
- 252x 0xF8 over 5.02 s, all decoded clean.
- Tempo accuracy: mean inter-0xF8 = **20.004 ms** vs 20.000 ms nominal (0.02%).
- Jitter: RMS **0.136 ms**, peak-to-peak **1.95 ms** (range 19.03-20.97 ms). The
  ~+/-1 ms extremes are the +/-1 FreeRTOS-tick (1 ms) quantization -- the measured
  floor behind the `CONFIG_FREERTOS_HZ=1000` choice, never quantified until now.
  RMS is far tighter than the tick. A hardware-timed output (RMT, the analog board)
  would beat this if sub-ms MIDI sync is ever needed.
- **Clock-to-downbeat alignment: +0.018 ms (18 us).** The 0xF8 at the bar sits 18us
  after the downbeat strobe -- same tick, code-ordering delay. The MIDI clock is
  effectively coincident with the Link bar.

The 0xFA launch-offset (pure ESP-011 web path) stays unmeasured: it needs
`s_stst_seen` false, which Ableton's Start/Stop broadcast keeps latching. Not
worth chasing -- the 18us clock alignment already answers "is the MIDI on the
grid?".
