## Problem

`/status` (`ks_status.c`) is telemetry only — bpm, peers, launch state — and never
carried the device's actual settings. The only place a client could read "is
output 2 enabled, what's its division" was by scraping the rendered HTML of
`GET /` (`ks_web.cpp`'s `root_handler`), which ties a client to the web UI's
markup instead of a contract.

This surfaced while starting Phase 1 of the iOS companion app
(`docs/plans/2026-07-14-ios-kitchensync-app-plan.md`, `kitchensync-ios_app`
repo): the app needs to show a device's real current config, not just telemetry,
and an HTML scraper is exactly the kind of brittle dependency this repo's own
web UI avoids internally (`ui_chrome.c`'s pure/tested pages).

## Design

New pure module `ks_config_json.{h,c}`, same shape and conventions as
`ks_status.c`: a single `ks_config_json(buf, len, const KsConfig*)` snprintf-style
encoder, no ESP-IDF dependency, host-tested (`test/test_ks_config_json.c`).

Field names match the `POST /save` / `POST /live` form grammar (`ks_config_set()`
in `ks_config.c`) 1:1, so a client reads and writes through one name per field.

**Security:** WiFi passwords are never included — the same rule `ks_web.cpp`'s
`build_wifi()` already applies to the HTML form (a saved password is never
echoed back, only a "keep current" placeholder). The JSON exposes `pass_set`
(bool) per WiFi slot instead of the password itself.

Wired as `GET /config.json` in `ks_web.cpp`, mirroring `status_handler`'s
mutex discipline: snapshot the whole `KsConfig` under `s_cfg_mutex` (so the
clock task never observes a torn read across a concurrent `/live` patch), then
format the JSON outside the lock — the same principle
`ks_web_config_persist_tick()` already uses for its own NVS-write snapshot.

## Acceptance criteria

- [x] `ks_config_json()` — pure, host-tested (`test/test_ks_config_json.c`, 7
      tests): every field round-trips, the `clock`/`wifi` arrays are in
      slot/output order, a saved password never appears in the output,
      `pass_set` reflects whether a slot has one, the return value follows
      `ks_status_json()`'s snprintf-style truncation convention, and a
      worst-case-valued config (692 bytes) fits the handler's 768-byte buffer
      with headroom.
- [x] `GET /config.json` route wired in `ks_web.cpp`, registered alongside
      `/status`.
- [x] Full host suite green (`make -C test all`, zero failures across every
      existing module — this is purely additive, nothing else changed).
- [ ] Hardware-validated: `curl http://<device-ip>/config.json` against a real
      KitchenSync unit. Not done in this session — no hardware attached.

## Explicitly out of scope

- Changing `GET /` or `/status` — both are unchanged; this is a new, additive
  route only.
- A write-side change to accept this JSON shape as a POST body — `/save`/`/live`
  keep their existing `x-www-form-urlencoded` grammar. This ticket is read-only.
