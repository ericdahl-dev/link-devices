## Goal

Replace the three loose `volatile` tempo globals — `g_current_bpm`,
`g_current_phase`, `g_phase_valid` — with a single **deep `tempo_snapshot`
seam** that publishes and reads an atomic `{bpm, phase, valid, quantum}`
snapshot. One writer (the `bpm_task`) publishes; every reader (web `/status`,
touch UI, LED) takes one consistent snapshot.

## Problem (from the architecture review)

Live tempo is three separate volatiles with **inconsistent, partly-missing
locking**:
- `X32Link.ino` protects `g_current_bpm` with `g_bpm_mutex` but writes
  `g_current_phase` / `g_phase_valid` with **no lock** (`X32Link.ino:160-172`).
- `X32MidiClock.ino` protects **nothing** (`:32-53`).
- `web_config.cpp handle_status()` reads all three with **no lock** (`:248-252`).

A read can tear across a write and emit a contradictory
`{"valid":true,"phase":-1.0}`. There is no module that owns "the current tempo
reading" — the consistency rule lives nowhere, so it can't be tested.

## Acceptance Criteria

- [ ] New pure-ish module `tempo_snapshot.{h,c}` (host-testable core) exposing:
      `tempo_snapshot_publish(bpm, phase, valid, quantum)` and
      `tempo_snapshot_read(TempoSnapshot* out)`, with an internal lock/seqlock so
      a reader never observes a torn `{valid,phase}` pair.
- [ ] Invariant enforced + host-tested: a read never returns
      `valid == true && phase < 0` (and `valid == false` ⇒ phase is the
      no-signal sentinel). Test in `esp32/test/test_tempo_snapshot.c`, wired into
      the Makefile `all` target.
- [ ] `X32Link.ino` and `X32MidiClock.ino` `bpm_task`s publish via
      `tempo_snapshot_publish()` instead of writing the three globals.
- [ ] `web_config.cpp` `handle_status()` and the touch status screen
      (`touch_display.cpp`) read via `tempo_snapshot_read()` — one snapshot, no
      per-field global reads.
- [ ] The three `g_current_bpm/phase/valid` externs are removed (or reduced to
      the snapshot's storage) — no reader touches them directly.
- [ ] Both firmwares compile; full host suite green; `/status` never emits a
      contradictory snapshot under load.

## Notes

- The ESP32 side still needs a real lock (FreeRTOS mutex or a portMUX
  critical section / seqlock); keep that glue thin and put the *invariant* in
  the host-testable core so the contradiction is impossible by construction.
- `tempo_snapshot.c` must stay Arduino-free (host-testable). The lock primitive
  can be injected or `#ifdef`-guarded, mirroring the pure/glue split used
  elsewhere (see `esp32/AGENTS.md`).
- Related: ARC-002 (measurement validity leak) folds in naturally — the
  publisher should derive `valid` from a committed xform, not `active`.
