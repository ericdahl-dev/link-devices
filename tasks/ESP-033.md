## Why

**Getting a box onto WiFi currently requires leaving your own WiFi.**

Today's flow (`wifi_link.c`, `start_ap()`): the device fails to associate → falls back to a SoftAP
called `KitchenSync-Setup` at 192.168.4.1 → **the user disconnects from their own network**, joins
the device's AP, fills in a web form, and rejoins. Per box.

At home that is merely annoying. At a venue, onto house WiFi, with three units on the bench and a
soundcheck running, it is miserable — and it is *precisely* the moment you cannot afford to drop off
the network you are working on.

BLE provisioning removes the whole detour: the app connects over BLE, hands the device credentials,
the device joins WiFi. The phone never leaves its network.

This is a well-trodden path, not a research project. ESP-IDF ships the `wifi_provisioning`
component with a BLE transport, and Espressif publishes reference apps for exactly this.

## Scope: BLE is for PROVISIONING ONLY

Record the boundary, so nobody relitigates it later:

| Use | Verdict | Why |
|---|---|---|
| **WiFi provisioning** | ✅ **Yes** | Removes the leave-your-network detour. Real pain, standard solution. |
| **Clock / MIDI transport** | ❌ **Never** | BLE MIDI jitter runs into the tens of milliseconds. This product exists to be *tight*: ESP-027 was a crisis because the DIN clock went silent for 138 s, and ESP-028 exists to prove pulses are actually on the wire. A clock over BLE is actively hostile to the thing the product is for. |
| **Discovery fallback** | ⚠️ **Probably not** | Tempting — the iOS app's empty state literally says *"Some guest networks block discovery."* But BLE only helps you FIND a box; you still need IP connectivity to do anything with it, and a network that blocks mDNS usually blocks client-to-client traffic too. You would find a device you still cannot talk to. Revisit only if mDNS blocking proves to be a real field problem. |

## ⚠️ Feasibility spike FIRST — the P4 has no native radio

On the **S3** (Touch), BLE is native and this is routine.

On the **ESP32-P4** (`KitchenSync`, the primary product) there is **no on-chip WiFi or BLE**. Both go
through the onboard **ESP32-C6 over `esp_hosted`** — the same arrangement `wifi_link.c` already
works around:

> *"On the P4 the WiFi MAC belongs to the onboard C6, not to this chip's efuse."*

BLE-over-hosted is supported, but it is more moving parts than the S3's native stack (HCI over the
host interface, and it shares the C6 with the WiFi stack that Link depends on). **Spike this before
promising it**, especially since ESP-032 points the whole product line at the P4.

Do not let a BLE stack destabilise the WiFi path. Link multicast is the product's lifeblood; a
provisioning convenience must never put it at risk.

## Approach

1. **Spike:** BLE provisioning on the P4 via `esp_hosted`/C6. Does it coexist with the Link WiFi
   stack without jitter or dropouts? If it does not, this task stops here for the P4 and stays an
   S3-only convenience.
2. Add ESP-IDF's `wifi_provisioning` with the BLE transport (`scheme_ble`). Keep the SoftAP path as
   a fallback — a phone with BLE off, or a laptop, still needs a way in. **Do not delete
   `KitchenSync-Setup`.**
3. Security: use a proof-of-possession secret. An open BLE endpoint that accepts WiFi credentials is
   an obvious attack surface at a venue full of strangers.
4. App side: a CoreBluetooth provisioning flow in KitchenSync-iOS_App (filed separately).
   Espressif ships an iOS provisioning library that speaks this protocol.

## Acceptance criteria

- [ ] Spike answers whether BLE coexists cleanly with the Link WiFi stack on the P4 — measured, not
      assumed. Clock jitter and Link peer stability checked with the analyzer, exactly as ESP-028 did.
- [ ] A device advertises for provisioning over BLE when it has no usable WiFi credentials
- [ ] The iOS app can hand it credentials **without the phone leaving its own network**
- [ ] Proof-of-possession is required — an unauthenticated box must not accept credentials
- [ ] The SoftAP fallback still works (do not strip it)
- [ ] BLE is OFF, or at least not advertising, once the device is provisioned and on WiFi — no
      permanently-open radio on a stage
- [ ] Clock output is unaffected: no added jitter, no dropped pulses, verified on the bench rig

## Notes

**BLE never carries clock or MIDI.** If a future ticket proposes it, this is the ticket that says no,
and why.

The Touch (S3) is the cheap place to prove the flow — native BLE, no hosted stack. But note ESP-032:
if Touch collapses into a P4 board, the S3-only version is throwaway. Spike the P4 first.
