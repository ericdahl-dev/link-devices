## Why

**Touch exists on Arduino/ESP32-S3 for a hardware reason that may no longer hold.** Waveshare now
ships ESP32-**P4** boards with a touch LCD — the same chip as KitchenSync, our primary product,
which is already ESP-IDF.

Candidates (checked 2026-07-14):

| | [P4 Touch LCD 4.3](https://www.waveshare.com/esp32-p4-wifi6-touch-lcd-4.3.htm) | [P4 Touch LCD 4B](https://www.waveshare.com/esp32-p4-wifi6-touch-lcd-4b.htm) |
|---|---|---|
| MCU | ESP32-P4 (RISC-V dual-core) | ESP32-P4 |
| Wi-Fi | **onboard ESP32-C6-MINI co-processor** | same |
| Memory | 32 MB PSRAM, 32 MB NOR flash | 32 MB PSRAM, 32 MB flash |
| Display | 4.3" 480×800 IPS, MIPI-DSI 2-lane | 4" 720×720 IPS |
| Other | USB OTG 2.0 HS, 40-pin GPIO, TF slot | **Ethernet** (on the -ETH-2RO variant), TF, MIPI-CSI |

**The P4-plus-onboard-C6 arrangement is the exact quirk KitchenSync's firmware already handles.**
`wifi_link.c` says so in as many words:

> *"On the P4 the WiFi MAC belongs to the onboard C6, not to this chip's efuse, so
> `esp_read_mac(ESP_MAC_WIFI_STA)` fails and every unit would name itself kitchensync-0000."*

We have already paid that tax. So a P4 touch product is not a new firmware — it is **KitchenSync
plus a display layer.**

## What this buys

**One codebase.** Today `KitchenSyncTouch/` symlinks **46 modules** from `X32Link/` — the clock
engine is already shared. What is NOT shared is the framework wrapper and the web layer, and that
is precisely where the drift lives (see ESP-029/ESP-030: `wbeats` vs `w_beats`, no `/config.json`,
no `launch[]`). Collapsing the products removes the seam that drift grows in.

**It also ends the duplicated-bug tax.** ESP-028: *"P4-039 did exactly this for the P4. The Touch
never got it."* The same `/status`-lies bug was fixed twice, and the Touch's copy went unfixed long
enough to cost 138 seconds of silent DIN clock.

**32 MB flash** (vs the S3's 16 MB) gives dual-OTA slots real headroom. The **Ethernet** variant is
genuinely interesting for stage use — a wired clock box does not care about venue Wi-Fi.

## What it costs — do not underestimate this

- **MIPI-DSI display bring-up** on the P4, a capacitive touch driver, and an LVGL UI on a 480×800
  panel. Real work. The S3's display stack (`axs5106l`, `ktouch_display`) does not port for free.
- **The S3 Touch ships today.** This is a direction, not a this-week decision, and nothing here
  justifies destabilising a working product.
- X32Link stays Arduino regardless — different product, different chip. This is not a
  "kill Arduino everywhere" task.

## Approach (if pursued)

1. Buy one board. Bring up display + touch under ESP-IDF, standalone, before touching product code.
2. Add a `display/` layer to `KitchenSync/` behind a compile-time flag, the way `config.h` already
   selects a board in Touch. The clock engine, web layer, and protocol come along for free — they
   are already the shared modules.
3. Retire `KitchenSyncTouch/` when the P4 variant reaches parity. Its pure modules already live in
   `X32Link/`; only the Arduino wrapper dies.

## Acceptance criteria

- [ ] A decision is recorded either way — pursue, or explicitly park with a reason
- [ ] If pursued: one P4 touch board boots KitchenSync firmware with a working display
- [ ] The clock engine, `/status`, `/config.json` and transport come from the SHARED modules, with
      no Touch-specific fork
- [ ] `KitchenSyncTouch/` has a retirement path, not an indefinite parallel life

## Notes

**This changes what ESP-029/ESP-030 should do**, and they have been written accordingly: *share*
the pure modules rather than reimplement the contract inside `ktouch_web.cpp`. The pure C survives
this migration; the framework wrapper does not. Do not invest in the Arduino web stack.

The iOS app (KitchenSync-iOS_App) is the forcing function that surfaced all of this. It cannot see
a Touch (its mDNS name is `kstouch-dfd0`, not `kitchensync-*` — ESP-031), cannot decode its
`/status`, and cannot read its config. A client is where "two products, two contracts" stops being
an internal tidiness question and becomes a user-visible defect.
