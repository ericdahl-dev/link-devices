## Goal

Hide the `active` / `xform.valid` distinction behind the measurement seam so
callers can't gate phase validity on the wrong flag. Expose one honest
question — `link_measurement_have_phase_estimate()` (a committed, non-stale
xform exists) — and make `active` an internal detail.

## Problem (from the architecture review)

`link_measurement.h:140-142` exposes two flags: `link_measurement_active()`
(an attempt is in flight) and `current_xform().valid` (a result is committed),
and makes the caller reconcile them. `tempo_source_phase_valid()` reconciled
them **wrong** — it gated on `active()` — which was the "flashing dot" Link bug
(fixed reactively in the Link-phase PR). The header documents the distinction
(`:127-137`) that the caller then ignored. Correctness depends on caller
discipline, not the interface — a leaky seam.

## Acceptance Criteria

- [ ] `link_measurement` exposes `bool link_measurement_have_phase_estimate(void)`
      meaning "a committed xform exists (and is fresh enough to trust)".
- [ ] `link_measurement_active()` is removed from the public header (or clearly
      marked internal); no caller outside the module reads it.
- [ ] `tempo_source_phase_valid()` (Link branch) calls
      `link_measurement_have_phase_estimate() && link_proto_timeline(&t)` —
      structurally unable to reproduce the `active`-gated bug.
- [ ] Optional staleness rule (xform older than N re-measure intervals ⇒ not
      an estimate) captured in `link_measurement.c` and host-tested.
- [ ] Host suite green; Link phase still holds valid on hardware.
