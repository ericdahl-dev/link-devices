## Why

ESP-024 needs a rig to measure on, and the product boards are awkward to probe. The rig is
**KitchenSyncTouch on a classic ESP32 DevKit** (WROOM-32, screw-terminal breakout): the same
clock engine, the same 1 ms writer, the same web UI. Only the transport *surface* differs —
two buttons instead of a touch LCD.

Both surfaces post the same `TransportLaunchIntent` into the same `ktouch_transport` mailbox,
so the writer neither knows nor cares which is fitted. **What the bench measures is what the
product runs.**

## What shipped

- `BOARD_ESP32_DEVKIT` profile for KitchenSyncTouch (no display, no USB peripheral, 4 MB flash)
- `buttons.{h,c}` — pure debounce, host-tested. A button held at boot cannot fire on power-up
- `transport_led.{h,c}` — lamp state: dark stopped, **blinking armed**, solid running
- `TL_INTENT_REALIGN` / `TL_RESTART` — re-anchor a running slave to the bar line (`0xFC`+`0xFA`)
- `tools/build-bench.sh` — the ONLY supported way to build it
- `POST /transport` (ESP-027) — headless transport, so timing runs need no human finger
- CI now compiles the rig (it previously had **zero** coverage)

## What it found

Standing it up surfaced four defects, all of which existed because the clock writer is written
twice (see ADR-0009 / ESP-026):

- **ESP-023** — transport emitted after clock. The fix existed and had never merged, in *either*
  firmware. Now analyzer-verified: `0xFA` precedes the downbeat `0xF8` by exactly 320.0 us.
- **`ks_tick` ternary** — every non-START action silently became STOP.
- **ESP-027** — the DIN clock went silent for 138 s while `/status` reported `sync:1`.
- **`build-bench.sh` was macOS-only** (`sed -i ''` is BSD syntax) — caught the moment CI first
  compiled the rig.

## Not done

**The physical button never worked.** Pin, screw terminal, internal pull-up and debounce are
all *proven good* — bridging GPIO32 straight to GND reads 0, the button never does. `/status`
carries `btn` / `btnlows` / `btnpress` so a replacement can be diagnosed without catching the
press inside a serial window. The fault is the switch or its two wires; if it has three legs,
use **COM + NO** (NO+NC never closes, which is exactly the symptom).

The rig does not need it: `POST /transport` drives the whole path, and a scripted trigger is
better than a finger for ESP-024 anyway.
