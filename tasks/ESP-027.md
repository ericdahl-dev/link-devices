## What was happening

Kill the only Link peer and let it rejoin, and **the DIN clock stops dead and never comes
back.** Not a stutter — permanent silence, only recoverable by rebooting.

The whole time, `/status` reports:

```json
{"bpm":120.0, "sync":1, "peers":1, "clock":1, "drop":0, "burst":0, "over":0}
```

Every counter green. **Zero bytes on the wire.** Found on the analyzer, not by ear —
nothing in any log or counter said a word.

For a clock box this is the worst failure that exists. Close the laptop, WiFi hiccups,
Ableton restarts — the drum machine stops forever and the box insists it is fine.

## Root cause — measured, not guessed

Instrumenting `beats` in `/status` across a kill/rejoin caught it:

```
SOLO     peers=0  locked=0  beats=  0.60      free-run, restarted at 0
REJOIN   peers=1  locked=1  beats=290.46      Link timeline
REJOIN   peers=1  locked=1  beats= 13.84   <-- STEPPED BACKWARDS 276 BEATS
REJOIN   peers=1  locked=1  beats= 18.93      climbing normally from there
```

A fresh GhostXForm commits and re-maps "now" to a different beat position. **`locked` never
changes** — it is `true` before and after — so `beat_source`, whose only re-prime rule was
`locked != was_locked` (a *basis switch*), fired **nothing**.

`clock_ticker` had primed its grid at beat 290. `beats` dropped to 14. The ticker then sat
waiting for beats to climb back to 290 — **276 beats ≈ 138 seconds** — emitting nothing.
And because it was never *dropping* pulses, just never finding any due, `drop` and `burst`
stayed at 0. The probe was blind by construction.

The same mechanism explains the smaller intermittent dropout seen earlier the same night:
a ~0.9-beat backstep → 19 pulses lost (and `drop:19` matched the analyzer exactly).

**Time does not run backwards. A backwards beat step IS a re-origin.**

## Three defects, three fixes

**1. `beat_source` re-primes on a backwards beat step** (the core fix, and it lands on the
P4 too — same shared module, same latent bug).

```c
if (s->have_last && beats < s->last_beats - BEAT_SOURCE_BACKSTEP_BEATS) reprime = true;
```

`BEAT_SOURCE_BACKSTEP_BEATS = 0.25` (6 pulses at 24 PPQN): sub-millisecond xform
refinements are absorbed, a real re-origin re-primes. Host-tested three ways — backstep
re-primes, tiny backstep does *not*, forward motion never does.

**2. The Touch now derives beats through `beat_source` + `master_clock`, like the P4.**
It was calling `tempo_source_beats_now()`, which returns `-1` the instant the xform is
invalid — and `clock_output`'s reset-on-invalid rule then emits nothing. So *any* loss of
the xform silenced the wire. `beat_source` free-runs through it; `master_clock_arbiter`
(P4-040) originates a clock when solo, seeded from whatever Link last showed.

`master_clock.{h,c}` moved out of `KitchenSync/main/` into the shared home. Its header said
*"KitchenSync (P4) only"* — written when the Touch was not yet the clock box. It is now the
thing driving the RC-505, so it originates too.

**3. `POST /transport`** — the Touch could only be driven from its LCD, so a headless unit
had no way to start the clock, and the analyzer runs needed a human finger. Now scriptable.

## Verified on hardware (Saleae, DIN TX @ 31250 8N1)

| | before | after |
|---|---|---|
| kill the only Link peer | clock dies within one beat | **runs continuously** |
| peer rejoins | 138 s of silence, never recovers | **runs continuously** |
| 40 s capture across kill+rejoin | — | **1925 clock bytes** (continuous = ~1920) |
| free-run jitter | — | **372.8 us stdev** — as tight as locked |
| `drop` / `burst` / `over` | 19 / 1 / 0 | **0 / 0 / 0** |

Residual: one lost pulse at the peer-timeout edge and ~half a pulse at rejoin — the grid
legitimately realigning to a new origin, not a dropout.

## The pattern (see ADR-0009 / ESP-026)

This is the **fourth** defect in one night that existed only because the clock writer is
implemented twice:

- ESP-023 — transport emitted after clock. Fixed on the P4, left broken in the Touch.
- `ks_tick` ternary — every non-START action silently became STOP.
- `beat_source` — the P4 had it, the Touch never got it.
- `master_clock` — the P4 originates a clock alone (P4-040), the Touch went silent.

**ESP-026 converges them.** Every one of these is a bug that got found once and fixed once.

## Follow-ups (not fixed here)

- [ ] **`/status` lies.** `sync:1, clock:1` with a dead wire. `sync` is
      `tempo_source_phase_valid()`, which says nothing about whether pulses are reaching the
      jack. It should report what the *writer* is actually doing. P4-039 did this for the P4.
- [ ] **The GhostXForm goes stale across a peer churn** rather than being invalidated and
      re-measured. The clock now survives it, but the underlying measurement-session
      recovery is still worth a look.
- [ ] **Smooth the re-origin realign** so the 1-pulse blip at the transition goes away.
