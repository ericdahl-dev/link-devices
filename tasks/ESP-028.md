## Why

Two defects fell out of ESP-027 and were deliberately left open. The clock now *survives* both,
so neither will stop the box mid-set — but both are still wrong, and the first one is what made
ESP-027 take a logic analyzer to find instead of a glance at `/status`.

## 1. `/status` lies about the clock

```c
int sync = tempo_source_phase_valid() ? 1 : (tempo_source_active() ? 0 : -1);
```

`sync` reports whether a **phase estimate exists**. It says *nothing* about whether pulses are
reaching the jack. During the ESP-027 failure the device reported:

```json
{"bpm":120.0, "sync":1, "peers":1, "clock":1, "drop":0, "burst":0, "over":0}
```

...with **zero bytes on the wire, for 138 seconds.** Every counter green. That is not a missing
feature, it is a *misleading* one: it actively told us the thing was healthy.

**It should report what the WRITER is doing.** The writer already knows — it has `bs.locked`,
`bs.active`, and the pulse count. P4-039 did exactly this for the P4 ("stops lying when solo").
The Touch never got it.

Suggested shape (name it honestly — it is not "sync"):

| value | meaning |
|---|---|
| `locked` | phase-locked to a Link session, pulses flowing |
| `free` | free-running (no xform, or solo via `master_clock`), pulses flowing |
| `silent` | **no pulses are being emitted** — the state that must never be invisible again |

A pulse-rate or a monotonic pulse counter in `/status` would make "is the wire alive?" answerable
without an analyzer. Cheap, and it is the single highest-value diagnostic this product lacks.

## 2. The GhostXForm goes stale across a peer churn

Kill the only Link peer and let it rejoin, and the committed xform is left **marked valid while
mapping to the old session's ghost time**. `link_measurement_have_phase_estimate()` returns
`s_xform.valid`, which stays true — so `/status` says `sync:1` — but `link_phase_beats_now()`
then produces a beat position from a stale mapping. That is what stepped `beats` backwards 276
beats and (pre-ESP-027) silenced the wire for 138 seconds.

ESP-027 made the clock *survive* it (`beat_source` re-primes on a backwards step). It did not
make the measurement layer *correct*.

`link_measurement_session.c` (LNK-031) targets a reference peer by `{ip, port}` and re-measures
every 2 s. When the peer disappears and returns on a **new ephemeral port**, it appears not to
re-target or re-measure — the estimate is neither invalidated nor refreshed. Worth confirming
with the `reprimes` counter (P4-038, landed in #105), which counts exactly this episode.

## Approach

1. **Report the writer's truth in `/status`** first — it is small, and it makes everything after
   it observable. Do not skip it to get to the "real" bug; the reason ESP-027 needed an analyzer
   is that this was missing.
2. Reproduce the stale xform with the bench rig (kill/restart `linkcli`; the analyzer and
   `POST /transport` are both scriptable) and watch `reprimes` + `beats`.
3. Fix the session policy: on peer-endpoint change, invalidate and re-measure rather than serving
   a stale mapping. Pure module, host-tested (`test_link_measurement_session.c`).

## Acceptance criteria

- [ ] `/status` reports whether **pulses are actually being emitted**, not whether a phase
      estimate exists. A silent wire must be visible in `/status`.
- [ ] The ESP-027 scenario (kill peer, rejoin) is reproduced and the stale-xform window is
      measured, not assumed
- [ ] Peer-endpoint change invalidates + re-measures instead of serving a stale mapping
- [ ] Host tests cover the re-target path
- [ ] Verified on the bench rig with the analyzer: clock continuous, and `/status` tells the
      truth throughout


## Resolution (2026-07-13) — hardware-verified

### Part 1: /status now reports the WRITER's truth

The writer is the only thing that knows whether bytes left the UART, so it is now the source.

| field | meaning |
|---|---|
| `clk` | `locked` / `free` / **`silent`** |
| `pulses` | lifetime 0xF8 count — "is the wire alive?" answerable by polling, no analyzer |
| `sync` | derived from `clk`, so it cannot claim a lock over a dead jack |
| `bpm` | from the arbiter's timeline, which **survives peer loss** |

`bpm` was the same lie one field over: it came from `link_proto_bpm()`, which zeroes when peers
drop — so a device free-running perfectly at 120 BPM reported **0.0 BPM while emitting a 120 BPM
clock**. Fixed the way P4-039 fixed it for the P4.

The web UI pills described *intent*, not reality: "Sync" showed whether a phase estimate existed
and "MIDI Clock" showed a **config flag**. Both read healthy through 138 s of a dead wire. They
now show Locked / Free Run / **SILENT** — and SILENT is not green.

### Part 2: the stale GhostXForm — and a second bug nobody had seen

The ticket's suspicion was **wrong**: the peer *does* re-target. The defect was that **nothing
invalidated the committed xform across the churn**, so a dead session's origin was served as a
valid phase estimate — `link_measurement_have_phase_estimate()` is just `return s_xform.valid`,
and nobody cleared it.

`link_session_on_trigger` now emits `LS_RESET_XFORM` when the reference endpoint **disappears**
or **moves** (a restarted peer returns on a new ephemeral port), before the fresh attempt.

**The second bug**, found while tracing: the P4-038 **slew clamp** (20 ms/commit) runs against
the *previous* xform. When that origin belongs to a session that no longer exists, the clamp
fights the new one — an epoch delta under 1 s would crawl the bar line for ~50 s with every
counter green. Our 138 s case escaped through the `>= 1 s` re-origin hatch and stepped whole,
which is why we saw a dramatic backstep rather than a slow drift. Clearing `valid` first closes
it, because the first commit then adopts the new origin whole.

**And one interaction worth naming:** `LS_RESET_XFORM` used to route through
`link_measurement_reset()`, which also **blanks the P4-038 phase-health gauge** (`commits`,
`max_step_us`, `rtt_min/max`) — documented as *lifetime*. A churn fires this, so the lifetime
record of how badly commits throw the origin would have been wiped on every peer bounce, i.e.
exactly when it is worth reading. Split into `link_measurement_invalidate_xform()`: drop the
mapping, keep the gauge.

### Verified (bench rig, Saleae on the DIN wire, 45 s across kill + rejoin)

```
LINKED    clk=locked  bpm=120.0  peers=1  beats=  27.6
SOLO      clk=free    bpm=120.0  peers=0  beats=   2.4
REJOIN    clk=locked  bpm=120.0  peers=1  beats=  12.1

clock bytes 2164 / 2160 expected   coverage 100.2%   jitter 373.6 us stdev
```

Continuous clock throughout. `beats` no longer steps backwards 276. 578 host tests, 0 failures.

## Known holes (not closed here)

- **A peer that restarts on the SAME port is still invisible** to the measurement module: new
  ghost epoch, unchanged endpoint. Covered — if at all — by `link_proto_epoch_reset_pending()` /
  the ARC-011 settled-timeline debounce, not here. Ephemeral ports make it unlikely, not
  impossible.
- **Multi-peer re-target costs a spurious invalidation.** `link_protocol.c` swap-removes peers,
  so `peer[0]` reshuffles when any peer leaves, and the pump targets "first peer with an
  endpoint". With 3+ peers, one leaving can move the reference to a *different peer in the same
  session* — identical ghost epoch — and we invalidate anyway, because `{ip,port}` is the only
  session identity this module can see. Cost: one re-measure landing within milliseconds of the
  old intercept (far under `beat_source`'s 0.25-beat re-prime threshold), with a free-run gap
  that ESP-027 made seamless. Serving stale cost 138 seconds. Trade taken deliberately.
  **To tighten it, expose the peer's 8-byte node id** (already in `s_peers[].id`) and invalidate
  only on a real session change.
