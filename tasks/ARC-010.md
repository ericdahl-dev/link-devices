## Problem (architecture review candidate E)

Two shallow spots surfaced by the deletion test:
- `clock_output` is a **one-line pass-through** over `clock_ticker` — its whole
  body is `clock_ticker_ticks_due(t, beats + phase_mbeats/1000.0, ...)`. Interface
  ≈ implementation. Deleting it moves ~zero complexity. **BUT** it's the intended
  home for swing (P4-013); folding it now and re-extracting later is churn.
- `wifi_link`'s wrappers are inconsistent glue — some functions add behavior, some
  are bare forwards to the ESP-Hosted API, so the seam doesn't pay for itself and a
  reader can't tell which calls matter.

## Solution

Decide `clock_output` **after P4-013**: if swing lands there, it becomes deep and
stays; if swing goes elsewhere, fold it into `clock_ticker`. For `wifi_link`, keep
only the wrappers that add behavior (retry, state, logging) and inline the bare
forwards so the seam means something.

## Deliverable

Either a recorded decision to keep `clock_output` (swing landed) or its fold into
`clock_ticker` with tests merged; `wifi_link` reduced to behavior-bearing wrappers.
Host suite + both builds green.

## Notes

- **Lowest priority / most speculative of the ARC set.** Do last, and only if
  P4-013's outcome makes the `clock_output` call clear. Don't pre-fold.
