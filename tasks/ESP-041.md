## Why

`/status` is now built by ONE shared module (`ks_status_json`, `X32Link/ks_status.c`) — ESP-029/035
fixed the field-name divergence (2 shared keys → 13). But the module is **shallow**: its interface is
nearly as complex as its implementation.

- `ks_status_json()` has **21 positional parameters**; the wire shape each device emits is decided by
  which args it passes non-NULL.
- Per-device keys ride in a free-text `extra` fragment appended *inside* the object
  (`ks_status.c`), **outside the contract and unvalidated**: X32Link's `phase/valid/quantum/batt_*`,
  the Touch's 14-key diagnostic block (`sync/clock/transport/beats/locked/…`).

So callers cross a wide interface (count 21 args, hand-format `extra`) for a thin body. This is a
shallow module — the deletion test barely moves.

## What to build

Replace the 21-arg call with a `StatusView` struct the caller fills; the serializer owns key names
and which optional blocks are present. Per-device keys become **declared optional fields**, not a
free-text tail.

```c
typedef struct {
    float bpm; int peers; bool playing; /* … shared core … */
    const StatusPhase* phase;   /* NULL = omit */
    const StatusTick*  tick;    /* NULL = omit */
    const StatusExtra* extra;   /* typed, per-device — not a raw string */
} StatusView;
int ks_status_json(const StatusView* v, char* out, int cap);
```

## Acceptance criteria

- [ ] `ks_status_json` takes a struct; the 21-arg signature is gone.
- [ ] Per-device keys are declared/typed, not an unvalidated `extra` string.
- [ ] All three firmwares emit through the struct; host tests assert each device's shape.
- [ ] CI green.

## Notes

**Weakest of the deepening candidates** (priority low). The union-by-NULL-args works today and the
key *names* are already shared. Only worth doing if `/status` keeps accreting arguments — otherwise
it is churn. Revisit before starting.
