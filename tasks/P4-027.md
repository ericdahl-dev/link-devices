## Problem

KitchenSync only started its `KitchenSync-Setup` SoftAP on a true first boot (empty
stored SSID). If it had credentials for a network it could no longer reach — moved
from home to a gig, guest WiFi changed — it retried `esp_wifi_connect()` forever
with no way back to a config screen. Dead in the water until a reflash. This is
exactly the provisioning gap the GTM design doc flagged (the device must be
recoverable on any stage).

## Fix

Parity with X32Link's `wifi_try_connect()` / `start_config_ap()`: give STA a fixed
30 s budget to associate; once it's blown, stop STA and fall back to the SoftAP so
the user can reconfigure.

- Pure `wifi_fallback_should_give_up(now_us, connect_start_us)` (host-tested,
  `test/test_wifi_fallback.c`, 5 cases incl. the inclusive 30 s boundary) — kept
  out of the hardware-coupled `wifi_link.c`.
- `wifi_link.c`: on `STA_DISCONNECTED`, check the budget → `give_up_and_start_ap()`
  (disconnect + stop STA + bring up the AP) instead of retrying forever. WiFi
  init + event registration hoisted into `wifi_link_start()` so the AP fallback
  path doesn't re-init the stack.
- Also fixed a latent rename bug: the AP `ssid_len` was still `11` after
  "P4Hub-Setup" → "KitchenSync-Setup" (17 chars), so the AP broadcast truncated;
  now `strlen(AP_SSID)`.

## Origin

Re-cut of PR #34 (`fix/p4hub-wifi-ap-fallback`) onto the post-rename tree. The
original was on the old `P4Hub/` paths and reused ticket numbers P4-018/P4-019,
which are now the LED work — hence this fresh P4-027. Verified on real P4-NANO
hardware in the original: gave up at ~35 s, brought up the setup AP with DHCP.
