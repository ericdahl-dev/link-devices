## Source

Design: `docs/plans/2026-07-19-rgb-led-output-subsystem-design.md` (slice 5 of 5). "Effect"
is the third style, defined-but-unbuilt; this lands only the seam so future Effect work
(WLED-style synced effects, sound-reactive) is additive, not a re-architecture.

## What to build

The Effect style's structural seam — no actual effects, no audio, no WLED board.

- Add `EFFECT` to the style enum and the third case to the dispatcher's `switch`.
- A pure effect-function-table shape: `effect_render` dispatches to a registered pure
  effect fn `(anim_clock, params, npix) -> fill out[])`, with an optional beat slice from
  the snapshot for beat-synced effects. Zero or one trivial placeholder effect (e.g. a
  solid-color fill) is enough to exercise the seam and its host test.
- `ks_config_valid` **rejects** selecting `EFFECT` on a build/board that does not implement
  a real effect (so the enum value is inert until a device brings it up).
- Effect-specific config fields (`effect_id, speed, intensity, palette`) are NOT added yet
  — they land with the first real effect device, keeping today's NVS/migration clean.

## Acceptance criteria

- [ ] `EFFECT` enum value + third dispatcher case exist; a placeholder effect renders through the effect-function-table seam in a host test.
- [ ] Adding a new effect is a drop-in pure fn + a host test, with no dispatcher edit.
- [ ] `ks_config_valid` rejects `EFFECT` where unimplemented; no speculative Effect config fields in NVS.
- [ ] No behavior change to Status or Metronome; host suites green.
