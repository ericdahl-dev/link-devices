## Source

Design: `docs/plans/2026-07-19-rgb-led-output-subsystem-design.md` (slice 3 of 5). This is
the original trigger — the Super Mini's onboard RGB has no runtime control today because
`KSTOUCH_HAS_LED` (external strip) is off and the ESP-043 status RGB is fixed.

## What to build

Bring the headless Super Mini's onboard WS2812 (GPIO48) into the per-output subsystem as a
**Status** output, replacing the ad-hoc ESP-043 wiring, with runtime dim + enable.

- Declare the Super Mini's descriptor `led_out[] = {{48, 1, ONBOARD_SINGLE, false}}`
  (per-board flag; GPIO48 is the panel on the Waveshare, so that board stays zero-output).
- `status_render(snap, npix, out[])` wrapping the pure `ktouch_status_rgb` to fill all
  `npix` (npix=1 here) with the one status color; brightness applied by the dispatcher.
- Arduino glue: the `ONBOARD_SINGLE` backend = `rgbLedWrite` (asserts npix==1). The status
  render runs on its own timer/task (not `loop()+delay(5)`) so its beat-pulse tempo tier
  stays tight.
- `led[0] = { enable, style=STATUS, bright }` surfaced in `/config.json` + the iOS app
  section (Status shows only enable + brightness — fixed palette).

## Acceptance criteria

- [ ] Super Mini onboard RGB driven through the subsystem's dispatcher + `ONBOARD_SINGLE` glue (ESP-043 ad-hoc path removed).
- [ ] `enable` and `bright` for the onboard RGB are settable live from the iOS app; dimming works on hardware.
- [ ] The status beat-pulse (green/cyan flash, amber armed-blink) remains phase-tight — flash edge not visibly late vs the beat.
- [ ] Waveshare (panel on GPIO48) still emits no LED capability; no regression to the touch product.
- [ ] Host suites green (`test_status_rgb` extended for `status_render` fill-npix + brightness owner).
