## Goal

Browse to `http://kitchensync.local` instead of hunting for whatever address
DHCP handed the board this boot. Today the only way to learn the IP is to
attach a serial cable and read the `got ip` line -- which defeats the point of
a device that is otherwise headless and OTA-updatable.

## Why

Found the hard way (2026-07-10): after a reboot the P4 was reachable at
192.168.1.201, then wasn't, and the debugging went looking at the firmware.
The firmware was fine -- the *laptop* had moved subnets. A stable name removes
a whole class of "is the device dead?" confusion, the same way ESP-009's
standby heartbeat does for "is the board alive?".

## Scope

- Add the ESP-IDF `mdns` component; advertise `kitchensync` + the `_http._tcp`
  service on port 80 once `wifi_link` reports `got ip`.
- Hostname should be derivable per unit, not hardcoded to one name -- two
  KitchenSyncs on one network must not collide. Suggest `kitchensync-XXXX`
  from the last two MAC bytes, with plain `kitchensync` as an alias, and log
  the chosen name at boot next to the IP.
- Advertise on the SoftAP fallback too (P4-027), so the first-boot config
  page is reachable by name as well.
- X32Link (Arduino) has the same problem and the same cheap fix (`ESPmDNS.h`).
  Out of scope here; file a sibling ticket if this lands well.

## Non-goals

- **This does not cross subnets.** mDNS is link-local by design. It fixes
  "which address today", not "the laptop is on another network". That was a
  double-NAT / second-DHCP-server problem on the bench LAN and belongs in the
  router, not the firmware.
- Not a static-IP feature. A wrong static IP strands the board off-network and
  forces the SoftAP recovery dance; if that is ever wanted it is a separate
  ticket, and it changes `KsConfig`'s layout (bump `KS_CONFIG_VERSION`, P4-014
  will catch a stale blob).

## Acceptance Criteria

- [x] `http://kitchensync.local` serves the config page on a fresh boot
- [x] Chosen hostname logged at boot alongside the IP
- [x] Two units on one LAN do not collide
- [ ] Name resolves in SoftAP fallback mode too  -- coded, NOT verified
- [ ] Survives a DHCP lease change (reboot the router, name still works)  -- NOT verified

## Notes

- Cheap: the ESP-IDF mdns component is a managed dependency and the wiring is
  ~15 lines in `wifi_link.c` behind the existing `got ip` event.
- Pairs with the OTA flow (P4-017): `curl -F ... http://kitchensync.local/update`
  is a materially nicer story than looking up an IP first.

## Progress

## Done 2026-07-10

Verified on the P4 (`kitchensync-242c`, on `test123`):

- `kitchensync-242c.local` and the `kitchensync.local` alias both resolve and
  serve `/status`.
- `dns-sd -B _http._tcp local` lists `KitchenSync`, and `-L` resolves it to
  `kitchensync-242c.local:80`.
- Hostname logged next to the IP at boot.

**Two criteria are coded but unverified**, and neither is worth pretending about:

- *SoftAP fallback* -- `start_ap()` calls `mdns_advertise()` with the AP netif's
  192.168.4.1, but proving it means deliberately stranding the board off-network
  and rejoining its AP. Untested.
- *DHCP lease change* -- `mdns_advertise()` re-registers the delegated alias on
  every `got ip`, which is the mechanism that should survive it. Nobody has
  rebooted the router.

One trap, found only on hardware. `esp_read_mac(mac, ESP_MAC_WIFI_STA)` FAILS on
the ESP32-P4: the WiFi MAC belongs to the onboard C6, not to this chip's efuse.
It fails softly, leaving a zeroed MAC, so the first build cheerfully named the
board `kitchensync-0000` -- precisely the collision the per-unit name exists to
prevent, and invisible to any host test. The fix asks the (remote) WiFi stack via
`esp_wifi_get_mac()`, falling back to `ESP_MAC_BASE`. Recorded in `AGENTS.md`.

**Flash cost: ~48 KB.** The app partition's free space went from 11% to 6% of
1 MB. That is not alarming yet, but the next feature of this size will need a
partition-table conversation, not a shrug.

Sibling filed as ESP-014 (X32Link / `ESPmDNS`), per this ticket's own note.
Notably `esp_read_mac(ESP_MAC_WIFI_STA)` *works* on the S3, so the P4 trap above
does not repeat there.
