## Goal

Surface battery state for QT Py units with Adafruit's "LiPo BFF" (MAX17048
fuel gauge, STEMMA QT) stacked underneath — cell voltage + charge percent in
`/status` and the web UI footer. Opt-in: most units don't have the board
attached.

## Solution

- `battery_gauge.{h,c}` — pure register decode (VCELL/SOC → volts/percent),
  host-tested (`test_battery_gauge.c`). No I2C in here.
- `battery_gauge_io.{h,cpp}` — thin `Wire`/I2C glue, gated behind
  `HAS_BATTERY_GAUGE` (separate flag from `BOARD_QTPY_ESP32S3` — not every QT
  Py has the LiPo BFF attached).
- `battery_snapshot.{h,c}` — publish/read seam (same shape as `tempo_snapshot`,
  ARC-001) so the poll task and `/status` never race on a torn reading.
- `battery_task` polls every 3s (the MAX17048's own ADC cycle is ~1Hz — no
  point polling faster) and publishes; I2C failure (no LiPo BFF attached)
  leaves `present=false` and `/status` omits the fields entirely.
- `web_status_json()` gains `has_batt`/`batt_v`/`batt_pct` params — omitted
  from the JSON schema entirely when `has_batt` is false, so boards without
  the gauge see the exact same payload as before this ticket.
- Web UI footer shows `BATT·NN%` when the field is present (feature-detected
  via `typeof d.batt_pct`).

## Deliverable

`battery_gauge.{h,c}`, `battery_gauge_io.{h,cpp}`, `battery_snapshot.{h,c}` +
`test_battery_gauge.c`; wiring in `X32Link.ino` (`battery_task`),
`web_config.cpp` (`handle_status()` + footer JS/CSS), `web_status_json.{h,c}`;
`build_opt.h` / `AGENTS.md` document the `HAS_BATTERY_GAUGE` flag; CI compiles
a QT Py + battery-gauge variant.
