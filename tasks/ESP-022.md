## Goal

A tiny standalone node that joins an Ableton Link session over WiFi and drives a
WS2812 strip as a **visual metronome** (beat strobe / bar-position chase) — the
LED half of the clock hub, on cheap generic hardware. The wedge: it can be
**flashed over WiFi onto an existing WLED controller**, no USB and no soldering,
so any WLED strip in a rig becomes Link-synced lighting.

Discovered example on the bench network: `WLED-Tent` (`wled-4ebda8.local`,
192.168.2.27) — **classic ESP32**, WLED 16.0.0, **144 WS2812** (single-wire, RGB,
no white), ~850 mA budget, ~43 fps. That is the reference target.

## Why

- The LED visual-metronome logic already exists on the P4 (P4-018 bar-position
  chase, P4-019 web customization, LNK-039 dot colours). This ticket productizes
  it as a **standalone ESP32 node** rather than a P4 peripheral.
- It's the visual output of **CLK-001** (the universal clock hub): "follow Link →
  paint the beat" on hardware that costs a few dollars.
- WLED hardware is everywhere and OTA-flashable, so distribution is trivial — no
  custom board, no cable, reflash in place.

## What already exists (reusable)

- **Link receive** — `link_protocol.c` (join session, tempo/phase). Portable.
- **Beat/phase → LED** — the visual-metronome logic from P4-018/019 + the phase
  math (`led_phase`, dot colours LNK-039). Pure, host-tested.
- **WS2812 driver** — RMT-based NeoPixel output already used on the QT Py path.

## What is new (the lift)

1. **Classic-ESP32 board target.** Current builds are ESP32-**S3**
   (`esp32:esp32:esp32s3`); WLED-class boards are classic **ESP32**
   (`esp32:esp32:esp32`). Add the target: no native USB, different pin map, no
   LovyanGFX/touch. The Link stack + LED path port cleanly; strip S3-only glue.
   (See ARC-018 — reference the shared timing core, don't fork it, when adding the
   target.)
2. **Match the WLED hardware config.** WS2812 data pin = whatever the WLED cfg
   uses (read `cfg.json` / WLED UI — commonly GPIO2/16 on ESP32), single-wire RGB,
   configurable count (144 on WLED-Tent) and a power/brightness cap (850 mA).
3. **WiFi OTA install path (no USB).** The WLED box exposes no USB data port —
   flash over the air: WLED `/update` (upload `.bin`) or ArduinoOTA/`espota`
   (port 3232). Produce an OTA-compatible image + document the one-command push.
4. **Minimal config surface.** Reuse `ui_chrome` for a small web page (WiFi, LED
   count/pin, brightness cap, pattern/colour) — per ADR-0008 mirror the P4 look.

## Acceptance Criteria

- [ ] `esp32:esp32:esp32` build target compiles a Link-LED-flasher sketch (WiFi +
      `link_protocol` + WS2812 visual metronome), host tests green for the shared
      pure logic.
- [ ] Drives a WS2812 strip in time with a Link session (beat strobe + bar chase),
      brightness/power-capped; verified with the logic analyzer / by eye against a
      known Link host.
- [ ] Flashes over WiFi onto a WLED-class ESP32 with **no USB** (documented
      `/update` or `espota` command), and boots into the flasher.
- [ ] Small web config page (WiFi, LED count/pin, brightness, pattern) styled off
      the P4 chrome.

## Notes / risks

- **Classic ESP32 ≠ S3.** No native USB (so serial is via a UART bridge if present),
  different RMT/pin setup, no display/touch. The Link + LED subset is portable; do
  not drag S3-only code along.
- **Reflashing WLED is destructive + roughly one-way.** It wipes WLED; getting WLED
  back means another OTA (or opening the case for UART/IO0 pads). Do NOT reflash an
  in-use installation (e.g. the actual `WLED-Tent`) — prototype on a spare board.
- **Alternative integration (no reflash):** instead of replacing WLED, one of our
  devices could *drive* WLED as a node over its realtime protocols (DDP / E1.31 /
  WARLS on UDP 21324). That keeps WLED and makes our device the Link→lighting
  source. Cheaper to prototype; worth comparing before committing to a reflash SKU.
- Rolls up under CLK-001 as the visual-output node. Independent of the MIDI/analog
  work.
