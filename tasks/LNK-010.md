## Goal

Collapse the two sketches into ONE firmware with both inputs compiled in and
the active one selected at runtime (web UI in LNK-012). One `.bin` to build
and flash; flip Link vs USB-MIDI in config instead of choosing a binary at
flash time. Eliminates the duplicated `.ino` (setup / wifi / AP / factory
reset / loop / `bpm_task`) entirely.

## Target board

**Adafruit QtPy ESP32-S3** — `esp32:esp32:adafruit_qtpy_esp32s3_n4r2`.
Chosen because we have 6 of them, MIDI mode is already proven there, it has a
BOOT button on GPIO0 (factory reset), and it can log over UART `Serial0` so the
USB CDC stays free for USB-MIDI. XIAO support is optional behind `#ifdef`.

## Current state (the duplication being removed)

`X32Link.ino` (167 lines) and `X32MidiClock.ino` (131 lines) share, nearly
verbatim: `setup()`, `wifi_try_connect()`, `start_config_ap()`,
`check_factory_reset()`, `loop()`, and most of `bpm_task()`. They differ only
in input wiring (now behind the LNK-008 seam), LED handling, and `Serial` vs
`Serial0`.

## Proposed change

1. **One sketch dir** (e.g. `X32Sync/`) containing: the single `.ino`, both
   TempoSource adapters (LNK-008), and symlinks to the shared modules
   (`osc_*`, `bpm_tracker`, `app_config*`, `web_config`, `tempo_source.*`,
   `link_*`, `midi_*`). Retire `X32Link/` and `X32MidiClock/` once green.
2. **Board abstraction** `board.h`: LED pin + on/off polarity + a `LOG` macro
   mapping to the right serial (QtPy → `Serial0`; XIAO `#ifdef` → `Serial`).
   Removes the hardcoded `LED_PIN`/`Serial0` divergence.
3. **Boot sequence** in one `setup()`:
   - `tempo_source_select(cfg.input_source)` (NVS field from LNK-012; default
     Link until set).
   - `tempo_source_pre_net()` — ALWAYS runs USB-MIDI init (harmless in Link
     mode), so MIDI enumerates before WiFi.
   - WiFi connect / AP fallback (unchanged).
   - `tempo_source_begin()` — Link joins multicast here.
   - spawn `bpm_task` (+ `led_task`); start web config.
4. **Unified `bpm_task`** drives everything through the seam:
   `tempo_source_poll()` → `tempo_source_bpm()` / `active()` / `beat()`, with
   the send-gate (ideally the LNK-011 `bpm_publisher`). One LED path using
   `tempo_source_beat()`.

## Constraints

- USB-MIDI device is always present (composite CDC+MIDI). Verify Link mode is
  unaffected by the idle MIDI interface.
- Keep `link_protocol.c` + host tests green.
- Don't regress the shipped Link multicast fix (`WiFi.setSleep(false)` stays).

## Acceptance Criteria

- [ ] Single sketch dir compiles:
      `arduino-cli compile --fqbn esp32:esp32:adafruit_qtpy_esp32s3_n4r2 X32Sync`
- [ ] No duplicated lifecycle code remains; `setup/loop/bpm_task` exist once.
- [ ] `board.h` abstracts LED + log serial; QtPy default, XIAO behind `#ifdef`.
- [ ] On-device, Link mode: joins session, `peers>0`, BPM tracks (regression
      guard for the multicast fix).
- [ ] On-device, MIDI mode: USB-MIDI clock from a DAW drives BPM; OSC reaches
      the mixer.
- [ ] Switching modes (via LNK-012, or a temporary compile-time default here)
      requires no re-flash — only a config change + reboot.
- [ ] Host tests still pass: `cd test && make`
- [ ] Old `X32Link/` and `X32MidiClock/` dirs retired (or clearly deprecated).

## Progress (2026-06-29) — DONE

`esp32/X32Link` is the unified firmware: both inputs compiled in, active one
chosen at runtime via the `input_source` NVS field (LNK-012 web selector) and
the TempoSource seam (LNK-008). Built, flashed, and hardware-verified this

the source-of-truth for the midi_* modules (symlinked into X32Link) and as a
standalone option; retiring that dir is optional, non-blocking cleanup.
