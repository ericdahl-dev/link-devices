Design: `docs/plans/2026-07-19-rgb-led-output-subsystem-design.md` (slice 2 of 5).

## What to build

Turn the P4's flat `led_*` config scalars into the per-output `led[]` array (mirroring
`clock[]`) and surface it end-to-end, so the P4 visual metronome is configured per output
live from the iOS app.

- `led[i] = { enable, style, bright, beat_color, accent_color, mode, fade }` in `KsConfig`
  (Status/Effect-only fields deferred; `style` enum includes METRONOME).
- Config-version bump with a frozen prior-struct migration mapping today's `led_*` scalars
  into `led[0]` with `style=METRONOME` (preserves current behavior); fix the `sizeof`
  static-assert; add the new fields to **both** `ks_config_live_safe_copy` (apply live, no
  reboot) and `ks_config_valid` (clamp `bright`, range-check `style`/`mode`).
- `/config.json` emits `led:[{…}]`, one object per output the board's descriptor declares;
  a board with zero LED outputs emits no `led` capability.
- iOS grows a per-output LED section rendered from the capability, covered by the existing
  live/reboot partition test.

## Acceptance criteria

- [ ] `led[]` array in `KsConfig`; migration from the current `led_*` scalars is loss-preserving (existing P4 config survives an OTA/reflash with identical behavior).
- [ ] Every `led[]` field is classified live-safe or reboot-required (never both/neither) and clamped in `ks_config_valid`.
- [ ] `/config.json` emits per-output `led[]`; boards with no LED output emit no `led` key.
- [ ] iOS shows a per-output LED section; changing color/brightness/pattern applies live on the P4.
- [ ] Host suites + the iOS live/reboot partition test green.
