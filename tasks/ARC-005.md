## Goal

Evaluate the near-pass-through glue modules and fold only the ones whose
removal genuinely concentrates complexity — without harming the pure/glue
split that keeps the core host-testable.

## Problem (from the architecture review)

A few glue files are near-pure pass-throughs:
- `link_listener.cpp:1-38` — forwards to `link_proto_*`; `poll()` is a read loop.
- `osc_sender.cpp:11-23` — `osc_sender_begin()` is one `udp.begin(2390)`.
- `midi_bpm.cpp:11-27` — orchestrates count + window + `midi_bpm_calc()`.

They pass the deletion test as "shallow" (deleting them just moves a line up a
layer).

## Acceptance Criteria

- [ ] For each candidate, a one-line verdict: fold or keep, with the reason.
- [ ] Any fold removes a file/indirection **without** pulling WiFiUDP/Arduino
      deps into a host-tested pure module.
- [ ] No behavioural change; host suites green.

## Notes

- **Speculative / lowest priority.** Several of these shallow seams exist
  specifically to keep WiFiUDP out of the host-tested core — that's a *good*
  shallow seam, not friction. Likely outcome: keep most, fold at most one.
