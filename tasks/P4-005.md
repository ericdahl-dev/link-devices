## Goal

Stream 24-PPQN MIDI clock (0xF8) derived from the Link session **out the USB host
port** to the enumerated USB-MIDI device — the headline P4 feature: tempo clock
straight to hardware (MIDIHub, synths, drum machines) with no computer in the path.

## Why

This is the P4 analogue of LNK-027 (Link → USB-MIDI clock OUT on the S3), but the
S3 writes as a USB *device*; the P4 writes as a USB *host* to attached gear —
strictly more capable.

## Reuse (the LNK-027/028 payoff)

The pure `clock_ticker` engine (24-PPQN, phase-locked to `tempo_source_beats_now()`,
re-primes on re-origin) is **unchanged**. The only new glue is the write path:
`midi_host_send()` (P4-003) instead of the TinyUSB *device* write. The
`midi_clock_out.{h,c}` thin 24-PPQN adapter over `clock_ticker` should port as-is.

## Depends on

- P4-003 — USB-MIDI host driver + `midi_host_send()`.
- P4-004 — Link tempo source (or free-run/MIDI-in as an interim test tempo).

## Deliverable

Attached USB-MIDI gear receives stable 24-PPQN clock locked to the Link session;
verify tempo accuracy (e.g. a synth arpeggiator / the MIDIHub monitor) as done for
LNK-027 on the S3 (127.08 vs 127.0 BPM check).

## Write-path spike (2026-07-04) — PROVEN on hardware ✅

Fixed-BPM ESP-IDF spike (`scratchpad/p4_midi_clock`, based on the P4-003
`usb_host` bring-up): the P4 claims the Midihub MIDIStreaming interface (if3),
streams 24-PPQN `0xF8` to bulk OUT `0x01`, and reads bulk IN `0x82`. With a
Midihub Editor preset routing **USB B → USB B** (loopback), the P4 sends **and
reads its own clock back**:

- TX ticks and RX `0xF8` track **1:1** (414/414 over ~8 s), **`out_err 0`,
  `missed 0`** — clean write path, no drops.
- Rate ~**48 ticks/s** = 120 BPM × 24 PPQN → tempo accurate.
- Loopback returned on **cable-mask 0x2 = cable 1**, confirming the mapping
  **USB port A/B/C/D = USB-MIDI cable 0/1/2/3**. Spike sends on all 4 cables so
  any routed port loops back.

So the USB-MIDI **host clock-OUT write path is validated end-to-end with real
gear.**

## Integrated in KitchenSync (2026-07-05) — DONE ✅

Folded into `KitchenSync/` as the real firmware. A live Ableton Link tempo now drives
24-PPQN MIDI clock out the USB-MIDI host. Data flow (ADR-0003, pure host-tested +
thin glue):

    wifi_link (Link gossip) -> LinkTimeline.micros_per_beat
      -> beat_clock    (pure, NEW, TDD'd) : integrate tempo over local clock -> beats
      -> clock_ticker  (pure, reused)      : quantize to 24 PPQN pulses
      -> usb_midi_pack (pure, NEW, TDD'd)  : encode 0xF8 as a USB-MIDI packet
      -> usb_midi_host (glue)              : bulk-OUT to the device

Two new pure modules were red-green TDD'd with Unity suites (`test_usb_midi_pack`,
`test_beat_clock`); everything else is reused unchanged (`clock_ticker`,
`link_protocol`) or thin glue (`wifi_link`, `usb_midi_host`). `beat_clock` is
local-clock-anchored (correct tempo/rate, always positive) — session-phase
alignment is deliberately deferred (needs the ghost-xform pipeline).

**Verified on hardware:** Ableton Note @ 132 BPM → `bpm 132.00`, `usb ready`,
**clock TX ~52.8/s = 132 × 24 ÷ 60**, and with the Midihub USB→USB loopback
preset the **TX/RX tracked 1:1** (264/257). Correct-tempo clock streams from a
live Link session to the Midihub, no computer in the path.

**Follow-ups (own tickets):** MIDI Start/Stop/Continue transport (the pure
`usb_midi_pack` already encodes them); phase-locked downbeat (ghost xform);
Multiclock-style per-output cable/division/phase/swing via web UI (see the KitchenSync
product vision); MIDI clock IN → Link.
