## Problem

P4Hub only starts its `P4Hub-Setup` SoftAP when NVS has no WiFi SSID stored at
all (true first boot / factory reset). If the board *has* credentials saved
but the configured network isn't reachable — moved off-site, network renamed,
travelling with the rig — `wifi_link.c`'s `WIFI_EVENT_STA_DISCONNECTED`
handler just calls `esp_wifi_connect()` again forever. No way back to a
config screen without physically re-flashing or wiping NVS.

X32Link already solves this: `wifi_try_connect()` gives the configured
network a 30s budget, and on timeout calls `start_config_ap()` to park in its
own `X32Link-Config` AP (terminal until reboot). This ticket brings P4Hub to
parity.

Found live: user was off-site (ND campus guest WiFi) with a P4Hub already
configured for a home network; needed to reconfigure but had no way to reach
a config screen without cutting power to force a fresh boot with cleared NVS.

## Implementation

- New pure module `P4Hub/main/wifi_fallback.{c,h}`: single function
  `wifi_fallback_should_give_up(now_us, connect_start_us)`, true once the
  same 30s budget as X32Link (`WIFI_FALLBACK_TIMEOUT_US`) has elapsed.
  Host-tested in `test/test_wifi_fallback.c` (5 cases: well before, zero
  elapsed, exact boundary, well past, and non-zero-origin timestamps) — kept
  separate from `wifi_link.c` because that file is ESP-IDF/hardware-coupled
  and not host-testable itself; a wrong inline comparison there would have no
  test coverage to catch it.
- `wifi_link.c`: `start_sta()` records `s_connect_start_us`. The
  `WIFI_EVENT_STA_DISCONNECTED` handler now checks
  `wifi_fallback_should_give_up()` before deciding to retry
  (`esp_wifi_connect()`) vs. give up (`give_up_and_start_ap()`: disconnect,
  stop STA, start the `P4Hub-Setup` AP — same terminal-until-reboot behavior
  as X32Link).
- Refactored `esp_wifi_init()` + event-handler registration to happen once in
  `wifi_link_start()` regardless of which path (STA or AP) is taken first,
  since the fallback path now needs to switch from STA to AP at runtime
  without re-initializing WiFi.

## Verified on hardware

Flashed to the P4-NANO in hand (already had a stale/unreachable SSID stored).
Serial log confirmed: retried STA every ~2.4s as before, then at ~35s
(30s budget + scheduling slack) logged `no connection after 30s — giving up
on STA, falling back to AP`, stopped STA, and brought up `P4Hub-Setup` with
its DHCP server — `/save` on the AP's web UI is reachable to enter new
credentials, which reboots into STA on save (existing `p4hub_web.cpp` path,
unchanged).
