## Goal

Run a FreeRTOS task that polls the Link session state every `LINK_POLL_MS`
milliseconds, detects BPM changes above `LINK_BPM_THRESHOLD`, and publishes
the new BPM to a mutex-protected shared variable. This decouples tempo
tracking from OSC output.

## Acceptance Criteria

- [ ] `link_task()` runs as a FreeRTOS task on core 0 (leave core 1 for WiFi/network)
- [ ] Polls Link session state every `LINK_POLL_MS` (default 50 ms)
- [ ] Only triggers on changes ≥ `LINK_BPM_THRESHOLD` (default 0.5 BPM)
- [ ] `g_bpm_changed` flag set atomically when a qualifying change is detected
- [ ] `g_current_bpm` updated under `g_bpm_mutex` before flag is set
- [ ] Serial confirms: `BPM changed: 120.00 → 130.00` within 200 ms of DAW change
- [ ] Task stack usage < 4 KB (verify with `uxTaskGetStackHighWaterMark`)

## Notes

**Firmware scope**: `esp32/X32Link/` only. Do NOT touch `esp32/X32_emulator/`.

**Shared state** (declare at file scope in X32Link.ino):
```cpp
static SemaphoreHandle_t g_bpm_mutex;
static volatile float    g_current_bpm = LINK_DEFAULT_BPM;
static volatile bool     g_bpm_changed = false;
```

**FreeRTOS task**:
```cpp
void link_task(void *arg) {
    double last_bpm = LINK_DEFAULT_BPM;
    for (;;) {
        double bpm = link_get_bpm();
        if (fabs(bpm - last_bpm) >= LINK_BPM_THRESHOLD) {
            Serial.printf("BPM changed: %.2f → %.2f\n", last_bpm, bpm);
            xSemaphoreTake(g_bpm_mutex, portMAX_DELAY);
            g_current_bpm = (float)bpm;
            g_bpm_changed  = true;
            xSemaphoreGive(g_bpm_mutex);
            last_bpm = bpm;
        }
        vTaskDelay(pdMS_TO_TICKS(LINK_POLL_MS));
    }
}
```

**Task creation in setup()**:
```cpp
g_bpm_mutex = xSemaphoreCreateMutex();
xTaskCreatePinnedToCore(link_task, "link", 4096, NULL, 1, NULL, 0);
```

**Core assignment**: Pin to core 0. The Arduino `loop()` and WiFi stack run
on core 1. The Link SDK's own internal threads may also run on core 0 —
this is fine; FreeRTOS schedules them.

**Poll interval tradeoff**: 50 ms gives 20 Hz resolution on BPM changes —
more than sufficient. Lower values waste CPU without improving perceived sync.
