## Problem

The P4 (KitchenSync) web UI got a round of improvements the S3's config page
(`X32Link/web_config.cpp`) never received. The two are drifting; the S3 should
carry the same UX so the device family feels consistent.

Concrete deltas found comparing `KitchenSync/main/ks_web.cpp` (source of truth)
against `X32Link/web_config.cpp`:

1. **Live-apply, no reboot (`/live`, P4-015).** The P4 POSTs each changed
   live-safe field to a `/live` endpoint and it takes effect immediately
   (`ks_web.cpp:179-195`, `live_handler` at `:360`), bumping a config-generation
   counter so the running task re-primes. The S3 still requires Save + reboot for
   every change. Carry the pattern for the S3's live-safe fields (e.g. quantum,
   input source — decide which are safe to change without a reconnect).
2. **Nudge ± steppers (up/down buttons).** The P4 wraps numeric fields with
   +/- steppers that bump by the field's step, clamp, and POST immediately
   (`ks_web.cpp:87` styling, `:196-206` JS, `:240-248` markup). Nicer than typing
   into a number box on a touch phone. Apply to the S3's numeric fields.
3. **Collapsible sections.** The P4 groups controls into collapsible sections
   with persisted open/closed state (`syncSect`, `ks_web.cpp:195`). The S3 page
   is a flat form. Group it the same way.
4. **Styling refresh.** Carry the P4's `<style>` block (`ks_web.cpp:40-165`) so
   the two pages share a look.

## Fix

Diff `ks_web.cpp` against `web_config.cpp` and port the applicable improvements,
adapting to the S3's own config fields (`AppConfig` — wifi, mixer_ip, model,
fx_slot, input_source, quantum_beats, midi_clock_out_enable, fdr_*). Don't port
P4-only controls (the WS2812 strip customization); do port the interaction
patterns and structure.

- Add a `/live` route + handler to the S3 web server mirroring the P4's
  `live_handler`, applying only the live-safe subset and persisting the rest via
  the existing Save path.
- Reuse the P4's client JS (poll `/status`, live-POST on change, nudge steppers,
  section toggles) largely as-is; it's vanilla and field-name driven.
- Keep the S3's existing `/status` polling and OTA (`/update`) intact.
