## Reported symptom

User, testing on real hardware: "when link host stopped, the dot acts weird.
kinda moves randomly."

**Reproduced on real hardware 2026-07-03** (see "Live reproduction" below):
scenario **(a) transport stop/start is the confirmed trigger**; scenario **(c)
peer disconnect was tested and is NOT erratic** (smooth coast, clean recovery).
The code-reading investigation below stands; the live data narrows it.

**2026-07-03 update — this is TWO bugs, BOTH now fixed & hardware-verified; full
handoff in `docs/plans/2026-07-03-lnk-026-phase-handoff.md`.** (1) Erratic garbage
during transport stop/start — new pure helper `link_phase_timeline_epoch_reset()` +
re-measure trigger in `link_measurement_io.cpp`. (2) Static ~2-beat phase offset —
**root cause was a stale pong in the WiFiUDP RX buffer being consumed at the top of
the next re-measure attempt** (echoed `host_time` ~2 s / one `REMEASURE_INTERVAL_US`
old → `sample_a` underestimates the GhostXForm by ~RTT/2 ≈ 1 s ≈ 2 beats). Fixed by
flushing the RX buffer in `start_attempt()` (root) + an RTT-sanity gate on samples
in the pure `link_measurement.c` (`LINK_MEASUREMENT_MAX_RTT_US`, host-tested).
Verified against a reference-truth Ableton Link SDK peer: device ghost error
−2.05 beats → −0.026 beats. Branch `fix/lnk-026-phase-restart`.

**"The dot" is most likely the LNK-015 touch-UI phase wheel** — the green
marker `touch_display.cpp:349-373` sweeps around a circle via
`ui_phase_angle(ts.phase, ts.quantum)` (`touch_ui.c:26`). The same underlying
computation also feeds the web `/status` beat dot (`web_config.cpp`) and the
LED/backlight beat strobe (`led_task()`, `X32Link.ino`) — if the root cause is
in `tempo_source.cpp` or below, all three are affected identically, so
confirm which one the user actually meant, but expect all three broken the
same way.

**Ambiguity to resolve on pickup:** "link host stopped" could mean (a) the
Link session's transport was stopped (Space-bar play/stop in Ableton or the
in-flight `linkcli` design) while the peer stays on the network, or (b) the
host app quit / Link was disabled, so the peer disappears entirely. These hit
different code paths below — confirm which (or both) before fixing.

## Important: this is filed against current master, not a stale snapshot

While investigating, found that **an earlier, related bug is already fixed**
and merged (PR #32, `esp32: fix Link phase never holding valid (LNK-018/019)`,
commit `c16b319`): `tempo_source_phase_valid()` used to gate on
`link_measurement_active()` (true only while a ping/pong attempt was
in-flight, false the instant it committed — "the flashing green dot" per that
commit's own message), which made phase valid for milliseconds and invalid
the rest of the time. It now correctly gates on
`link_measurement_current_xform().valid` (`tempo_source.cpp:103-115` on
master). **Do not rediscover or re-fix that bug** — this ticket is about
what's still wrong (or newly exposed) on top of that fix. PR #32 also added
unconditional periodic re-measurement every 2s (`REMEASURE_INTERVAL_US`,
`link_measurement_io.cpp:44`), which didn't exist when LNK-018/019 were
originally written and is a new variable worth considering below.

## Investigation (current master)

Traced: `touch_display.cpp:349-373` (phase wheel) →
`tempo_snapshot_read()` (`tempo_snapshot.c`, ARC-001 atomic-read seam — rules
out a torn-read explanation, ships a coherent `{bpm,phase,valid,quantum}`
struct) → published by the bpm task from `tempo_source_phase()` /
`tempo_source_phase_valid()` (`tempo_source.cpp:88-115`) → `link_protocol.c`
/ `link_measurement.c` / `link_measurement_io.cpp`.

### (a) No start/stop (isPlaying) tracking anywhere — likely the actual gap

Neither `link_protocol.c`'s TLV parsing nor `tempo_source_phase()` track
Link's start/stop state at all — there's no code path for it. `tempo_source_phase()`
(`tempo_source.cpp:88-101`) always extrapolates forward from elapsed
wall-clock time through the committed `GhostXForm`, regardless of whether the
host's transport is playing or stopped. Real Ableton Link's own Timeline
behaves the same way in principle (phase is a continuously-advancing virtual
clock; "isPlaying"/start-stop sync is a separate, opt-in signal most Link
integrations don't gate rendering on) — so a **smoothly continuing** sweep
after a host stop would be expected-per-spec, not a bug. But "moves randomly"
describes something less orderly than "keeps sweeping at the same rate" —
confirm with the user whether they mean (i) it doesn't freeze (expected, a
product decision, not a bug) or (ii) it visibly jumps/stutters (a real bug,
see (b) and (c)).

### (b) Periodic re-measurement (new since PR #32) could commit a jump

`link_measurement_io.cpp:89-118`'s `check_trigger()` now re-triggers a fresh
measurement attempt against the *same* peer every 2s unconditionally
(`REMEASURE_INTERVAL_US`), on top of the original "different peer appeared"
trigger. Each successful re-measurement calls `link_measurement_attempt_end(true)`,
which recomputes `xform.intercept_us` from a fresh median
(`link_measurement.c:217-228`) and **replaces** the committed xform outright —
there's no smoothing/slew between old and new `intercept_us`, so
`tempo_source_phase()`'s next read after a recommit uses the new offset
immediately. If measurement noise or network jitter is larger while the host
is stopped than while playing (plausible if a stopped/idle DAW gets
deprioritized by the OS, or if `linkcli`/Ableton's own ping-response timing
varies with transport state), each ~2s recommit could produce a small but
visible discontinuity in the wheel position — repeated every 2 seconds, which
could read as "random" rather than a single one-off glitch. Worth
instrumenting: log `xform.intercept_us` on every successful commit
(`link_measurement_attempt_end(true)` call site) and compare magnitude/
frequency of jumps while the host is playing vs. stopped.

### (c) Stale timeline data survives full peer loss (still present, unrelated to PR #32)

`link_protocol.c`'s `remove_peer()` / `link_proto_tick()` (`:76-101`) reset
`s_bpm` to `0.0` once peer count reaches zero, but **never** reset
`s_timeline_seen` or `s_timeline`. `link_proto_timeline()` keeps returning
`true` with the last-known (frozen, stale) `micros_per_beat` /
`beat_origin_micro` / `time_origin_us` indefinitely after a host fully
disconnects. Only relevant to the "(b) host app quit" reading of "stopped" —
if the peer fully vanishes, `tempo_source_phase_valid()` should probably also
eventually go false (it currently doesn't: `link_measurement_current_xform().valid`
stays true forever once committed, nothing invalidates it on peer loss), so
the wheel would keep sweeping on stale data with no live peer at all. Whether
this reads as "random" depends on how stale vs. current tempo compares —
worth checking independently of (b).

## Suggested fix direction (not prescriptive — confirm root cause first)

- If (b): consider a sanity/slew bound on `intercept_us` jumps between
  consecutive successful re-measurements (e.g. reject or log-and-average a
  recommit that differs from the previous value by more than some threshold),
  or at minimum add the instrumentation described above before changing
  behavior blind.
- If (c): invalidate `s_timeline`/committed xform when peer count drops to
  zero (reset `s_timeline_seen = false` in `link_protocol.c`'s tick/remove
  path, and consider exposing a way to invalidate `link_measurement`'s
  committed xform on total peer loss too — currently nothing does this).
- If (a) turns out to be the actual complaint (expects freeze-on-stop): that's
  a product decision requiring real Link start/stop-sync TLV support, a much
  bigger change — don't build this speculatively, confirm intent first.
- Keep any new validity/staleness decision in a small, pure, host-testable
  helper (same pattern as `led_phase.c` / `tempo_snapshot.c`) rather than
  inline in `tempo_source.cpp` or `link_measurement_io.cpp`, which are
  Arduino/C++-coupled and not host-tested.

## Live reproduction (2026-07-03, real hardware)

Waveshare ESP32-S3-Touch-LCD-1.47 on WiFi, Link peer = Ableton Note (iOS) on
the user's phone, transport playing. Method: poll `GET /status` at 2 Hz while
the user drives the phone + watches the phase wheel. Findings:

- **Scenario (c) — peer disconnect (Link off ~5-8s, then on):** `valid` **never
  drops to false** (confirmed across 3 runs, incl. 3 Hz polling — nothing catches
  a `valid=false` window). BPM holds; phase keeps sweeping at the correct rate off
  the last committed xform. User confirms the **wheel coasts smoothly** during the
  gap and recovers cleanly on rejoin. So (c)'s stale-timeline path exists in code
  but is **not the erratic symptom** — coasting is graceful here. Not the bug.

- **Scenario (a) — transport stop/start: THIS is the bug.** In `/status`, on
  **stop** the phase **freezes** (held ~1.0 for the ~6.5s stopped, `valid` still
  true — the wheel dot sits still, not "random"); on **start** sweeping resumes at
  the correct rate but locked to a **stale beat origin** — the user reports the dot
  is visibly **off the downbeat and does NOT self-correct** (still off after
  watching several seconds; the #32 2 s re-measure fixes clock *offset*, not the
  beat *origin*). So the defect is a **phase-origin staleness** across a transport
  stop/start: the peer republishes a new `beatOrigin`/`timeOrigin` on restart, but
  the device keeps `valid=true` on the old origin and never re-aligns.

- **Scenario (b) — periodic re-measure jump:** not separately instrumented this
  pass; the (a) misalignment persisting (not jumping every 2 s) argues the visible
  fault is (a), not (b). Leave (b) instrumentation as a nice-to-have.

**Refined fix direction:** on a timeline discontinuity (transport stop→start →
new `beatOrigin`/`timeOrigin`), the device must re-derive phase from the *new*
origin, not coast on the committed one. Candidate: detect a `beatOrigin`/
`timeOrigin` change in `link_protocol.c` and invalidate/refresh the phase basis
(and consider whether `tempo_source_phase_valid()` should briefly drop to false
so the wheel shows "syncing" rather than a confidently-wrong position). Keep the
decision in a pure host-testable helper per the existing pattern.
