## Goal

Start and stop the Link session's transport from KitchenSync's web UI --
a play/stop control on the status page, so the rig can be started from the
box (or a phone) instead of only from the DAW.

## The catch: our Link stack is receive-only

This is not a UI ticket with a small backend. Today KitchenSync **observes**
transport and never **proposes** it:

- `X32Link/link_protocol.h` exposes `link_proto_playing()` /
  `link_proto_start_stop_seen()` -- both read the session's StartStopState.
- Nothing in the tree transmits a Link packet. `wifi_link.c` joins the
  multicast group and receives; there is no send path, no peer announce, no
  StartStopState proposal. The whole hand-rolled Link implementation is a
  listener.

So this ticket's real content is: **make KitchenSync a first-class Link peer
that can propose StartStopState.** That means encoding and sending Link
protocol packets, adopting session-timeline ownership rules, and handling the
proposal/adoption dance when peers disagree -- meaningfully harder than the
receive path, and the part of the protocol we have never exercised.

## Two routes

1. **Extend the hand-rolled stack.** Add a send path + StartStopState
   proposal to `link_protocol.c`. Keeps the pure/host-tested split (ADR-0003)
   and the small footprint. Risk: transport proposal is exactly where a
   half-correct Link implementation misbehaves against real peers, and our
   only ground truth is bench testing against Ableton.

2. **Use the official Ableton SDK**, which `enableStartStopSync()` +
   `setIsPlaying()` handle correctly for free. P4-032 proved the SDK builds
   and runs on the P4 (see `LinkAudioPoC/`). Risk: GPLv2 -- the same
   licensing gate that blocks Link Audio productization
   (`docs/plans/2026-07-08-p4-link-audio-feasibility.md`). Adopting the SDK
   for transport drags that gate onto the *core* firmware, not just the audio
   feature.

Route 2 is less code and more correct; route 1 keeps the licensing surface
where it is. **Decide only after the Ableton licensing answer comes back** --
the same email that gates P4-032/ESP-008 also gates this choice. Do not start
this ticket before that.

## Scope (once the route is chosen)

- Play/stop control on the web status page, reflecting live transport state.
- Proposal is session-wide: pressing play here starts Ableton too.
- Respect the existing gate -- `ks_tick.c:56` already keys the metronome and
  LED off `tp_playing`, so they follow automatically with no changes.
- Pairs naturally with ESP-009 (standby heartbeat): the box shows it is
  waiting, and now you can also start it.

## Acceptance Criteria

- [ ] Web UI shows current session transport state, live
- [ ] Pressing play from the web UI starts transport in Ableton
- [ ] Pressing stop from the web UI stops transport in Ableton
- [ ] Metronome / LED / MIDI transport out follow, unchanged
- [ ] Transport proposal logic host-tested (pure module, ADR-0003)
- [ ] No regression when another peer proposes transport
