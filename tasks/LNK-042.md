## Problem

The Waveshare ESP32-S3-Touch-LCD-1.47 has a **16 MB** flash chip. X32Link on it runs
a **4 MB-flash** partition table (`min_spiffs`, see LNK-041), so:

- the app slot is capped at **1.9 MB** (X32Link uses 55% of it), and
- roughly **12 MB of the chip is simply unused**.

KitchenSync Touch already solved this in ESP-016: a sketch-local `partitions.csv`
(16 MB, dual-OTA, **4 MB app slots**) selected with `PartitionScheme=custom`.
**X32Link has no `partitions.csv` at all.**

At 55% there is no capacity emergency. The reason this ticket exists is the
**deadline**, not the headroom.

## Why this has a deadline (the whole point)

**A partition table cannot be changed over OTA.** OTA writes the *app slot*; it does
not rewrite the partition table. Repartitioning requires a **USB flash**.

And per AGENTS.md (hardware-verified): changing the partition scheme **relocates NVS
→ config + WiFi credentials reset to first-boot defaults (AP fallback)**.

So repartitioning **wipes the saved config**. Note carefully: the ARC-020 blob
migrator **cannot** rescue this. ARC-020 migrates a config that is *stale*; a
repartition makes it *gone*. Different failure, no overlap.

Therefore: **repartition a unit before it leaves the bench, or accept that changing
it later means getting the box back** (or talking a musician through a USB flash and
a full reconfigure). Dan's Touch already ships on the 16 MB/custom table (ESP-016) —
it is X32Link touch units that are still on the wrong table.

## Solution

Give X32Link a sketch-local `partitions.csv` (16 MB, dual-OTA, 4 MB app slots),
mirroring `KitchenSyncTouch/partitions.csv`, and select it with
`PartitionScheme=custom` **for the Waveshare/16 MB variant only**.

X32Link is multi-board (QT Py S3 4 MB, S3 Super Mini, Waveshare 16 MB), and one
sketch-local table cannot serve a 4 MB and a 16 MB chip. That is fine: the core only
reads `partitions.csv` when `PartitionScheme=custom` is selected, so the file is
inert for the other builds. Pick the scheme per board:

| Board | Flash | Scheme |
|---|---|---|
| QT Py S3 / Super Mini | 4 MB | `min_spiffs` (1.9 MB app, OTA-capable) |
| Waveshare S3-Touch | 16 MB | `custom` + `partitions.csv` (4 MB app) |

## Acceptance Criteria

- [ ] `X32Link/partitions.csv` exists: 16 MB, dual-OTA, 4 MB app slots, NVS + coredump,
      modelled on `KitchenSyncTouch/partitions.csv`.
- [ ] The Waveshare X32Link build uses `PartitionScheme=custom`; the 4 MB builds keep
      `min_spiffs` (LNK-041) and are unaffected by the new file.
- [ ] CI compiles the Waveshare variant against the 16 MB table and reports usage
      against a 4 MB slot.
- [ ] `/update` OTA still works on the repartitioned board (dual slots present) —
      verify on hardware, this is the whole reason for the second app slot.
- [ ] **Hardware:** flash a Waveshare unit over USB with the new table, confirm it
      boots, then confirm the expected NVS wipe (AP fallback) and reconfigure it.
      Confirm OTA works afterwards.
- [ ] AGENTS.md notes the per-board scheme so nobody flashes the wrong table.

## Notes

- **Ship gate.** Any X32Link touch unit that goes to a person should already be on
  its final partition table. Repartitioning after it ships is a USB job + a config
  wipe, and no firmware update can undo that.
- Expect the NVS wipe on the repartition flash. That is not a bug — back the config
  up off the device's `/` page first, then one `/save` restores it (AGENTS.md already
  documents this dance for the min_spiffs move).
- Depends on LNK-041 only for tidiness: pin the shipping scheme in CI first, so the
  before/after numbers in this ticket are measured against something real.
