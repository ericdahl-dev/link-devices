## Goal

Implement the Ableton Link **measurement protocol** as a unicast UDP client:
ping a peer's `mep4` endpoint (from `LNK-017`), exchange a few ping/pong round
trips, and derive this device's clock offset relative to the session's shared
virtual clock (**GHostTime**). This offset is what `LNK-019` uses to compute
real beat/bar phase. Without it, `Timeline.timeOrigin` (parsed in `LNK-017`)
is meaningless — it's expressed in a clock domain we don't yet have a mapping
into.

**We only need the client/pinger role.** This firmware never broadcasts its
own Alive/Response packets (it's a passive multicast listener — see
`link_listener.cpp`), so it never advertises a measurement endpoint of its
own and no other peer will ever ping us. We do not need to implement the
PingResponder (reply) side at all.

Reference: wire format and algorithm confirmed by reading
`esp32/lib/link/include/ableton/link/{Measurement,MeasurementService,PingResponder,GhostXForm,Median}.hpp`
and `esp32/lib/link/include/ableton/link/v1/Messages.hpp` (submodule —
`git submodule update --init esp32/lib/link` if needed). As with LNK-017,
this is a byte-format/algorithm spec only — we are not linking the SDK's
asio-based networking (confirmed dead end, see LNK-002/LNK-003).

## Wire format

**Different protocol header than discovery** (`link_protocol.c`'s multicast
parser uses `"_asdp_v\x01"` + a 12-byte header with ttl/groupId/NodeId — this
is unrelated and separate):

```
magic       = "_link_v\x01"   (8 bytes)
messageType = uint8           kPing = 1, kPong = 2
... TLV payload follows, same TLV envelope as discovery (key u32 BE, size u32 BE, value) ...
```

TLV keys used here (same TLV framing as `link_protocol.c` already implements):

```
HostTime      key 0x5f5f6874 ('__ht')  value: int64 BE microseconds
GHostTime     key 0x5f5f6774 ('__gt')  value: int64 BE microseconds
PrevGHostTime key 0x5f706774 ('_pgt')  value: int64 BE microseconds
SessionMembership key 0x73657373 ('sess') value: 8 raw bytes (NodeId/SessionId) — present in pongs, can be ignored for our purposes (we're not tracking session identity changes here)
```

**Ping** (what we send):
- Round 1: `HostTime{now}` only.
- Round N>1: `HostTime{now}` + `PrevGHostTime{g}`, where `g` is the
  `GHostTime` value we extracted from the **previous** pong we received.

**Pong** (what we receive — sent by the real Link peer's own built-in
responder, nothing we implement): `SessionMembership` + `GHostTime{...}`,
followed by **our own ping payload echoed back verbatim** — so a received
pong, after TLV-scanning, yields up to 4 values: the responder's
`GHostTime`, plus our own echoed `HostTime` and (round 2+) `PrevGHostTime`.
Parse it with the same generic TLV scanner `link_protocol.c` already has;
order of TLVs in the blob is not guaranteed, look up by key.

## Offset algorithm (confirmed: NOT a Kalman filter / PLL — simple closed-form + median)

On each received pong, given:
- `h_recv` = **our own** local clock reading taken at the moment this pong
  arrived (read locally, not from the wire)
- `g`      = `GHostTime` extracted from this pong
- `h_sent` = `HostTime` extracted from this pong (our own echoed send-time
  for the ping this pong replies to)
- `prev_g` = `PrevGHostTime` extracted from this pong, if present (round 2+)

compute up to two offset samples (`double`, units = microseconds) and push
both into a flat sample buffer:

```
sample_a = g - (h_recv + h_sent) / 2.0                 // always, if h_sent present
sample_b = (g + prev_g) / 2.0 - h_sent                  // only if prev_g present (round 2+)
```

After collecting samples, immediately send the next ping (`HostTime{now}` +
`PrevGHostTime{g}`) to keep the round-trip cadence going — don't wait for a
timer; the SDK's real cadence is response-driven (back-to-back at RTT speed),
the 50ms timer is only a silence watchdog.

**Final offset** = `median(all samples)`. Use a simple sort/`nth_element`
over a small fixed buffer — the official SDK accumulates up to 100 samples
before trusting the estimate, but that threshold is conservative, not a
protocol minimum; **32 samples is enough** for our purposes (cap the buffer
at 32, evict oldest on overflow, take the median of whatever's accumulated
once a "good enough" count is reached — see Acceptance Criteria for the
exact target).

**GhostXForm**: a single signed microsecond constant per device (`slope` is
hardcoded `1.0` in the real protocol too — Link does not model clock drift in
this path):

```c
typedef struct { int64_t intercept_us; bool valid; } LinkGhostXForm;
int64_t link_ghost_xform_host_to_ghost(LinkGhostXForm xf, int64_t host_us) {
    return host_us + xf.intercept_us;
}
```

**Failure handling**: if no pong arrives for 5 consecutive 50ms silence
windows (250ms of cumulative silence since the last received pong), abandon
this measurement attempt — leave the existing `GhostXForm` (if any) as-is
rather than overwriting it with garbage, and retry against the current (or a
newly-selected) peer on the next trigger.

## Implementation

- New module: `esp32/X32Link/link_measurement.h` / `link_measurement.c`
  (pure logic — sample buffer, median, sample-formula math, ping/pong TLV
  build/parse using the existing TLV helpers in `link_protocol.c`) +
  `link_measurement_io.cpp` (the thin Arduino `WiFiUDP` unicast send/receive
  glue, mirroring how `link_listener.cpp` is the thin Arduino shim around
  pure-C `link_protocol.c`).
- Clock source: use `esp_timer_get_time()` for all local timestamps in this
  path (not `millis()`/Arduino `micros()` — see LNK-019 for why precision
  matters here specifically).
- Peer selection: target the first peer in `s_peers[]` that has a recorded
  `mep4` endpoint (from LNK-017). Don't build multi-peer consensus — single
  reference peer is sufficient for our read-only use case.
- Trigger a measurement attempt: on first peer discovered (peer count 0→1
  transition), and re-trigger if the reference peer disappears (TTL expiry)
  and a different peer becomes available. A periodic refresh (e.g. every few
  minutes) is reasonable but not required for this task — note it as a
  follow-up if skipped.

## Acceptance Criteria

- [ ] `link_measurement.c`'s sample math (`sample_a`/`sample_b` formulas,
      median calculation, `GhostXForm` derivation) is pure and host-tested
      in `esp32/test/test_link_measurement.c` with synthetic ping/pong
      timestamp sequences and known-correct expected offsets.
- [ ] TLV build/parse for ping and pong messages round-trips correctly
      (build a ping, parse it back, build a pong wrapping it, parse that) —
      host-tested.
- [ ] On real hardware: a measurement attempt against a live Link peer
      (Ableton Live / AUM / etc. on the same LAN) completes and produces a
      `GhostXForm` with `valid = true` within a few seconds of the peer
      appearing.
- [ ] Failure path: if the target peer is unreachable (wrong port / firewall
      / peer vanishes mid-measurement), the attempt times out per the
      250ms / 5-window rule above without hanging or crashing, and a retry
      is scheduled.
- [ ] `cd esp32/test && make` passes including new test target.

## Notes

- This task does not yet expose phase to the rest of the firmware — that's
  `LNK-019`. This task's deliverable is `LinkGhostXForm
  link_measurement_current_xform(void)` (or equivalent) plus
  `bool link_measurement_active(void)`.
- Keep `link_measurement_io.cpp` on a separate UDP socket/port from
  `link_listener.cpp`'s multicast socket — these are two independent UDP
  conversations (multicast discovery vs. unicast measurement).

**Deviations / judgment calls (ticket ambiguities):**
1. `LINK_MEASUREMENT_READY_SAMPLES = 8` — the ticket's sample-math section
   says "see Acceptance Criteria for the exact target" for when a "good
   enough" sample count should stop an attempt and commit a GhostXForm,
   but the Acceptance Criteria section doesn't actually give a number.
   Picked 8 (≈4 round trips once PrevGHostTime echoing kicks in) as a
   judgment call — comfortably converges well under the 250ms failure
   window at LAN RTT speed, documented inline in `link_measurement.h`.
2. Round-counting / watchdog timer / consecutive-timeout state live in
   `link_measurement_io.cpp` per the ticket's explicit module split, even
   though the arithmetic itself doesn't actually require hardware — kept
   there (not host-tested) to follow the ticket's instruction literally
   ("don't try to unit test it... math lives in link_measurement.c").
3. Skipped periodic re-measurement refresh (ticket explicitly allows
   skipping this and calling it a follow-up) — only the 0→1 / reference-
   peer-vanished triggers are implemented.
4. `IPAddress` construction in `link_measurement_io.cpp` deliberately uses
   the 4-octet constructor, not `IPAddress(uint32_t)` — the latter would
   silently byte-reverse the address on this platform (Arduino's
   single-arg ctor assigns the raw word with no byteswap against our
   wire-format's big-endian-octet uint32 convention).
