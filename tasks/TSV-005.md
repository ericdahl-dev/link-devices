## Goal

Harden the WiFi connection (reconnect on drop, status indicator), add the /xremote keepalive loop, and wire up the web config page so the device can be configured at the venue without reflashing.

## WiFi and keepalive

The sketch already connects in `setup()`. This task makes the connection robust for live use:

- **Reconnect on drop**: check `WiFi.status()` each `loop()`. If disconnected, pause detection, attempt reconnect (up to 20 seconds), then resume. Print `[TSV] WiFi lost, reconnecting...` and `[TSV] WiFi reconnected`.
- **/xremote keepalive**: send `/xremote` to `g_config.mixer_ip:10024` every `TSV_XREMOTE_INTERVAL` ms (9 seconds). The XR18 doesn't need xremote for TEQ control (only for meter subscriptions), but sending it maintains the logical session. Use `millis()` comparison — do not block.
- **Connection indicator**: set a `bool g_mixer_connected` global; display task (TSV-006) reads it.

## Web config page

Reuse `web_config.cpp` symlinked from X32Link. That module already serves a config page at `http://<device-ip>/` and persists settings to NVS.

Extend the config page to expose TSV-specific parameters. Add to `AppConfig` (in `app_config.h`):

```c
/* Toast Saver — add to AppConfig struct */
float tsv_threshold_db;    /* default: TSV_THRESHOLD_DB (20.0) */
float tsv_cut_db;          /* default: TSV_CUT_DB (-6.0) */
float tsv_release_sec;     /* default: TSV_RELEASE_SEC (10.0) */
int   tsv_input_channel;   /* XR18 channel 1–18 for display labeling (default: 1) */
```

Add defaults in `config_defaults()` in `app_config.c`:
```c
cfg->tsv_threshold_db  = TSV_THRESHOLD_DB;
cfg->tsv_cut_db        = TSV_CUT_DB;
cfg->tsv_release_sec   = TSV_RELEASE_SEC;
cfg->tsv_input_channel = 1;
```

Add the four fields to the web config HTML form in `web_config.cpp` following the existing pattern for `fx_slot`. Labels:
- "Threshold (dB above noise)" — number input, range 5–40, default 20
- "Cut depth (dB)" — number input, range -15 to -3, default -6
- "Release time (s)" — number input, range 2–60, default 10
- "Toast mic channel (1-18)" — number input, range 1–18, default 1

## Acceptance Criteria

- [ ] `AppConfig` gains the four TSV fields; `config_defaults()` initializes them.
- [ ] Web config page shows and saves the four new fields.
- [ ] NVS persistence: after config save and device reboot, the new values are restored.
- [ ] WiFi reconnect loop is active. Tested by toggling the AP off and on: device prints reconnect log and resumes detection within 30 seconds.
- [ ] `/xremote` sent every 9 seconds; confirmed with Serial print `[TSV] xremote sent`.
- [ ] `g_mixer_connected` is true after `/xinfo` handshake and false during WiFi loss.
- [ ] `fft_detector` and `teq_notch` are paused (not called) while WiFi is disconnected.

## Notes

- `web_config.cpp` is symlinked from X32Link and **shared** — modifications here affect both X32Link and X32MidiClock. Keep TSV additions behind a `#ifdef TSV_BUILD` guard or add them to a local `tsv_web_config.cpp` that extends the shared page. The latter is cleaner.
- The existing `config_load()` / `config_save()` NVS functions use a fixed namespace. New fields are added as new NVS keys — existing devices that don't have them will get defaults on first load.
- On the Waveshare ESP32-S3-Touch-LCD-4.3, the device IP is shown on the display (TSV-006). For this task, print it to Serial on WiFi connect: `[TSV] IP: 192.168.x.x`.
