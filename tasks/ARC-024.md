## Problem

KitchenSync and KitchenSync Touch both carry a tick-health probe: `max_gap`, `max_work`,
`overruns`, `bursts`, `dropped`, and the per-stage split. **X32Link carries none of it.** A grep
for `max_gap` / `s_overruns` across `X32Link/*.c*` returns nothing.

So the firmware we have measured the *least* is the one that has been running the *longest*.

## Why this is not merely a tidiness ticket

ESP-018 was not an exotic bug. The MIDI writer ran at **priority 6**, below lwIP's `tcpip_task`
at **18**. The network stack legally preempted the clock generator. Under web load, the writer's
call into the Link engine blew up from **62 µs to 5038 µs** -- an 80× blowup with no lock
anywhere in that path. It was never blocking; it was being *preempted*.

**X32Link is an ESP32-S3.** It runs WiFi on-die, exactly like the Touch. It has the same lwIP
priority, the same core-1 contention with `loopTask`, and it serves a web config UI over the same
stack. Every structural precondition for ESP-018 is present.

Whether X32Link's writer sits above or below lwIP today, **nobody has measured it**, and the
whole point of this cycle is that measuring is the only thing that ever produced a true answer.
The three silent-drop bugs found this cycle were all invisible on serial and all lived in
untested glue. X32Link's glue is the same glue.

And `clock_ticker.dropped` -- the counter that makes a stall past `MAX_BURST` visible at all --
**already exists in X32Link**; it is the shared module where it was added. The pulses are being
counted right now. Nothing reads them out.

## Scope

- [ ] Add the gap/work probe to X32Link's MIDI writer, mirroring `ktouch_midi_out.cpp`. Published
      to volatile scalars, **never logged** -- an `ESP_LOGx` in a 1 ms RT task is a blocking UART
      write, and the first version of this probe logged on every overrun and manufactured 3,945
      phantom overruns in 40 s.
- [ ] Surface `dropped` (already counted by the shared `clock_ticker`, currently discarded). Bank
      the lifetime total before `reset()` zeroes it -- `reset()` *must* zero it, since callers
      stack-allocate.
- [ ] Report `core` -- confirm which core the writer actually lands on rather than assuming. The
      ESP-018 ticket's original premise was **wrong** about exactly this, and only measurement
      caught it.
- [ ] Expose on X32Link's existing status/web surface.
- [ ] **Then read the numbers under load.** Hammer the web UI with a Link session running and see
      whether X32Link has been quietly living with ESP-018 the whole time. If its writer is below
      lwIP, fix it the same way (priority 19: above lwIP at 18, below WiFi at 23) and confirm on
      the analyzer.

## Parity (ESP-017)

The parity principle is about features, but it applies to *instrumentation* just as hard: a bug
class found and fixed on two devices, on a third device that shares the silicon and the network
stack, with no way to even ask the question. Feeds ARC-023, which has nothing to display from
X32Link until this exists.
