## Why

The owner has an **iConnectivity mio10** on the bench. Two separate opportunities, and they
should not be conflated:

1. **A second supported USB-MIDI interface**, alongside the Blokas Midihub. Today the
   product effectively assumes one specific box.
2. **The first real demonstration of the P4's headline feature.** This is the bigger one.

## The feature has never actually been shown working

P4-010 gives every clock output its own division, phase nudge and swing. P4-034 then found
that **multi-output USB-MIDI had never worked at all** -- `usb_midi_host_send()` kept a
single in-flight transfer and silently dropped anything offered while busy, so with 4
outputs enabled roughly 3 of every 4 packets were discarded and **only the first output
ever reached USB gear**.

That is fixed (batching, PR #63). But it was only proven across **TWO** outputs, and only
after shuffling cable assignments -- because the **Midihub's USB B -> MIDI B route was not
configured**, which cost several confused analyzer captures chasing a firmware ghost that
was not there.

So the actual product claim -- *four independent clock outputs, each with its own division,
swing and nudge, driving four separate pieces of gear* -- **has never once been
demonstrated end to end.** It is the pitch, and it is unproven.

The mio10's DIN outs are what make that demo possible.

## Do not repeat the Midihub lesson

The mio10 needs its internal routing configured (Auracle: USB port N -> DIN OUT N). It will
**not** "just work" any more than the Midihub did. Confirm the routing BEFORE concluding
anything about the firmware -- that mistake already burned a chunk of a bench session once.

## Scope

- [ ] Does the P4's USB host stack enumerate the mio10? (iConnectivity is class-compliant --
      it works driverless on iOS -- so this is expected, but expected is not measured.)
- [ ] Does the mio10 present multiple USB-MIDI virtual cables, and do they map to its DIN
      outs the way the Midihub's A..D do?
- [ ] **The demo:** enable all 4 outputs at DIFFERENT divisions (e.g. 24 / 12 / 48 / 4 PPQN),
      route each to its own DIN out, and verify on the analyzer that all four appear
      simultaneously with the correct ratios. This is the P4-034 fix's real acceptance test.
- [ ] Confirm `usbdrop` stays 0 with four outputs live (it is in /status; a non-zero value
      means the endpoint is refusing packets again).
- [ ] Per-output swing/nudge audible and correct across separate boxes.
- [ ] Document the mio10 alongside the Midihub as a supported interface.

## Also relevant to RTP-MIDI (P4-030 / P4-031)

The mio10 has Ethernet + RTP-MIDI, so it is a natural endpoint to test the network-MIDI path
against. But see the ESP-018 warning added to P4-031: wired Ethernet would put lwIP back on
the P4's own CPU, and that is exactly what preempted the S3's clock generator. Do not assume
the P4's 331us jitter survives the move. Separate ticket; noted here only so the connection
is not lost.

## Rig

The analyzer setup from P4-034 already exists: CH0 on the P4's DIN mirror (GPIO20), CH1 on
the downbeat strobe (GPIO21), a probe on a MIDI OUT jack's **pin 5** (data; pin 2 = ground --
pin 4 is +5V and reads flat high, which cost a capture last time).
