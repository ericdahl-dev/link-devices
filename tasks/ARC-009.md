## Problem (architecture review candidate D)

The pure engine (`beat_clock`, `clock_ticker`, `clock_output`, `transport`,
`metronome`, `metronome_voice`, `usb_midi_pack`, the four `link_*` modules) is the
**shared heart of both firmwares** but physically lives inside one target's dir,
`X32Link/`. KitchenSync reaches it via `#include "../../X32Link/foo.h"` and CMake
`../../X32Link` include-dirs; the test suite compiles it from there too. The
directory name lies about ownership — a reader sees "X32Link code" and doesn't know
KitchenSync depends on it. ADR-0005 says pure C is shared across targets; the layout
doesn't reflect it.

## Solution

Move the pure modules to a neutral `shared/` root (or `firmware/shared/`). Both
`KitchenSync/main/CMakeLists.txt` and `X32Link/`'s build reference `shared/`; the two
socket-glue files stay with their targets. Test Makefile `VPATH`/includes point at
`shared/`. Directory name now tells the truth: shared engine vs per-target glue.

## Deliverable

`shared/` holds the pure modules + tests; both firmware builds + host suite green
off the new paths; no `../../X32Link` include escapes remain. Pure move — zero
logic change.

## Notes

- Sequence with ARC-008: land the `link/` grouping first, then hoist `link/` +
  the rest together, so files move once, not twice.
- Watch the `test/Makefile` `all:`-prereq rule (every `test_*` in BOTH the
  prereq line and the recipe) when include paths change.

## Decision (2026-07-05): declined — see ADR-0006

Same blocker as ARC-008: arduino-cli compiles only the sketch root, so a neutral
`shared/` root would require symlinking every shared module back into `X32Link/`
(and `X32MidiClock/`). The shared pure C therefore stays in the `X32Link/` sketch
root by design; ownership is documented in `AGENTS.md`, not encoded in the path.
Reopen only if X32Link migrates off arduino-cli.

## Reconsidered (2026-07-16): see [ADR-0015](../docs/adr/0015-extract-ks-core-in-repo.md)

The 2026-07-05 decline weighed only the arduino-cli symlink cost. Three facts have since
shifted the calculus, so this is reopened as a **proposed** ADR (not yet accepted):

1. The **P4 (flagship) is ESP-IDF** and decouples for *free* — the CMake `X32LINK_DIR`
   becomes `SHARED_DIR`, one line, no symlinks. The decline never weighted the flagship
   building against a niche product's folder.
2. The **Touch is migrating to ESP-IDF** (ADR-0009 / ESP-026), so its 53 symlinks disappear
   regardless — bundling the move with that convergence (ADR-0015 Option A) is symlink-neutral.
3. The conflation has since **caused real defects** (ESP-040 X32Link `/save`, ESP-042 config),
   and the owner has clarified **X32Link is a separate product for a separate audience**.

ADR-0015 carries the trade-off analysis, sequencing options, and a migration plan.
