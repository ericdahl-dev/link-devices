## Problem (architecture review candidate D)

The pure engine (`beat_clock`, `clock_ticker`, `clock_output`, `transport`,
`metronome`, `metronome_voice`, `usb_midi_pack`, the four `link_*` modules) is the
**shared heart of both firmwares** but physically lives inside one target's dir,
`X32Link/`. P4Hub reaches it via `#include "../../X32Link/foo.h"` and CMake
`../../X32Link` include-dirs; the test suite compiles it from there too. The
directory name lies about ownership — a reader sees "X32Link code" and doesn't know
P4Hub depends on it. ADR-0005 says pure C is shared across targets; the layout
doesn't reflect it.

## Solution

Move the pure modules to a neutral `shared/` root (or `firmware/shared/`). Both
`P4Hub/main/CMakeLists.txt` and `X32Link/`'s build reference `shared/`; the two
socket-glue files stay with their targets. Test Makefile `VPATH`/includes point at
`shared/`. Directory name now tells the truth: shared engine vs per-target glue.

## Deliverable

`shared/` holds the pure modules + tests; both firmware builds + host suite green
off the new paths; no `../../X32Link` include escapes remain. Pure move — zero
logic change.
