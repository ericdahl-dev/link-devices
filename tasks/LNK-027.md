## Goal

When the device is following **Ableton Link** (tempo source = Link), also emit
**24-PPQN USB-MIDI clock** to a connected host, so a DAW or MIDI hardware plugged
into the device's USB port slaves to the Link session. This makes X32Link a
**Link → USB-MIDI clock bridge** — the complement of its two existing paths:
Link → OSC (mixer FX delay) and USB-MIDI-in → OSC. Same box, tempo flows out to
MIDI gear instead of only in.

"Link master mode" in the request: Ableton Link is peer-to-peer with no master,
so read this as *"whenever the runtime tempo source is Link, the device is the
MIDI-clock authority for whatever's downstream on USB."*

## Why

Today the only way to get Link tempo onto MIDI gear is a laptop running Link +
a MIDI interface. A tiny ESP32-S3 that already joins the Link session and already
enumerates as a USB-MIDI device could hand 0xF8 clock straight to a DAW / drum
machine / pedal, phase-locked to the bar via the LNK-018/019 measurement work.

## Design sketch (not binding)

- **Tick source:** derive the emit schedule from the existing seam —
  `tempo_source_bpm()` + `tempo_source_phase()` (LNK-019). Emit `0xF8` at 24
  ticks/beat; align the first tick to phase-zero so downstream gear locks to the
  bar, not just the tempo. Reuse the phase-valid gate (`tempo_source_phase_valid()`)
  so we don't spray ticks before Link is measured.
- **Pure, host-tested scheduler:** a `midi_clock_out.c` (mirrors `bpm_publisher.c`
  / `midi_bpm_calc.c`) that, given `(bpm, phase, now_us)`, returns how many ticks
  to emit this call and the next-due timestamp. Glue (`*_io.cpp`) does the actual
  TinyUSB MIDI write. Keep all timing math off the board so it's Unity-testable.
- **Jitter discipline:** MIDI-clock consumers are sensitive to per-tick jitter.
  Drive from a dedicated high-priority FreeRTOS task or a hardware timer, not from
  `loop()` polling — same lesson as the MIDI-clock-*in* waver fix (#33: decouple
  timing from loop jitter, widen/average where needed).
- **Transport (open):** `0xF8` free-run is the MVP. `Start`/`Stop`/`Continue`
  (0xFA/0xFC/0xFB) need a play-state source — Link's optional **StartStopSync** —
  which is out of scope for a first cut. Decide: clock-only, or wire StartStopSync.
- **Config:** a new NVS flag + web-UI toggle + touch-UI toggle, e.g.
  `midi_clock_out_enable`, only meaningful when `input_source == Link`.

## Open questions

1. **USB endpoint coexistence.** The USB-MIDI device is currently set up for the
   MIDI-*in* path (MCK-002). Is the MIDI IN (device→host) endpoint available/
   enumerated in **Link** mode? Enumeration order is handled (`tempo_source_pre_net()`
   runs USB before WiFi), but confirm Link mode actually brings USB-MIDI up, and
   that TX-to-host + RX-from-host can coexist if both are ever on.
2. **Transport scope:** clock-only (0xF8) vs full StartStopSync (Start/Stop/Continue)?
3. **Accuracy target:** what tempo/jitter tolerance counts as pass (e.g. a DAW
   slaved to the device tracks within ±0.1 BPM and no audible flam)?

## Acceptance Criteria

- [ ] New config: `midi_clock_out_enable` in NVS, exposed in the web config UI and
      the touch settings screen (only actionable when tempo source = Link).
- [ ] Emits 24-PPQN `0xF8` over USB-MIDI locked to `tempo_source_phase()`; first
      tick aligned to phase-zero.
- [ ] Tick-schedule math lives in a pure, host-tested module (`midi_clock_out.c`
      + `test_midi_clock_out.c` wired into `esp32/test/Makefile`).
- [ ] Verified on hardware: a DAW (Ableton/Logic) slaved to the device's USB clock
      tracks the Link session tempo within the agreed tolerance, no audible flam.
- [ ] No regression to USB-MIDI-*in* mode or to the Link → OSC path.
- [ ] `esp32/AGENTS.md` module table updated for the new module.

## Notes

- Complements, doesn't replace, the OSC output — a user could drive the mixer FX
  delay *and* clock external MIDI gear from one Link session simultaneously.
- Related: #33 (MIDI-in BPM stability) for the jitter mindset; LNK-018/019 for the
  phase API this rides on; LNK-021 (phase-locked LED) as the existing consumer of
  the same phase-zero crossing this would tick from.
