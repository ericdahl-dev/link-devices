## Goal

Wire `LNK-017`'s parsed `Timeline` and `LNK-018`'s `GhostXForm` together into
a real, bar-quantized **phase** reading exposed through the existing
`tempo_source` seam, so callers (the touch UI in `LNK-015`, and any future
consumer like the `docs/ideas.md` A1 "bar-quantized scene recall" idea) get
one clean API regardless of whether the active source is Link or MIDI.

## Phase math (confirmed: trivial once GhostXForm exists — no SDK-level `SessionState`/`Controller` machinery needed)

```c
// 1. Map our local clock into the session's shared GHostTime domain.
int64_t ghost_now_us = link_ghost_xform_host_to_ghost(xform, esp_timer_get_time());

// 2. Beats elapsed at this instant (Timeline.toBeats(), per LNK-017's parsed struct).
double beats_now = (timeline.beat_origin_micro / 1e6)
                  + (double)(ghost_now_us - timeline.time_origin_us) / (double)timeline.micros_per_beat;

// 3. Phase within a bar — modulo by quantum, with correct handling of negative beats_now.
double phase_now = fmod(beats_now, quantum);
if (phase_now < 0) phase_now += quantum;   // fmod() can return negative; shift into [0, quantum)
```

`quantum` (beats per bar, e.g. `4.0`) **never gossiped over wire** — local
const per app (confirmed `ableton/link/Phase.hpp`/`Controller.hpp`: per-call
arg, not session state). **Make it user-configurable now** (not hardcoded —
later UI work needs it):

- New `AppConfig` field `app_config.h`: `int quantum_beats;` range 1-16,
  default 4.
- `config_defaults()` sets 4. `config_validate()` rejects outside 1-16.
- NVS key `quantum_beats`, same load/save path as existing fields
  (`app_config_nvs.cpp`).
- Callers pass `g_config.quantum_beats` (not a `#define`) to
  `tempo_source_phase()`.
- `LNK-021`'s LED stays fixed at `tempo_source_phase(1.0f)` (per-beat, not
  per-bar) — quantum config doesn't affect it, don't wire it in there.

## API (`tempo_source.h`)

```c
// Phase within the current quantum (bar), range [0, quantum). Returns -1.0f
// if no valid phase reading is available yet (no GhostXForm for Link; no
// pulses received yet for MIDI).
float tempo_source_phase(float quantum);

// True once tempo_source_phase() can return a real value.
bool  tempo_source_phase_valid(void);
```

### Link adapter

Backed by LNK-017's `link_proto_timeline()` + LNK-018's
`link_measurement_current_xform()` via the math above.
`tempo_source_phase_valid()` is `link_measurement_active()` (a completed,
non-stale `GhostXForm`) AND `link_proto_timeline()` returning `true`.

### MIDI adapter

USB MIDI clock is **24 pulses per quarter note (PPQN)**, and unlike Link
this is an exact, locally-counted pulse stream — no clock-sync estimation
needed at all. `midi_clock.*` already tracks pulses for BPM derivation
(`midi_bpm_calc`); extend it to also track **pulse count modulo
`24 * quantum`** (i.e. modulo pulses-per-bar) so
`tempo_source_phase(quantum)` for MIDI is `(pulse_count % (24*quantum)) / 24.0`.
This is *more* precise than Link's phase (which carries clock-sync jitter)
since MIDI clock pulses are directly observed, not estimated — call this out
explicitly in code comments so it isn't assumed to be the lesser of the two.
`tempo_source_phase_valid()` for MIDI is true once at least one full bar's
worth of pulses has been observed since clock start (avoids a misleading
phase reading from a partial bar at stream start).

### `tempo_source_beat()` — leave as-is, but note its new relationship to phase

`tempo_source_beat()` (the existing per-beat blink, "MIDI forwards the real
clock's beat flag; Link synthesizes from BPM locally") remains unchanged —
it's still useful as a cheap, always-available heartbeat indicator that
doesn't require a completed Link measurement. Once `tempo_source_phase_valid()`
is true, a consumer (e.g. `LNK-015`'s touch UI) can derive a *more accurate*
beat-crossing event by watching `tempo_source_phase()` cross zero, instead of
relying on the locally-synthesized blink — document this distinction where
`tempo_source.h` declares both functions so future readers don't treat them
as redundant.

      this task just needs it to be directionally correct and not jump
      around erratically).
- [ ] Switching `input_source` between Link and MIDI at runtime (existing
      NVS field, LNK-012) correctly switches which phase implementation is
      active, with `tempo_source_phase_valid()` correctly reporting `false`

- [ ] `quantum_beats` field: defaults to 4, persists via NVS, rejected
      outside 1-16 by `config_validate()`. Host-tested in
      `test_app_config.c`.

## Notes

- This task changes the **clock source** used in the Link measurement path
  from `millis()`/`micros()` to `esp_timer_get_time()` (already established
  in LNK-018) — do not introduce a second, inconsistent clock read anywhere
  in the phase pipeline. `tempo_source_beat()`'s existing `millis()`-based
  synthesis is untouched and can keep using `millis()`; it's a separate,
  lower-precision indicator by design.
- Don't implement `requestBeatAtTime`/phase-encoded-beats scheduling
  machinery (`Phase.hpp`'s `toPhaseEncodedBeats`) — that's write-side
  scheduling for a Link *controller*, not needed for a read-only "what's the
  phase right now" client.
