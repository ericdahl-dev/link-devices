## Goal

Create the second firmware target's home: a `P4Hub/` sketch/app dir in this repo
that **compiles + flashes on the ESP32-P4-NANO** and **reuses the pure modules
unchanged** (the ADR-0003 payoff). This is the scaffold every other P4 feature
(P4-003…006) builds on — no product feature yet, just the skeleton + a proven
shared-module build.

## Why

P4-001 answered the toolchain question (Arduino-ESP32 3.3.8 `esp32:esp32:esp32p4`
compiles/flashes on the P4-NANO). The next foundation is a real dir with the
build wiring so the pure C engines (`clock_ticker`, `beat_synth`, `link_phase`,
`link_protocol`, `link_measurement*`, `app_config`, `bar`, `bpm_*`) come along
for free and keep their host Unity suites.

## Scope

- Decide dir layout: `P4Hub/` Arduino sketch (flat dir like `X32Link/`), with the
  shared C files **symlinked** in the same way `X32MidiClock/` shares X32Link's
  files. Confirm the symlink pattern works for a second consumer.
- Bare `P4Hub.ino`: boot, serial (`USBMode=hwcdc,CDCOnBoot=cdc` per the P4-001
  serial note), print firmware banner + which pure modules linked.
- Compile at least one pure engine into the P4 build (e.g. `clock_ticker`) to
  prove the shared-module reuse concretely — tick a fake beat clock, log pulses.
- CI: add a P4 firmware-compile job (arduino-cli `esp32:esp32:esp32p4`) mirroring
  the S3 compile job; host Unity suites already cover the pure modules.

## Deliverable

`P4Hub/` builds + flashes on the P4-NANO, links a pure engine, and a green CI
compile job. Follow-on features (P4-003 USB-MIDI host, P4-004 Link-over-WiFi,
P4-005 clock-out, P4-006 metronome) land in this dir.

## Progress (2026-07-05) — scaffold done ✅ (CI remaining)

Stood up `P4Hub/` and it builds + flashes + runs on the P4-NANO, driving the
**reused pure `clock_ticker` compiled unchanged from `../X32Link`** (single
source, relative-path SRCS — no symlink, no fork): logs 48 pulses/s at 120 BPM ×
24 PPQN, exactly as expected.

**Framework decision (supersedes the "Arduino sketch" assumption above):**
P4Hub is a **native ESP-IDF app**, not an Arduino sketch — the P4 hub features
(USB-MIDI host P4-003/005, WiFi-via-C6 P4-004) need ESP-IDF components the Arduino
core doesn't expose (`usb_host`, `esp_hosted`/`esp_wifi_remote`). The S3 X32Link
stays Arduino. ADR-0003 still holds (pure C shared unchanged; only glue differs).
Recorded in ADR-0005.

**CI (done):** added a `p4hub-compile` job to `.github/workflows/ci.yml` —
`espressif/esp-idf-ci-action` builds `P4Hub/` for `esp32p4` (ESP-IDF
`release-v5.5`), alongside the X32Link arduino-cli job. `idf.py` auto-pulls
managed components once the P4-004/005 glue lands.

**P4-002 done** — scaffold builds/runs on hardware + green CI. Feature work
(P4-004 listener + P4-005 clock-out) folds into `P4Hub/` next.
