## Why

The one safety-critical ordering rule in the clock engine — a MIDI **Start (0xFA) must
precede the downbeat clock (0xF8)** so a slave that starts from stopped is not one 24-PPQN
tick late (ESP-023, measured on the analyzer at 20.8 ms @ 120 BPM) — is the one thing the
pure plan does **not** own.

`ks_tick_step` returns two *separate* arrays, `plan.transport[]` and `plan.pulses[]`
(`ks_tick.h`), with no ordering encoded between them. The rule "transport first, then clock"
exists only as the **physical order of two `for` loops** inside each 1 ms writer task:

- P4: `ks_main.c` — transport loop, then clock loop.
- Touch: `ktouch_midi_out.cpp` — same rule re-expressed as loop order.

There is **no host test** asserting the order — `test_ks_tick.c` checks `transport[]` and
`pulses[]` *values*, never that Start is written before Clock, because the plan struct
doesn't model emission order. So the invariant is enforced twice, in prose + loop order, on
two frameworks, with zero coverage. It already diverged once: ESP-023 fixed the P4 and the
Touch silently kept the bug until it was found by reading the code (see ADR-0009). It is the
canonical "found once, fixed once, other copy keeps it" surface.

## What to build

Give `ks_tick` a pure, host-tested emitter that owns the order:

```c
// Ordered System Real-Time bytes for output `out` this tick: the 0xFA/0xFC transport
// edge (if any) FIRST, then plan.pulses[out] x 0xF8. ESP-023: the ORDER is the invariant.
// Returns byte count written (<= cap). Pure — the order lives here, tested here, once.
int ks_tick_out_bytes(const KsTickPlan* p, int out, uint8_t* buf, int cap);
```

Rewire both 1 ms writer glue tasks to call it and blast the returned bytes (still staged
into the P4's one USB batch, still mirrored to DIN). The two-loops-in-the-right-order
footgun disappears: glue becomes one loop over one ordered list.

This is **complementary to ADR-0009**, not a reopening of it: the ADR converges the whole
writer via a Touch→ESP-IDF migration; this makes the *ordering* pure and tested now, so that
migration has one less real-time invariant to re-implement by hand.

## Acceptance criteria

- [x] `ks_tick_out_bytes` exists, pure C, no framework headers.
- [x] Host test: a START tick with pulses emits `0xFA` strictly before the first `0xF8`.
- [x] Host test: NONE + 0 pulses emits nothing; STOP emits `0xFC`; `cap` never overflowed.
- [x] `ks_main.c` (P4) emits via the helper — its two order-dependent loops collapse to one
      routing loop; the invariant is now owned + tested, not implied by loop order.
- [ ] ~~`ktouch_midi_out.cpp` emits via the helper~~ — **deferred to ADR-0009 (see below).**
- [x] CI green for esp32 / esp32p4 / esp32s3.

## Scope note (2026-07-15)

Landed for the **P4 only**. The Touch has no `KsTickPlan` — it emits from
`transport_launch_step`'s `TransportLaunchOut.action` (`TL_START/STOP/RESTART/NONE`, including a
`RESTART` = Stop-then-Start case the P4 plan lacks), so `ks_tick_out_bytes` does not fit it.

Rather than build a shared cross-framework primitive and wire the Touch's `ktouch_midi_out.cpp`
to it, the Touch conversion folds into **ADR-0009**: that migration makes the Touch a board
target of KitchenSync and **deletes `ktouch_midi_out.cpp`**, routing the Touch through
`ks_main.c`'s writer — which, after this ticket, already owns the tested ordering invariant. So
the Touch inherits the fix at convergence instead of getting a throwaway shim now. The ESP-023
fix currently mirrored by hand in `ktouch_midi_out.cpp` stays correct until then.
