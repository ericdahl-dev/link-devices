## Problem

KitchenSync persists config as a **versioned blob decoded by a pure module**;
Touch and X32 **hand-serialize every field into NVS twice**. The gap is stark:

- **KitchenSync (deep seam).** `ks_config_save` is `nvs_set_blob(h, KEY, c, sizeof(*c))`
  + a version u32 (`ks_config_nvs.c:64-66`); load hands the raw bytes to the pure
  `ks_config_decode` (`ks_config.c:248-277`), which owns versioning, v1→v2
  migration (`ks_config.c:217-242`), and fail-closed validation. Adding a field
  touches struct/defaults/validate/set and **zero NVS lines**; a
  `_Static_assert(sizeof(KsConfig)==436)` (`ks_config.h:86`) turns a layout change
  into a compile error. Because persistence is a pure decode, the round-trip is
  **host-tested at the pure seam** (`test_ks_config.c:240-347`: current/legacy/
  size-mismatch/absent/invalid → defaults, and v1→v2 preserving creds) with no NVS
  fake.

- **Touch / X32 (shallow, drift-prone, untested).** Every field is hand-written
  twice — `config_load` (`prefs.getInt("quantum",…)`) and `config_save`
  (`prefs.putInt("quantum",…)`) — in `KitchenSyncTouch/app_config_nvs.cpp:9-34`
  and `X32Link/app_config_nvs.cpp:6-47`. A field therefore lives in **7 places**:
  struct, `DEFAULT_*`, `config_validate`, `app_config_set` switch,
  `AppConfigField` enum, `config_load`, `config_save`. Miss one `get`/`put` pair
  and the field silently fails to persist — no compile error, and **no round-trip
  test can exist on host** because `.cpp` NVS glue is excluded from the test
  harness and there is no Preferences fake (grep: no fake/stub/mock anywhere).

**Deletion test:** the persistence behaviour must stay, but the field-by-field
enumeration inside `app_config_nvs.cpp` is removable duplication — KitchenSync
proves the same job needs zero per-field NVS lines.

## Solution

Adopt KitchenSync's model on Touch and X32: save the config struct as one
versioned blob, and move all accept/reject/migrate judgement into a pure
`config_decode(blob, len, ver)` compiled into the host suite. The `.cpp` NVS glue
shrinks to blob get/set + commit and "does no judging of its own". Add the
`_Static_assert(sizeof(...) == N)` guard so a layout change is a compile error.
Round-trip and version/size-mismatch/absent → defaults become host tests at the
pure seam, mirroring `test_ks_config.c`.

## Notes

- Legitimately per-firmware and NOT in scope: the three `AppConfig`/`KsConfig`
  structs are different domains (mixer/model/fx vs MIDI-only vs multi-output
  clock+LED) — ADR-0008-adjacent; keep them separate.
- Migration reality: a blob format change from the current field-keyed layout is
  itself a v1→v2 transition. Either write a one-time migrator (read the old
  per-field keys once, write the blob) or accept a defaults-reset on first boot
  after the change — decide explicitly and note it, don't strand saved configs.
- Related idiom fork (out of scope here, flag for later): KitchenSync uses a
  string-keyed `ks_config_set(key,val)` that re-implements range checks
  `ks_config_valid` already owns (`ks_config.c:84-184`), while Touch/X32 use the
  deeper enum-keyed `app_config_set` with `config_validate` as the single range
  owner. Converging the setter idiom is a separate deepening.

## Done 2026-07-11

Both firmwares now persist config as **one versioned blob**, decoded by a pure,
host-tested `config_decode()`. A config field now lives in **4 places** (struct,
`DEFAULT_*`, `config_validate`, `app_config_set`/enum) instead of 7 -- the two NVS
lines per field are **gone**, and with them the silent no-persist bug: the old code
listed every field twice, so a missed `put`/`get` pair failed to save with no
compile error and no test that could ever catch it.

**The seam.** `config_decode(out, blob, len, version_present, version)` is the one
owner of "is this persisted blob safe to load?", fail-closed on every gate:
absent / unversioned / wrong-version / wrong-size / fails `config_validate` -> clean
defaults. `app_config_nvs.cpp` does no judging of its own; it hands over whatever
NVS held. Straight copy of the `ks_config_decode` model (P4-014).

**Migration -- the part that mattered.** The triage decision was *migrate, do not
defaults-reset*, because Dan's Touch (ESP-021) OTAs across this change and a silent
wipe of his WiFi creds and nudge trim is unrecoverable in the field. So
`config_load` reads the blob; if there is no blob but the **old field-keys** are
present, it reads them once, runs them through `config_validate` (a migration is not
a licence to skip the guard), and writes the blob back. Next boot takes the blob path
and the legacy keys are never read again. A fresh flash -- no blob, no legacy keys --
just gets defaults.

This migrator could NOT go in the pure decode: the old format is not a blob, it is a
set of Preferences keys, so only the glue can see it. Noted in both headers so nobody
"fixes" that later.

**Tests (TDD, pure seam):** 6 new decode tests per firmware -- round-trip, absent
blob, no-version, wrong-version, size-mismatch, invalid-contents. All written red
first. Touch 12/12, X32 44/44, full suite **490 tests / 0 failures**.

**One real bug the host tests could never have caught.** `_Static_assert` is C11;
these headers are compiled as C by the host suite *and as C++* by the Arduino sketch,
where g++ rejects that spelling. The C tests were green while the firmware would not
build. Only compiling the firmware found it -- fixed with an `#ifdef __cplusplus`
macro that spells the assert both ways. Worth remembering: **green host tests do not
mean the firmware compiles.**

Compiles: KitchenSyncTouch 6% flash, X32Link 82% (headless + Waveshare variants).

## Hardware verification 2026-07-11 — the migrator, on real silicon

Run on the bench Touch board (ESP32-S3, 16MB, MAC `a4:cb:8f:da:df:d0`), which was
already carrying a **pre-ARC-020 config in the old field-keyed NVS** (fw 2.2.0,
joined to WiFi `Skeyelab`). Flashed with the SAME `PartitionScheme=custom` table, so
NVS was NOT relocated — a scheme change would have wiped it and made the test
meaningless.

**1. Migration preserves the config.** Stamped a distinctive fingerprint through the
OLD firmware first (quantum 3 bars, nudge **-35**, bright **45**, transport/clock/
play-on-release all on), then flashed the ARC-020 build:

| field | before (field-keys) | after (blob) |
|---|---|---|
| wifi_ssid + **wifi_pass** | `Skeyelab` | rejoined the LAN in 3s |
| quantum / nudge / bright | 3 / -35 / 45 | 3 / -35 / 45 |
| clock / transport / play_rel | 1 / 1 / 1 | 1 / 1 / 1 |

The **password is the real proof**: nobody typed it in, it was only ever in the old
NVS keys, and the unit came straight back onto `Skeyelab`. Had the migrator failed,
the creds would be gone and the board would have dropped to `KSTouch-Config` AP mode
— which is exactly the silent, unrecoverable-in-the-field reset this ticket existed
to prevent.

**2. The blob path actually takes over (the migrator does not re-run).** A plain
reboot proves nothing here: the legacy keys are still sitting in NVS, so a firmware
that re-migrated on every boot would look identical. So instead: saved **new** values
through the ARC-020 firmware (quantum 5, nudge **+77**, bright **60**) — which go to
the blob — and rebooted. The legacy keys still said -35/45/3.

Result: **77 / 60 / 5**. It read the blob. A re-migrating build would have reverted to
the legacy values. It did not.

Both halves hold. The remaining risk for Dan's unit (ESP-021) is not this change —
it is **repartitioning**, which relocates NVS and *would* wipe the config, and cannot
be done over OTA. Do any partition change before the unit ships, not after.
