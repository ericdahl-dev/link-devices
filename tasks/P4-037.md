## Symptom

The KitchenSync web UI stops answering. The device is otherwise **fine** — it pings, the
clock keeps running, serial keeps printing `link peers 1 bpm 120.00 phase locked ... play 1`.
Only HTTP is dead, and it stays dead until reboot. Serial shows the tell:

```
E (831000) httpd: httpd_accept_conn: error in accept (23)
W (831000) httpd: httpd_server: error accepting new connection
```

errno 23 is `ENFILE` — out of sockets.

Found while verifying ESP-019 on hardware. Reproduced in seconds: open `/` in a desktop
Chromium **once**, and the server is gone.

## Cause

**The device's whole socket budget is too small for its tenants.** `CONFIG_LWIP_MAX_SOCKETS`
was **10**, shared by everything on the box:

| tenant | sockets |
|---|---|
| httpd (1 listener + `max_open_sockets` 7) | 8 |
| Link multicast (`wifi_link.c`) | 1 |
| Link measurement (`link_measure_io.c`) | 1 |
| mDNS | ~2 |
| **total** | **~12** — against a budget of 10 |

A desktop browser opens up to **6 parallel keep-alive connections** to one host, so a single
visit is enough to walk LWIP off the end. `accept()` then fails with `ENFILE` and the server
never recovers. The phone never showed it: mobile Safari opens far fewer parallel connections.

## The wrong fix (tried first, does NOT work — verified on hardware)

`hcfg.lru_purge_enable = true` alone. It looks right and it is not sufficient: **LRU purge
only fires when httpd's *own* pool is full**, and here LWIP runs dry *before* that, so
`accept()` errors out and the purge path is never reached. Flashed it, and one Chromium visit
still killed the server.

Raising `max_open_sockets` is also not available — it is capped at `CONFIG_LWIP_MAX_SOCKETS - 3`,
so with a budget of 10 the existing 7 was already the ceiling. The pool was never the problem.

## The actual fix

`CONFIG_LWIP_MAX_SOCKETS=16` in `KitchenSync/sdkconfig.defaults` — give the box enough sockets
for its real tenants. `lru_purge_enable = true` is kept alongside it as defence in depth: it
is what stops httpd's own pool from refusing a client once the pool (not LWIP) is the thing
that fills.

## Acceptance criteria

- [x] Socket budget covers the real tenants (httpd + Link + Link-measure + mDNS)
- [x] `lru_purge_enable = true` on the KitchenSync httpd
- [x] The server survives a desktop-Chromium page load and keeps answering `/status`
- [x] The server survives repeated aborted requests (curl killed mid-page) without wedging
- [x] No `accept (23)` in the serial log through the above

## Outcome (2026-07-11)

Verified on the P4 (FW 2.2.0, Link-locked to Ableton at 120 BPM).

**Before:** fresh boot answered `/status`; **one** Chromium page load then wedged the server
permanently — `curl` got nothing, the device still pinged, the clock still ran, and serial
printed `httpd_accept_conn: error in accept (23)`.

**After (`LWIP_MAX_SOCKETS=16`):** the same Chromium page load leaves the server answering
`/status` normally, the page keeps live-polling (BPM readout updating), and a follow-up burst
of 10 deliberately-aborted `curl`s does not wedge it either. No `accept (23)`.

Related: same shape as P4-034 — a real failure hiding behind a device that looks healthy on
every easy check (pings, clocks, phase-locks). The web UI was the only thing dead, and the web
UI is now (ESP-019) a performance transport surface, so this is a stage failure, not a nuisance.
