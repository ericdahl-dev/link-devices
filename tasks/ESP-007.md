## Goal

Stage 2 of P4-020's original vision: when Follow Beat's detected tempo is confident,
let it actually drive the Link session (and every clock output), not just display it.
This is the "differentiating feature" P4-020 originally described — clap/drum into
the mic and the whole rig follows.

## Context

P4-020 shipped v1 (detect + display only) on 2026-07-09, hardware-validated within ~1%
accuracy across 108-183 BPM. That work deliberately stopped short of touching
`beat_source`/the clock pipeline — see
`docs/plans/2026-07-09-p4-020-follow-beat-design.md`'s "Explicitly out of scope (v1)"
section. This ticket picks up from there.

Depends on P4-011 (MIDI clock IN → Link) because both are the same shape of problem:
an external tempo source needs to become the session's master, arbitrated against
Link's own peers. Whatever tempo-source-arbiter seam P4-011 builds, this should
reuse rather than inventing a second one.

## Scope

- Wire `follow_beat_io_status()`'s confident BPM into whatever tempo-source arbiter
  P4-011 establishes (or `beat_source`, if P4-011 lands differently than expected).
- Web UI: a source selector (Link / MIDI-in / Follow-Beat) instead of Follow Beat
  being a passive status-only row.
- Decide fallback behavior: what happens when Follow Beat's confidence drops mid-session
  (does it hold last tempo, fall back to Link, require explicit reselection?).
- Real hardware validation: clap/drum into the mic and confirm the Link session tempo
  actually tracks it, with other Link peers and clock outputs following.

## Acceptance Criteria

- [ ] Confident Follow Beat tempo can become the Link session's master tempo
- [ ] Web UI has a tempo-source selector including Follow Beat
- [ ] Defined, tested fallback behavior when confidence drops
- [ ] Hardware-validated: real drumming/clapping drives the whole rig's tempo

## Notes

- Known algorithmic limitation carried over from P4-020: the octave-error guard in
  `follow_beat.c` only corrects toward shorter lags, so strongly syncopated/backbeat-
  heavy audio can bias toward reporting 2x tempo. Worth stress-testing before this
  becomes the actual session master (a wrong-but-displayed BPM is annoying; a
  wrong-and-driving BPM is disruptive).
