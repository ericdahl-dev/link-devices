## Problem

Both `P4Hub` and `X32Link` store exactly one WiFi network
(`P4HubConfig.wifi_ssid`/`.wifi_pass`, `AppConfig.wifi_ssid`/`.wifi_pass` —
single fixed-size fields, not a list). The user regularly moves these devices
between (at least) three networks:

- **home** ("Skeyelab")
- **work** ("ND-guest" — open network, no password)
- **live sound rig** ("Skeyelab-Gigs" — WiFi only, no actual internet uplink)

Today, switching locations means reconfiguring via the web UI and rebooting
every time — and until P4-018 landed, P4Hub couldn't even reach its config
screen again without a factory-reset/NVS wipe if the previously-configured
network wasn't reachable. P4-018 fixes the *get back to config* half of this;
this ticket is the *don't have to reconfigure every time* half.

## Goal

Store a short list of known networks (SSID + password, maybe 3-5 slots — this
is a config problem, not a firmware-updates-daily problem, so no need to
over-engineer capacity) and, at boot, try them — connecting to whichever one
is actually in range, without user intervention.

## Design questions to resolve before implementing

- **Selection strategy**: scan for visible SSIDs first and only attempt
  networks that are actually in range (avoids a slow serial timeout-per-entry
  loop), vs. just trying each stored entry in order with a short per-entry
  timeout. Scanning first is very likely the right call — cleaner UX, and
  P4-018's `wifi_fallback` timeout logic would still apply if *none* of the
  stored networks are visible.
- **Config UI**: the existing single-SSID form (`p4hub_form.c`/`p4hub_web.cpp`
  `/save`, and X32Link's `web_config.cpp` equivalent) needs multiple
  SSID/password slots instead of one — needs a small UI redesign (list + add
  new), not just a bigger struct.
- **Storage**: NVS blob size for N `{ssid[33], pass[64]}` entries is trivial,
  but `P4HubConfig`/`AppConfig`'s `nvs_get_blob`/`_set_blob` round-trip and
  `config_validate()`-style checks need to handle a variable-ish list rather
  than two fixed scalar fields — check whether the existing blob-versioning
  approach (if any) needs a migration path for boards already in the field
  with the old single-SSID config shape.
- **Cross-firmware scope**: this affects both `P4Hub` (`p4hub_config.h`,
  `wifi_link.c`) and `X32Link` (`app_config.h`, the wifi connect path in
  `X32Link.ino`). Reuse of a shared pure "pick a network from a list of
  {known SSIDs} × {scan results}" helper across both firmwares would fit this
  repo's existing reuse pattern (link_protocol.c, link_measurement.c etc. are
  already shared between them) — worth designing once, not twice. Consider
  filing a matching `LNK-035` ticket (next free LNK id at time of writing)
  when this is actually picked up, rather than trying to cover both firmwares'
  full implementation in one ticket.

## Acceptance Criteria (implementation not started — ticket only)

- [ ] Design decision recorded on selection strategy (scan-then-connect vs.
      try-each-in-order) and storage shape (list size, NVS layout, migration
      from existing single-SSID configs already deployed).
- [ ] Config web UI updated on at least one firmware to add/edit/remove
      multiple network entries.
- [ ] Boot sequence tries stored networks (via whichever strategy was chosen)
      before falling through to P4-018/X32Link's existing AP-config fallback
      if none are reachable.
- [ ] Host-tested wherever the selection logic can be made pure (matching
      this repo's pure-logic/thin-glue convention) — the WiFi scan/connect
      calls themselves stay hardware-coupled and untested, same as today.
- [ ] Verified on real hardware across at least two of the three known
      networks (e.g. confirm it picks "ND-guest" over "Skeyelab" when only
      the former is in range, and vice versa at home).

## Notes

Filed directly from the user's own use case, not speculative: they carry
these boards between exactly the three networks named above. No urgency to
gold-plate beyond that — a fixed small list (e.g. 4 slots) comfortably covers
it without needing a "sync from phone" or "connect via BLE" style feature.
