## Goal

Scope what the device could do with mixer FX beyond today's single hardcoded
target. Today (LNK-005) we send BPM → `/fx/<slot>/par/01 ,f` (delay-time param)
to **one** configured slot, send-only, assuming the slot holds a delay. This
ticket researches four questions before any implementation:

1. **What can we read from the mixer about the FX slots?**
2. **What other parameters make sense to tempo-control?**
3. **How to handle multiple tempo-syncable effects at once** (e.g. a delay in one
   slot and another effect with its own tap-tempo/rate in another)?
4. **Should we control more than one FX slot?**

Output is a findings doc + a design recommendation, **not** code.

## What we already know (grounding)

- `/fx/N/type ,i` — the loaded **algorithm** per slot, index `[0,60]`
  (`docs/xr18-xair-osc-cheatsheet.md`). This is the key introspection hook.
- `/fx/N/par/MM ,f` — per-algorithm parameters, `MM = 01..64`. Which `MM` is the
  tempo/delay-time param **depends on the algorithm** (par/01 happens to be delay
  time for the delay algorithms we target now).
- We are currently **send-only** and never read `/fx/N/type` — so we're blind to
  what's actually loaded and just assume par/01.
- XR18 has 4 FX slots, X32 has 8 (`config_model_slot_max`).
- The receive/subscribe plumbing exists (`osc_in`, xremote — LNK-013), so reading
  back mixer state is feasible, we just don't use it for FX.

## Research questions

### Q1 — FX-slot introspection
- Read `/fx/N/type` for each slot → know the algorithm. Can we also get a
  human-readable name, or only the index? Build the **index → algorithm** table
  for XR18 and X32 (they may differ).
- For each tempo-relevant algorithm, which `par/MM` is the tempo/delay-time param,
  and what are its **units/encoding** (normalized float → ms? a note-division
  enum? a raw ms value?). Does the XR/X32 FX expose a "tempo sync on/off" or a
  "division" param we should set instead of writing a raw time?
- Can/should we **subscribe** (xremote) so we track slot type changes live and
  avoid writing a delay-time to a slot the user just changed to reverb?

### Q2 — which params are worth controlling
- Catalog tempo-syncable params across algorithms: delay time (delays), LFO rate
  (chorus/flanger/phaser), reverb pre-delay, tremolo/auto-pan rate, gate/ducker
  timing. Note which are musically useful to lock to Link tempo vs noise.
- Non-tempo params worth a look while we're here? (probably out of scope — keep
  this ticket tempo-focused, list others as future.)

### Q3 — multiple tempo-syncable effects at once
- If a user has a delay in slot 1 and, say, a tremolo/tap-tempo effect in slot 3,
  both want the tempo. The device is the tempo authority — we'd write each slot's
  tempo param directly (overriding the mixer's own tap-tempo). Confirm that's the
  right model vs trying to drive the mixer's tap-tempo input.
- Different targets may want **different note divisions** (1/4 vs 1/8-dotted vs
  1/2). Implies a per-target division/multiplier, not one global BPM→ms.
- Encoding differs per algorithm (ms float vs division enum) — the per-target
  mapping must carry the algorithm's param index + unit conversion.

### Q4 — multi-slot control
- Extend config from a single `fx_slot` to a set of targets: each `{slot,
  enabled, division, (auto-detected type → param index + unit)}`.
- Config/UI implications: the web + touch UIs, NVS schema, and validation grow
  from one slot to N. How much of the mapping can be **auto-derived** from
  `/fx/N/type` so the user just picks slots + divisions?
- Rate-limit / OSC-flood concerns when writing several params on every tempo
  change (reuse LNK-011 bpm_publisher send-policy).
