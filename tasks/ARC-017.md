## Problem

ADR-0005 says the two firmware frameworks are fine because "the glue is thin."
That holds everywhere except the web layer, which is the one place where
duplicated *content* — not duplicated glue — has accumulated.

`X32Link/web_config.cpp` (485 lines, Arduino `WebServer`) and
`KitchenSync/main/ks_web.cpp` (583 lines, `esp_http_server`) each carry a full
copy of the rack-panel look and the client-side plumbing. `ks_web.cpp:5` says so
outright: *"CSS/aesthetic lifted from X32Link/web_config.cpp."*

To be precise about what is and isn't duplicated, because this is easy to
overstate:

**Not duplicated — leave alone.** The two `<form>` bodies are different
products. X32Link's is tempo source / mixer model / FX slot / bar quantum /
mixer IP; KitchenSync's is 4× clock output (cable, PPQN, nudge, swing), a
metronome, and an LED strip, plus a 4-across status meter bridge. There is no
shared config page to extract, and pretending otherwise would produce a
parameterized monster. Likewise the two servers (`WebServer` vs
`esp_http_server`) and the two OTA paths (`Update.h` multipart vs `esp_ota_ops`
raw-body streaming) are correctly framework-specific.

**Duplicated — extract.** Roughly 260 lines:

| What | X32Link | KitchenSync |
|---|---|---|
| CSS block (`:root`…`@media prefers-reduced-motion`) | `web_config.cpp:34-109` | `ks_web.cpp:44-130` |
| `send_result()` dark page + reboot-poll script | `web_config.cpp:339-359` | `ks_web.cpp:365-381` |
| `/update` page card | `web_config.cpp:271-295` | `ks_web.cpp:455-491` |
| `setBeat` / `showBpm` / `poll` / `postLive` JS | `web_config.cpp:187-236` | `ks_web.cpp:204-226` |

It has already drifted. `--mut` is `#717a82` on the S3 and `#838d95` on the P4.
The `.sw .swlbl::after` content is `"Off"` on one and `" Off"` (leading space) on
the other. Nobody decided either of those. Nothing in CI compares them.

This is the same shape as the `/status` JSON, which we already fixed correctly:
`web_status_json.c` (10 lines) and `ks_status.c` (9 lines) are pure, host-tested
builders, and the glue just calls them. The markup never got the same treatment.

## Solution

A pure C `ui_chrome.{h,c}` in `X32Link/`, picked up by KitchenSync through the
existing relative-path `SRCS` + `X32LINK_DIR` include path in
`KitchenSync/main/CMakeLists.txt` — the same way it already compiles
`clock_ticker.c`, `link_protocol.c`, and the rest of the shared pure C. No
Arduino headers, no ESP-IDF headers, just `snprintf`. Host-tested in
`test/test_ui_chrome.c`.

Sketch of the interface — each function is a `snprintf`-shaped builder returning
what `snprintf` returns, so the caller can detect truncation. Same contract as
`ks_status_json()`:

```c
// ui_chrome.h — shared web chrome for both firmwares' config UIs (ARC-017).
// Pure: no Arduino/ESP-IDF deps. The *forms* stay per-firmware; this owns the
// look and the client-side plumbing that both pages share.

// The panel stylesheet, contents of a <style> block (no tags). Static storage,
// never NULL. One definition of --mut, .unit, .sw, .write, @keyframes breathe.
const char* ui_chrome_css(void);

// The shared client-side JS, contents of a <script> block (no tags):
// setBeat / showBpm / postLive, and a poll() that fetches /status and calls a
// page-supplied onStatus(d) hook. Each firmware appends its own <script> with
// the hook + any page-specific wiring (X32Link's phase-lock anchor, the P4's
// syncSect + nudge steppers). Static storage, never NULL.
const char* ui_chrome_js(void);

// Result page shown after Write & Reboot / OTA. `reboot` appends the
// come-back-and-redirect-to-/ poller. Replaces both send_result()s.
int ui_result_page(char* buf, size_t len,
                   const char* title, const char* body, bool reboot);

// GET /update page. `product` is the <title> / heading prefix ("X32·SYNC",
// "KitchenSync"); `fw`/`build` are FW_VERSION / FW_BUILD, passed in so the
// builder stays pure (same reason ks_status_json takes `fw`).
// `multipart` picks the form: true -> <form enctype=multipart> (Arduino
// Update.h), false -> the fetch(body:file) uploader (esp_ota_ops).
int ui_update_page(char* buf, size_t len, const char* product,
                   const char* fw, const char* build, bool multipart);
```

The two `<style>`/`<script>` blocks are no longer inside the page template, so
each firmware assembles its page from parts and sends it **chunked** — Arduino's
`server.sendContent()` / IDF's `httpd_resp_send_chunk()`. That is a side benefit
worth naming: `build_html()` today copies a ~9 KB template into an Arduino
`String` and runs eight `.replace()` passes over it, each a realloc. Chunked
assembly drops the largest heap allocation on the S3's web path entirely.

      Arduino/ESP-IDF headers.
- [ ] `test/test_ui_chrome.c` wired into `test/Makefile`; asserts each builder's
      truncation return, that `ui_result_page(reboot=true)` contains the retry

- [ ] `KitchenSync/main/CMakeLists.txt` lists `${X32LINK_DIR}/ui_chrome.c`.
- [ ] `check_docs.sh` passes; `ui_chrome.c` named in the `AGENTS.md` module map.
- [ ] Host suite green.
- [ ] Hardware check on both targets: config page renders identically to today
      (screenshot diff by eye), `/status` still drives the beat dot, `/live`
      still applies without reboot, `/update` still flashes and reboots.

## Notes

- **Resolve the drift explicitly, don't just pick one.** `--mut`: the S3's
  `#717a82` vs the P4's `#838d95`. The P4's is the later value and lighter —
  likely a deliberate legibility fix on the bigger 900px card that never got
  back-ported. Confirm on hardware, then one value wins. Same for the `.swlbl`
  leading space.
- The P4's CSS has a `@media (min-width:760px)` desktop block the S3 lacks, and
  the S3 has `rise` entry animations the P4 lacks. Neither is shared chrome —
  keep them in the per-firmware `<style>` that follows `ui_chrome_css()`.
  Cascade order: shared block first, page block second.
- `poll()` differs meaningfully: X32Link phase-locks off `{phase,valid,quantum}`
  and interpolates with rAF (LNK-022); KitchenSync free-runs off `bpm` and also
  paints peers/usb/min/tx. The shared part is `setBeat`/`showBpm`/`postLive` and
  the fetch loop; the divergence is the `onStatus(d)` body. Hence the hook, not
  a flag. Resist growing `ui_chrome_js()` a `product` switch.
- Deliberately **not** in scope: unifying the servers, the OTA transports, or the
  form bodies. Those are the framework-specific glue ADR-0005 says we accept.
- This does not change ADR-0005. It restores its premise: after this, the
  duplicated surface between the two firmwares is ~250 lines of genuinely
  irreducible IO glue (`app_config_nvs.cpp` / `ks_config_nvs.c`,
  `link_measurement_io.cpp` / `link_measure_io.c`, `midi_clock.cpp` /
  `midi_clock_in.c`+`usb_midi_host.c`), which is what "two thin glue layers"
  was supposed to mean.

## Progress

  sharing (1,024 lines compiled into both via `X32LINK_DIR`) is holding; the web
  chrome is the one place ADR-0003's discipline was never applied, and the
  `--mut` colour drift is the first symptom.
