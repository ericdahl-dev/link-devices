## Goal

`web_config.cpp`'s `.beat` dot is free-running JS, same flaw the LED had
pre-`LNK-021`: `setBeat(bpm)` does `setInterval(..., 60000/bpm)`, no real
phase, just BPM-rate blinking from an arbitrary local start. Bring web UI to
parity with `LNK-021` (LED) and `LNK-015` (touch wheel) ‚Äî real
`tempo_source_phase()`, not synthesized.

Also add the new `quantum_beats` field (LNK-019) to the web settings form,
matching the touch UI's control (LNK-015) ‚Äî same config, both front ends.

## Design ‚Äî poll-correct + client-side interpolate

HTTP poll (`/status`, currently 1Hz) is too coarse to *drive* a smooth
beat-synced dot directly ‚Äî animate locally, resync each poll:

- `/status` JSON adds `"phase": F, "valid": bool, "quantum": F` (alongside
  existing `"bpm"`).
- JS: on each poll, if `valid`, snap a local `phaseAtPollMs` anchor (phase +
  `Date.now()`). Between polls, extrapolate: `phase_now = (phaseAtPoll +
  (Date.now()-pollMs)/1000 * bpm/60) % quantum`. Drive the dot from this
  extrapolated value via `requestAnimationFrame`, not `setInterval`.
- If `valid` is `false`: fall back to existing `setBeat(bpm)` free-running
  blink (same as today) ‚Äî don't show a fake phase dot.

## Implementation

- `handle_status()` in `web_config.cpp`: add `phase`/`valid`/`quantum` to
  the JSON. Needs `tempo_source_phase()`/`tempo_source_phase_valid()`
  (LNK-019) accessible here ‚Äî same pattern as existing `g_current_bpm`
  extern.
- Settings form: add a `quantum_beats` field (number input or a small
  segmented control, 1-16) next to FX Slot; `handle_save()` validates and
  writes it through `config_validate()`/`config_save()`, same as other
  fields.
- `.beat` dot CSS/JS: replace `setBeat()`'s `setInterval` with a
  `requestAnimationFrame` loop computing extrapolated phase per the design
  above. Keep the dot's visual (size/color/glow) unchanged ‚Äî only the
  timing source changes.

## Acceptance Criteria

- [ ] `/status` returns `phase`/`valid`/`quantum` correctly.
- [ ] Web beat dot visibly phase-locks to a real Link/MIDI session (same
      manual check as `LNK-020` step 7, just for the web page instead of
      the LED) ‚Äî verify side-by-side against the touch UI's phase wheel on
      the same unit, both should visually agree.
- [ ] Settings form's quantum field round-trips through save+reboot,
      matches the touch UI's value (shared NVS).
- [ ] Pre-sync fallback (free-running blink) still works exactly as before
      when `valid` is `false`.
- [ ] No regression to existing `/status` consumers ‚Äî `bpm` field unchanged
      in meaning/units.

## Notes

- Don't tighten the `/status` poll interval below 1s just for this ‚Äî the
  client-side extrapolation is what makes 1Hz polling look smooth; polling
  faster adds server load for no visual gain.
- Mirrors `LNK-021`'s "fallback during sync gap, snap to real phase once
  valid" pattern ‚Äî same shape, different layer (JS instead of FreeRTOS
  task).
