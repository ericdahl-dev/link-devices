## Why

**A client cannot speak to both devices.** The new iOS companion app
([KitchenSync-iOS_App](https://github.com/ericdahl-dev/KitchenSync-iOS_App)) was built against
`KitchenSync/main/ks_web.cpp`. Pointed at a real Touch on the bench (`kstouch-dfd0`, 2026-07-14),
its `KsStatus` decoder **throws on every required key**. Not degrades — throws.

### This is drift, and we can prove it

`KitchenSyncTouch/` already symlinks **46 modules** from `X32Link/` — `beat_clock`, `beat_source`,
`clock_output`, `link_measurement`, `transport_led`, `config_persist`, `swing`, `link_phase`…
**The clock engine is already one codebase.** The two products are not different in the thing that
matters.

The **only** layer that diverged is web/status presentation — because it is the one piece nobody
shared. `ktouch_web.cpp:252` hand-rolls its own `snprintf`; `ks_status.c` has its own. Two
implementations of the same document.

The tell: both emit the same nine tick-health fields, but Touch spells them `wbeats`/`wclock` and
KitchenSync spells them `w_beats`/`w_clock`. Nobody chooses that. And `ks_status.c` states the
intent out loud:

> *"in X32Link's exact key names (ARC-024) — one shape across the fleet, so one script audits every
> device."*

**We have already paid for this twice.** ESP-028's own text: *"P4-039 did exactly this for the P4.
The Touch never got it."* The same `/status`-lies-about-the-clock bug had to be found and fixed
separately on each firmware, and the Touch's copy went unfixed long enough to cost **138 seconds of
silent DIN clock**. That is the duplication tax, paid in a real failure.

## Approach: share the module, don't reimplement the document

**Do NOT add the missing fields to `ktouch_web.cpp`.** That is a second implementation and it will
drift again — it is exactly how we got here.

`ks_status.c` is **pure C**: includes only `<stdio.h>`, takes plain parameters, writes a buffer. No
ESP-IDF. Already host-tested (`test/test_ks_status.c`). `arduino-cli` compiles plain C fine, and
the 46 existing symlinks prove the mechanism works across the framework boundary.

1. Move `ks_status.c` / `ks_status.h` into `X32Link/` (the shared-module home — it already hosts
   `web_status_json.c`, which `ks_status.h` includes).
2. Symlink into `KitchenSyncTouch/`, exactly like the other 46. KitchenSync already compiles with
   `X32Link/` on its include path (ADR-0003).
3. `ktouch_web.cpp`'s status handler calls `ks_status_json(...)` and **deletes its own snprintf**.
4. Feed it the Touch's real values. Where the Touch genuinely lacks the hardware, report the honest
   value — do NOT fake one. A `usb:true` on a device with no USB-MIDI is the same class of lie as
   ESP-028's `sync:1` over a dead wire.

Result: one implementation, one host-test suite, identical JSON on both devices. Drift becomes
**impossible**, not merely discouraged.

### `launch[]` on a one-output device

Emit `launch` with a length equal to the ACTUAL output count — `launch:[0]` on Touch,
`launch:[0,0,0,0]` on KitchenSync. **Do not pad Touch to four.** A client renders a card per
element and would draw three dead outputs. (`ks_status_json`'s signature currently takes
`const int launch[4]`; it needs a count parameter.)

The app is being changed to drive off array length rather than a hardcoded 4
(KitchenSync-iOS_App T-015). Emitting the honest length is what makes that possible.

**Good news:** Touch's `/transport` ALREADY has the quantized model —
`POST /transport?play=1  PLAY (arms; fires on the next bar line)` / `play=0  STOP (immediate)`. So
this is *exposing state the firmware already tracks*, not inventing a transport concept.

### KitchenSync should gain `clk` / `pulses`

Converge on the **union**, not on either side. Touch has ESP-028's writer's-truth fields (`clk`:
locked / free / **silent**, and `pulses`); the P4 does not. Those are the highest-value diagnostic
in the fleet — they make "is the wire actually alive?" answerable without a logic analyzer. Sharing
`ks_status.c` is what lets both get them for free.

## Acceptance criteria

- [ ] `ks_status.c` lives in `X32Link/` and is symlinked into `KitchenSyncTouch/`
- [ ] `ktouch_web.cpp` has NO status `snprintf` of its own — it calls `ks_status_json()`
- [ ] One client decodes BOTH devices' `/status` with no per-device special-casing
- [ ] `launch[]` length == actual output count (1 on Touch, 4 on KitchenSync) — never padded
- [ ] Tick-health keys identical on both (`w_beats`, `w_clock`, `reprime`)
- [ ] `clk` / `pulses` present on BOTH devices
- [ ] Touch's own web UI still works — its JS reads these keys, so renaming `wbeats` → `w_beats`
      breaks it silently (the pill just stops updating). Change both in the same commit.
- [ ] Fields for hardware a device lacks are reported honestly, never faked
- [ ] Host tests pass (`test/test_ks_status.c`) — one suite now covers both firmwares
- [ ] Verified against the real iOS app on the bench: the Touch appears, decodes, and its transport
      button drives a real output

## Notes

**The bench unit `kstouch-dfd0` was reflashed to current `master` on 2026-07-14** and now matches
source — `/status` carries `clk:"silent"` and `pulses:0` (ESP-028 live, and telling the truth: no
Link peer, so no clock). It is a trustworthy test target now. It was NOT before.

Build/flash for the record:

```sh
arduino-cli compile --fqbn 'esp32:esp32:esp32s3:USBMode=hwcdc,CDCOnBoot=cdc,FlashSize=16M,PSRAM=opi,PartitionScheme=custom' KitchenSyncTouch
arduino-cli upload  --fqbn '<same>' -p /dev/cu.usbmodem2101 KitchenSyncTouch
```

⚠️ **Identify the board before flashing.** Several ESP32-S3s may be attached at once. The Touch is
the one whose MAC's last two bytes match its mDNS name: `a4:cb:8f:da:df:d0` → `kstouch-dfd0`.
Confirm with `esptool --port <p> chip_id`. Flashing the product's 16 MB partition table onto the
wrong board boot-loops with no app and no banner, and looks exactly like a brick (AGENTS.md).

**Longer term this whole layer may go away** — see **ESP-032**. A P4 board with a touch LCD would
let Touch collapse into KitchenSync outright. That is another reason not to invest in a second web
implementation: the pure modules survive that migration; the framework wrappers do not.
