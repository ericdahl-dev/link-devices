## Why

**A client cannot speak to both devices.** The new iOS companion app
([KitchenSync-iOS_App](https://github.com/ericdahl-dev/KitchenSync-iOS_App)) was built against
`KitchenSync/main/ks_web.cpp`. Pointed at a real Touch on the bench (`kstouch-dfd0`, 2026-07-14),
its `KsStatus` decoder **throws on every required key**. Not degrades ŌĆö throws. Touch does not
serve one field the app needs.

This is drift, not design. The tell: both firmwares emit the same nine tick-health fields, but
Touch spells them `wbeats`/`wclock` while KitchenSync spells them `w_beats`/`w_clock`. Nobody
chooses that. And `ks_status.c` states the intent out loud:

> *"in X32Link's exact key names (ARC-024) ŌĆö one shape across the fleet, so one script audits
> every device."*

One shape across the fleet is the stated goal. Touch violates it, and a client is where that
finally bites.

The design intent is that Touch is *"the 1-MIDI-output, touchscreen version"* ŌĆö not a different
protocol. The firmware should say so.

## The actual gap (read from source, 2026-07-14)

`KitchenSyncTouch/ktouch_web.cpp:252` emits:

```
bpm, sync, peers, clock, transport, cue, tfail, tzero, ccancel,
drop, burst, gap, work, over, core, wbeats, wclock, wtport,
beats, locked, bsactive, btn, btnlows, btnpress, clk, pulses
```

`KitchenSync/main/ks_status.c:8` emits:

```
bpm, min, peers, usb, tx, fw, follow_enabled, follow_bpm, follow_confidence,
follow_valid, launch[4], playing, link_owns,
drop, burst, gap, work, over, core, w_beats, w_clock, reprime,
xf, xf_step, xf_max, rtt_min, rtt_max
```

| | KitchenSync | Touch | Consequence for a client |
|---|---|---|---|
| `launch[N]` | Ō£ģ per-output launch state | ŌØī **absent** | The whole quantized transport UI (arm ŌåÆ bar line ŌåÆ run) has no data. This is the app's main screen. |
| `playing`, `link_owns` | Ō£ģ | ŌØī | Cannot tell who owns transport, or whether anything is running. |
| `fw` | Ō£ģ | ŌØī | Cannot tell what firmware a unit runs ŌĆö see ESP-030, where that matters. |
| `usb`, `tx`, `min` | Ō£ģ | ŌØī | No USB-MIDI state, no clock-pulse count, no MIDI-clock-in tempo. |
| `follow_*` | Ō£ģ (4 fields) | ŌØī | No follow-beat telemetry. |
| tick health | `w_beats`, `w_clock`, `reprime` | `wbeats`, `wclock`, **no reprime** | Same data, different spelling. One audit script cannot read both ŌĆö the exact thing ARC-024 exists to prevent. |
| phase health | `xf`, `xf_step`, `xf_max`, `rtt_min/max` | ŌØī | No P4-038 gauge. `xf_max` is the number that, unread, cost 138 s of silent DIN clock (ESP-027). |
| `clk`, `pulses` | ŌØī | Ō£ģ (ESP-028) | **Touch is AHEAD here.** This is the writer's-truth fix. KitchenSync should gain it, not Touch lose it. |

## Approach

**Converge on the union, not on either side.** Neither firmware is simply "right":

1. Touch **gains** the KitchenSync contract: `min, usb, tx, fw, follow_*, launch[], playing,
   link_owns`, and the P4-038 phase-health block.
2. Touch **renames** `wbeats`/`wclock` ŌåÆ `w_beats`/`w_clock` and adds `reprime`. (Breaking for
   Touch's own web UI JS ŌĆö update `ktouch_web.cpp`'s page in the same commit.)
3. KitchenSync **gains** `clk` and `pulses` (ESP-028's writer's-truth fix, which the Touch has and
   the P4 does not). File separately if it grows.
4. Touch **keeps** its extras ŌĆö `beats`, `bsactive`, `btn*`, `cue`, `tfail`, `tzero`, `ccancel`.
   A client ignores unknown keys; that's fine and costs nothing.

### `launch[]` on a one-output device

**Emit `launch` as an array whose length IS the output count** ŌĆö `launch:[0]` on Touch,
`launch:[0,0,0,0]` on KitchenSync. Do NOT pad Touch to four with dummies: a client that renders a
card per element would draw three dead outputs.

The iOS app currently hardcodes `KsConfig.outputCount = 4`; it will be changed to drive off the
array length (tracked app-side). Emitting the honest length is what makes that possible.

If Touch has no per-output launch concept at all today, that is the real work here ŌĆö it needs the
same arm ŌåÆ bar-line ŌåÆ run model (`transport_launch.h`), because quantized start is the product's
whole point and the app's primary control.

## Acceptance criteria

- [ ] One client decodes BOTH devices' `/status` with no per-device special-casing
- [ ] Touch emits `launch[]` with length == its actual clock-output count (1), plus `playing` and
      `link_owns`
- [ ] Touch emits `fw` (from `fw_version.h` ŌĆö LNK-038, already shared)
- [ ] Touch emits `min`, `usb`, `tx`, `follow_*`
- [ ] Tick-health keys are spelled identically on both (`w_beats`, `w_clock`, `reprime`)
- [ ] Touch's own web UI still works (its JS reads these keys ŌĆö update it in the same commit)
- [ ] `clk` / `pulses` survive on Touch; ideally KitchenSync gains them
- [ ] Verified against the real iOS app on the bench: the Touch appears, decodes, and its
      transport button drives a real output

## Notes

**ŌÜĀ’ĖÅ The bench unit `kstouch-dfd0` is running STALE firmware** (verified 2026-07-14). Its `/status`
stops at `wtport` ŌĆö no `beats`, `locked`, `bsactive`, `btn*`, `clk`, `pulses`. That means it
predates ESP-028. **Reflash it before using it to verify anything**, or you will be testing against
a contract that no longer exists in source.

The web UI's own JS is the second consumer of these keys (`ktouch_web.cpp`'s page). Renaming
`wbeats` ŌåÆ `w_beats` breaks it silently ŌĆö the pill just stops updating. Change both together.

Touch's `/status` has a documented history of lying (ESP-027, ESP-028: *"the SAME lie, one field
over"*, *"status still cheerfully reports sync:1 peers:1 clock:1. Zero bytes in 6 s"*). When adding
fields, report **what the writer is doing**, not what the config says it should be doing. That
principle is the whole lesson of ESP-028 and it applies to every field added here.
