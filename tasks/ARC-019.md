## Problem

The "given session phase + output config, emit these 0xF8 pulses now" decision is
**re-derived in three FreeRTOS tick tasks** instead of living behind one deep
interface. KitchenSync is the deep exemplar; the other two re-implement a stripped
subset in untested glue:

| Firmware | Tick task | Derivation |
|---|---|---|
| KitchenSync | `KitchenSync/main/ks_tick.c:54-60` | `clock_output_due` → full swing + phase-nudge + `clock_ticker` (**pure, host-tested**) |
| Touch | `KitchenSyncTouch/ktouch_midi_out.cpp:41-43` | re-derives nudge inline; `midi_clock_out_ticks_due` (ppqn-24 only); **no swing, untested** |
| X32Link | `X32Link/midi_clock_out_io.cpp:26-34` | `midi_clock_out_ticks_due` direct; **no swing, no nudge, no per-output phase, untested** |

Consequences:

- **Swing / per-output phase / division is a KitchenSync-only capability.** Touch
  and X32 physically cannot do it because they bypass `clock_output_due`.
- **The load-bearing invariants live in untested `.cpp` tasks.** Catch-up-burst
  suppression (`max_burst`) is copy-pasted three times
  (`ks_tick.c` `KS_TICK_MAX_BURST`, `midi_clock_out_io.cpp:15`,
  `ktouch_midi_out.cpp:24` — the last literally commented "Same as X32Link"). The
  reset-on-invalid-phase decision is re-typed in each task. The pure engines
  underneath are tested, so the suite gives false confidence that Touch/X32 clock
  output is covered when only the KitchenSync *wiring* is.
- X32Link doesn't use `beat_source.c` at all (grep: only KitchenSync references
  it), so its re-prime-on-basis-switch logic — tested in `test_beat_source.c` —
  protects one firmware and not the others.

**Deletion test:** delete the two inline derivations and route their tasks through
the same module KitchenSync uses — complexity concentrates in one tested place;
what vanishes is three copies of `max_burst` and three reset-on-invalid re-types.

## Solution

Promote the KitchenSync stack (`clock_output_due` over `clock_ticker`, fed by
`beat_source_step`) to **the** clock-out derivation module and route all three
tick tasks through it. Each task keeps only its hardware emit — USB+DIN
(`midi_clock_out_io.cpp`), DIN-only (`ktouch_midi_out.cpp`), the KitchenSync plan
loop — and its arbitration where it has one. The interface is a pure
"session phase + per-output config → pulses due now" call; the interface is the
test surface, so Touch/X32 clock output becomes host-testable for the first time.

Depends on ARC-018 — the engines must be one source before three callers can
share them.

## Acceptance Criteria

- [ ] `ktouch_midi_out.cpp` and `midi_clock_out_io.cpp` call the shared derivation
      (`clock_output_due` / `beat_source_step`) instead of re-deriving nudge/reset
      and calling the ppqn-24-only path directly.
- [ ] `max_burst` and the reset-on-invalid-phase rule have exactly one definition.
- [ ] Host test exercises the derivation for a Touch/X32-shaped config (nudge
      applied, swing off) at the module interface — not just the KitchenSync path.
- [ ] Swing / phase-nudge are reachable on Touch and X32 (or explicitly deferred
      with a one-line reason if a config field is missing — don't silently drop).
- [ ] All three firmwares compile; host suite green.
- [ ] Hardware: DIN clock still ticks on Touch; USB+DIN on X32; KitchenSync
      unchanged. Measure with the logic analyzer, don't trust ears.

## Agent Brief

**Category:** enhancement (refactor — concentrate the clock-out derivation)
**Summary:** route all three firmwares' tick tasks through the one host-tested
derivation (`clock_output_due` over `clock_ticker`), so swing / division /
phase-nudge and the burst/reset invariants have exactly one tested owner.

**Current behavior:** three tick tasks derive Link→pulses independently; swing
and division work only on KitchenSync; `max_burst` and reset-on-invalid-phase
are re-typed three times in untested glue.

**Desired behavior:** each tick task keeps only its hardware emit (USB+DIN /
DIN-only / plan loop) and calls the shared pure derivation. Touch and X32Link
clock output become host-testable at the module interface.

**Method — TDD, vertical slices:** one caller at a time. For each migration
(Touch first, then X32Link): RED — host test at the derivation interface for
that firmware's config shape (nudge applied, swing off, division 24) → GREEN —
route the task through `clock_output_due` → delete the inline derivation →
suite green → next caller. Do not batch-write all tests up front; each test
covers the caller being migrated.

**Acceptance criteria:** the six checkboxes above. The hardware checkbox
(analyzer on Touch DIN / X32 USB+DIN) is an owner bench step — mark NOT
VERIFIED in the PR and leave the box unchecked rather than claiming it.

**Out of scope:** the emit adapters themselves, per-firmware transport
arbitration ("never both masters"), and the `test_transport_intent` /
`test_tempo_source` gap (separate tickets — file them, don't fix them here).
