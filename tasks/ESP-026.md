## SPIKE RESULT 2026-07-13 — the hard part is done, and one assumption was WRONG

**KitchenSync now builds for `esp32p4`, `esp32s3` AND `esp32` from one ESP-IDF project.**
All three, exit 0, images created. The migration's core premise is no longer an assumption.

It took **102 lines**:

1. **Target-gate the P4-only managed components** (`esp_hosted`, `esp_wifi_remote`, `es8311`)
   with `rules: - if: "target in [esp32p4]"` in `idf_component.yml`.
2. **Target-gate the P4-only sources** in `main/CMakeLists.txt` (`usb_midi_host.c`,
   `metronome_audio.c`, `i2s_audio_bus.c`, `follow_beat_io.c`).
3. **No-op stubs in the four P4-only headers** when the hardware is not fitted — so the app
   core (`ks_main.c`, `ks_web.cpp`) compiles **completely unchanged**, with no `#ifdef`s
   sprayed through it. Same shape as `HAS_USB_MIDI` / `HAS_TOUCH_DISPLAY` / `LED_NONE`
   elsewhere in this repo (ADR-0003: thin per-target glue, pure shared logic).

The P4 still builds with all four P4-only objects compiled in — the product is untouched.
CI now compiles all three targets, so this cannot rot.

### The assumption that was wrong

This ticket implied the WiFi stack would be a major port: the P4 has no radio and gets WiFi
from the C6 over ESP-Hosted, so five of the eight managed components exist purely for that.

**`wifi_link.c` compiled for the S3 and the classic ESP32 unchanged**, against native
`esp_wifi`. It was never ESP-Hosted-specific. The feared blocker was not a blocker.

### What is actually left

- **The display.** LovyanGFX + AXS5106L on ESP-IDF is now the ONLY real unknown, and it still
  needs the S3 touch board on USB. Spike it before deleting anything (unchanged advice).
- The transport surface (touch UI / ESP-025 lit buttons) posting into `transport_intent.c`.
- Moving the shared pure C out of `X32Link/` (ADR-0009) — symlink-neutral once the Touch's 41
  symlinks go.
- Deleting `KitchenSyncTouch/` and `tools/build-bench.sh`.

**The risk profile has changed.** "Can the ESP-IDF app even run on non-P4 silicon?" was the
question that made this a scary multi-day migration. It is answered, and it was 102 lines.

## Problem

**The 1 ms clock writer exists twice**, and the copies have already diverged.

| | Writer | Framework |
|---|---|---|
| KitchenSync (P4) | `ks_main.c` + `ks_tick.c` | ESP-IDF |
| KitchenSyncTouch (S3, bench rig) | `ktouch_midi_out.cpp` | Arduino |

ESP-023 found — and measured on the analyzer — that transport (`0xFA`) was emitted
**after** the downbeat clock (`0xF8`), so every slave began playing one 24-PPQN tick late:
**20.8 ms at 120 BPM, on every start, forever.** A slave starts on the first clock it
receives *after* a Start, so a clock-first tick is simply discarded.

**The fix landed on the P4 only.** The Touch carried the identical bug, unfixed, until it
was found by reading the code on 2026-07-12 while building the ESP-025 bench rig. (It is
fixed there now, in the ESP-025 branch.) That is the bill for a duplicate implementation:
the bug gets found once and fixed once, and the other copy keeps it silently.

See [ADR-0009](../docs/adr/0009-converge-the-clock-box-on-esp-idf.md).

## Why now — ESP-024 is the forcing function

ESP-024's whole thesis is **hardware-timed RMT edges and `IRAM_ATTR` placement**. The
Arduino core ships a **fixed sdkconfig** — there is no `menuconfig` — so the knobs that
actually decide jitter (IRAM, cache behaviour, interrupt allocation) are not reachable
from it.

Doing that timing work on the framework that hides the controls, and then doing it a
**second time** on the P4, is the worst of both. **This lands before ESP-024, or the clock
gets tuned twice and the two copies drift again.**

## The port is bounded

KitchenSync already has an ESP-IDF implementation of nearly every piece of glue the Touch
gets from Arduino:

| Touch needs (Arduino) | KitchenSync already has (ESP-IDF) |
|---|---|
| `WiFi.h` | `wifi_link.c` |
| `WebServer.h` + `Update.h` | `ks_web.cpp` (esp_http_server + OTA) |
| `Preferences.h` | `ks_config_nvs.c` |
| `ESPmDNS.h` | `ks_hostname.c` |
| DIN MIDI UART | `midi_uart_out.c` |
| USB MIDI | `usb_midi_host.c` |
| transport mailbox | `transport_intent.c` |
| beat LED | `ks_led.c` |
| the 1 ms writer | `ks_main.c` + `ks_tick.c` |

**The only genuinely new work is the display**: LovyanGFX (which supports ESP-IDF natively)
and the AXS5106L touch controller over I²C. `axs5106l.c` is already pure C and host-tested —
only the I²C transport is Arduino-bound.

## Approach

One ESP-IDF project, three board targets (`idf.py set-target` supports all three):

| Target | Board | Transport surface |
|---|---|---|
| `esp32p4` | P4-NANO | web UI (existing) |
| `esp32s3` | Waveshare S3-Touch-LCD-1.47 | touch UI |
| `esp32` | classic DevKit (**ESP-025 bench rig**) | two lit buttons |

Board differences go behind **Kconfig**, not `#define` comment-swaps — which also retires
`tools/build-bench.sh` and the partition-table swap it exists to make safe.

1. Add `esp32s3` + `esp32` targets and a Kconfig board menu to the KitchenSync project.
2. Port the display (LovyanGFX + AXS5106L I²C) as an ESP-IDF component, gated on the S3
   board.
3. Move the ESP-025 button/lamp surface (`buttons.c`, `transport_led.c` — already pure)
   behind the DevKit board option. Both surfaces post `TransportLaunchIntent` into
   `transport_intent.c`, so the writer neither knows nor cares which is fitted.
4. **Move the shared pure C out of `X32Link/`** into a KitchenSync ESP-IDF component
   (ADR-0009). `X32Link/`, `LoraLink/` and `X32_emulator/` — still arduino-cli — symlink it
   into their flat roots. Repoint `test/Makefile`.
   - Do this **as part of** this migration, not before. Standalone it *adds* ~40 symlinks to
     `X32Link/` on top of KitchenSyncTouch's 41; bundled here, the Touch's 41 disappear
     (ESP-IDF compiles by path) and the move is roughly **symlink-neutral**.
5. Delete `KitchenSyncTouch/` and `tools/build-bench.sh`.
6. Handle `TL_RESTART` properly in `ks_tick.c` — today it is an explicit no-op with a
   comment (ESP-025), because a restart is **two bytes in one tick** (`0xFC` then `0xFA`)
   and `plan.transport[]` holds one action per output. The P4 needs a plan-level change to
   carry it.

## Risks — say these out loud

- **KitchenSyncTouch works today** and is close to shipping. This risks regressions on
  working hardware to buy a mostly future-facing benefit. Absent ESP-024, leaving it alone
  would be defensible. **Do not start this without intending to finish it.**
- The touch display is the only real unknown. **Spike LovyanGFX-on-ESP-IDF first** — if that
  does not come up on the bench S3 within a day, stop and re-plan rather than half-migrating.
- The bench rig (ESP-025) must keep working throughout: it is the fixture ESP-024 depends on.

## Acceptance criteria

- [ ] LovyanGFX + AXS5106L spiked on ESP-IDF on the S3 **before** anything is deleted
- [ ] One ESP-IDF project builds for `esp32p4`, `esp32s3` and `esp32`
- [ ] **One** clock writer. `ktouch_midi_out.cpp` is gone, not forked
- [ ] Touch UI and the ESP-025 lit buttons both post intents into `transport_intent.c`
- [ ] Board selection is Kconfig; `tools/build-bench.sh` and the partition-table swap are gone
