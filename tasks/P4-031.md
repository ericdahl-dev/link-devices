## Problem / opportunity

Once RTP-MIDI works over WiFi ([[P4-030]]), run the same session over the P4-NANO's
**wired Ethernet + PoE**. RTP-MIDI is transport-agnostic (UDP/IP), so the pure
`rtp_midi_session` is unchanged — this ticket is the Ethernet transport + bring-up.

Why wired matters here specifically: MIDI clock (`0xF8`) is timing-sensitive and WiFi
adds jitter. Wired Ethernet is meaningfully lower-jitter and rock-solid for a stage,
and **PoE** means a single cable for power + data — a strong story for the
phase-accurate KitchenSync hub ([[p4hub-product-vision]]).

## Scope

- The **ESP32-P4 has an internal EMAC**; the **Waveshare P4-NANO populates Ethernet +
  PoE** (per the board spec — see [[waveshare-p4-nano-board.md]]). Bring up
  ESP-IDF `esp_eth` against the board's PHY (identify the PHY + its MDIO/RMII pins from
  the Waveshare schematic; confirm the PoE splitter powers the board).
- Give the RTP-MIDI UDP glue ([[P4-030]]) a second `esp_netif` (the Ethernet
  interface) so the AppleMIDI session + mDNS advertise on the wired LAN. Ideally the
  session is netif-agnostic — bind to the active interface(s).
- Decide the interface policy: Ethernet-preferred when the cable is up, WiFi fallback
  (or both advertised). The [[wifi-conn-policy]] pattern (a pure lifecycle machine) is
  a good model if this needs its own state.

## Notes / risks

- Early-silicon P4 (rev v1.3) + ESP-IDF min-rev flags already sorted
  ([[waveshare-p4-nano-board.md]]); confirm EMAC works on this rev.
- Verify PoE power budget is enough for the P4 + C6 + peripherals under load.
- Depends on [[P4-030]] proving the protocol/UX first; this is purely the wired
  transport + interface selection.
