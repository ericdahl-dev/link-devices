## Problem / opportunity

Once RTP-MIDI works over WiFi ([[P4-030]]), run the same session over the P4-NANO's
**wired Ethernet + PoE**. RTP-MIDI is transport-agnostic (UDP/IP), so the pure
`rtp_midi_session` is unchanged — this ticket is the Ethernet transport + bring-up.

Why wired matters here specifically: MIDI clock (`0xF8`) is timing-sensitive and WiFi
adds jitter. Wired Ethernet is meaningfully lower-jitter and rock-solid for a stage,
and **PoE** means a single cable for power + data — a strong story for the
phase-accurate KitchenSync hub ([[p4hub-product-vision]]).

## Scope

- The **ESP32-P4 has an internal EMAC**; the **Waveshare P4-NANO populates Ethernet +
  PoE** (per the board spec — see [[waveshare-p4-nano-board.md]]). Bring up
  ESP-IDF `esp_eth` against the board's PHY (identify the PHY + its MDIO/RMII pins from
  the Waveshare schematic; confirm the PoE splitter powers the board).
- Give the RTP-MIDI UDP glue ([[P4-030]]) a second `esp_netif` (the Ethernet
  interface) so the AppleMIDI session + mDNS advertise on the wired LAN. Ideally the
  session is netif-agnostic — bind to the active interface(s).
- Decide the interface policy: Ethernet-preferred when the cable is up, WiFi fallback
  (or both advertised). The [[wifi-conn-policy]] pattern (a pure lifecycle machine) is
  a good model if this needs its own state.

## Notes / risks

- Early-silicon P4 (rev v1.3) + ESP-IDF min-rev flags already sorted
  ([[waveshare-p4-nano-board.md]]); confirm EMAC works on this rev.
- Verify PoE power budget is enough for the P4 + C6 + peripherals under load.
- Depends on [[P4-030]] proving the protocol/UX first; this is purely the wired
  transport + interface selection.

## ⚠️ Risk found in ESP-018 (2026-07-11): wired does NOT automatically mean low-jitter

This ticket's premise is "WiFi jitters the clock, so run it over wired Ethernet instead."
That is right about the NETWORK and wrong about the CLOCK TASK, and the difference matters.

The P4 gets its tight timing (331us jitter vs the S3's ~500us) for a reason that has
nothing to do with wired vs wireless: **its WiFi lives on a separate C6 co-processor over
SDIO, so the P4's own CPU never runs a network stack at all.**

Wired Ethernet would put lwIP back ON the P4's CPU.

That is exactly the bug ESP-018 turned out to be on the S3. The MIDI writer ran at priority
6 -- below lwIP's tcpip_task (18) and WiFi (23) -- so under network load the stack legally
preempted the real-time clock generator mid-tick. Measured: the writer's call into the Link
engine went from 62us idle to 5038us under load, with NO lock anywhere in that path. It was
never blocking; it was being PREEMPTED. On the wire that produced a 40ms stall and a
catch-up burst. Fix was priority 19 (above lwIP, below the radio): 0 stalls, 0 bursts.

So P4-031 must NOT assume the P4's current jitter survives the move to Ethernet. Before
trusting it:

- [ ] Raise the clock task above lwIP's tcpip_task, as ESP-018 did on the S3
- [ ] Re-measure jitter on the analyzer WITH Ethernet + RTP-MIDI under load -- not idle.
      Idle proves nothing; ESP-018 looked perfectly healthy at idle and stalled under load.
- [ ] Watch the `dropped` counter (clock_ticker, ESP-018): a stall past MAX_BURST discards
      pulses SILENTLY -- no burst, no gap, no trace on the wire.

An iConnectivity mio10 (owner has one) is a natural RTP-MIDI endpoint to test against.
